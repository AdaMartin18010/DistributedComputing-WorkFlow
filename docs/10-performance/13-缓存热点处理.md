# 缓存热点处理：本地缓存与副本

## 核心概念

### 1. 缓存热点问题

缓存热点是指少数key被大量请求访问，导致缓存节点压力集中，甚至成为系统瓶颈。

**热点问题表现**：
```
┌─────────────────────────────────────────────────────────────┐
│                   缓存热点问题表现                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   场景：微博热点事件，单条微博被千万用户访问                 │
│                                                             │
│   正常情况：                                                │
│   请求1 ──→ Redis Node1                                     │
│   请求2 ──→ Redis Node2                                     │
│   请求3 ──→ Redis Node3                                     │
│   （负载均衡）                                              │
│                                                             │
│   热点情况：                                                │
│   请求1 ──┐                                                 │
│   请求2 ──┼──→ Redis Node1 ← 热点Key:weibo:12345            │
│   请求3 ──┘                                                 │
│   请求4 ──→ Redis Node1                                     │
│   ...                                                       │
│   请求N ──→ Redis Node1                                     │
│   （单节点CPU/带宽打满）                                    │
│                                                             │
│   后果：                                                     │
│   1. 热点节点CPU使用率100%                                  │
│   2. 网络带宽耗尽                                           │
│   3. 响应延迟飙升                                           │
│   4. 可能引发级联故障                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**热点识别方法**：
```
1. 访问频次统计
   - 单位时间内key的访问次数
   - 超过阈值判定为热点

2. 流量占比分析
   - Top 1%的key贡献90%流量
   - 头部效应明显

3. Redis监控
   - redis-cli --hotkeys (需要配置)
   - slowlog分析
   - 连接数监控
```

### 2. 热点分级

```
┌─────────────────────────────────────────────────────────────┐
│                   缓存热点分级                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Level 1: 一般热点                                          │
│   - QPS: 1万-10万/秒                                        │
│   - 影响：单节点负载较高                                    │
│   - 处理：本地缓存 + 限流                                   │
│                                                             │
│   Level 2: 严重热点                                          │
│   - QPS: 10万-100万/秒                                      │
│   - 影响：单节点可能过载                                    │
│   - 处理：多副本 + 本地缓存 + 限流                          │
│                                                             │
│   Level 3: 极端热点                                          │
│   - QPS: 100万+/秒                                          │
│   - 影响：系统整体风险                                      │
│   - 处理：静态化 + CDN + 多副本 + 限流                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. 热点解决方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 本地缓存 | 进程内缓存热点数据 | 零网络开销 | 一致性问题 | 允许短暂不一致 |
| 多副本 | 热点key在多个节点存储 | 负载分散 | 内存增加 | 极端热点 |
| 读写分离 | 从节点分担读压力 | 简单易实现 | 有延迟 | 读多写少 |
| 限流降级 | 限制对热点访问 | 保护系统 | 有损服务 | 所有场景 |
| 静态化 | 内容预生成 | 完全无压力 | 实时性差 | 页面内容 |

## 实践方法

### 本地缓存方案

**1. 本地缓存实现**：
```go
package cache

import (
    "sync"
    "time"
    
    "github.com/allegro/bigcache/v3"
)

// HotKeyLocalCache 热点key本地缓存
type HotKeyLocalCache struct {
    // 普通缓存（所有key）
    localCache *bigcache.BigCache
    
    // 热点key检测
    hotKeyDetector *HotKeyDetector
    
    // 分布式缓存
    redis *RedisClient
    
    // 热点key集合
    hotKeys sync.Map
    
    // 配置
    hotThreshold int           // 热点阈值（每秒访问次数）
    hotKeyTTL    time.Duration // 热点key本地缓存TTL
}

func NewHotKeyLocalCache(redis *RedisClient) *HotKeyLocalCache {
    cache, _ := bigcache.New(bigcache.Config{
        Shards:       256,
        LifeWindow:   10 * time.Minute,
        CleanWindow:  5 * time.Minute,
        MaxEntrySize: 500,
    })
    
    return &HotKeyLocalCache{
        localCache:     cache,
        redis:          redis,
        hotKeyDetector: NewHotKeyDetector(1000),
        hotThreshold:   1000,           // 每秒1000次访问视为热点
        hotKeyTTL:      1 * time.Minute, // 本地缓存1分钟
    }
}

func (c *HotKeyLocalCache) Get(ctx context.Context, key string) ([]byte, error) {
    // 1. 检查是否是热点key
    if _, isHot := c.hotKeys.Load(key); isHot {
        // 2. 先查本地缓存
        if val, err := c.localCache.Get(key); err == nil {
            return val, nil
        }
        
        // 3. 本地未命中，查Redis并缓存到本地
        val, err := c.redis.Get(ctx, key).Bytes()
        if err != nil {
            return nil, err
        }
        
        c.localCache.Set(key, val)
        return val, nil
    }
    
    // 4. 非热点key，直接查Redis
    c.hotKeyDetector.Record(key)
    return c.redis.Get(ctx, key).Bytes()
}

func (c *HotKeyLocalCache) RefreshHotKeys() {
    // 定时刷新热点key列表
    hotKeys := c.hotKeyDetector.GetHotKeys(c.hotThreshold)
    
    // 清空旧的热点标记
    c.hotKeys = sync.Map{}
    
    // 标记新的热点key
    for _, key := range hotKeys {
        c.hotKeys.Store(key, true)
        
        // 预加载到本地缓存
        if val, err := c.redis.Get(context.Background(), key).Bytes(); err == nil {
            c.localCache.Set(key, val)
        }
    }
}
```

**2. 热点检测器**：
```go
// HotKeyDetector 热点key检测
type HotKeyDetector struct {
    counters     map[string]*atomic.Int64
    mu           sync.RWMutex
    windowSize   time.Duration
    maxKeyCount  int
}

func NewHotKeyDetector(maxKeyCount int) *HotKeyDetector {
    d := &HotKeyDetector{
        counters:    make(map[string]*atomic.Int64),
        windowSize:  1 * time.Second,
        maxKeyCount: maxKeyCount,
    }
    go d.cleanupLoop()
    return d
}

func (d *HotKeyDetector) Record(key string) {
    d.mu.RLock()
    counter, exists := d.counters[key]
    d.mu.RUnlock()
    
    if !exists {
        d.mu.Lock()
        // 双重检查
        if counter, exists = d.counters[key]; !exists {
            if len(d.counters) >= d.maxKeyCount {
                d.mu.Unlock()
                return // 超过上限不再记录新key
            }
            counter = &atomic.Int64{}
            d.counters[key] = counter
        }
        d.mu.Unlock()
    }
    
    counter.Add(1)
}

func (d *HotKeyDetector) GetHotKeys(threshold int64) []string {
    d.mu.RLock()
    defer d.mu.RUnlock()
    
    var hotKeys []string
    for key, counter := range d.counters {
        if counter.Load() >= threshold {
            hotKeys = append(hotKeys, key)
        }
    }
    
    return hotKeys
}

func (d *HotKeyDetector) cleanupLoop() {
    ticker := time.NewTicker(d.windowSize)
    defer ticker.Stop()
    
    for range ticker.C {
        d.mu.Lock()
        d.counters = make(map[string]*atomic.Int64)
        d.mu.Unlock()
    }
}
```

### 多副本方案

**1. 热点key多副本**：
```go
// HotKeyReplica 热点key副本管理
type HotKeyReplica struct {
    redis       *RedisCluster
    replicaCount int    // 副本数
    keyPrefix   string
}

// 获取副本key
func (r *HotKeyReplica) getReplicaKey(key string, index int) string {
    return fmt.Sprintf("%s:%s:%d", r.keyPrefix, key, index)
}

// 写入多副本
func (r *HotKeyReplica) Set(ctx context.Context, key string, value []byte, ttl time.Duration) error {
    pipe := r.redis.Pipeline()
    
    for i := 0; i < r.replicaCount; i++ {
        replicaKey := r.getReplicaKey(key, i)
        pipe.Set(ctx, replicaKey, value, ttl)
    }
    
    _, err := pipe.Exec(ctx)
    return err
}

// 读取（随机选择副本）
func (r *HotKeyReplica) Get(ctx context.Context, key string) ([]byte, error) {
    // 随机选择副本，分散读压力
    index := rand.Intn(r.replicaCount)
    replicaKey := r.getReplicaKey(key, index)
    
    return r.redis.Get(ctx, replicaKey).Bytes()
}

// 删除所有副本
func (r *HotKeyReplica) Del(ctx context.Context, key string) error {
    keys := make([]string, r.replicaCount)
    for i := 0; i < r.replicaCount; i++ {
        keys[i] = r.getReplicaKey(key, i)
    }
    return r.redis.Del(ctx, keys...).Err()
}
```

**2. 读写分离 + 副本结合**：
```go
// 完整的缓存客户端
type CacheClient struct {
    // 普通key直接访问
    redis *RedisClient
    
    // 热点key本地缓存
    localCache *HotKeyLocalCache
    
    // 极端热点key多副本
    hotKeyReplica *HotKeyReplica
    
    // 热点key集合（由监控系统推送）
    extremeHotKeys sync.Map
}

func (c *CacheClient) Get(ctx context.Context, key string) ([]byte, error) {
    // 1. 检查是否极端热点（使用多副本）
    if _, isExtreme := c.extremeHotKeys.Load(key); isExtreme {
        return c.hotKeyReplica.Get(ctx, key)
    }
    
    // 2. 使用本地缓存（自动检测热点）
    return c.localCache.Get(ctx, key)
}

func (c *CacheClient) Set(ctx context.Context, key string, value []byte, ttl time.Duration) error {
    // 写入时，如果是极端热点，写入多副本
    if _, isExtreme := c.extremeHotKeys.Load(key); isExtreme {
        return c.hotKeyReplica.Set(ctx, key, value, ttl)
    }
    
    return c.redis.Set(ctx, key, value, ttl).Err()
}
```

### 限流保护

```go
// 热点key限流
type HotKeyRateLimiter struct {
    limiters sync.Map  // key -> *rate.Limiter
    r        rate.Limit // 每秒允许的请求数
    b        int        // 桶大小
}

func NewHotKeyRateLimiter(r rate.Limit, b int) *HotKeyRateLimiter {
    return &HotKeyRateLimiter{r: r, b: b}
}

func (l *HotKeyRateLimiter) Allow(key string) bool {
    limiter, _ := l.limiters.LoadOrStore(key, rate.NewLimiter(l.r, l.b))
    return limiter.(*rate.Limiter).Allow()
}

// 使用
func (c *CacheClient) GetWithProtection(ctx context.Context, key string) ([]byte, error) {
    // 1. 检查是否是已知热点
    if _, isHot := c.knownHotKeys.Load(key); isHot {
        // 2. 限流检查
        if !c.rateLimiter.Allow(key) {
            // 3. 降级处理：返回本地缓存或默认值
            if val, err := c.localCache.Get(key); err == nil {
                return val, nil
            }
            return c.getDefaultValue(key), nil
        }
    }
    
    return c.Get(ctx, key)
}
```

## 案例数据

### 微博热点事件处理

**场景**：某明星发布微博，10分钟内阅读量过亿

**处理前**：
```
单条微博key: weibo:123456
QPS: 100万/秒
Redis单节点CPU: 100%
延迟: 500ms-2s
大量请求超时
```

**处理方案**：
```
1. 自动检测热点
   - 5秒内检测到key访问超过5000次
   - 判定为热点key

2. 本地缓存激活
   - 本地缓存weibo:123456
   - TTL: 10秒
   - 本地QPS处理能力: 10万/秒

3. 多副本部署
   - 创建10个副本: weibo:123456:0 ~ weibo:123456:9
   - 读请求随机路由到副本

4. 限流保护
   - 单key限流: 1万/秒
   - 超限返回本地缓存数据
```

**处理后**：
```
Redis单节点CPU: 30%
延迟: 5ms-10ms
处理能力: 1000万/秒（本地缓存+多副本）
服务稳定性: 100%
```

### 电商大促商品热点

**场景**：秒杀商品详情页

**方案**：
```
1. 商品信息静态化
   - 商品标题、图片、描述 → CDN
   
2. 库存信息多级缓存
   - L1: 本地缓存（1秒TTL）
   - L2: Redis多副本（5秒TTL）
   - DB: 实际库存（异步同步）

3. 请求分层
   - 90%请求 → CDN + 本地缓存
   - 10%请求 → Redis
   - <1%请求 → DB
```

**效果**：
| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 单商品QPS | 10万 | 500万 |
| 数据库压力 | 100% | <1% |
| 响应延迟 | 500ms | 10ms |
| 系统稳定性 | 经常宕机 | 稳定运行 |

## 优化前后对比

| 方案 | 单机QPS | 网络开销 | 一致性 | 复杂度 |
|------|---------|----------|--------|--------|
| 无优化 | 10万 | 高 | 强 | 低 |
| 本地缓存 | 50万 | 低 | 最终 | 中 |
| 多副本 | 100万 | 中 | 强 | 中 |
| 本地+副本 | 500万 | 低 | 最终 | 高 |

---

**总结**：缓存热点是分布式系统中的常见挑战。通过本地缓存、多副本、限流降级等手段的组合使用，可以有效应对各种级别的热点问题，保障系统在高并发场景下的稳定性和可用性。
