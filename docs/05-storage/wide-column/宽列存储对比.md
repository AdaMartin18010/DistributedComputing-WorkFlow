## 📋 执行摘要

宽列存储（Wide-Column Store）是一种 NoSQL 数据库模型，以列族（Column Family）为单位组织数据，适合大规模分布式存储场景。本文档深入对比 Cassandra、HBase、ScyllaDB 等主流宽列存储系统，并分析 Bigtable 架构设计原理。

---

## 一、核心概念

### 1.1 定义与原理

**宽列存储** 是一种介于关系型数据库和键值存储之间的数据模型：
- **行键（Row Key）**：唯一标识一行数据
- **列族（Column Family）**：列的集合，物理上分开存储
- **列（Column）**：动态列，每行可有不同列
- **时间戳（Timestamp）**：多版本数据支持

**核心原理**：
1. **稀疏表存储**：空值不占空间
2. **列式组织**：同列数据连续存放，利于压缩
3. **分布式设计**：原生支持数据分片和复制
4. **最终一致性**：优先保证可用性和分区容错

### 1.2 关键特性

| 特性 | 说明 |
|------|------|
| **动态 Schema** | 每行可有不同列，无需预定义 |
| **高吞吐写入** | 日志结构合并树（LSM-Tree）优化 |
| **水平扩展** | 线性扩展至数百节点 |
| **多数据中心** | 跨地域复制支持 |
| **时间戳版本** | 自动维护多版本数据 |
| **可调一致性** | 读写一致性级别可调 |

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 大规模日志存储 | ⭐⭐⭐⭐⭐ | 写入密集型场景 |
| 物联网数据 | ⭐⭐⭐⭐⭐ | 海量设备时序数据 |
| 用户画像/特征存储 | ⭐⭐⭐⭐⭐ | 稀疏矩阵存储 |
| 推荐系统 | ⭐⭐⭐⭐⭐ | 实时特征读写 |
| 消息/Feed 流 | ⭐⭐⭐⭐ | 高并发读写 |
| 复杂事务处理 | ⭐⭐ | 不支持 ACID 事务 |

**Cassandra 核心特性**：

| 组件 | 说明 |
|------|------|
| **Gossip 协议** | 去中心化节点通信，传播集群状态 |
| **一致哈希** | 数据分区，支持虚拟节点（VNode） |
| **复制策略** | SimpleStrategy（单 DC）/ NetworkTopologyStrategy（多 DC） |
| **读写路径** | 写：CommitLog → MemTable → SSTable；读：Bloom Filter → 合并多版本 |
| **修复机制** | Hinted Handoff / Read Repair / Anti-Entropy Repair |

**HBase 核心特性**：

| 组件 | 说明 |
|------|------|
| **ZooKeeper** | 协调服务，Master 选举、元数据存储 |
| **HMaster** | 管理 Region 分配、负载均衡 |
| **RegionServer** | 处理读写请求，管理 Region |
| **HDFS** | 底层存储，保证数据可靠性 |
| **LSM-Tree** | 日志结构合并，顺序写入优化 |


## 四、实践指南

### 4.1 部署配置

**Cassandra 关键配置（cassandra.yaml）**：
```yaml
# 集群名称
cluster_name: 'Production Cluster'

# 数据目录
data_file_directories:
  - /var/lib/cassandra/data
commitlog_directory: /var/lib/cassandra/commitlog
saved_caches_directory: /var/lib/cassandra/saved_caches

# 分区器（一致性哈希）
partitioner: org.apache.cassandra.dht.Murmur3Partitioner

# 种子节点
seed_provider:
  - class_name: org.apache.cassandra.locator.SimpleSeedProvider
    parameters:
      - seeds: "10.0.0.1,10.0.0.2,10.0.0.3"

# 多数据中心配置
endpoint_snitch: GossipingPropertyFileSnitch

# 并发控制
concurrent_reads: 32
concurrent_writes: 32

# JVM 调优（jvm.options）
-Xms31G
-Xmx31G
```

### 4.2 数据建模最佳实践

**Cassandra 建模原则**：
```sql
-- 反例：按用户查询订单（需要 ALLOW FILTERING）
CREATE TABLE orders_bad (
    order_id uuid PRIMARY KEY,
    user_id uuid,
    order_date timestamp,
    amount decimal
);

-- 正例：按用户分区，支持高效查询
CREATE TABLE orders_by_user (
    user_id uuid,
    order_date timestamp,
    order_id uuid,
    amount decimal,
    PRIMARY KEY (user_id, order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC);

-- 物化视图支持其他查询模式
CREATE MATERIALIZED VIEW orders_by_date AS
    SELECT * FROM orders_by_user
    PRIMARY KEY (order_date, user_id, order_id);
```

```
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. 散列前缀（避免热点）                                                    │
│      Row Key: MD5(user_id)[0:4] + user_id + timestamp                       │
│      例: a3f2_12345_1649097600                                              │
│                                                                             │
│   2. 反转时间戳（最新数据在前）                                              │
│      Row Key: user_id + (Long.MAX_VALUE - timestamp)                        │
│      例: 12345_9223372036854775807                                          │
│                                                                             │
│   3. 复合键（多维度查询）                                                    │
│      Row Key: region_id + bucket + user_id                                  │
│      例: bj_07_12345  （北京区域，07 号桶）                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 4.3 常见问题

**Q1: Cassandra 的 "最后一英里" 问题是什么？**
A:
- 指删除数据后，直到所有副本都执行 compaction 才能真正释放空间
- 解决方案：使用 TTL 自动过期，定期执行 `nodetool repair`

**Q2: HBase Region 热点问题如何解决？**
A:
- 预分区（Pre-splitting）：创建表时预先分好 Region
- 散列 Row Key：避免连续递增的 Row Key
- 监控 Region 大小，及时调整 `hbase.hregion.max.filesize`

**Q3: ScyllaDB 与 Cassandra 的兼容性如何？**
A:
- ScyllaDB 完全兼容 Cassandra 的 CQL 协议和 SSTable 格式
- 可以逐步迁移：混合部署期间两者可互操作
- 注意事项：某些 Cassandra 特性（如 SASI 索引）不支持
