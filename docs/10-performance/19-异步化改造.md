# 异步化改造：消息队列与协程

## 核心概念

### 1. 同步 vs 异步

同步调用会阻塞线程等待结果，异步调用立即返回，通过回调或事件获取结果。

**对比分析**：

```
┌─────────────────────────────────────────────────────────────┐
│                   同步 vs 异步                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   同步调用：                                                 │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  调用方 ──→ 被调用方执行 ──→ 等待结果 ──→ 继续     │  │
│   │     │          10ms           10ms        │        │  │
│   │     └────────────────────────────────────┘        │  │
│   │                  阻塞20ms                          │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   异步调用：                                                 │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  调用方 ──→ 提交任务 ──→ 立即返回                   │  │
│   │     │          <1ms                                 │  │
│   │     │                                               │  │
│   │     │        被调用方执行 ──→ 回调/事件通知         │  │
│   │     │          10ms              │                  │  │
│   │     └───────────────────────────┘                   │  │
│   │                  阻塞<1ms                           │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   收益：                                                     │
│   - 提高系统吞吐量                                          │
│   - 降低响应延迟                                            │
│   - 资源更高效利用                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 异步化场景

**适用场景**：

| 场景 | 说明 | 异步方案 |
|------|------|----------|
| 非核心流程 | 日志记录、通知发送 | 消息队列 |
| 耗时操作 | 文件上传、图片处理 | 异步任务 |
| 外部调用 | 第三方API、支付 | 异步HTTP |
| 批量处理 | 数据导入、报表生成 | 任务队列 |
| 定时任务 | 数据清理、统计 | 调度系统 |

```java
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean("taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(200);
        executor.setThreadNamePrefix("async-");
        executor.initialize();
        return executor;
    }
}

@Service
public class OrderService {

    @Autowired
    private MessageProducer messageProducer;

    // 同步核心流程
    public Order createOrder(OrderRequest request) {
        // 1. 保存订单（同步）
        Order order = orderRepository.save(request.toOrder());

        // 2. 扣减库存（同步）
        inventoryService.deduct(order);

        // 3. 发送通知（异步）
        messageProducer.sendOrderCreatedEvent(order);

        // 4. 更新统计（异步）

        return order;
```
```java
    }

// 定时扫描未投递消息
@Component
public class MessageResendJob {

    @Scheduled(fixedRate = 60000)
    public void resendPendingMessages() {
        List<Message> pendingMessages = messageRepository
            .findByStatusAndCreateTimeBefore("PENDING",
                LocalDateTime.now().minusMinutes(5));

        for (Message message : pendingMessages) {
            try {
                messageProducer.send(message.getTopic(), message.getBody());
                message.setStatus("SENT");
                messageRepository.save(message);
            } catch (Exception e) {
                // 重试次数+1
                message.setRetryCount(message.getRetryCount() + 1);
                if (message.getRetryCount() > 3) {
                    message.setStatus("FAILED");
                    alert.send("Message send failed: " + message.getId());
                }
                messageRepository.save(message);
            }
        }
    }
}
```

### 协程异步化

**1. Go协程模式**：

```go
package main

import (
    "context"
    "sync"
    "time"
)

// 异步处理订单
func processOrderAsync(order Order) {
    // 启动多个协程并行处理
    var wg sync.WaitGroup

    // 1. 发送通知
    wg.Add(1)
    go func() {
        defer wg.Done()
        sendNotification(order)
    }()

    // 2. 更新统计
    wg.Add(1)
    go func() {
        defer wg.Done()
        updateStats(order)
    }()

    // 3. 记录日志
    wg.Add(1)
    go func() {
        defer wg.Done()
        writeLog(order)
    }()

    // 不等待，立即返回（或者wg.Wait()根据需求）
}

// 带超时的异步调用
func callExternalAPI(ctx context.Context, req Request) (Response, error) {
    ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    resultCh := make(chan Response, 1)
    errCh := make(chan error, 1)

    go func() {
        resp, err := httpClient.Do(req)
        if err != nil {
            errCh <- err
            return
        }
        resultCh <- resp
    }()

    select {
    case resp := <-resultCh:
        return resp, nil
    case err := <-errCh:
        return Response{}, err
    case <-ctx.Done():
        return Response{}, ctx.Err()
    }
}

// Pipeline模式
func processPipeline(inputs []Input) []Output {
    // 阶段1：解析
    parseCh := make(chan Parsed, 100)
    go func() {
        for _, input := range inputs {
            parseCh <- parse(input)
        }
        close(parseCh)
    }()

    // 阶段2：处理
    processCh := make(chan Processed, 100)
    go func() {
        for parsed := range parseCh {
            processCh <- process(parsed)
        }
        close(processCh)
    }()

    // 阶段3：输出
    var outputs []Output
    for processed := range processCh {
        outputs = append(outputs, format(processed))
    }

    return outputs
}
```

**2. Java CompletableFuture**：

```java
@Service
public class AsyncOrderService {

    public CompletableFuture<Order> createOrderAsync(OrderRequest request) {
        return CompletableFuture.supplyAsync(() -> {
            // 1. 保存订单
            return orderRepository.save(request.toOrder());
        }, taskExecutor).thenApplyAsync(order -> {
            // 2. 扣减库存
            inventoryService.deduct(order);
            return order;
        }).thenComposeAsync(order -> {
            // 3. 异步发送通知
            return sendNotificationAsync(order)
                .thenApply(v -> order);
        }).exceptionally(ex -> {
            // 异常处理
            log.error("Create order failed", ex);
            throw new OrderException("Create order failed", ex);
        });
    }

    private CompletableFuture<Void> sendNotificationAsync(Order order) {
        return CompletableFuture.runAsync(() -> {
            emailService.sendOrderConfirmation(order);
        }, taskExecutor);
    }

    // 并行调用多个服务
    public OrderDetails getOrderDetails(Long orderId) {
        CompletableFuture<Order> orderFuture = CompletableFuture
            .supplyAsync(() -> orderRepository.findById(orderId), taskExecutor);

        CompletableFuture<List<OrderItem>> itemsFuture = CompletableFuture
            .supplyAsync(() -> itemRepository.findByOrderId(orderId), taskExecutor);

        CompletableFuture<PaymentInfo> paymentFuture = CompletableFuture
            .supplyAsync(() -> paymentService.getPaymentInfo(orderId), taskExecutor);

        // 等待所有完成
        CompletableFuture.allOf(orderFuture, itemsFuture, paymentFuture).join();

        return new OrderDetails(
            orderFuture.join(),
            itemsFuture.join(),
            paymentFuture.join()
        );
    }
}
```

## 案例数据

### 订单系统异步化改造

**改造前**：

```
创建订单流程（同步）：
1. 保存订单  10ms
2. 扣减库存  20ms
3. 发送通知  50ms
4. 更新统计  30ms
5. 记录日志  10ms
总计：120ms
吞吐量：100 TPS
```

**改造后**：

```
创建订单流程（异步）：
1. 保存订单  10ms
2. 扣减库存  20ms
3. 发送通知  0ms（异步）
4. 更新统计  0ms（异步）
5. 记录日志  0ms（异步）
总计：30ms
吞吐量：1000 TPS
```

**效果**：

| 指标 | 同步 | 异步 | 改进 |
|------|------|------|------|
| 平均RT | 120ms | 30ms | 75%↓ |
| 吞吐量 | 100 TPS | 1000 TPS | 10倍↑ |
| 用户体验 | 明显延迟 | 即时响应 | 显著提升 |

---

**总结**：异步化改造是提升系统性能的有效手段。通过将非核心流程异步化，可以显著降低响应时间，提升吞吐量。需要注意消息可靠性、异常处理和监控告警。
