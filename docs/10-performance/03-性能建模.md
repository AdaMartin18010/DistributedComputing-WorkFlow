# 性能建模：排队论与Little定律

## 核心概念

### 1. 排队论基础

排队论（Queuing Theory）是研究服务系统中排队现象的数学理论，为分布式系统性能分析提供理论基础。

**基本排队系统模型**：

```
        到达率 λ                    服务率 μ
            ↓                          ↓
    ┌──────────────┐    ┌─────────────────────────┐
    │   客户到达    │───→│  队列  │  服务台（c个）  │
    │  (泊松过程)   │    │  (等待) │  (指数分布)    │
    └──────────────┘    └─────────────────────────┘
                                    │
                                    ↓
                              服务完成后离开
```

**Kendall记号**：

```
A/B/c/K/m/Z

A - 到达间隔时间分布
    M: 指数分布 (Markovian)
    D: 确定性分布 (Deterministic)
    G: 一般分布 (General)

B - 服务时间分布
    M: 指数分布
    D: 确定性分布
    G: 一般分布

c - 服务台数量
K - 系统容量（队列+服务台）
m - 顾客源数量
Z - 服务规则（FIFO/LIFO/PRIORITY）

常见模型：
M/M/1 - 单服务台，到达和服务均为指数分布
M/M/c - 多服务台
M/G/1 - 一般服务时间分布
```

### 2. Little定律

Little定律是排队论中最基础且最重要的定律，建立了系统中平均顾客数、到达率和平均逗留时间之间的关系。

**定律表述**：

```
L = λ × W

其中：
L = 系统中平均顾客数（队列中+正被服务）
λ = 平均到达率（单位时间到达的顾客数）
W = 顾客在系统中的平均逗留时间

推论：
Lq = λ × Wq

Lq = 队列中平均顾客数
Wq = 顾客平均等待时间
```

**直观理解**：

```
场景：高速公路收费站

假设：
- 每分钟到达10辆车（λ = 10辆/分钟）
- 每辆车平均在系统中停留2分钟（W = 2分钟）

根据Little定律：
L = 10 × 2 = 20辆

即：系统中平均有20辆车（排队+正在缴费）
```

### 3. 分布式系统中的排队模型

**Web服务器排队模型**：

```
                    用户请求
                       ↓
    ┌──────────────────────────────────────────────┐
    │                  Nginx                        │
    │  ┌──────────────────────────────────────┐    │
    │  │  请求队列 (Kendall: M/M/c队列)        │    │
    │  │  - 到达率: λ requests/second         │    │
    │  │  - 服务率: μ requests/second/worker  │    │
    │  │  - 工作线程: c                        │    │
    │  └──────────────────────────────────────┘    │
    └──────────────────────────────────────────────┘
                       ↓
    ┌──────────────────────────────────────────────┐
    │              Application Server               │
    │  ┌──────────────────────────────────────┐    │
    │  │  业务处理队列                         │    │
    │  │  - Tomcat线程池                       │    │
    │  │  - 数据库连接池                       │    │
    │  └──────────────────────────────────────┘    │
    └──────────────────────────────────────────────┘
```

**数据库连接池模型（M/M/c/K）**：

```
参数定义：
- λ：应用请求连接的速率
- μ：连接使用的平均时间的倒数
- c：连接池大小
- K：连接池大小（无排队，拒绝策略）

关键指标：
- 连接利用率: ρ = λ / (c×μ)
- 连接拒绝概率: P_reject
- 平均等待时间: Wq
```

## 实践方法

### 基于排队论的性能估算

**1. 单服务台系统（M/M/1）**：

```
系统利用率: ρ = λ/μ  (必须 < 1，否则队列无限增长)

系统中平均顾客数: L = ρ / (1 - ρ) = λ / (μ - λ)

平均逗留时间: W = 1 / (μ - λ)

平均等待时间: Wq = ρ / (μ - λ) = λ / (μ × (μ - λ))

队列中平均顾客数: Lq = ρ² / (1 - ρ) = λ² / (μ × (μ - λ))

系统中有n个顾客的概率: Pn = (1-ρ) × ρⁿ
```

**2. 多服务台系统（M/M/c）**：

```
系统利用率: ρ = λ / (c×μ)

服务台全忙的概率（Erlang-C公式）：

         ( (c×ρ)^c / c! ) × ( 1 / (1-ρ) )
Pc = ─────────────────────────────────────────────
     Σ(k=0 to c-1) (c×ρ)^k/k! + (c×ρ)^c/(c!×(1-ρ))

平均等待时间: Wq = Pc / (c×μ - λ)

平均队列长度: Lq = λ × Wq
```

**3. 系统容量规划计算**：

```python
import math

class QueueingModel:
    """排队论计算工具类"""

    @staticmethod
    def mm1_metrics(arrival_rate, service_rate):
        """M/M/1队列性能指标"""
        rho = arrival_rate / service_rate

        if rho >= 1:
            return {"error": "系统不稳定，ρ >= 1"}

        return {
            "utilization": rho,
            "avg_in_system": rho / (1 - rho),          # L
            "avg_in_queue": rho**2 / (1 - rho),        # Lq
            "avg_wait_time": rho / (service_rate - arrival_rate),  # Wq
            "avg_system_time": 1 / (service_rate - arrival_rate),  # W
            "prob_empty": 1 - rho,                      # P0
        }

    @staticmethod
    def mmc_metrics(arrival_rate, service_rate, servers):
        """M/M/c队列性能指标"""
        rho = arrival_rate / (servers * service_rate)

        if rho >= 1:
            return {"error": "系统不稳定，ρ >= 1"}

        # 计算P0（系统空闲概率）
        sum_term = sum(
            (arrival_rate / service_rate)**k / math.factorial(k)
            for k in range(servers)
        )

        last_term = ((arrival_rate / service_rate)**servers / math.factorial(servers)) * (1 / (1 - rho))
        P0 = 1 / (sum_term + last_term)

        # Erlang-C公式：所有服务台忙的概率
        Pc = last_term * P0

        # 队列中的平均顾客数
        Lq = Pc * rho / (1 - rho)

        return {
            "utilization": rho,
            "P0": P0,
            "prob_all_busy": Pc,
            "avg_in_queue": Lq,
            "avg_wait_time": Lq / arrival_rate,
            "avg_in_system": Lq + arrival_rate / service_rate,
            "avg_system_time": Lq / arrival_rate + 1 / service_rate,
        }

    @staticmethod
    def required_servers(arrival_rate, service_rate, max_wait_time, max_utilization=0.8):
        """计算所需服务台数量"""
        c = max(1, int(arrival_rate / service_rate) + 1)

        while c <= 1000:  # 设置上限
            metrics = QueueingModel.mmc_metrics(arrival_rate, service_rate, c)

            if "error" not in metrics:
                if metrics["avg_wait_time"] <= max_wait_time and metrics["utilization"] <= max_utilization:
                    return {
                        "servers": c,
                        "metrics": metrics
                    }
            c += 1

        return {"error": "无法满足性能要求"}

# 使用示例
model = QueueingModel()

# API网关场景
# 峰值QPS: 10000, 平均处理时间: 10ms
result = model.required_servers(
    arrival_rate=10000,      # 10000 req/s
    service_rate=100,        # 1/0.01 = 100 req/s per worker
    max_wait_time=0.05,      # 最大等待50ms
    max_utilization=0.7
)

print(f"需要工作线程数: {result['servers']}")
print(f"系统利用率: {result['metrics']['utilization']:.2%}")
print(f"平均等待时间: {result['metrics']['avg_wait_time']*1000:.2f}ms")
```

### Little定律在容量规划中的应用

**案例：电商订单系统容量评估**

```
业务场景：
- 双11峰值订单：50万单/小时 = 139单/秒
- 订单处理流程：创建→扣库存→支付→发货
- 每个订单平均在系统停留时间：30分钟（W）

根据Little定律计算并发订单数：
L = λ × W = 139 × (30 × 60) = 250,200 单

即：系统中同时存在约25万笔处于各种状态的订单

各阶段并发计算：
- 待支付：λ_pay × W_pay = 139 × 600 = 83,400单
- 已支付待发货：λ_ship × W_ship = 139 × 1200 = 166,800单
```

**连接池大小估算**：

```
场景：应用服务器连接池配置

已知：
- 应用QPS: 10000
- 每个请求平均执行时间: 50ms
- 数据库查询占比: 80%
- 每个请求平均查询次数: 5次

计算：
1. 数据库操作到达率：λ = 10000 × 0.8 × 5 = 40000次/秒
2. 平均服务时间：1/μ = 50ms / 5 = 10ms
3. 服务率：μ = 100次/秒/连接

目标P99等待时间 < 5ms
根据M/M/c模型反推：
- 需要连接数 c ≈ 500
- 系统利用率 ρ ≈ 80%

推荐配置：
- 最小连接数: 100
- 最大连接数: 600
- 连接超时: 5s
```

```python
import numpy as np
import matplotlib.pyplot as plt

class SystemCapacityModel:
    """系统容量规划模型"""

    def __init__(self, base_qps, growth_rate, target_latency):
        self.base_qps = base_qps
        self.growth_rate = growth_rate
        self.target_latency = target_latency

    def project_capacity(self, months=12):
        """预测未来容量需求"""
        months_range = np.arange(0, months + 1)
        projected_qps = self.base_qps * ((1 + self.growth_rate) ** months_range)

        # 根据排队论计算所需服务器数量
        servers_needed = []
        for qps in projected_qps:
            # 假设单机能处理1000 QPS，利用率<70%
            servers = max(2, int(qps / 700) + 1)
            servers_needed.append(servers)

        return {
            "months": months_range.tolist(),
            "projected_qps": projected_qps.tolist(),
            "servers_needed": servers_needed
        }

    def plot_projection(self, data):
        """可视化容量规划"""
        fig, (ax1, ax2) = plt.subplots(2, 1, figsize=(10, 8))

        # QPS增长
        ax1.plot(data["months"], data["projected_qps"], 'b-', marker='o')
        ax1.set_xlabel('Month')
        ax1.set_ylabel('QPS')
        ax1.set_title('Traffic Growth Projection')
        ax1.grid(True)

        # 服务器需求
        ax2.bar(data["months"], data["servers_needed"], color='green', alpha=0.7)
        ax2.set_xlabel('Month')
        ax2.set_ylabel('Servers Needed')
        ax2.set_title('Server Capacity Planning')
        ax2.grid(True, axis='y')

        plt.tight_layout()
        plt.savefig('capacity_planning.png')

# 使用示例
model = SystemCapacityModel(
    base_qps=10000,
    growth_rate=0.15,  # 月增长15%
    target_latency=0.1
)

projection = model.project_capacity(months=12)
model.plot_projection(projection)
```

def plot_latency_curve():
    """绘制利用率与延迟关系曲线"""
    import matplotlib.pyplot as plt

    # M/M/1队列理论曲线
    rho_values = np.linspace(0.1, 0.99, 100)
    service_time = 0.01  # 10ms服务时间

    # 理论延迟公式: W = 1/(μ(1-ρ))
    latencies = [service_time / (1 - rho) for rho in rho_values]

    plt.figure(figsize=(10, 6))
    plt.plot(rho_values * 100, [l * 1000 for l in latencies], 'b-', linewidth=2)
    plt.axhline(y=50, color='r', linestyle='--', label='Target P99 (50ms)')
    plt.axvline(x=80, color='g', linestyle='--', label='Safe Utilization (80%)')

    plt.xlabel('Utilization (%)')
    plt.ylabel('Latency (ms)')
    plt.title('M/M/1 Queue: Utilization vs Latency')
    plt.legend()
    plt.grid(True)
    plt.yscale('log')
    plt.savefig('utilization_latency.png')

### 缓存命中率对性能的影响

**缓存排队模型**：

```
缓存系统：M/M/1队列
- 命中时服务率：μ_hit = 10000/s（缓存读取）
- 未命中时服务率：μ_miss = 100/s（DB查询）
- 命中率：h

等效服务率：
μ_effective = 1 / (h/μ_hit + (1-h)/μ_miss)

不同命中率下的等效服务率：
┌───────────┬──────────────────┐
│ 命中率(h) │ 等效服务率(req/s) │
├───────────┼──────────────────┤
│    50%    │       198         │
│    80%    │       495         │
│    90%    │       917         │
│    95%    │      1695         │
│    99%    │      5025         │
└───────────┴──────────────────┘

结论：缓存命中率从90%提升到99%，
     系统处理能力提升5.5倍
```

## 优化前后对比

### 基于排队论的优化决策

**案例：消息队列消费者优化**

```
优化前：
- 消息到达率：λ = 500 msg/s
- 消费者数量：c = 4
- 单消费者处理率：μ = 150 msg/s
- 系统利用率：ρ = 500/(4×150) = 83.3%
- 平均等待时间：Wq = 15ms
- 消息积压：Lq = 7.5条

问题：高峰期消息积压严重

优化方案（基于排队论计算）：
目标：ρ < 70%，Wq < 5ms

计算所需消费者数：
c ≥ λ / (μ × 0.7) = 500 / 105 = 4.76

优化后：
- 消费者数量：c = 6
- 系统利用率：ρ = 500/(6×150) = 55.6%
- 平均等待时间：Wq = 2.1ms
- 消息积压：Lq = 1.05条

效果：
- 等待时间下降：86%
- 积压消息下降：86%
- 成本增加：50%（4→6消费者）
```
