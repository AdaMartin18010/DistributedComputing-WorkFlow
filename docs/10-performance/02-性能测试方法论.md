# 性能测试方法论：负载/压力/稳定性

## 核心概念

### 1. 负载测试（Load Testing）

负载测试用于验证系统在正常和预期峰值负载下的行为表现。

**测试目标**：

- 确定系统在预期负载下的性能基线
- 验证系统是否满足SLA要求
- 发现性能瓶颈和容量上限

**负载模型类型**：

```
┌─────────────────────────────────────────────────────────┐
│                  负载模型对比                           │
├──────────────┬──────────────────────────────────────────┤
│   模型类型   │   特征描述                               │
├──────────────┼──────────────────────────────────────────┤
│  恒定负载    │  固定并发用户数，持续稳定压力            │
│  阶梯负载    │  逐步增加负载，寻找拐点                  │
│  波浪负载    │  模拟周期性波动，如早晚高峰              │
│  突发负载    │  瞬间增加大流量，测试弹性                │
│  混合负载    │  多业务场景按比例混合                    │
└──────────────┴──────────────────────────────────────────┘
```

**负载测试设计示例**：

```yaml
# 电商系统负载测试场景
test_scenarios:
  browse_products:
    weight: 60          # 60%用户浏览商品
    think_time: 3s      # 思考时间3秒

  add_to_cart:
    weight: 25          # 25%用户加购物车
    think_time: 5s

  place_order:
    weight: 10          # 10%用户下单
    think_time: 10s

  payment:
    weight: 5           # 5%用户支付
    think_time: 15s

  ramp_up: 5min         # 5分钟逐步加压到目标
  steady_state: 30min   # 稳定运行30分钟
  ramp_down: 5min       # 5分钟逐步减压
```

### 2. 压力测试（Stress Testing）

压力测试通过超过系统容量的负载，验证系统的极限和恢复能力。

**测试类型**：

**容量压力测试**：

```
目标：找到系统崩溃的临界点
方法：持续增加负载直到系统失败
关注点：最大承载量、失败模式、数据一致性
```

**尖峰压力测试**：

```
目标：验证突发流量处理能力
方法：瞬间投入2-5倍于容量的请求
关注点：自动扩缩容、熔断降级、队列积压
```

**耐久压力测试**：

```
目标：长时间高负载下的稳定性
方法：以80-90%容量运行24-72小时
关注点：内存泄漏、连接池耗尽、磁盘空间
```

**压力测试曲线**：

```
并发数
  │
  │        ┌──────┐
  │       /        \
  │      /          \
  │     /            \
  │    /              \
  │   /                \
  │  /                  \
  │ /                    \
  │/                      \
  └────────────────────────→ 时间
    预热    高压    恢复
    5min   10min   5min
```

### 3. 稳定性测试（Stability Testing）

稳定性测试验证系统在长时间运行下的可靠性。

**测试重点**：

- **内存稳定性**：无内存泄漏，GC正常
- **连接稳定性**：无连接泄漏，池化正常
- **资源稳定性**：文件句柄、线程数稳定
- **数据一致性**：无脏数据，事务完整

**稳定性测试检查清单**：

```
□ 7×24小时持续运行
□ 模拟真实的日/周/月周期波动
□ 定期进行故障注入（混沌工程）
□ 监控资源使用趋势
□ 验证数据完整性
□ 测试备份恢复流程
```

## 实践方法

### 性能测试流程

**完整测试生命周期**：

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ 需求分析 │ → │ 场景设计 │ → │ 环境搭建 │ → │ 脚本开发 │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
                                               ↓
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ 报告输出 │ ← │ 结果分析 │ ← │ 测试执行 │ ← │ 基线测试 │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
```

**各阶段输出物**：

| 阶段 | 输出物 | 关键检查点 |
|------|--------|------------|
| 需求分析 | 测试方案 | 业务场景、性能目标、通过标准 |
| 场景设计 | 测试场景 | 用户行为模型、数据模型 |
| 环境搭建 | 环境文档 | 与生产1:1或按比例缩放 |
| 脚本开发 | 测试脚本 | 参数化、关联、断言 |
| 基线测试 | 基线报告 | 单用户验证、数据准备 |
| 测试执行 | 测试日志 | 监控数据、错误日志 |
| 结果分析 | 分析报告 | 瓶颈定位、优化建议 |
| 报告输出 | 测试报告 | 结论、风险、建议 |

### 测试数据准备

**数据量级设计**：

```python
# 基于生产数据推算测试数据量
def calculate_test_data_scale(
    production_users: int,
    production_orders: int,
    test_concurrency: int,
    production_concurrency: int
) -> dict:
    """
    按比例缩放原则：
    1. 并发数按比例缩放
    2. 数据量至少保留1年热数据
    3. 索引和数据分布与生产一致
    """
    scale_factor = test_concurrency / production_concurrency

    return {
        "user_count": max(100000, int(production_users * scale_factor)),
        "order_count": max(1000000, int(production_orders * scale_factor)),
        "product_count": 50000,
        "inventory_records": 200000,
        "hot_data_ratio": 0.2,  # 20%热点数据
    }
```

**数据构造策略**：

```sql
-- 构造真实分布的测试数据

```

### 测试场景设计

**真实用户行为建模**：

```javascript
// k6 用户行为脚本示例
import http from 'k6/http';
import { check, sleep, group } from 'k6';
import { Rate, Trend } from 'k6/metrics';

// 自定义指标
const loginErrorRate = new Rate('login_errors');
const apiLatency = new Trend('api_latency');

export const options = {
  scenarios: {
    // 正常负载场景
    normal_load: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '5m', target: 100 },   // 预热
        { duration: '30m', target: 100 },  // 稳定
        { duration: '5m', target: 0 },     // 收尾
      ],
      gracefulRampDown: '30s',
    },
    // 峰值场景
    peak_load: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '2m', target: 500 },
        { duration: '10m', target: 500 },
        { duration: '2m', target: 0 },
      ],
      startTime: '45m',  // 延迟启动
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<200', 'p(99)<500'],
    http_req_failed: ['rate<0.01'],
    login_errors: ['rate<0.05'],
  },
};

export default function () {
  group('用户登录流程', () => {
    // 1. 获取登录页
    let loginPage = http.get('https://api.example.com/auth/captcha');
    check(loginPage, {
      'captcha loaded': (r) => r.status === 200,
    });

    // 2. 登录
    let loginRes = http.post('https://api.example.com/auth/login', {
      username: `user_${__VU}`,
      password: 'test123',
      captcha: '1234',
    });

    let loginSuccess = check(loginRes, {
      'login successful': (r) => r.status === 200 && r.json('token') !== '',
    });

    loginErrorRate.add(!loginSuccess);
    apiLatency.add(loginRes.timings.duration);

    if (loginSuccess) {
      let token = loginRes.json('token');

      group('核心业务操作', () => {
        // 浏览商品（60%概率）
        if (Math.random() < 0.6) {
          let products = http.get('https://api.example.com/products', {
            headers: { Authorization: `Bearer ${token}` },
          });
          check(products, { 'products loaded': (r) => r.status === 200 });
          sleep(Math.random() * 3 + 1);
        }

        // 加购物车（30%概率）
        if (Math.random() < 0.3) {
          let cart = http.post('https://api.example.com/cart/items', {
            product_id: Math.floor(Math.random() * 10000),
            quantity: 1,
          }, {
            headers: { Authorization: `Bearer ${token}` },
          });
          check(cart, { 'added to cart': (r) => r.status === 200 });
        }
      });
    }
  });

  sleep(Math.random() * 2 + 1);
}
```

### 电商大促压测案例

**全链路压测方案**：

```
压测流量：500万QPS（生产峰值120%）
压测时长：72小时（模拟双11周期）
数据隔离：影子库+影子表
流量染色：Header标记区分压测流量
```

**关键发现与优化**：

```
问题1：库存扣减热点
  - 现象：单SKU扣减RT飙升到2s
  - 根因：行锁竞争
  - 优化：分段锁+缓存预扣
  - 效果：RT降至50ms

问题2：订单号生成瓶颈
  - 现象：DB自增ID成为热点
  - 根因：集中式ID生成
  - 优化：雪花算法+本地号段
  - 效果：支持100万TPS

问题3：支付回调延迟
  - 现象：支付成功通知延迟5分钟
  - 根因：MQ消费能力不足
  - 优化：增加消费者+批量消费
  - 效果：延迟<100ms
```
