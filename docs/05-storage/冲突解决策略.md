## 📋 执行摘要

分布式系统中的数据冲突是多主复制、无主复制和最终一致性架构中的核心挑战。冲突解决策略决定了系统的行为和数据正确性。本文档系统阐述冲突检测方法、解决策略以及CRDT等无冲突数据结构的应用。

## 一、核心概念

### 1.1 冲突类型

```
┌─────────────────────────────────────────────────────────────┐
│                    分布式冲突类型                            │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 写写冲突（Write-Write Conflict）                        │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Client A              Client B                       │ │
│  │     │                     │                           │ │
│  │     ↓                     ↓                           │ │
│  │  Write X=1            Write X=2                       │ │
│  │     │                     │                           │ │
│  │     ↓                     ↓                           │ │
│  │  Node 1 ←───────────→ Node 2                          │ │
│  │     │                     │                           │ │
│  │  X=1                  X=2                             │ │
│  │     │                     │                           │ │
│  │     └──────────┬──────────┘                           │ │
│  │                ↓ Conflict!                            │ │
│  │         X=? (1 or 2)                                  │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. 读写冲突（Read-Write Conflict）                         │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Client A              Client B                       │ │
│  │     │                     │                           │ │
│  │     ↓ Read X            │                           │ │
│  │     X=1                 ↓ Write X=2                   │ │
│  │     │                     │                           │ │
│  │     ↓ (基于X=1计算)     │                           │ │
│  │  Write X=X+1          │                           │ │
│  │     │                     │                           │ │
│  │     ↓                     │                           │ │
│  │  X=2 (丢失更新)                                         │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. 时钟漂移导致的冲突                                      │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Node A (时钟快)       Node B (时钟慢)                 │ │
│  │     │                     │                           │ │
│  │  T=100: Write X=1       │                           │ │
│  │     │                     │                           │ │
│  │     │                  T=95: Write X=2               │ │
│  │     │                     │                           │ │
│  │  如果按LWW: X=2 (错误的旧值)                           │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 冲突检测方法

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **版本向量** | 记录每个节点的版本号 | 精确检测因果 | 向量增长 |
| **时间戳** | 物理/逻辑时间戳 | 简单 | 时钟漂移问题 |
| **哈希校验** | 比较数据指纹 | 快速 | 只能检测不同 |
| **向量时钟** | Dynamo模型 | 判断并发 | 复杂度高 |

## 二、技术细节

### 2.1 冲突解决策略

```
┌─────────────────────────────────────────────────────────────┐
│                    冲突解决策略                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. Last-Write-Wins (LWW)                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  策略：保留时间戳最新的写入                              │ │
│  │  实现：每个写入附带时间戳，取max(timestamp)             │ │
│  │  优点：简单，无需人工干预                                │ │
│  │  缺点：时钟漂移导致数据丢失                              │ │
│  │  适用：日志、缓存等对丢失不敏感的场景                     │ │
│  │                                                       │ │
│  │  改进：逻辑时钟（Lamport Clock）                        │ │
│  │  ├─ 单调递增计数器                                      │ │
│  │  └─ 避免物理时钟问题                                    │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. 应用层解决                                              │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  策略：保留所有版本，由应用决定如何合并                   │ │
│  │  实现：多版本存储，读取时返回所有冲突版本                 │ │
│  │  优点：灵活性高，可业务定制                              │ │
│  │  缺点：应用逻辑复杂                                      │ │
│  │  例：购物车合并、文档协作                                 │ │
│  │                                                       │ │
│  │  实现示例：                                             │ │
│  │  ┌─────────────────────────────────────────────────┐   │ │
│  │  │  {                                              │   │ │
│  │  │    "key": "cart:123",                          │   │ │
│  │  │    "values": [                                  │   │ │
│  │  │      {"data": {"item1": 2}, "version": [A:1]}, │   │ │
│  │  │      {"data": {"item2": 1}, "version": [B:1]}  │   │ │
│  │  │    ]                                            │   │ │
│  │  │  }                                              │   │ │
│  │  │  客户端合并: {"item1": 2, "item2": 1}           │   │ │
│  │  └─────────────────────────────────────────────────┘   │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. CRDT (Conflict-free Replicated Data Types)            │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  策略：数学上保证无冲突的数据结构                        │ │
│  │  特点：                                               │ │
│  │  ├─ 满足交换律、结合律、幂等性                           │ │
│  │  ├─ 无需锁定，自动合并                                   │ │
│  │  └─ 最终一致保证                                        │ │
│  │                                                       │ │
│  │  类型：                                               │ │
│  │  ├─ State-based CRDT: 传输完整状态                      │ │
│  │  └─ Operation-based CRDT: 传输操作                      │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 CRDT详解

```
┌─────────────────────────────────────────────────────────────┐
│                    CRDT 类型详解                             │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. G-Counter (Grow-only Counter)                          │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  只能递增的计数器                                        │ │
│  │  结构: 每个节点维护一个计数器数组                         │ │
│  │                                                       │ │
│  │  Node A: [5, 0]  (A增加5次)                            │ │
│  │  Node B: [0, 3]  (B增加3次)                            │ │
│  │  合并: [5, 3]                                          │ │
│  │  值: 5 + 3 = 8                                         │ │
│  │                                                       │ │
│  │  操作:                                                │ │
│  │  ├─ increment(): 本地计数器+1                          │ │
│  │  └─ merge(other): 取各节点max                          │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. PN-Counter (Positive-Negative Counter)                 │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  可增可减的计数器                                        │ │
│  │  结构: 两个G-Counter（增计数器 + 减计数器）               │ │
│  │  值: P - N                                             │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. G-Set (Grow-only Set)                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  只能添加元素的集合                                      │ │
│  │  合并: 集合取并集                                        │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  4. 2P-Set (Two-Phase Set)                                 │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  可添加可删除的集合                                      │ │
│  │  结构: G-Set(添加) + G-Set(删除，Tombstone)             │ │
│  │  元素在删除集合中即视为已删除                            │ │
│  │  限制: 删除后不能再添加同一元素（可用LWW-Element-Set）   │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  5. OR-Set (Observed-Removed Set)                          │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  观察删除集合，支持删除后再次添加                         │ │
│  │  每个元素附带唯一标签（tag）                             │ │
│  │  删除时记录被删除的tag                                   │ │
│  │  重新添加使用新tag                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  6. LWW-Register (Last-Write-Wins Register)                │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  简单的键值存储，LWW语义                                 │ │
│  │  结构: (timestamp, value)                              │ │
│  │  合并: 保留timestamp较大的                              │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 三、实践指南

### 3.1 CRDT实现示例

```java
// G-Counter 实现
public class GCounter {
    private final Map<String, Long> counters = new ConcurrentHashMap<>();
    private final String nodeId;

    public GCounter(String nodeId) {
        this.nodeId = nodeId;
        counters.put(nodeId, 0L);
    }

    public void increment() {
        counters.merge(nodeId, 1L, Long::sum);
    }

    public long get() {
        return counters.values().stream().mapToLong(Long::longValue).sum();
    }

    public void merge(GCounter other) {
        other.counters.forEach((node, count) ->
            counters.merge(node, count, Math::max));
    }
}

// PN-Counter 实现
public class PNCounter {
    private final GCounter increments;
    private final GCounter decrements;

    public void increment() {
        increments.increment();
    }

    public void decrement() {
        decrements.increment();
    }

    public long get() {
        return increments.get() - decrements.get();
    }

    public void merge(PNCounter other) {
        increments.merge(other.increments);
        decrements.merge(other.decrements);
    }
}

// OR-Set 实现
public class ORSet<T> {
    private final Map<T, Set<UUID>> additions = new HashMap<>();
    private final Map<T, Set<UUID>> removals = new HashMap<>();

    public void add(T element) {
        additions.computeIfAbsent(element, k -> new HashSet<>())
                 .add(UUID.randomUUID());
    }

    public void remove(T element) {
        Set<UUID> tags = additions.get(element);
        if (tags != null) {
            removals.computeIfAbsent(element, k -> new HashSet<>())
                    .addAll(tags);
        }
    }

    public Set<T> get() {
        Set<T> result = new HashSet<>();
        additions.forEach((elem, addTags) -> {
            Set<UUID> remTags = removals.getOrDefault(elem, Set.of());
            if (!addTags.equals(remTags)) {
                result.add(elem);
            }
        });
        return result;
    }

    public void merge(ORSet<T> other) {
        other.additions.forEach((elem, tags) ->
            additions.merge(elem, tags, (a, b) -> {
                a.addAll(b); return a;
            }));
        other.removals.forEach((elem, tags) ->
            removals.merge(elem, tags, (a, b) -> {
                a.addAll(b); return a;
            }));
    }
}
```

## 四、常见问题

**Q1: CRDT有什么限制？**
A: 1) 只有特定数据结构；2) 可能存储开销大（保留多版本）；3) 不是所有操作都可CRDT化。

**Q2: 如何设计避免冲突的架构？**
A: 1) 单Key单写者；2) 按用户分片；3) 使用不可变数据；4) 异步最终一致操作。

**Q3: 时钟同步重要吗？**
A: LWW策略下很重要。建议使用逻辑时钟或TrueTime（Spanner）。NTP只能减小时钟漂移，不能保证。
