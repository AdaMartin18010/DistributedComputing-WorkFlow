## 📋 执行摘要

多主复制（Multi-Master Replication）允许多个节点同时接受写入，提高可用性和写入性能，但也带来了数据冲突的复杂性。MySQL Group Replication、Galera Cluster和PostgreSQL BDR是主流实现方案，适用于需要多地写入或高可用写入的场景。

## 一、核心概念

### 1.1 多主复制架构

```
┌─────────────────────────────────────────────────────────────┐
│                    多主复制架构对比                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  双主（Active-Standby）：                                    │
│  ┌─────────┐           ┌─────────┐                         │
│  │ Master1 │←─────────→│ Master2 │                         │
│  │ (Write) │  Binlog   │ (Standby)│                         │
│  │  ↓      │   复制    │         │                         │
│  │ Slave   │           │  (Read) │                         │
│  └─────────┘           └─────────┘                         │
│  特点：主备模式，同一时间只有一台可写，故障时切换              │
│                                                            │
│  ────────────────────────────────────────────────────────  │
│                                                            │
│  真多主（Active-Active）：                                    │
│  ┌─────────┐           ┌─────────┐                         │
│  │ Node A  │←─────────→│ Node B  │                         │
│  │ (Write) │  同步复制  │ (Write) │                         │
│  │  ↓      │           │  ↓      │                         │
│  │ Slave   │           │ Slave   │                         │
│  └─────────┘           └─────────┘                         │
│       ↑                     ↑                              │
│       └──────────┬──────────┘                              │
│                  ↓                                         │
│            ┌─────────┐                                     │
│            │ Node C  │                                     │
│            │ (Write) │                                     │
│            └─────────┘                                     │
│  特点：所有节点可写，需处理冲突                               │
│                                                            │
│  ────────────────────────────────────────────────────────  │
│                                                            │
│  主从+多主混合：                                             │
│  ┌─────────┐           ┌─────────┐                         │
│  │ Master1 │←─────────→│ Master2 │                         │
│  │  ↓  ↓   │  同步复制  │  ↓  ↓   │                         │
│  │ S1  S2  │           │ S3  S4  │                         │
│  └─────────┘           └─────────┘                         │
│       ↑                     ↑                              │
│       └──────────┬──────────┘                              │
│                  ↓ 异地多活                                 │
│            ┌─────────┐                                     │
│            │ Master3 │                                     │
│            └─────────┘                                     │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 方案对比

| 方案 | 一致性 | 性能 | 冲突处理 | 适用场景 |
|------|--------|------|----------|----------|
| **MySQL双主** | 最终一致 | 高 | 应用层处理 | 主备切换 |
| **Group Replication** | 强一致 | 中 | 组内冲突检测 | 高可用集群 |
| **Galera Cluster** | 强一致 | 中 | 认证机制 | 多地写入 |
| **PostgreSQL BDR** | 逻辑一致 | 中 | 冲突回调 | PG多主 |

## 二、技术细节

### 2.1 MySQL Group Replication

```
┌─────────────────────────────────────────────────────────────┐
│                MySQL Group Replication                       │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  架构特点：                                                  │
│  ├─ 基于Paxos变种（Mencius）的组通信                         │
│  ├─ 单主模式（Single Primary）或多主模式（Multi Primary）    │
│  ├─ 自动故障检测和切换                                       │
│  └─ 强一致性（多数派确认）                                    │
│                                                            │
│  工作流程：                                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Primary → 提交事务 → 发送到Group                      │ │
│  │                           ↓                           │ │
│  │                    Certification                        │ │
│  │                    (冲突检测)                           │ │
│  │                           ↓                           │ │
│  │                    多数派确认                            │ │
│  │                           ↓                           │ │
│  │                    本地执行 + Binlog传播                │ │
│  │                           ↓                           │ │
│  │                    返回客户端                            │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  Certification过程：                                         │
│  ├─ 检测并发事务的写集冲突                                    │
│  ├─ 冲突事务回滚（第一个提交者成功）                           │
│  └─ 无需全局锁，性能优于传统两阶段提交                          │
│                                                            │
│  限制：                                                      │
│  ├─ 仅支持InnoDB                                            │
│  ├─ 表必须有主键                                            │
│  ├─ 不支持SERIALIZABLE隔离级别                               │
│  └─ 网络分区时少数派不可用                                    │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Galera Cluster

```
┌─────────────────────────────────────────────────────────────┐
│                  Galera Cluster (MariaDB/Percona)           │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  特点：                                                      │
│  ├─ 基于wsrep（Write Set Replication）                      │
│  ├─ 真正的多点写入（Certification-based Replication）       │
│  ├─ 同步复制，所有节点数据一致                               │
│  └─ 自动节点加入和恢复                                       │
│                                                            │
│  复制流程：                                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  本地执行 → 生成Write Set → GCache                     │ │
│  │                              ↓                        │ │
│  │                    Certification (Galera Layer)       │ │
│  │                    ├─ 全局排序                          │ │
│  │                    ├─ 冲突检测                          │ │
│  │                    └─ 认证通过                          │ │
│  │                              ↓                        │ │
│  │                    Apply to Local InnoDB              │ │
│  │                              ↓                        │ │
│  │                    广播到其他节点                        │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  状态传输（SST/IST）：                                        │
│  ├─ SST（State Snapshot Transfer）：全量同步                 │
│  ├─ IST（Incremental State Transfer）：增量同步               │
│  └─ 使用GCache缓存近期事务，支持增量恢复                      │
│                                                            │
│  流控（Flow Control）：                                       │ │
│  └─ 防止慢节点拖垮整个集群                                    │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 三、实践指南

### 3.1 Group Replication配置

```ini
# my.cnf
server_id = 1
gtid_mode = ON
enforce_gtid_consistency = ON
binlog_checksum = NONE

# Group Replication
plugin_load_add = group_replication.so
group_replication_group_name = "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"
group_replication_start_on_boot = off
group_replication_local_address = "node1:33061"
group_replication_group_seeds = "node1:33061,node2:33061,node3:33061"
group_replication_bootstrap_group = off

# 单主/多主模式
group_replication_single_primary_mode = OFF  # ON=单主, OFF=多主
group_replication_enforce_update_everywhere_checks = ON
```

```sql
-- 初始化组复制
SET GLOBAL group_replication_bootstrap_group=ON;
START GROUP_REPLICATION;
SET GLOBAL group_replication_bootstrap_group=OFF;

-- 其他节点加入
START GROUP_REPLICATION;

-- 查看组成员
SELECT * FROM performance_schema.replication_group_members;
```

### 3.2 多主复制最佳实践

```
┌─────────────────────────────────────────────────────────────┐
│                    多主复制最佳实践                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 冲突避免策略：                                           │ │
│  ├─ 按业务划分写节点（不同表不同主）                          │ │
│  ├─ 按地域划分写节点（用户就近写入）                          │ │
│  ├─ 使用全局ID避免主键冲突（不同节点不同ID段）                 │ │
│  └─ 避免同时修改同一行                                       │ │
│                                                            │
│  2. 数据一致性：                                             │ │
│  ├─ 读已提交使用本地读                                       │ │
│  ├─ 关键读使用全局读（wsrep_sync_wait）                      │ │
│  └─ 定期数据校验（pt-table-checksum）                        │ │
│                                                            │
│  3. 网络分区处理：                                           │ │
│  ├─ 多数派继续服务                                           │ │
│  ├─ 少数派自动停写（避免脑裂）                                │ │
│  └─ 配置仲裁节点（ witness node）                            │ │
│                                                            │
│  4. 监控告警：                                               │ │
│  ├─ 集群节点数量                                             │ │
│  ├─ 流控状态（Flow Control Pause）                           │ │
│  ├─ 冲突率                                                   │ │
│  └─ 复制延迟                                                 │ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 四、常见问题

**Q1: 多主复制适合什么场景？**
A: 多地写入、就近访问、高可用写入、需要自动故障转移的场景。不适合强一致性要求高的金融交易。

**Q2: 如何处理冲突？**
A: 1) 应用层避免冲突（推荐）；2) 使用Last-Write-Wins；3) 自定义冲突解决函数。

**Q3: 多主与分布式事务如何选择？**
A: 多主复制是最终一致方案，分布式事务（XA/2PC）是强一致但性能差。根据业务一致性要求选择。

**Q4: 写入性能如何优化？**
A: 1) 本地事务优先；2) 批量写入；3) 流控参数调优；4) 网络优化。
