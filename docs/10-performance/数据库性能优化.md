## 📋 执行摘要

数据库性能优化是系统性能调优的核心环节。通过合理的索引设计、SQL优化、连接池配置、分库分表和读写分离策略，可显著提升数据库吞吐量和降低查询延迟。

## 一、核心概念

### 1.1 定义与原理

**数据库性能优化**：通过架构设计、参数调优、SQL优化等手段，提升数据库系统的响应速度、吞吐量和资源利用率的过程。

**核心原理**：

- **减少IO**：索引减少磁盘扫描，缓存减少磁盘访问
- **减少计算**：优化SQL执行计划，减少排序和关联
- **并行处理**：读写分离、分片并行提升吞吐量
- **资源隔离**：连接池限制、限流防止资源耗尽

### 1.2 关键特性

- **系统性**：涉及应用、中间件、数据库多个层面
- **数据驱动**：基于慢查询、执行计划等客观数据
- **渐进式**：持续监控、分析、优化迭代
- **权衡性**：在一致性、可用性、性能间取舍

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 高并发OLTP | ⭐⭐⭐⭐⭐ | 优化响应时间和并发能力 |
| 海量数据查询 | ⭐⭐⭐⭐⭐ | 索引优化、分库分表 |
| 慢查询问题 | ⭐⭐⭐⭐⭐ | SQL优化、索引调整 |
| 大促峰值 | ⭐⭐⭐⭐⭐ | 读写分离、缓存预热 |
| 复杂分析报表 | ⭐⭐⭐⭐ | OLAP优化、列式存储 |

```
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤

```

#### 2.1.3 索引设计原则

```sql
-- 1. 最左前缀原则
-- 组合索引 (a,b,c) 的有效查询
WHERE a=1                    -- ✓ 使用索引
WHERE a=1 AND b=2            -- ✓ 使用索引
WHERE a=1 AND b=2 AND c=3    -- ✓ 使用索引
WHERE b=2                    -- ✗ 不使用索引（缺少a）
WHERE a=1 AND c=3            -- △ 部分使用（只用a）
WHERE a=1 AND b>2 AND c=3    -- △ 部分使用（只用a,b）

-- 2. 覆盖索引设计
-- 原查询
SELECT user_id, username, email FROM users WHERE status=1;
-- 创建覆盖索引
CREATE INDEX idx_status_cover ON users(status, user_id, username, email);
-- 结果：Extra列显示"Using index"，无需回表

-- 3. 避免冗余索引
-- 已有(a,b)索引时，单独创建(a)索引是冗余的
-- 但(b)索引不是冗余的（不满足最左前缀）

-- 4. 控制索引数量
-- 单表索引不超过5个
-- 单个索引列数不超过5列
```

#### 2.1.4 索引优化Checklist

```
□ 为WHERE、JOIN、ORDER BY、GROUP BY字段加索引
□ 优先使用组合索引减少单列索引数量
□ 区分度高的列放在组合索引前面
□ 避免对频繁更新的列创建索引
□ 长字符串使用前缀索引
□ 定期分析冗余和未使用索引
□ 大数据量导入时先删除索引，导入后重建
□ 使用EXPLAIN验证索引使用
```

#### 2.2.2 SQL优化技巧

```sql
-- 1. 避免SELECT *
-- 坏
SELECT * FROM users WHERE user_id = 1;
-- 好
SELECT user_id, username, email FROM users WHERE user_id = 1;

-- 2. 分页优化
-- 坏：深度分页
SELECT * FROM orders ORDER BY order_id LIMIT 1000000, 10;
-- 好：使用覆盖索引+子查询
SELECT o.* FROM orders o
JOIN (SELECT order_id FROM orders ORDER BY order_id LIMIT 1000000, 10) tmp
ON o.order_id = tmp.order_id;
-- 更好：基于游标/书签
SELECT * FROM orders WHERE order_id > 1000000 ORDER BY order_id LIMIT 10;

-- 3. 避免隐式类型转换
-- 坏：索引失效
SELECT * FROM users WHERE phone = 13800138000;  -- phone是varchar
-- 好
SELECT * FROM users WHERE phone = '13800138000';

-- 4. 避免在索引列上使用函数
-- 坏：索引失效
SELECT * FROM users WHERE DATE(created_at) = '2024-01-01';
-- 好：范围查询
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2024-01-02';

-- 5. OR优化
-- 坏
SELECT * FROM users WHERE user_id = 1 OR user_id = 2;
-- 好：使用IN
SELECT * FROM users WHERE user_id IN (1, 2);
-- 或UNION ALL（大数据量时更高效）
SELECT * FROM users WHERE user_id = 1
UNION ALL
SELECT * FROM users WHERE user_id = 2;

-- 6. 关联优化
-- 确保关联字段有索引
-- 小表驱动大表（MySQL优化器通常会自动调整）
SELECT * FROM small_table s
JOIN large_table l ON s.id = l.small_id;

-- 7. 批量插入优化
-- 坏：逐条插入
INSERT INTO users (name) VALUES ('a');
INSERT INTO users (name) VALUES ('b');
-- 好：批量插入
INSERT INTO users (name) VALUES ('a'), ('b'), ('c');
```

#### 2.3.2 HikariCP配置（推荐）

```yaml
# application.yml - Spring Boot + HikariCP
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mydb?useSSL=false&serverTimezone=UTC
    username: user
    password: pass
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      # 核心参数
      minimum-idle: 10           # 最小空闲连接数
      maximum-pool-size: 50      # 最大连接数
      connection-timeout: 30000  # 获取连接超时(30秒)
      idle-timeout: 600000       # 空闲连接超时(10分钟)
      max-lifetime: 1800000      # 连接最大生命周期(30分钟)

      # 性能优化
      pool-name: HikariPool-Primary
      auto-commit: true
      connection-test-query: SELECT 1
      validation-timeout: 5000

      # 监控
      register-mbeans: true
      metric-registry: myMetricRegistry

# 连接数计算公式：
# 连接数 = ((核心数 * 2) + 有效磁盘数)
# 例如：8核服务器，推荐连接数 = 8 * 2 + 1 = 17
# 但通常设置为 10-50 之间，根据实际负载调整
```

#### 2.4.3 ShardingSphere配置示例

```yaml
# application.yml - ShardingSphere-JDBC
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/db0
        username: root
        password: root
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/db1
        username: root
        password: root

    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds${0..1}.t_order_${0..15}
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: table-inline
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: database-inline
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake

        sharding-algorithms:
          table-inline:
            type: INLINE
            props:
              algorithm-expression: t_order_${order_id % 16}
          database-inline:
            type: INLINE
            props:
              algorithm-expression: ds${user_id % 2}

        key-generators:
          snowflake:
            type: SNOWFLAKE
            props:
              worker-id: 0

    props:
      sql-show: true
```

// 1. 配置多数据源
@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties("spring.datasource.master")
    public DataSource masterDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.slave")
    public DataSource slaveDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    public DataSource routingDataSource(
            @Qualifier("masterDataSource") DataSource master,
            @Qualifier("slaveDataSource") DataSource slave) {
        DynamicRoutingDataSource routing = new DynamicRoutingDataSource();
        Map<Object, Object> target = new HashMap<>();
        target.put("master", master);
        target.put("slave", slave);
        routing.setTargetDataSources(target);
        routing.setDefaultTargetDataSource(master);
        return routing;
    }
}

// 2. 上下文持有者
public class DataSourceContext {
    private static final ThreadLocal<String> CONTEXT = new ThreadLocal<>();

    public static void set(String dataSource) {
        CONTEXT.set(dataSource);
    }

    public static String get() {
        return CONTEXT.get();
    }

    public static void clear() {
        CONTEXT.remove();
    }
}

// 3. 动态数据源
public class DynamicRoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContext.get();
    }
}

// 4. AOP拦截
@Aspect
@Component
public class DataSourceAspect {

    @Before("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void beforeTransaction() {
        DataSourceContext.set("master");  // 事务走主库
    }

    @Before("execution(* com.example.service..*.get*(..)) || " +
            "execution(* com.example.service..*.find*(..)) || " +
            "execution(* com.example.service..*.query*(..))")
    public void beforeRead() {
        if (DataSourceContext.get() == null) {
            DataSourceContext.set("slave");  // 读操作走从库
        }
    }

    @After("@within(org.springframework.stereotype.Service)")
    public void afterService() {
        DataSourceContext.clear();
    }
}

## 四、实践指南

### 4.1 性能优化Checklist

```
□ 架构层
  □ 评估是否需要读写分离
  □ 评估是否需要分库分表
  □ 引入缓存层（Redis）
  □ 异步写入（消息队列）

□ 数据库层
  □ 选择合适的存储引擎（InnoDB）
  □ 合理配置缓冲池大小
  □ 开启慢查询日志
  □ 定期分析表和优化表

□ 应用层
  □ 使用连接池（HikariCP）
  □ 批量操作代替循环单条
  □ 事务尽量短小
  □ 预编译语句（PreparedStatement）

□ SQL层
  □ 为WHERE/JOIN字段加索引
  □ 避免SELECT *
  □ 避免深度分页
  □ 定期分析慢查询并优化
```

### 4.2 最佳实践

1. **索引设计**
   - 优先使用组合索引
   - 定期清理无用索引
   - 避免过度索引影响写入

2. **查询优化**
   - 使用EXPLAIN分析每个慢查询
   - 覆盖索引减少回表
   - 控制JOIN表数量

3. **连接管理**
   - 连接池大小按公式配置
   - 设置合理的超时时间
   - 监控连接池使用情况

4. **大表处理**
   - 提前规划分库分表
   - 归档历史数据
   - 使用分区表

### 4.3 常见问题

**Q1: 加了索引查询还是很慢？**
A: 检查是否使用了索引（EXPLAIN），是否存在隐式类型转换，是否有函数操作。

**Q2: 写入性能差？**
A: 检查索引数量，是否批量插入，事务是否过大，是否有锁竞争。

**Q3: 主从延迟严重？**
A: 检查从库性能，并行复制配置，大事务拆分，必要时强制走主库。
