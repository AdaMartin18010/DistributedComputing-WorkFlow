## 📋 执行摘要

分布式缓存是提升系统性能、降低数据库负载的核心技术，通过将热点数据存储在内存中，实现毫秒级响应和高并发支持。

---

## 一、核心概念

### 1.1 定义与原理

**缓存**是一种通过保存数据副本以提高读取性能的技术。分布式缓存则是将缓存数据分散存储在多个节点上，以支持大规模系统的性能需求。

**核心原理**：

- **局部性原理**：利用时间局部性（最近访问的数据可能再次被访问）和空间局部性（相邻数据可能被一起访问）
- **内存访问速度**：内存访问（~100ns）比磁盘访问（~10ms）快约10万倍
- **分布式存储**：通过数据分片将缓存分布到多个节点，实现水平扩展

### 1.2 关键特性

- **高性能**：内存操作，微秒级延迟
- **高并发**：支持数万到百万级QPS
- **分布式**：支持多节点集群，数据自动分片
- **持久化**：可选的RDB/AOF持久化机制
- **高可用**：主从复制、哨兵、集群模式

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 热点数据读取 | ⭐⭐⭐⭐⭐ | 用户信息、商品详情等高频读取场景 |
| 会话存储 | ⭐⭐⭐⭐⭐ | 分布式Session、用户登录状态 |
| 计算结果缓存 | ⭐⭐⭐⭐⭐ | 复杂查询结果、聚合计算结果 |
| 限流计数 | ⭐⭐⭐⭐ | 接口限流、滑动窗口计数 |
| 数据一致性要求极高 | ⭐⭐⭐ | 需要配合其他机制保证强一致 |

## 二、技术细节

### 2.1 本地缓存

#### Caffeine

```java
// 基础配置
Caffeine.newBuilder()
    .maximumSize(10_000)           // 最大条目数
    .expireAfterWrite(10, MINUTES) // 写入后过期时间
    .recordStats()                  // 开启统计
    .build(key -> loadData(key));
```

**核心特性**：

- **Window TinyLFU**：高效的缓存淘汰算法
- **异步加载**：支持异步数据加载
- **刷新机制**：支持定时刷新而非过期

#### Guava Cache

```java
LoadingCache<String, User> cache = CacheBuilder.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .removalListener(notification ->
        System.out.println("Removed: " + notification.getKey()))
    .build(new CacheLoader<String, User>() {
        @Override
        public User load(String key) {
            return loadUserFromDB(key);
        }
    });
```

### 2.3 多级缓存架构

```
┌─────────────────────────────────────────────────────────┐
│                      多级缓存架构                        │
├─────────────────────────────────────────────────────────┤
│  ┌─────────────┐                                        │
│  │  L1: 浏览器  │  HTTP Cache / Service Worker          │
│  └─────────────┘                                        │
│         ↓                                               │
│  ┌─────────────┐                                        │
│  │  L2: CDN    │  静态资源缓存                          │
│  └─────────────┘                                        │
│         ↓                                               │
│  ┌─────────────┐                                        │
│  │  L3: 本地   │  Caffeine / Guava Cache               │
│  └─────────────┘                                        │
│         ↓                                               │
│  ┌─────────────┐                                        │
│  │  L4: 远程   │  Redis / Memcached                    │
│  └─────────────┘                                        │
│         ↓                                               │
│  ┌─────────────┐                                        │
│  │  L5: 数据库 │  MySQL / PostgreSQL / Elasticsearch   │
│  └─────────────┘                                        │
└─────────────────────────────────────────────────────────┘
```

**访问策略**：

1. 先查本地缓存（L3）
2. 未命中查分布式缓存（L4）
3. 仍未命中查询数据库（L5）
4. 回写缓存时遵循：L4 → L3

### 2.4 缓存一致性策略

#### Cache Aside（旁路缓存）

```
读取流程：
┌──────┐     ┌──────┐     ┌──────┐
│Client│────→│Cache │     │  DB  │
└──┬───┘     └──┬───┘     └──────┘
   │            │
   │1.get(key)  │
   │───────────→│
   │            │
   │2.miss      │
   │←───────────│
   │            │
   │3.get(key)  │
   │────────────────────────────→│
   │            │
   │4.set(key)  │
   │───────────→│
   └────────────┘

写入流程：
┌──────┐     ┌──────┐     ┌──────┐
│Client│────→│  DB  │     │Cache │
└──┬───┘     └──┬───┘     └──┬───┘
   │            │            │
   │1.write data│            │
   │───────────→│            │
   │            │            │
   │2.OK        │            │
   │←───────────│            │
   │            │            │
   │3.del(key)  │            │
   │────────────────────────→│
   └────────────┴────────────┘
```

**优缺点**：

- ✅ 简单直观，DB是唯一数据源
- ❌ 存在短暂不一致窗口

#### Read/Write Through（直写/直读）

```
┌──────┐     ┌─────────────┐     ┌──────┐
│Client│────→│Cache Service│────→│  DB  │
└──┬───┘     └──────┬──────┘     └──────┘
   │                │
   │1.get(key)      │
   │───────────────→│
   │                │2.若miss则从DB加载
   │                │3.返回数据
   │←───────────────│
   └────────────────┘

写入：
   │1.put(key,value)│
   │───────────────→│
   │                │2.同步写入DB
   │                │3.更新缓存
```

#### Write Behind（异步回写）

```
写入流程：
┌──────┐     ┌──────┐     ┌─────────┐     ┌──────┐
│Client│────→│Cache │────→│ Write   │────→│  DB  │
└──┬───┘     └──┬───┘     │ Queue   │     └──────┘
   │            │         └─────────┘
   │1.write     │              │
   │───────────→│              │
   │            │2.立即返回    │
   │←───────────│              │
   │            │              │3.异步批量写入
   │            │              │───────────────→
```

### 2.5 缓存问题及解决方案

#### 缓存穿透（Cache Penetration）

**问题**：查询不存在的数据，每次都会打到数据库

**解决方案**：

```java
// 1. 布隆过滤器
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1000000,  // 预期插入数量
    0.01      // 误判率
);

// 2. 缓存空值
public Object getData(String key) {
    Object value = redis.get(key);
    if (value != null) {
        // 可能是空值标记
        return value.equals(NULL_VALUE) ? null : value;
    }

    value = db.get(key);
    if (value == null) {
        // 缓存空值，短过期时间
        redis.setex(key, 60, NULL_VALUE);
    } else {
        redis.setex(key, 3600, value);
    }
    return value;
}
```

#### 缓存击穿（Cache Breakdown）

**问题**：热点key突然过期，大量请求同时打到数据库

**解决方案**：

```java
// 1. 互斥锁
public Object getHotData(String key) {
    Object value = redis.get(key);
    if (value != null) return value;

    // 获取锁
    if (redis.setnx(key + "_lock", "1", 10)) {
        try {
            value = db.get(key);
            redis.setex(key, 3600, value);
        } finally {
            redis.del(key + "_lock");
        }
    } else {
        // 未获取到锁，短暂等待后重试
        Thread.sleep(100);
        return getHotData(key);
    }
    return value;
}

// 2. 逻辑过期（永不过期）
public Object getWithLogicalExpire(String key) {
    Object value = redis.get(key);
    if (value == null) return null;

    RedisData redisData = (RedisData) value;
    if (redisData.getExpireTime().isAfter(LocalDateTime.now())) {
        return redisData.getData(); // 未过期
    }

    // 已过期，尝试重建
    if (redis.setnx(key + "_lock", "1", 10)) {
        // 异步重建缓存
        executor.submit(() -> rebuildCache(key));
    }
    return redisData.getData(); // 返回过期数据
}
```

#### 缓存雪崩（Cache Avalanche）

**问题**：大量key同时过期，或Redis宕机，导致数据库压力剧增

**解决方案**：

```java
// 1. 过期时间加随机值
int baseExpire = 3600;
int randomExpire = ThreadLocalRandom.current().nextInt(300);
redis.setex(key, baseExpire + randomExpire, value);

// 2. 多级缓存降级
public Object getWithFallback(String key) {
    // L1: 本地缓存
    Object value = localCache.get(key);
    if (value != null) return value;

    // L2: Redis
    try {
        value = redis.get(key);
        if (value != null) {
            localCache.put(key, value);
            return value;
        }
    } catch (Exception e) {
        // Redis故障，降级到数据库
        log.warn("Redis不可用，降级到数据库");
    }

    // L3: 数据库（限流保护）
    if (rateLimiter.tryAcquire()) {
        value = db.get(key);
        if (value != null && redisAvailable) {
            redis.setex(key, 3600, value);
        }
    } else {
        throw new ServiceException("系统繁忙，请稍后再试");
    }
    return value;
}

// 3. 熔断器配置
@CircuitBreaker(name = "redis", fallbackMethod = "getFromDB")
public Object getWithCircuitBreaker(String key) {
    return redis.get(key);
}
```

@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))
            .serializeKeysWith(
                RedisSerializationContext.SerializationPair.fromSerializer(

                RedisSerializationContext.SerializationPair.fromSerializer(

### 4.3 最佳实践

1. **合理设置过期时间**
   - 热点数据：30分钟~1小时
   - 普通数据：5~10分钟
   - 空值：1~5分钟
   - 添加随机偏移避免同时过期

2. **Key设计规范**

### 4.4 常见问题

**Q1: 缓存和数据库如何保持强一致？**
A: 强一致需要付出性能代价，常用方案：

- 使用Write Through模式
- 分布式锁+事务
- 监听Binlog异步更新缓存

**Q2: 大Key问题如何处理？**
A:

- 拆分：将大Hash拆分为多个小Hash
- 压缩：使用Snappy/LZ4压缩
- 删除：使用UNLINK异步删除
- 监控：设置告警阈值（如>10KB）

**Q3: 热Key如何发现和处理？**
A:

- 发现：Redis 4.0+的hotkeys命令，或代理层统计
- 处理：本地缓存+打散（key_1, key_2...）

## 五、形式化分析

### 5.1 缓存一致性模型

**定理**：在Cache Aside模式下，系统满足最终一致性。

**证明概要**：

1. 设写操作W在时刻t完成DB更新
2. 缓存删除操作在t+δ完成
3. 在此时间窗口内，读操作可能读取到旧缓存数据
4. 当缓存删除完成后，后续读操作必从DB获取最新数据
5. 因此系统最终达到一致状态

### 5.2 复杂度分析

| 操作 | Cache Aside | Read Through | Write Through |
|------|-------------|--------------|---------------|
| 读命中 | O(1) | O(1) | O(1) |
| 读未命中 | O(1)+DB | O(1)+DB | O(1)+DB |
| 写 | O(1)+DB | O(1)+DB | O(1)+DB+Cache |
| 一致性 | 最终一致 | 最终一致 | 强一致 |
