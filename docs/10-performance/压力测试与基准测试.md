## 📋 执行摘要

压力测试与基准测试是评估系统性能、发现瓶颈、验证容量规划的核心手段。通过模拟真实或极限负载，获取系统在不同压力下的性能表现数据。

---

## 一、核心概念

### 1.1 定义与原理

**压力测试（Stress Testing）**：通过施加超出正常负载的压力，测试系统在极限条件下的稳定性和恢复能力。

**基准测试（Benchmark Testing）**：在标准条件下测量系统性能指标，建立可比较的基准数据。

**负载测试（Load Testing）**：模拟预期负载，验证系统在正常和峰值负载下的表现。

**核心原理**：
- **可控变量**：每次只改变一个变量，准确归因性能变化
- **可重复性**：相同条件下结果可复现
- **渐进加压**：从低负载逐步增加，识别性能拐点
- **监控全覆盖**：系统、应用、业务多维度监控

### 1.2 关键特性

- **真实性**：模拟真实用户行为和场景
- **可度量**：关键指标可量化、可对比
- **可诊断**：定位性能瓶颈和问题根因
- **可预测**：基于数据预测系统容量

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 上线前验证 | ⭐⭐⭐⭐⭐ | 确保系统可承载预期流量 |
| 容量规划验证 | ⭐⭐⭐⭐⭐ | 验证容量估算的准确性 |
| 性能优化 | ⭐⭐⭐⭐⭐ | 发现瓶颈，指导优化方向 |
| 回归测试 | ⭐⭐⭐⭐ | 验证优化效果，防止性能退化 |
| 竞品对比 | ⭐⭐⭐ | 建立性能基准对比 |

### 2.1 测试方法论

#### 2.1.1 测试类型矩阵

| 测试类型 | 目标 | 负载特征 | 持续时间 |
|---------|------|---------|---------|
| 基准测试 | 建立基线 | 固定低负载 | 短（10-30分钟） |
| 负载测试 | 验证正常负载 | 逐步增加到目标 | 中（1-4小时） |
| 压力测试 | 找到极限 | 持续增加到失败 | 短-中 |
|  spike测试 | 突发流量 | 瞬间激增 | 短 |
|   soak测试 | 稳定性 | 持续高负载 | 长（8-24小时） |
|  断点测试 | 找到极限 | 阶梯式增加 | 中 |

// basic_test.js
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Counter, Rate, Trend } from 'k6/metrics';

// 自定义指标
const errorRate = new Rate('errors');
const apiLatency = new Trend('api_latency');
const successCount = new Counter('success_count');

// 配置选项
export const options = {
  stages: [
    { duration: '2m', target: 100 },    // 渐进到100用户
    { duration: '5m', target: 100 },    // 保持100用户
    { duration: '2m', target: 500 },    // 增加到500用户
    { duration: '5m', target: 500 },    // 保持500用户
    { duration: '2m', target: 1000 },   // 增加到1000用户
    { duration: '5m', target: 1000 },   // 保持1000用户
    { duration: '2m', target: 0 },      // 逐渐停止
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],    // 95%请求<500ms
    http_req_failed: ['rate<0.01'],      // 错误率<1%
    errors: ['rate<0.05'],               // 自定义错误率<5%
  },
  ext: {
    loadimpact: {
      projectID: 12345,
      name: 'API Stress Test',
    },
  },
};

// 初始化（每个VU执行一次）
export function setup() {
  // 登录获取token
  const loginRes = http.post('https://api.example.com/login', {
    username: 'testuser',
    password: 'testpass',
  });
  
  check(loginRes, {
    'login successful': (r) => r.status === 200,
  });
  
  return { token: loginRes.json('token') };
}

// 主测试函数（每个VU循环执行）
export default function (data) {
  const params = {
    headers: {
      'Authorization': `Bearer ${data.token}`,
      'Content-Type': 'application/json',
    },
  };
  
  // 执行请求
  const start = Date.now();
  const response = http.get('https://api.example.com/orders', params);
  const latency = Date.now() - start;
  
  // 记录自定义指标
  apiLatency.add(latency);
  
  // 验证响应
  const success = check(response, {
    'status is 200': (r) => r.status === 200,
    'response time < 500ms': (r) => r.timings.duration < 500,
    'has orders': (r) => r.json('data.orders').length > 0,
  });
  
  if (success) {
    successCount.add(1);
  }
  errorRate.add(!success);
  
  // 模拟思考时间
  sleep(Math.random() * 2 + 1); // 1-3秒随机等待
}

// 清理（测试结束后执行一次）
export function teardown(data) {
  console.log('Test completed. Total success:', successCount.name);
}

# advanced_locust.py
from locust import HttpUser, task, between, LoadTestShape
from locust.contrib.fasthttp import FastHttpUser
import math

# 自定义负载形状（按时间调整并发）
class StepLoadShape(LoadTestShape):
    """阶梯式负载：每30秒增加100用户"""
    step_time = 30
    step_load = 100
    spawn_rate = 50
    time_limit = 600
    
    def tick(self):
        run_time = self.get_run_time()
        
        if run_time > self.time_limit:
            return None
        
        current_step = math.floor(run_time / self.step_time) + 1
        user_count = current_step * self.step_load
        
        return (user_count, self.spawn_rate)

# 自定义波浪形负载
class WaveLoadShape(LoadTestShape):
    """正弦波浪形负载"""
    min_users = 100
    max_users = 1000
    wave_duration = 300  # 5分钟一个周期
    
    def tick(self):
        run_time = self.get_run_time()
        
        # 正弦波计算
        wave_position = (run_time % self.wave_duration) / self.wave_duration
        angle = wave_position * 2 * math.pi
        
        user_count = int(self.min_users + 
                        (self.max_users - self.min_users) * 
                        (0.5 + 0.5 * math.sin(angle)))
        
        return (user_count, 100)

### 2.5 性能指标分析

#### 2.5.1 关键指标定义

| 指标 | 说明 | 健康阈值参考 |
|-----|------|------------|
| **吞吐量(Throughput)** | 每秒处理请求数(RPS/QPS) | 视业务而定 |
| **响应时间(Response Time)** | 请求从发起到响应的时间 | |
| - 平均(Avg) | 所有请求的平均值 | < 200ms |
| - 中位数(P50) | 50%请求低于此值 | < 100ms |
| - P95 | 95%请求低于此值 | < 500ms |
| - P99 | 99%请求低于此值 | < 1000ms |
| **错误率(Error Rate)** | 失败请求占比 | < 0.1% |
| **并发数(Concurrency)** | 同时处理的请求数 | |
| **CPU使用率** | 系统CPU利用率 | < 70% |
| **内存使用率** | 系统内存利用率 | < 80% |
| **网络IO** | 网络吞吐量 | < 带宽80% |

### 4.2 最佳实践

1. **渐进式压测**
   - 从单用户开始，逐步增加负载
   - 每个阶段稳定运行后再增加
   - 记录每个阶段的性能数据

2. **数据参数化**
   - 避免缓存命中导致结果失真
   - 使用真实分布的数据
   - 敏感数据脱敏处理

3. **思考时间模拟**
   - 加入随机思考时间模拟真实用户
   - 不同操作设置不同的思考时间

4. **断言验证**
   - 不仅验证HTTP状态码
   - 验证业务返回数据的正确性
   - 发现隐性错误

5. **结果可信性**
   - 多次执行取平均值
   - 排除预热期的数据
   - 记录环境配置信息

### 4.3 常见问题

**Q1: 压测结果与生产差异大？**
A: 检查环境一致性（数据量、缓存状态、网络延迟），确认流量模型是否真实。

**Q2: 压测机成为瓶颈？**
A: 使用更高效的工具（如k6替代JMeter），采用分布式部署，优化压测脚本。

**Q3: 如何模拟真实用户行为？**
A: 分析生产日志提取用户行为模式，使用混合场景，加入思考时间和随机性。

## 五、形式化分析

### 5.1 利特尔法则(Little's Law)

```
在稳定系统中：
L = λ × W

其中：
L = 系统中的平均请求数（并发数）
λ = 请求到达率（QPS）
W = 请求平均处理时间（响应时间）

应用示例：
- 已知：目标QPS=1000，目标RT=100ms
- 求：系统需要支持的并发数
- 计算：L = 1000 × 0.1 = 100
- 结论：系统需要支持100并发
```
