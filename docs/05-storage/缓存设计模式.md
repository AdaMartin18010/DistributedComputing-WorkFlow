## 📋 执行摘要

缓存设计模式是解决系统性能瓶颈的关键架构模式，包括旁路缓存、穿透缓存、回写缓存等多种策略。合理选择和应用这些模式可以显著提升系统吞吐量、降低延迟，同时需要权衡数据一致性、系统复杂度和运维成本。

## 一、核心概念

### 1.1 缓存设计模式概览

```
┌─────────────────────────────────────────────────────────────┐
│                    缓存设计模式分类                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  按读写策略分类：                                            │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 旁路缓存 (Cache Aside / Lazy Loading)                 │ │
│  │ ├─ 读：先查缓存，未命中查DB，回填缓存                 │ │
│  │ └─ 写：先写DB，再删缓存                               │ │
│  ├───────────────────────────────────────────────────────┤ │
│  │ 穿透缓存 (Read Through / Cache Through)               │ │
│  │ ├─ 读：缓存未命中自动从DB加载                         │ │
│  │ └─ 写：直接写缓存，由缓存层同步DB                     │ │
│  ├───────────────────────────────────────────────────────┤ │
│  │ 回写缓存 (Write Behind / Write Back)                  │ │
│  │ ├─ 写：先写缓存，异步批量回写DB                       │ │
│  │ └─ 读：优先读缓存                                     │ │
│  ├───────────────────────────────────────────────────────┤ │
│  │ 直写缓存 (Write Through)                              │ │
│  │ ├─ 写：同步写缓存和DB                                 │ │
│  │ └─ 读：读缓存                                         │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  按缓存位置分类：                                            │
│  ├─ 客户端缓存: LocalStorage, 应用内存                     │
│  ├─ CDN缓存: 边缘节点缓存                                   │
│  ├─ 反向代理缓存: Nginx, Varnish                           │
│  ├─ 应用缓存: Caffeine, Guava Cache                        │
│  ├─ 分布式缓存: Redis, Memcached                           │
│  └─ 数据库缓存: Buffer Pool, 查询缓存                       │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 模式对比矩阵

| 模式 | 数据一致性 | 复杂度 | 适用场景 | 风险 |
|------|------------|--------|----------|------|
| **Cache Aside** | 最终一致 | 低 | 读多写少，通用场景 | 并发更新不一致 |
| **Read Through** | 较高 | 中 | 封装缓存层，简化应用 | 缓存穿透风险 |
| **Write Through** | 强一致 | 中 | 写后立即读，强一致性 | 写延迟增加 |
| **Write Behind** | 最终一致 | 高 | 写密集，可容忍延迟 | 数据丢失风险 |

#### 代码实现

```java
@Service
public class CacheAsideService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private UserRepository userRepository;

    private static final String USER_KEY_PREFIX = "user:";
    private static final long CACHE_TTL = 30; // 分钟

    /**
     * 读取：Cache Aside模式
     */
    public User getUser(Long userId) {
        String key = USER_KEY_PREFIX + userId;

        // 1. 先查缓存
        User user = (User) redisTemplate.opsForValue().get(key);
        if (user != null) {
            return user;
        }

        // 2. 缓存未命中，查数据库
        user = userRepository.findById(userId).orElse(null);

        // 3. 回填缓存（即使为空也缓存，防止缓存穿透）
        if (user != null) {
            redisTemplate.opsForValue().set(key, user, CACHE_TTL, TimeUnit.MINUTES);
        } else {
            // 缓存空值，短时间过期
            redisTemplate.opsForValue().set(key, "", 5, TimeUnit.MINUTES);
        }

        return user;
    }

    /**
     * 更新：先更新DB，再删除缓存
     */
    @Transactional
    public void updateUser(User user) {
        // 1. 更新数据库
        userRepository.save(user);

        // 2. 删除缓存（不是更新缓存，避免并发写覆盖）
        String key = USER_KEY_PREFIX + user.getId();
        redisTemplate.delete(key);
    }

    /**
     * 删除：先删DB，再删缓存
     */
    @Transactional
    public void deleteUser(Long userId) {
        // 1. 删除数据库
        userRepository.deleteById(userId);

        // 2. 删除缓存
        String key = USER_KEY_PREFIX + userId;
        redisTemplate.delete(key);
    }
}
```

#### 并发安全问题

```
并发更新问题（Race Condition）：

时间点  线程A(写)          线程B(写)          线程C(读)
─────────────────────────────────────────────────────────
T1      更新DB: v=1
T2                       更新DB: v=2
T3                       删缓存
T4      删缓存
T5                                          读缓存miss
T6                                          读DB: v=1
T7                                          写缓存: v=1
─────────────────────────────────────────────────────────
结果：缓存中是旧值v=1，数据库是v=2，不一致！

解决方案：
1. 延迟双删（Delayed Double Delete）
2. 分布式锁
3. 版本号/CAS
4. 接受最终一致性，设置短过期时间
```

#### 实现示例（使用Spring Cache）

```java
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))
            .serializeKeysWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new GenericJackson2JsonRedisSerializer()));

        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .build();
    }
}

@Service
public class UserService {

    @Autowired
    private UserRepository userRepository;

    /**
     * Read Through: Spring Cache自动处理缓存未命中
     * @Cacheable: 有缓存读缓存，无缓存执行方法并缓存结果
     */
    @Cacheable(value = "users", key = "#userId", unless = "#result == null")
    public User getUser(Long userId) {
        // 仅缓存未命中时执行
        return userRepository.findById(userId).orElse(null);
    }

    /**
     * Write Through: 更新DB并刷新缓存
     * @CachePut: 执行方法并更新缓存
     */
    @CachePut(value = "users", key = "#user.id")
    public User updateUser(User user) {
        return userRepository.save(user);
    }

    /**
     * 删除并清缓存
     * @CacheEvict: 删除缓存
     */
    @CacheEvict(value = "users", key = "#userId")
    public void deleteUser(Long userId) {
        userRepository.deleteById(userId);
    }
}
```

#### 实现示例

```java
@Component
public class WriteBehindCache {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private UserRepository userRepository;

    @Autowired
    private WriteBehindProcessor processor;

    private static final String WRITE_QUEUE = "write:queue";
    private static final String CACHE_PREFIX = "user:";

    /**
     * 写操作：只写缓存，入队异步处理
     */
    public void write(User user) {
        String key = CACHE_PREFIX + user.getId();

        // 1. 写缓存（带版本号）
        user.setVersion(System.currentTimeMillis());
        redisTemplate.opsForValue().set(key, user, 1, TimeUnit.HOURS);

        // 2. 入队待回写
        WriteTask task = new WriteTask(user.getId(), user, WriteType.UPDATE);
        redisTemplate.opsForList().leftPush(WRITE_QUEUE, task);

        // 3. 立即返回
    }

    /**
     * 读操作：读缓存
     */
    public User read(Long userId) {
        String key = CACHE_PREFIX + userId;
        return (User) redisTemplate.opsForValue().get(key);
    }
}

@Component
public class WriteBehindProcessor {

    @Autowired
    private UserRepository userRepository;

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private static final String WRITE_QUEUE = "write:queue";

    /**
     * 定时批量处理回写任务
     */
    @Scheduled(fixedDelay = 5000) // 5秒批量处理一次
    public void processBatch() {
        List<WriteTask> batch = new ArrayList<>();

        // 1. 批量取任务（最多100个）
        for (int i = 0; i < 100; i++) {
            Object task = redisTemplate.opsForList().rightPop(WRITE_QUEUE);
            if (task == null) break;
            batch.add((WriteTask) task);
        }

        if (batch.isEmpty()) return;

        // 2. 按用户ID去重（保留最新）
        Map<Long, WriteTask> merged = batch.stream()
            .collect(Collectors.toMap(
                WriteTask::getUserId,
                Function.identity(),
                (old, newer) -> newer.getTimestamp() > old.getTimestamp() ? newer : old
            ));

        // 3. 批量写入DB
        List<User> users = merged.values().stream()
            .map(WriteTask::getUser)
            .collect(Collectors.toList());

        userRepository.saveAll(users);

        // 4. 确认后更新缓存状态或删除标记
    }
}
```

### 3.1 模式选择决策树

```
缓存模式选择决策树：

开始
  │
  ├── 数据一致性要求？
  │     ├── 强一致（金融交易）
  │     │     └── Write Through + 分布式锁
  │     │
  │     └── 最终一致（大部分场景）
  │           │
  │           ├── 读写比例？
  │                 │
  │                 ├── 读多写少 (>10:1)
  │                 │     └── Cache Aside（推荐）
  │                 │
  │                 ├── 写多读少 (<1:10)
  │                 │     └── Write Behind
  │                 │
  │                 └── 读写均衡
  │                       └── Read Through（简化代码）
  │
  └── 是否需要封装缓存层？
        ├── 是 → Read Through / Write Through
        └── 否 → Cache Aside
```

|------|--------|--------|--------|--------|--------|
| **Cache Aside** | 低(hit) / 高(miss) | 低 | 高 | 最终一致 | 低 |

### 4.2 延迟双删实现

```java
@Service
public class DelayedDoubleDeleteService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private UserRepository userRepository;

    @Autowired
    private ThreadPoolExecutor executor;

    /**
     * 延迟双删更新
     */
    @Transactional
    public void updateWithDelayedDelete(User user) {
        String key = "user:" + user.getId();

        // 1. 先删缓存
        redisTemplate.delete(key);

        // 2. 更新数据库
        userRepository.save(user);

        // 3. 延迟再次删除（异步）
        executor.execute(() -> {
            try {
                Thread.sleep(500); // 延迟500ms
                redisTemplate.delete(key);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
    }
}
```

### 4.3 常见问题

**Q1: 为什么更新时推荐删缓存而不是更新缓存？**
A: 避免并发写导致的数据不一致。两个线程同时更新同一数据，由于网络延迟等因素，后写缓存的可能覆盖先写的，导致缓存与DB不一致。

**Q2: 缓存穿透、击穿、雪崩的区别？**
A:

- 穿透：查询不存在的数据，绕过缓存直达DB
- 击穿：热点key过期瞬间大量请求打到DB
- 雪崩：大量key同时过期，DB压力剧增

**Q3: 如何选择缓存过期时间？**
A: 根据数据变更频率：静态数据可长期缓存；配置类数据几分钟到几小时；动态业务数据几秒到几分钟。配合随机偏移防止集中过期。

**Q4: Write Behind模式如何保证数据不丢？**
A: 1) 缓存持久化（AOF）；2) 消息队列持久化；3) 定期全量同步；4) 监控未回写队列长度。
