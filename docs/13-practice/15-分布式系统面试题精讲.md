# 分布式系统面试题精讲

## 一、分布式理论基础

### Q1：CAP定理如何权衡

**答案：** CAP只能满足其二，P必须满足，实际在C和A之间选择。

- CP系统：ZooKeeper、HBase（强一致性场景）
- AP系统：Cassandra、Eureka（高可用场景）

### Q2：BASE与ACID区别

BASE：基本可用、软状态、最终一致
ACID：原子性、一致性、隔离性、持久性

### Q3：Raft共识算法

1. 节点启动为Follower
2. 超时变Candidate
3. 发起投票选举
4. 多数票成Leader
5. 心跳维持地位

## 二、分布式事务

### Q4：2PC的问题与改进

**问题：**

- 同步阻塞
- 单点故障
- 数据不一致

**改进：**

- 3PC增加预检查
- 超时机制
- 协调者多副本

### Q5：TCC与2PC区别

| 特性 | 2PC | TCC |
|------|-----|-----|
| 阶段 | Prepare/Commit | Try/Confirm/Cancel |
| 锁定 | 资源锁定 | 业务预留 |
| 性能 | 低 | 高 |
| 侵入性 | 低 | 高 |

### Q6：Saga模式适用场景

长事务拆分，通过补偿实现最终一致。适用：

- 业务流程长
- 需要异步执行
- 可接受最终一致

## 三、分布式锁

### Q7：Redis分布式锁实现

```java
// 加锁
String result = jedis.set(key, value, "NX", "EX", expireTime);

// 解锁（Lua脚本保证原子性）
String script =
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "return redis.call('del', KEYS[1]) else return 0 end";
jedis.eval(script, Collections.singletonList(key),
    Collections.singletonList(value));
```

**问题与解决：**

- 主从延迟：RedLock算法
- 锁续期：看门狗机制
- 可重入：ThreadLocal记录

### Q8：ZooKeeper分布式锁原理

基于临时顺序节点：

1. 创建临时顺序节点
2. 判断是否为最小序号
3. 是最小则获取锁
4. 否则监听前一个节点
5. 释放时删除节点

### Q10：Paxos与Raft区别

| 特性 | Paxos | Raft |
|------|-------|------|
| 复杂度 | 高 | 低 |

## 六、缓存

### Q14：缓存穿透、击穿、雪崩

| 问题 | 原因 | 解决 |
|------|------|------|
| 穿透 | 查询不存在数据 | 布隆过滤器 |
| 击穿 | 热点key过期 | 互斥锁/逻辑过期 |
| 雪崩 | 大量key同时过期 | 随机过期时间 |

### Q15：缓存与数据库一致性

**Cache Aside：**

1. 读：先读缓存，miss则读库并写缓存
2. 写：先写库，再删缓存

**问题：** 延迟双删、 Canal订阅binlog

## 十二、面试技巧

### 回答结构

1. **概念解释**：先说明基本概念
2. **原理阐述**：讲解工作原理
3. **优缺点**：分析利弊
4. **适用场景**：什么时候用
5. **实际经验**：结合项目

### 常见误区

- 只背概念不理解原理
- 不结合实际场景
- 忽视边界条件
- 缺乏性能意识
