# 分库分表策略

## 核心概念

### 1. 分库分表的必要性

当单库单表的数据量和访问量达到瓶颈时，需要通过分库分表来水平扩展系统。

**单库瓶颈表现**：
```
┌─────────────────────────────────────────────────────────────┐
│                   单库瓶颈表现                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 数据量瓶颈                                              │
│      - MySQL单表推荐 < 1000万行                              │
│      - 单表过大导致B+树层级增加                              │
│      - 查询性能下降                                          │
│                                                             │
│   2. 并发瓶颈                                                │
│      - 单机连接数有限（通常几千）                            │
│      - CPU/IO成为瓶颈                                        │
│      - 锁竞争加剧                                            │
│                                                             │
│   3. 存储瓶颈                                                │
│      - 单机磁盘容量有限                                      │
│      - 单库备份恢复时间过长                                  │
│      - 故障恢复RTO无法满足                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 分库分表类型

**垂直拆分 vs 水平拆分**：

| 类型 | 拆分维度 | 特点 | 适用场景 |
|------|----------|------|----------|
| 垂直分库 | 按业务 | 业务解耦 | 微服务架构 |
| 垂直分表 | 按字段 | 冷热分离 | 字段过多 |
| 水平分库 | 按数据 | 数据分散 | 高并发 |
| 水平分表 | 按数据 | 表结构相同 | 大数据量 |

**拆分策略对比**：
```
垂直分库：
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  用户库      │      │  订单库      │      │  商品库      │
│  users      │      │  orders      │      │  products    │
│  accounts   │      │  order_items │      │  inventory   │
│  profiles   │      │  payments    │      │  categories  │
└─────────────┘      └─────────────┘      └─────────────┘

垂直分表：
┌─────────────────────────────────────────────────────────┐
│                    商品大表                              │
│  id, name, price, description, specs, images, reviews   │
│                          │                              │
│          ┌───────────────┼───────────────┐              │
│          ▼               ▼               ▼              │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│   │ 商品基础表   │  │ 商品详情表   │  │ 商品统计表   │    │
│   │ id,name,price│  │ id,desc,specs│  │ id,sales,score│   │
│   └─────────────┘  └─────────────┘  └─────────────┘    │
│   （热数据，内存缓存） （冷数据，磁盘）  （统计，可异步）   │
└─────────────────────────────────────────────────────────┘

水平分表：
┌─────────────────────────────────────────────────────────┐
│                   订单大表（按user_id分片）               │
│                                                         │
│   user_id % 8 = 0  →  order_0                          │
│   user_id % 8 = 1  →  order_1                          │
│   user_id % 8 = 2  →  order_2                          │
│   ...                                                   │
│   user_id % 8 = 7  →  order_7                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3. 分片算法

**常用分片算法对比**：

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 取模 | hash(key) % N | 分布均匀 | 扩容难 | 固定分片数 |
| 范围 | 按ID范围 | 扩容容易 | 热点问题 | 时间序列 |
| 一致性哈希 | 哈希环 | 平滑扩容 | 实现复杂 | 缓存分片 |
| 路由表 | 映射表 | 灵活 | 需维护 | 复杂规则 |
| 标签 | 按业务属性 | 业务清晰 | 分布不均 | 多租户 |

**一致性哈希详解**：
```
┌─────────────────────────────────────────────────────────────┐
│                   一致性哈希环                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   0° ───────────────────────────────────────────────── 360° │
│        │              │              │                      │
│      Node1          Node2          Node3                    │
│       ↓              ↓              ↓                       │
│     hash=30       hash=150       hash=280                   │
│                                                             │
│   Key定位规则：顺时针找到第一个大于等于key hash的节点        │
│                                                             │
│   Key A (hash=50)  →  Node2                                 │
│   Key B (hash=200) →  Node3                                 │
│   Key C (hash=300) →  Node1 (回环)                          │
│                                                             │
│   扩容时（增加Node4 at hash=200）：                          │
│   - 只需要迁移Node3中hash<200的数据到Node4                  │
│   - 其他数据不动                                            │
│                                                             │
│   虚拟节点（解决数据倾斜）：                                  │
│   - 每个物理节点映射为N个虚拟节点                           │
│   - Node1 → V1_1, V1_2, V1_3...                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 实践方法

### 分库分表设计

**1. 分片键选择**：
```
分片键选择原则：
1. 高频查询条件
2. 数据分布均匀
3. 避免跨分片查询
4. 业务可预测

常见选择：
- 用户表：user_id
- 订单表：user_id（查用户订单）或order_id（查订单详情）
- 商品表：shop_id（店铺商品）或category_id（类目商品）

方案对比：
┌─────────────────────────────────────────────────────────────┐
│  订单表按user_id分片 vs 按order_id分片                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  按user_id分片：                                            │
│  ✓ 查询用户订单列表：单分片查询                             │
│  ✗ 查询订单详情：需先路由到user_id                          │
│  ✗ 全局ID生成：需考虑分片                                   │
│                                                             │
│  按order_id分片：                                           │
│  ✓ 查询订单详情：直接路由                                   │
│  ✗ 查询用户订单：广播所有分片                               │
│  ✗ 数据倾斜：某些用户订单多                                 │
│                                                             │
│  推荐：用户端按user_id，运营端按时间索引表                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**2. 分片规则配置**：
```yaml

    orders:
      actualDataNodes: ds${0..7}.orders_${0..15}

          shardingColumn: user_id
```

**3. 全局ID生成**：
```java
// 雪花算法实现
public class SnowflakeIdGenerator {
    // 起始时间戳
    private final long twepoch = 1609459200000L; // 2021-01-01
    
    // 各部分位数
    private final long workerIdBits = 5L;
    private final long datacenterIdBits = 5L;
    private final long sequenceBits = 12L;
    
    // 最大值
    private final long maxWorkerId = ~(-1L << workerIdBits);
    private final long maxDatacenterId = ~(-1L << datacenterIdBits);
    
    // 位移
    private final long datacenterIdShift = sequenceBits;
    private final long workerIdShift = sequenceBits + datacenterIdBits;
    private final long timestampShift = sequenceBits + datacenterIdBits + workerIdBits;
    
    private long workerId;
    private long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;
    
    public synchronized long nextId() {
        long timestamp = timeGen();
        
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards");
        }
        
        if (lastTimestamp == timestamp) {
            sequence = (sequence + 1) & ((1 << sequenceBits) - 1);
            if (sequence == 0) {
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        
        lastTimestamp = timestamp;
        
        return ((timestamp - twepoch) << timestampShift)
            | (datacenterId << datacenterIdShift)
            | (workerId << workerIdShift)
            | sequence;
    }
    
    // ID结构：
    // 1bit(符号) + 41bit(时间戳) + 5bit(数据中心) + 5bit(机器) + 12bit(序列号)
    // 每秒可生成约400万个ID
}
```

### 分布式事务

**1. 柔性事务（最终一致性）**：
```java
// Saga模式实现
public class OrderSaga {
    
    @Compensable(compensationMethod = "cancelOrder")
    public void createOrder(Order order) {
        // 1. 创建订单
        orderService.create(order);
        
        // 2. 扣减库存
        inventoryService.deduct(order);
        
        // 3. 扣减余额
        accountService.debit(order);
    }
    
    public void cancelOrder(Order order) {
        // 补偿操作
        accountService.credit(order);
        inventoryService.restore(order);
        orderService.cancel(order);
    }
}
```

**2. 最大努力通知**：

```java
// 消息队列保证最终一致
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        // 1. 本地事务：创建订单
        orderDao.insert(order);
        
        // 2. 发送消息
        messageProducer.send(
            new OrderCreatedEvent(order),
            new SendCallback() {
                @Override
                public void onSuccess(SendResult sendResult) {
                    // 发送成功，更新订单状态
                    orderDao.updateStatus(order.getId(), "MESSAGE_SENT");
                }
                
                @Override
                public void onException(Throwable e) {
                    // 发送失败，定时重试
                    retryScheduler.schedule(order);
                }
            }
        );
    }
}
```

### 跨分片查询

**1. 分页查询优化**：
```java
// 全局分页查询
public class ShardingPagination {
    
    public PageResult<Order> queryOrders(int pageNum, int pageSize) {
        // 1. 计算需要查询的分片范围
        int shardCount = 16;
        int offset = (pageNum - 1) * pageSize;
        
        // 2. 在每个分片查询
        List<Future<List<Order>>> futures = new ArrayList<>();
        for (int i = 0; i < shardCount; i++) {
            final int shardIndex = i;
            futures.add(executor.submit(() -> {
                return orderDao.queryByShard(shardIndex, offset, pageSize);
            }));
        }
        
        // 3. 合并结果
        List<Order> allOrders = new ArrayList<>();
        for (Future<List<Order>> future : futures) {
            allOrders.addAll(future.get());
        }
        
        // 4. 内存排序分页（不推荐大数据量）
        allOrders.sort(Comparator.comparing(Order::getCreatedAt).reversed());
        
        // 5. 返回分页结果
        int fromIndex = offset;
        int toIndex = Math.min(fromIndex + pageSize, allOrders.size());
        return new PageResult<>(
            allOrders.subList(fromIndex, toIndex),
            allOrders.size()
        );
    }
}
```

**2. 聚合查询优化**：
```sql
-- 跨分片COUNT优化：使用汇总表
-- 1. 实时统计（近实时）
CREATE TABLE order_stats (
    stat_date DATE,
    shard_id INT,
    order_count BIGINT,
    total_amount DECIMAL(15,2),
    PRIMARY KEY (stat_date, shard_id)
);

-- 2. 定时任务汇总
INSERT INTO order_stats 
SELECT 
    DATE(created_at),
    ${shardId},
    COUNT(*),
    SUM(amount)
FROM orders
WHERE created_at >= CURDATE()
GROUP BY DATE(created_at)
ON DUPLICATE KEY UPDATE
    order_count = VALUES(order_count),
    total_amount = VALUES(total_amount);

-- 3. 查询时直接读汇总表
SELECT SUM(order_count), SUM(total_amount) 
FROM order_stats 
WHERE stat_date = '2024-01-01';
```

## 案例数据

### 电商平台分库分表

**背景**：
- 订单表：10亿+数据
- 日增：1000万订单
- 查询：用户订单列表（高频）、订单详情（高频）、运营统计（低频）

**方案**：
```
1. 分库：8个库
2. 分表：每库16张表，共128张表
3. 分片键：user_id
4. 单表数据：约800万行

路由规则：
- 库索引 = user_id % 128 % 8
- 表索引 = user_id % 128 / 8

ID生成：雪花算法
- 嵌入user_id分片信息，避免数据迁移时重新计算
```

**效果**：
| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 单表数据量 | 10亿 | 800万 |
| 查询RT | 2s | 20ms |
| 写入RT | 500ms | 10ms |
| 扩容能力 | 无 | 可水平扩展 |

## 优化前后对比

| 指标 | 单库单表 | 分库分表 | 改进 |
|------|----------|----------|------|
| 最大数据量 | 1亿 | 无上限 | ∞ |
| 并发能力 | 1000/s | 10万/s | 100倍 |
| 查询RT | 2s | 20ms | 99%↓ |
| 扩容难度 | 高 | 低 | 显著提升 |

---

**总结**：分库分表是应对海量数据和高并发的终极手段。设计时需要仔细选择分片键、规划分片数量、处理好分布式事务和跨分片查询问题。推荐采用成熟的分库分表中间件，避免重复造轮子。
