## 📋 执行摘要

时序数据库（Time-Series Database, TSDB）是专为处理时间序列数据而优化的数据库系统，广泛应用于监控、IoT、金融等领域。本文档深入分析时序数据特点，并对比 InfluxDB、TimescaleDB、Prometheus TSDB、OpenTSDB 等主流系统的架构与选型要点。

---

## 一、核心概念

### 1.1 定义与原理

**时序数据（Time-Series Data）** 是指按照时间顺序记录的数据点序列，每个数据点包含：

- **时间戳（Timestamp）**：数据产生的时间
- **度量（Metric）**：被测量的数值
- **标签/维度（Tags/Dimensions）**：描述数据的元数据

**核心原理**：

1. **时间局部性**：相邻时间的数据通常一起被查询
2. **写入模式**： predominantly append-only，极少更新删除
3. **高压缩率**：同类数据具有高度相似性，适合压缩
4. **降采样**：历史数据通常只需聚合视图

### 1.2 关键特性

| 特性 | 说明 |
|------|------|
| **高写入吞吐** | 支持每秒百万级数据点写入 |
| **高效时间范围查询** | 针对时间范围扫描优化 |
| **自动过期** | 支持 TTL（Time-To-Live）自动清理 |
| **连续聚合** | 内置降采样和预聚合能力 |
| **标签索引** | 支持多维度标签的高效检索 |
| **高压缩比** | 典型压缩率 10:1 到 100:1 |

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 系统监控（Metrics） | ⭐⭐⭐⭐⭐ | CPU、内存、磁盘、网络指标采集 |
| IoT 传感器数据 | ⭐⭐⭐⭐⭐ | 海量设备实时数据写入 |
| 金融行情数据 | ⭐⭐⭐⭐⭐ | 高频交易数据、K线存储 |
| APM 应用性能监控 | ⭐⭐⭐⭐⭐ | 分布式追踪、日志指标 |
| 日志分析 | ⭐⭐⭐⭐ | 时间序日志，但需权衡与 ELK 方案 |
| 通用 OLTP | ⭐⭐ | 不适合事务性业务数据 |

## 二、技术细节

### 2.1 时序数据特点

```
┌─────────────────────────────────────────────────────────────┐
│                    时序数据核心特点                          │
├─────────────────────────────────────────────────────────────┤
│  1. 数据按时间顺序到达，天然有序                              │
│  2. 写入为主，更新删除极少（Immutable）                      │
│  3. 查询通常基于时间范围 + 标签过滤                          │
│  4. 数据量巨大，需要高效压缩                                 │
│  5. 冷热数据分层，历史数据可降采样                           │
│  6. 高基数问题：标签组合可能导致指数级序列增长                │
└─────────────────────────────────────────────────────────────┘
```

**数据模型示例**：

```
Measurement: cpu_usage
Tags: host=server01, region=beijing, env=prod
Fields: usage_user=45.2, usage_system=12.1, usage_idle=42.7
Timestamp: 2026-04-03T10:30:00Z
```

### 2.2 InfluxDB 架构

```
┌─────────────────────────────────────────────────────────────┐
│                      InfluxDB 架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │   HTTP API   │   │   Flux Query │   │   Line Protocol│  │
│  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘  │
│         │                   │                    │          │
│         └───────────────────┼────────────────────┘          │
│                             ▼                               │
│  ┌─────────────────────────────────────────────────────────┐│
│  │                    Storage Engine                        ││
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ ││
│  │  │  WAL (写前日志)│  │  Cache (热数据)│  │  TSM Files (冷数据)│ ││
│  │  └─────────────┘  └─────────────┘  └─────────────────┘ ││
│  └─────────────────────────────────────────────────────────┘│
│                             │                               │
│                             ▼                               │
│  ┌─────────────────────────────────────────────────────────┐│
│  │              Index Engine (索引引擎)                     ││
│  │         (TSI - Time Series Index)                       ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心组件**：

| 组件 | 功能 | 说明 |
|------|------|------|
| WAL | 写前日志 | 保证数据持久性，顺序写入 |
| Cache | 内存缓存 | 最新热数据，支持快速查询 |
| TSM | 时间序列树形合并 | 列式存储，高压缩率 |
| TSI | 时间序列索引 | 倒排索引，支持高基数标签 |

**TSM 存储格式**：

- 按时间范围分片（Shard）
- 列式存储，相同类型的数据连续存放
- 多种压缩算法：Gorilla（浮点）、Snappy（字符串）、RLE（整数）

### 2.3 TimescaleDB（PostgreSQL 扩展）

```
┌─────────────────────────────────────────────────────────────┐
│                   TimescaleDB 架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              PostgreSQL 内核                         │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │         TimescaleDB 扩展层                   │   │   │
│  │  │  ┌───────────┐  ┌───────────┐  ┌──────────┐ │   │   │
│  │  │  │ Hypertable│  │  Chunk    │  │ Contin.  │ │   │   │
│  │  │  │  (虚拟表)  │  │ (数据分区) │  │ Aggregate│ │   │   │
│  │  │  └─────┬─────┘  └───────────┘  └──────────┘ │   │   │
│  │  │        │                                     │   │   │
│  │  │  ┌─────┴─────────────────────────────────┐   │   │   │
│  │  │  │      Chunk 管理器                      │   │   │   │
│  │  │  │  - 自动分区 (按时间/空间)               │   │   │   │
│  │  │  │  - 自动压缩旧数据                      │   │   │   │
│  │  │  │  - 数据保留策略                        │   │   │   │
│  │  │  └─────────────────────────────────────────┘   │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心概念**：

| 概念 | 说明 |
|------|------|
| **Hypertable** | 虚拟表，透明分布在多个 Chunk 上 |
| **Chunk** | 实际数据分区，按时间间隔自动创建 |
| **Continuous Aggregate** | 物化视图，自动维护预聚合数据 |
| **Compression** | 列式压缩，支持多种压缩算法 |

**分区策略**：

```sql
-- 创建 Hypertable，按时间分区
SELECT create_hypertable('metrics', 'time', chunk_time_interval => INTERVAL '1 day');

-- 添加空间分区（可选）
SELECT create_hypertable('metrics', 'time', 'device_id', 4);
```

```

│                                                             │
│  ┌─────────────────────────────────────────────────────────┐│
│  │                    Head Block (内存)                     ││

│                           │                                 │
│                           │  压缩/持久化                     │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────────┐│
│  │              Persistent Blocks (磁盘)                    ││
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ ││
│  │  │  Chunk Files │  │  Index Files │  │  Meta.json      │ ││
│  │  │  (数据块)    │  │  (倒排索引)  │  │  (元数据)       │ ││
│  │  └─────────────┘  └─────────────┘  └─────────────────┘ ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键特性**：

| 特性 | 实现 |
|------|------|
| **存储格式** | 自定义二进制格式，高压缩率 |
| **索引** | 倒排索引，支持标签匹配 |
| **抓取模型** | Pull 模式，主动抓取目标端点 |
| **本地存储** | 单机设计，水平扩展通过 Federation 或 Thanos |
| **PromQL** | 强大的查询语言，支持范围向量 |

**数据保留**：

- 默认 15 天本地存储
- 通过 `--storage.tsdb.retention.time` 配置
- 长期存储需对接远程存储（如 Thanos、Cortex、VictoriaMetrics）

```
│                   OpenTSDB 架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐│
│  │   TSD Server │     │   TSD Server │     │   TSD Server ││
│  │   (Region 1) │     │   (Region 2) │     │   (Region N) ││

│                              │                              │
│                              ▼                              │
│  ┌─────────────────────────────────────────────────────────┐│
│  │                   Apache HBase                           ││
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ ││
│  │  │  tsdb-uid   │  │  tsdb       │  │  tsdb-tree      │ ││
│  │  │  (UID映射)  │  │  (数据表)   │  │  (树形索引)     │ ││
│  │  └─────────────┘  └─────────────┘  └─────────────────┘ ││
│  └─────────────────────────────────────────────────────────┘│
│                              │                              │
│                              ▼                              │
│  ┌─────────────────────────────────────────────────────────┐│
│  │                   HDFS (底层存储)                        ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心设计**：

| 组件 | 说明 |
|------|------|
| **UID 映射** | 将长字符串标签映射为 3 字节 UID，节省存储 |
| **Row Key 设计** | `metric_uid + timestamp(小时) + tagk_uid + tagv_uid` |
| **HBase 依赖** | 利用 HBase 的高可用和水平扩展能力 |
| **异步写入** | 支持批量写入，提高吞吐量 |

**Row Key 结构**：

```
┌──────────────┬──────────────────┬──────────────┬──────────────┐
│  Metric UID  │  Base Timestamp  │  TagK UID    │  TagV UID    │
│   (3 bytes)  │   (4 bytes)      │  (3 bytes)   │  (3 bytes)   │
└──────────────┴──────────────────┴──────────────┴──────────────┘
```

## 三、系统对比

### 3.1 主流时序数据库对比矩阵

| 维度 | InfluxDB | TimescaleDB | Prometheus | OpenTSDB |
|------|----------|-------------|------------|----------|
| **底层存储** | 自定义 TSM | PostgreSQL | 自定义 TSDB | HBase |
| **部署模式** | 单机/集群 | 单机/流复制 | 单机 | 分布式 |
| **协议支持** | HTTP/UDP/Line Protocol | SQL/PostgreSQL | HTTP Pull/Push Gateway | HTTP/ Telnet |
| **查询语言** | Flux (类 SQL) | SQL | PromQL | 类 SQL |
| **高可用** | 企业版支持 | 流复制 | 联邦/Thanos | HBase 原生 |
| **水平扩展** | 企业版/InfluxDB IOx | 有限 | Thanos/Cortex | 优秀 |
| **压缩率** | 高 (10-100x) | 高 (PG 原生 + 列式) | 高 | 中等 |
| **生态工具** | Telegraf, Chronograf | 丰富 PG 生态 | Grafana, Alertmanager | 较少 |

### 3.2 选型决策树

```
需求分析
│
├── 需要 SQL 查询和事务支持？
│   ├── 是 → TimescaleDB（PostgreSQL 生态兼容）
│   └── 否 → 继续判断
│
├── 云原生监控场景，与 K8s 深度集成？
│   ├── 是 → Prometheus（事实标准）
│   └── 否 → 继续判断
│
├── 超大规模数据（PB级）？
│   ├── 是 → OpenTSDB（基于 HBase）或 InfluxDB IOx
│   └── 否 → 继续判断
│
├── 需要简单易用的单机部署？
│   ├── 是 → InfluxDB（开源版）或 Prometheus
│   └── 否 →
│
└── 最终推荐
    ├── 通用时序数据 → InfluxDB
    ├── 已有 PostgreSQL 基础设施 → TimescaleDB
    └── 云原生可观测性 → Prometheus
```

### 3.3 性能基准

| 指标 | InfluxDB | TimescaleDB | Prometheus | OpenTSDB |
|------|----------|-------------|------------|----------|
| **写入吞吐** | ~700K points/s | ~300K points/s | ~100K samples/s | ~1M+ points/s |
| **查询延迟 (p99)** | < 100ms | < 50ms | < 10ms | < 200ms |
| **可扩展性** | 中等 | 中等 | 有限（需外部方案） | 优秀 |
| **资源占用** | 中等 | 较高 | 低 | 高 |

## 四、实践指南

### 4.1 部署配置

**InfluxDB 示例配置**：

```toml
[meta]
  dir = "/var/lib/influxdb/meta"

[data]
  dir = "/var/lib/influxdb/data"
  wal-dir = "/var/lib/influxdb/wal"
  max-series-per-database = 1000000
  max-values-per-tag = 100000

[coordinator]
  write-timeout = "10s"
  max-concurrent-queries = 0
  query-timeout = "0s"

[retention]
  enabled = true
  check-interval = "30m"

[http]
  enabled = true
  bind-address = ":8086"
  auth-enabled = true
```

**TimescaleDB 最佳实践 SQL**：

```sql
-- 创建带压缩策略的 Hypertable
CREATE TABLE metrics (
    time TIMESTAMPTZ NOT NULL,
    device_id TEXT NOT NULL,
    temperature DOUBLE PRECISION,
    humidity DOUBLE PRECISION
);

SELECT create_hypertable('metrics', 'time', chunk_time_interval => INTERVAL '7 days');

-- 启用自动压缩（7天前的数据）
ALTER TABLE metrics SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id'
);

SELECT add_compression_policy('metrics', INTERVAL '7 days');

-- 设置数据保留策略
SELECT add_retention_policy('metrics', INTERVAL '1 year');

-- 创建连续聚合（降采样）
CREATE MATERIALIZED VIEW metrics_hourly
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 hour', time) AS bucket,
    device_id,
    avg(temperature) AS avg_temp,
    max(temperature) AS max_temp
FROM metrics
GROUP BY bucket, device_id;
```

### 4.2 最佳实践

1. **标签设计原则**
   - 避免高基数标签（如用户ID、订单ID）
   - 使用有意义的标签名，便于查询过滤
   - 控制标签数量，通常不超过 10 个

2. **写入优化**
   - 使用批量写入（Batch Insert）
   - 合理设置时间精度（秒级 vs 毫秒级）
   - 避免单点写入热点

3. **查询优化**
   - 优先使用预聚合（Continuous Aggregate）
   - 限制查询时间范围
   - 避免全表扫描

4. **数据生命周期管理**
   - 设置合理的 TTL/保留策略
   - 对历史数据进行降采样
   - 冷热数据分层存储

### 4.3 常见问题

**Q1: 高基数问题如何解决？**
A:

- 避免将高唯一性值（如 UUID、用户ID）作为标签
- 使用 InfluxDB IOx 或 TimescaleDB 处理高基数场景
- 考虑将高基数字段转换为普通字段而非标签

**Q2: 时序数据库 vs 传统关系型数据库？**
A:

- 时序数据库针对时间范围查询和写入优化
- 传统数据库适合复杂事务和关联查询
- 对于混合场景，TimescaleDB 是较好的折中方案

**Q3: 如何选择单机和集群部署？**
A:

- 单机适合数据量 < 1TB、写入 < 100K points/s
- 集群适合多租户、高可用、水平扩展需求
- 考虑运维复杂度，优先选择托管服务
