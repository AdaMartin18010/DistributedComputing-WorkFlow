## 📋 执行摘要

无主复制（Leaderless Replication）是一种去中心化的复制架构，所有节点地位平等，都可接受读写请求。Dynamo、Cassandra和Voldemort是该架构的代表实现。通过向量时钟、读修复和提示移交等机制实现最终一致性，适用于高可用、高写入场景。

## 一、核心概念

### 1.1 无主复制架构

```
┌─────────────────────────────────────────────────────────────┐
│                    无主复制架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  架构特点：                                                  │
│  ┌─────────┐         ┌─────────┐         ┌─────────┐       │
│  │ Node A  │←───────→│ Node B  │←───────→│ Node C  │       │
│  │ (R/W)   │   Gossip│  (R/W)  │  Gossip │  (R/W)  │       │
│  │         │         │         │         │         │       │
│  │ ┌─────┐ │         │ ┌─────┐ │         │ ┌─────┐ │       │
│  │ │Rep1 │ │         │ │Rep2 │ │         │ │Rep3 │ │       │
│  │ └─────┘ │         │ └─────┘ │         │ └─────┘ │       │
│  └─────────┘         └─────────┘         └─────────┘       │
│       ↑                                              ↑       │
│       └──────────────┬───────────────────────────────┘       │
│                      ↓                                       │
│                   Client                                     │
│                   ├─ 可连接任意节点                          │
│                   ├─ Quorum读写 (W+R>N)                      │
│                   └─ 冲突解决机制                            │
│                                                            │
│  Quorum机制：                                                │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  N = 副本数 (Replication Factor)                       │ │
│  │  W = 写入确认数 (Write Quorum)                         │ │
│  │  R = 读取节点数 (Read Quorum)                          │ │
│  │                                                       │ │
│  │  规则: W + R > N (保证读写至少有一个节点重叠)            │ │
│  │                                                       │ │
│  │  例: N=3, W=2, R=2                                    │ │
│  │  ├─ 写入时等待2个副本确认                              │ │
│  │  ├─ 读取时读取2个副本，比较版本                        │ │
│  │  └─ 保证至少有一个副本是最新的                         │ │
│  │                                                       │ │
│  │  调整策略:                                            │ │
│  │  ├─ W=1, R=N: 快速写，慢读（读修复）                    │ │
│  │  ├─ W=N, R=1: 慢写，快速读                              │ │
│  │  └─ W=R=(N+1)/2: 平衡模式（常用）                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 与主从架构对比

| 特性 | 主从复制 | 无主复制 |
|------|----------|----------|
| **节点角色** | 主+从 | 全对等 |
| **写入路径** | 必须经过主 | 任意节点 |
| **可用性** | 主故障需切换 | 部分故障可继续服务 |
| **一致性** | 易于强一致 | 最终一致 |
| **冲突** | 主节点串行 | 需冲突解决机制 |
| **延迟** | 单向复制 | 多向复制 |

## 二、技术细节

### 2.1 Dynamo模型

```
┌─────────────────────────────────────────────────────────────┐
│                    Dynamo 架构详解                           │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  关键技术：                                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 1. 一致性哈希（Consistent Hashing）                    │ │
│  │    ├─ 节点和数据映射到哈希环                           │ │
│  │    ├─ 虚拟节点解决数据倾斜                             │ │
│  │    └─ 扩容只影响相邻节点                               │ │
│  │                                                       │ │
│  │ 2. 向量时钟（Vector Clocks）                           │ │
│  │    ├─ [A:1, B:2, C:1] 表示各节点版本                   │ │
│  │    ├─ 用于冲突检测和因果关系判断                       │ │
│  │    └─ 无法合并时交给客户端解决                         │ │
│  │                                                       │ │
│  │ 3. 读修复（Read Repair）                               │ │
│  │    ├─ 读取时比较多个副本                               │ │
│  │    ├─ 发现过时数据时异步更新                           │ │
│  │    └─ 后台反熵（Anti-Entropy）持续修复                  │ │
│  │                                                       │ │
│  │ 4. 提示移交（Hinted Handoff）                          │ │
│  │    ├─ 临时故障节点写入邻居                             │ │
│  │    ├─ 故障恢复后移交数据                               │ │
│  │    └─ 保证高可用性                                     │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  写操作流程：                                                │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Client                                               │ │
│  │    ↓ 1. 计算Key的哈希，确定Coordinator节点              │ │
│  │  Coordinator (Node A)                                 │ │
│  │    ↓ 2. 向N个副本节点发送写请求                        │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐               │ │
│  │  │ Node A  │  │ Node B  │  │ Node C  │               │ │
│  │  │ (本地)  │  │ (副本1) │  │ (副本2) │               │ │
│  │  │ 写入    │  │ 写入    │  │ 写入    │               │ │
│  │  └───┬─────┘  └───┬─────┘  └───┬─────┘               │ │
│  │      └─────────────┴─────────────┘                    │ │
│  │                    ↓ 等待W个确认                       │ │
│  │                 返回成功                               │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  读操作流程：                                                │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Client                                               │ │
│  │    ↓ 1. 向R个节点发送读请求                            │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐               │ │
│  │  │ Node A  │  │ Node B  │  │ Node C  │               │ │
│  │  │ 返回V1  │  │ 返回V2  │  │ 返回V1  │               │ │
│  │  └─────────┘  └─────────┘  └─────────┘               │ │
│  │       └─────────────┬─────────────┘                   │ │
│  │                     ↓ 2. 比较版本                      │ │
│  │                V2较新，返回V2                          │ │
│  │                     ↓ 3. 读修复                        │ │
│  │                异步更新A、C到V2                        │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Cassandra实现

```
┌─────────────────────────────────────────────────────────────┐
│                  Cassandra 实现细节                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  数据模型：                                                  │
│  ├─ 列族（Column Family）/ 表（Table）                      │ │
│  ├─ 宽行存储（Wide Row）                                    │ │
│  └─ 动态列（Dynamic Columns）                               │ │
│                                                            │
│  存储引擎：                                                  │ │
│  ├─ LSM-Tree（MemTable + SSTable）                         │ │
│  ├─ Bloom Filter加速查询                                    │ │
│  └─ 压缩（Snappy/LZ4/Zstd）                                 │ │
│                                                            │
│  协调节点（Coordinator）：                                    │ │
│  ├─ 接收客户端请求                                          │ │
│  ├─ 路由到副本节点                                          │ │
│  ├─ 收集响应并返回                                          │ │
│  └─ 触发读修复                                              │ │
│                                                            │
│  一致性级别（Consistency Level）：                            │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  写一致性：                                            │ │
│  │  ├─ ANY: 任意一个节点（包括Hint）                      │ │
│  │  ├─ ONE: 一个副本确认                                  │ │
│  │  ├─ TWO/THREE: 2/3个副本确认                           │ │
│  │  ├─ QUORUM: 多数派 (N/2+1)                             │ │
│  │  ├─ ALL: 所有副本确认                                  │ │
│  │  └─ LOCAL_QUORUM: 本地数据中心多数派                   │ │
│  │                                                       │ │
│  │  读一致性：                                            │ │
│  │  ├─ ONE: 读一个副本（可能旧数据）                       │ │
│  │  ├─ QUORUM: 读多数派（最新数据）                        │ │
│  │  └─ ALL: 读所有副本（最慢但最准确）                     │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

```yaml
  - /var/lib/cassandra/data

  - class_name: org.apache.cassandra.locator.SimpleSeedProvider

      - seeds: "10.0.0.1,10.0.0.2,10.0.0.3"

# 副本策略
# NetworkTopologyStrategy用于多数据中心

  'DC1': 3,
  'DC2': 2
};
```

## 四、常见问题

**Q1: 无主复制适合什么场景？**
A: 高可用要求高、可容忍最终一致性、需要多地写入、高吞吐写入场景（如IoT、日志、社交Feed）。

**Q2: 如何解决冲突？**
A: 1) LWW（时间戳）；2) 向量时钟判断因果；3) 应用层合并；4) CRDT数据结构。

**Q3: 节点故障如何恢复？**
A: 1) Hinted Handoff临时保存写；2) 故障恢复后数据移交；3) 反熵修复同步差异。

**Q4: 与主从架构如何选择？**
A: 需要强一致、事务支持选主从；需要极致可用、全球分布选无主。
