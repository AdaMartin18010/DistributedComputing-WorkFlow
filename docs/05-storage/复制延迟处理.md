## 📋 执行摘要

复制延迟是主从架构中不可避免的问题，会影响读写一致性和业务正确性。本文档系统阐述复制延迟的产生原因、监控方法以及延迟隐藏、强制路由、缓存标记等多种应对策略，帮助构建高可用的分布式数据架构。

## 一、核心概念

### 1.1 复制延迟产生原因

```
┌─────────────────────────────────────────────────────────────┐
│                    复制延迟来源分析                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 网络延迟                                                │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  主从异地部署 → 网络RTT增加                              │ │
│  │  带宽不足 → Binlog传输阻塞                               │ │
│  │  网络抖动 → 丢包重传                                     │ │
│  │                                                       │ │
│  │  典型值: 同机房 <1ms, 同城<5ms, 异地<50ms               │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. 从库执行延迟                                            │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Master并发执行 → Slave串行执行                         │ │
│  │                                                       │ │
│  │  Master:                    Slave:                    │ │
│  │  T1: UPDATE ...            T1: UPDATE ... (执行)      │ │
│  │  T2: UPDATE ...    ──→     (等待T1完成)               │ │
│  │  T3: UPDATE ...            (等待)                     │ │
│  │                                                       │ │
│  │  原因:                                                │ │
│  │  ├─ SQL Thread单线程                                    │ │
│  │  ├─ 大事务（批量UPDATE/DELETE）                         │ │
│  │  └─ DDL操作（ALTER TABLE）                              │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. 资源竞争                                                │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  从库同时承担:                                          │ │
│  │  ├─ 复制SQL执行                                         │ │
│  │  ├─ 业务读查询                                          │ │
│  │  └─ 备份任务                                            │ │
│  │                                                       │ │
│  │  结果: CPU/IO竞争，复制变慢                              │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  4. 主库高写入压力                                          │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  TPS超过从库处理能力                                    │ │
│  │  大批量写入（LOAD DATA）                                │ │
│  │  Binlog格式ROW导致日志量大                              │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 延迟的影响

| 场景 | 影响 | 严重程度 |
|------|------|----------|
| 读写分离读从库 | 读到旧数据 | 中 |
| 写入后读取 | 读不到自己写的 | 高 |
| 会话一致性 | 操作结果不一致 | 高 |
| 数据迁移 | 数据不一致窗口 | 中 |

### 2.2 延迟处理策略

```
┌─────────────────────────────────────────────────────────────┐
│                    复制延迟处理策略                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 强制主库读取（强制路由）                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  场景: 用户刚修改资料，立即查看                          │ │
│  │                                                       │ │
│  │  实现:                                                │ │
│  │  ├─ 写入时返回影响的主键/标识                            │ │
│  │  ├─ 后续读取强制路由到主库                               │ │
│  │  └─ 或使用Hint标记强制走主库                             │ │
│  │                                                       │ │
│  │  示例:                                                │ │
│  │  ┌─────────────────────────────────────────────────┐ │ │
│  │  │  // 写入后读取同一会话                            │ │ │
│  │  │  userService.update(user);                      │ │ │
│  │  │  User updated = userService.getById(user.getId(),│ │ │
│  │  │      ReadHint.MASTER); // 强制读主库               │ │ │
│  │  └─────────────────────────────────────────────────┘ │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. 缓存标记法                                               │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  原理: 写入后缓存标记，读时检查标记                      │ │
│  │                                                       │ │
│  │  流程:                                                │ │
│  │  Write: DB → Cache写标记(user_id, expire=5s)          │ │
│  │  Read: 检查Cache标记存在 → 读主库                      │ │
│  │        标记不存在 → 读从库                             │ │
│  │                                                       │ │
│  │  优点: 大部分读仍走从库，仅可能延迟的读走主库            │ │
│  │  缺点: 引入缓存依赖                                     │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. 延迟阈值路由                                             │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  原理: 监控延迟，超过阈值时自动切换                       │ │
│  │                                                       │ │
│  │  策略:                                                │ │
│  │  ├─ 延迟 < 1s: 读从库                                  │ │
│  │  ├─ 延迟 1-10s: 敏感读走主库                            │ │
│  │  └─ 延迟 > 10s: 所有读走主库                            │ │
│  │                                                       │ │
│  │  实现: 动态数据源切换                                   │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  4. 半同步复制 + 超时降级                                    │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  配置: rpl_semi_sync_master_timeout = 1000            │ │
│  │                                                       │ │
│  │  正常情况: 等待至少一个从库ACK后返回                     │ │
│  │  超时情况: 降级为异步复制                                │ │
│  │                                                       │ │
│  │  适合: 对一致性要求高，可接受偶尔延迟的场景               │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  5. 并行复制优化                                             │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  MySQL MTS参数:                                       │ │
│  │  ├─ slave_parallel_type = LOGICAL_CLOCK               │ │
│  │  ├─ slave_parallel_workers = 16                       │ │
│  │  └─ slave_preserve_commit_order = 1                   │ │
│  │                                                       │ │
│  │  适用场景: 高并发写入，无冲突事务可并行                  │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 三、实践指南

### 3.1 动态路由实现

```java
@Component
public class DynamicRoutingDataSource extends AbstractRoutingDataSource {

    @Autowired
    private ReplicationLagMonitor lagMonitor;

    private static final ThreadLocal<DataSourceType> context =
        new ThreadLocal<>();

    public static void setMaster() {
        context.set(DataSourceType.MASTER);
    }

    public static void setSlave() {
        // 检查延迟
        if (lagMonitor.getLagSeconds() > 5) {
            context.set(DataSourceType.MASTER);
        } else {
            context.set(DataSourceType.SLAVE);
        }
    }

    @Override
    protected Object determineCurrentLookupKey() {
        return context.get();
    }

    @AfterMethod
    public void clear() {
        context.remove();
    }
}

// AOP自动切换
@Aspect
@Component
public class DataSourceAspect {

    @Around("@annotation(readOnly)")
    public Object around(ProceedingJoinPoint point, ReadOnly readOnly)
            throws Throwable {
        try {
            DynamicRoutingDataSource.setSlave();
            return point.proceed();
        } finally {
            DynamicRoutingDataSource.clear();
        }
    }

    @Around("execution(* *..*Service.update*(..))")
    public Object aroundUpdate(ProceedingJoinPoint point) throws Throwable {
        try {
            DynamicRoutingDataSource.setMaster();
            return point.proceed();
        } finally {
            DynamicRoutingDataSource.clear();
        }
    }
}
```

### 3.2 缓存标记实现

```java
@Service
public class ReadAfterWriteService {

    @Autowired
    private RedisTemplate<String, String> redis;

    @Autowired
    private DataSource masterDataSource;

    @Autowired
    private DataSource slaveDataSource;

    private static final String WRITE_MARK_PREFIX = "write:mark:";
    private static final long MARK_TTL = 5; // 5秒

    // 写入后打标记
    public void updateUser(User user) {
        // 1. 写入主库
        jdbcTemplate.update("UPDATE users SET ... WHERE id = ?", user.getId());

        // 2. 打缓存标记
        String markKey = WRITE_MARK_PREFIX + user.getId();
        redis.opsForValue().set(markKey, "1", MARK_TTL, TimeUnit.SECONDS);
    }

    // 读取时检查标记
    public User getUser(Long userId) {
        String markKey = WRITE_MARK_PREFIX + userId;
        Boolean hasMark = redis.hasKey(markKey);

        DataSource ds = Boolean.TRUE.equals(hasMark) ?
            masterDataSource : slaveDataSource;

        // 使用指定数据源查询
        return queryWithDataSource(ds, userId);
    }
}
```

## 四、常见问题

**Q1: Seconds_Behind_Master为0但实际有延迟？**
A: 网络中断时该值不准确。应同时监控Exec_Master_Log_Pos与主库位置的差距。

**Q2: 并行复制为什么仍有延迟？**
A: 1) 事务有依赖关系必须串行；2) 单表热点；3) 大事务阻塞。可调整slave_parallel_workers和binlog_group_commit_sync_delay。

**Q3: 如何做到读写完全一致性？**
A: 1) 全同步复制（性能差）；2) 强制主库读（扩展性差）；3) 使用分布式协议（Paxos/Raft）如TiDB/CockroachDB。
