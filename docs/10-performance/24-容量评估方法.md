# 容量评估方法

## 核心概念

### 1. 容量评估的重要性

容量评估是系统可扩展性的基础，确保系统能够支撑业务增长。

**容量评估维度**：

```
┌─────────────────────────────────────────────────────────────┐
│                   容量评估维度                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 业务容量                                                │
│      - 支持多少用户同时在线                                  │
│      - 支持多少TPS/QPS                                       │
│      - 数据存储容量                                          │
│                                                             │
│   2. 系统容量                                                │
│      - CPU利用率上限                                         │
│      - 内存容量                                              │
│      - 网络带宽                                              │
│      - 磁盘IO                                                │
│                                                             │
│   3. 依赖容量                                                │
│      - 数据库连接数                                          │
│      - 缓存容量                                              │
│      - 消息队列吞吐量                                        │
│      - 第三方服务限流                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 容量评估方法

**常用方法对比**：

| 方法 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 经验估算 | 基于历史经验 | 快速 | 不准确 | 初期规划 |
| 业务建模 | 基于业务指标 | 贴合业务 | 需要数据 | 电商、社交 |
| 排队论 | 数学建模 | 理论支撑 | 简化假设 | 系统架构 |
| 压力测试 | 实际验证 | 最准确 | 成本高 | 上线前验证 |
| 监控预测 | 趋势外推 | 数据驱动 | 依赖历史 | 运维阶段 |

## 实践方法

### 业务容量评估

**1. 用户模型法**：

```
评估公式：
并发用户数 = 日活用户 × 峰值系数 × 在线率

示例：
- 日活用户(DAU)：100万
- 峰值系数（峰值/平均）：3
- 在线率（同时在线比例）：10%

并发用户数 = 100万 × 3 × 10% = 30万

QPS估算：
QPS = 并发用户数 × 人均请求频率 / 时间窗口
    = 30万 × 10次/分钟 / 60
    = 5万 QPS
```

**2. 业务活动法**：

```java
public class CapacityCalculator {

    // 计算订单系统容量
    public CapacityResult calculateOrderCapacity(BusinessScenario scenario) {
        // 1. 业务指标
        long dailyOrders = scenario.getDailyOrders();      // 日订单量
        double peakRatio = scenario.getPeakRatio();        // 峰值比例
        int peakDurationMinutes = scenario.getPeakDuration(); // 峰值持续

        // 2. 计算峰值订单量
        double peakOrdersPerSecond = (dailyOrders * peakRatio) / (peakDurationMinutes * 60);

        // 3. 考虑业务复杂度
        double businessComplexity = 1.5;  // 每订单涉及的操作数
        double systemCalls = 3;           // 每操作涉及的系统调用

        // 4. 计算系统QPS
        double systemQps = peakOrdersPerSecond * businessComplexity * systemCalls;

        // 5. 计算资源需求
        int requiredInstances = (int) Math.ceil(systemQps / 1000);  // 假设单机1000QPS

        return CapacityResult.builder()
            .peakQps((long) systemQps)
            .requiredInstances(requiredInstances)
            .build();
    }
}
```

### 系统容量评估

**1. 资源瓶颈分析**：

```
评估步骤：

1. 单实例压测
   - CPU瓶颈：找出CPU 80%时的QPS
   - 内存瓶颈：找出内存瓶颈时的并发数
   - IO瓶颈：找出磁盘/网络饱和时的吞吐量

2. 计算扩展系数
   扩展系数 = 目标QPS / 单机QPS × 冗余系数(1.3)

3. 考虑依赖限制
   - 数据库连接数限制
   - 缓存容量限制
   - 消息队列限制

4. 综合评估
   最终容量 = min(CPU容量, 内存容量, IO容量, 依赖容量)
```

**2. 容量评估工具**：

```python
class CapacityEvaluator:
    """容量评估器"""

    def __init__(self, metrics_collector):
        self.metrics = metrics_collector

    def evaluate(self, service_name, target_qps):
        """评估服务容量"""
        # 1. 获取当前负载和容量
        current_qps = self.metrics.get_qps(service_name)
        current_cpu = self.metrics.get_cpu_usage(service_name)

        # 2. 计算CPU瓶颈
        cpu_capacity = current_qps * (0.8 / current_cpu)  # 目标CPU 80%

        # 3. 计算内存瓶颈
        memory_capacity = self.evaluate_memory_limit(service_name)

        # 4. 计算依赖瓶颈
        dependency_capacity = self.evaluate_dependencies(service_name)

        # 5. 综合评估
        total_capacity = min(cpu_capacity, memory_capacity, dependency_capacity)

        # 6. 计算缺口
        gap = target_qps - total_capacity
        required_instances = math.ceil(target_qps / (total_capacity / current_instances))

        return {
            'current_capacity': total_capacity,
            'target_qps': target_qps,
            'gap': gap,
            'required_instances': required_instances,
            'bottleneck': self.identify_bottleneck(cpu_capacity, memory_capacity, dependency_capacity)
        }

    def forecast(self, service_name, months=6):
        """预测未来容量需求"""
        # 获取历史增长趋势
        growth_rate = self.metrics.get_growth_rate(service_name, months=3)
        current_capacity = self.evaluate(service_name, 0)['current_capacity']

        forecasts = []
        for i in range(1, months + 1):
            projected = current_capacity * ((1 + growth_rate) ** i)
            required = self.calculate_required_resources(projected)
            forecasts.append({
                'month': i,
                'projected_qps': projected,
                'required_instances': required
            })

        return forecasts
```

## 案例数据

### 电商系统容量评估

**业务场景**：双11大促容量规划

**评估过程**：

```
1. 业务预测
   - 去年双11订单：1000万
   - 预期增长：50%
   - 今年目标：1500万订单
   - 峰值时段：2小时
   - 峰值订单率：60%订单在峰值产生

2. 计算峰值
   峰值订单数 = 1500万 × 60% = 900万单
   峰值QPS = 900万 / (2 × 3600) = 1250单/秒

3. 系统容量计算
   订单创建涉及：
   - 订单服务：1调用
   - 库存服务：1调用
   - 支付服务：1调用
   - 消息通知：1调用（异步）

   同步调用：3次
   系统QPS = 1250 × 3 = 3750 QPS

   加上查询流量（10:1）
   总QPS = 3750 × 11 = 41250 QPS

4. 资源评估
   单机能力：1000 QPS @ 70% CPU
   需要实例：41250 / 1000 × 1.3(冗余) = 54台

5. 依赖评估
   数据库连接：54 × 20 = 1080（需DB支持）
   Redis容量：热点数据100GB
   MQ吞吐量：1250 msg/s
```

**评估结果**：

| 组件 | 当前容量 | 需求容量 | 缺口 | 建议 |
|------|----------|----------|------|------|
| 应用服务器 | 30台 | 54台 | 24台 | 扩容 |
| 数据库 | 500连接 | 1080连接 | 580连接 | 升级 |
| Redis | 50GB | 100GB | 50GB | 扩容 |
| MQ | 500/s | 1250/s | 750/s | 扩容 |

## 优化前后对比

| 成本控制 | 差 | 好 | 优化 |

---

**总结**：容量评估是系统稳定运行的基础。通过业务建模、系统压测、依赖分析等方法，可以准确评估系统容量，提前规划资源，避免系统过载或资源浪费。
