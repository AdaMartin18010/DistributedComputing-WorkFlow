# 索引设计最佳实践

## 核心概念

### 1. 索引的本质

索引是数据库中用于加速数据查询的数据结构，以额外的存储空间和维护开销换取查询性能。

**索引的代价与收益**：
```
┌─────────────────────────────────────────────────────────────┐
│                   索引的成本收益分析                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   收益（查询加速）：                                          │
│   ┌─────────────────────────────────────────────────────┐  │
│   │ 全表扫描            索引扫描                          │  │
│   │    O(N)      →      O(logN)                         │  │
│   │    100万行          20次IO                          │  │
│   │    10秒             10ms                            │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   成本（存储与维护）：                                        │
│   ┌─────────────────────────────────────────────────────┐  │
│   │ 存储成本                                            │  │
│   │   - B+树索引通常占表大小的10-30%                    │  │
│   │   - 二级索引还要存储主键指针                        │  │
│   │                                                     │  │
│   │ 写入成本                                            │  │
│   │   - INSERT: 维护所有索引                            │  │
│   │   - UPDATE: 可能涉及索引重建                        │  │
│   │   - DELETE: 标记删除，延迟清理                      │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   权衡原则：                                                 │
│   - 读多写少的表：多建索引                                   │
│   - 写多读少的表：少建索引                                   │
│   - 核心查询路径必须有索引                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 索引类型详解

**MySQL索引类型**：

| 类型 | 结构 | 特点 | 适用场景 |
|------|------|------|----------|
| 主键索引 | B+树 | 唯一、非空、聚簇 | 表必须有 |
| 唯一索引 | B+树 | 值唯一 | 业务唯一字段 |
| 普通索引 | B+树 | 无限制 | 普通查询加速 |
| 前缀索引 | B+树 | 只索引前N个字符 | 长字符串 |
| 联合索引 | B+树 | 多列组合 | 多条件查询 |
| 覆盖索引 | B+树 | 包含查询所有字段 | 避免回表 |
| 全文索引 | 倒排表 | 分词索引 | 文本搜索 |
| 空间索引 | R树 | 空间数据 | GIS应用 |
| 哈希索引 | 哈希表 | O(1)查询 | 精确匹配 |

**聚簇索引 vs 非聚簇索引**：
```
┌─────────────────────────────────────────────────────────────┐
│                   聚簇索引结构                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   聚簇索引（InnoDB主键）：                                    │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  主键值 → 完整行数据                                  │  │
│   │                                                     │  │
│   │  [10] → {id:10, name:'Alice', age:25, ...}          │  │
│   │  [20] → {id:20, name:'Bob', age:30, ...}            │  │
│   │  [30] → {id:30, name:'Carol', age:28, ...}          │  │
│   └─────────────────────────────────────────────────────┘  │
│   特点：数据按主键顺序存储，主键查询最快                      │
│                                                             │
│   非聚簇索引（二级索引）：                                    │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  索引列值 → 主键值                                    │  │
│   │                                                     │  │
│   │  [Alice] → 10                                       │  │
│   │  [Bob]   → 20                                       │  │
│   │  [Carol] → 30                                       │  │
│   └─────────────────────────────────────────────────────┘  │
│   查询时需要回表：先查二级索引获取主键，再查聚簇索引          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. 最左前缀原则

联合索引的使用必须遵循最左前缀原则。

**原则说明**：
```sql
-- 创建联合索引 (a, b, c)
CREATE INDEX idx_abc ON table_name(a, b, c);

-- 可以使用索引的情况：
WHERE a = 1                    -- 使用索引第1列
WHERE a = 1 AND b = 2          -- 使用索引第1、2列
WHERE a = 1 AND b = 2 AND c = 3  -- 使用索引全部列
WHERE a = 1 AND c = 3          -- 使用索引第1列（c跳过b，只能用a）
WHERE a = 1 AND b > 2 AND c = 3  -- 使用索引第1、2列（b是范围，c无法使用）

-- 不能使用索引的情况：
WHERE b = 2                    -- 跳过a，无法使用
WHERE b = 2 AND c = 3          -- 跳过a，无法使用
WHERE c = 3                    -- 跳过a、b，无法使用
```

**索引列顺序设计**：
```
设计原则：
1. 等值查询列放前面
2. 区分度高的列放前面
3. 范围查询列放后面
4. 排序列按需放置

示例：订单表查询
常见查询：WHERE user_id = ? AND status = ? AND created_at > ?

方案1: (created_at, user_id, status)  -- 差
方案2: (user_id, status, created_at)  -- 好
方案3: (status, user_id, created_at)  -- 一般（status区分度低）

分析：
- user_id区分度高，先过滤大部分数据
- status等值查询，放在中间
- created_at范围查询，放最后
```

## 实践方法

### 索引设计流程

**1. 分析查询模式**：
```sql
-- 收集慢查询
SELECT 
    DIGEST_TEXT,
    COUNT_STAR,
    AVG_TIMER_WAIT,
    SUM_ROWS_SENT,
    SUM_ROWS_EXAMINED,
    FIRST_SEEN,
    LAST_SEEN
FROM performance_schema.events_statements_summary_by_digest
WHERE SUM_ROWS_EXAMINED / SUM_ROWS_SENT > 100  -- 扫描行数远大于返回行数
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 20;
```

**2. 设计索引方案**：
```sql
-- 示例：电商订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(32) NOT NULL,
    status TINYINT NOT NULL DEFAULT 0,
    amount DECIMAL(10,2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    -- 业务唯一索引
    UNIQUE INDEX uk_order_no (order_no),
    
    -- 用户查询索引
    INDEX idx_user_status_created (user_id, status, created_at),
    
    -- 后台管理查询索引
    INDEX idx_status_created (status, created_at),
    
    -- 订单号前缀搜索索引（如有需求）
    INDEX idx_order_no_prefix (order_no(8))
) ENGINE=InnoDB;
```

**3. 索引优化检查清单**：
```
□ 是否有主键？
□ 外键是否有索引？
□ WHERE条件列是否有索引？
□ ORDER BY列是否有索引？
□ JOIN列是否有索引？
□ 是否存在冗余索引？
□ 是否存在从未使用的索引？
□ 索引列选择性是否足够高？
□ 联合索引顺序是否符合最左前缀？
□ 是否存在索引失效的场景？
```

### 高效索引设计

**1. 覆盖索引设计**：
```sql
-- 场景：查询用户的订单数量
-- 原SQL
SELECT COUNT(*) FROM orders WHERE user_id = 100;

-- 索引方案1：只在user_id建索引
CREATE INDEX idx_user ON orders(user_id);
-- 执行计划：使用idx_user，但需要回表统计（如果COUNT需要判断NULL）

-- 索引方案2：覆盖索引
CREATE INDEX idx_user_id ON orders(user_id, id);
-- 执行计划：Using index，无需回表

-- 更优方案
-- 如果InnoDB且id是主键，方案1已经足够，因为二级索引包含主键
```

**2. 延迟关联优化深分页**：
```sql
-- 问题：深分页性能差
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY created_at DESC 
LIMIT 1000000, 10;
-- 需要扫描1000010行，返回10行

-- 优化：延迟关联
SELECT o.* 
FROM orders o
JOIN (
    SELECT id 
    FROM orders 
    WHERE user_id = 100 
    ORDER BY created_at DESC 
    LIMIT 1000000, 10
) tmp ON o.id = tmp.id;

-- 需要索引：
CREATE INDEX idx_user_created_id ON orders(user_id, created_at, id);
-- 子查询可以覆盖索引，只返回id，再关联查询完整数据
```

**3. 索引下推优化**：
```sql
-- MySQL 5.6+ 索引下推（ICP）
-- 场景
SELECT * FROM orders 
WHERE user_id = 100 
  AND amount > 500;  -- amount不在联合索引中

-- 索引：idx_user_created (user_id, created_at)

-- 无ICP：
-- 1. 从索引查出所有user_id=100的记录
-- 2. 回表获取完整数据
-- 3. 过滤amount > 500

-- 有ICP：
-- 1. 从索引查出user_id=100的记录
-- 2. 检查WHERE所有条件（包括amount，虽然不在索引中但可以减少回表）
-- 3. 满足条件的才回表
```

### 索引维护

**1. 索引碎片整理**：
```sql
-- 查看索引碎片
SELECT 
    table_name,
    index_name,
    round(data_free/1024/1024, 2) as data_free_mb,
    round(data_length/1024/1024, 2) as data_length_mb,
    round(index_length/1024/1024, 2) as index_length_mb,
    round(data_free/(data_length+index_length)*100, 2) as frag_pct
FROM information_schema.tables
WHERE table_schema = 'mydb'
  AND data_free > 0
ORDER BY data_free DESC;

-- 重建索引（MySQL 5.7+）
ALTER TABLE orders ENGINE=InnoDB;

-- 或在线重建（MySQL 8.0）
ALTER TABLE orders ADD INDEX idx_new (...), 
ALGORITHM=INPLACE, LOCK=NONE;
```

**2. 索引统计信息更新**：
```sql
-- 更新统计信息
ANALYZE TABLE orders;

-- 查看统计信息
SHOW INDEX FROM orders;

-- 直方图统计（MySQL 8.0）
ANALYZE TABLE orders UPDATE HISTOGRAM ON status, user_id;
```

**3. 冗余索引识别**：
```sql
-- 查找冗余索引
SELECT 
    t.TABLE_SCHEMA,
    t.TABLE_NAME,
    s1.INDEX_NAME as 'redundant_index',
    s1.COLUMN_NAME as 'redundant_column',
    s2.INDEX_NAME as 'dominant_index',
    s2.COLUMN_NAME as 'dominant_column'
FROM information_schema.STATISTICS s1
JOIN information_schema.STATISTICS s2 
    ON s1.TABLE_SCHEMA = s2.TABLE_SCHEMA 
    AND s1.TABLE_NAME = s2.TABLE_NAME
    AND s1.SEQ_IN_INDEX = s2.SEQ_IN_INDEX
    AND s1.COLUMN_NAME = s2.COLUMN_NAME
    AND s1.INDEX_NAME != s2.INDEX_NAME
WHERE s1.SEQ_IN_INDEX = 1
  AND s1.CARDINALITY <= s2.CARDINALITY
  AND s1.TABLE_SCHEMA = 'mydb'
ORDER BY s1.TABLE_SCHEMA, s1.TABLE_NAME;
```

## 案例数据

### 电商订单索引优化

**优化前**：
```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    status INT,
    created_at TIMESTAMP,
    INDEX idx_user (user_id),
    INDEX idx_status (status),
    INDEX idx_created (created_at)
);

-- 查询1：用户订单列表
SELECT * FROM orders WHERE user_id = 100 ORDER BY created_at DESC;
-- Using where; Using filesort

-- 查询2：状态统计
SELECT status, COUNT(*) FROM orders WHERE created_at > '2024-01-01' GROUP BY status;
-- Using where; Using temporary; Using filesort
```

**优化后**：
```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    status INT,
    created_at TIMESTAMP,
    -- 删除单个索引，改为联合索引
    INDEX idx_user_created (user_id, created_at),
    INDEX idx_created_status (created_at, status)
);

-- 查询1：
-- Using index（覆盖索引，无需回表）

-- 查询2：
-- Using index（覆盖索引，无需临时表和排序）
```

**效果**：
| 查询 | 优化前RT | 优化后RT | 优化前扫描行 | 优化后扫描行 |
|------|----------|----------|--------------|--------------|
| 用户订单 | 500ms | 5ms | 100万 | 100 |
| 状态统计 | 2s | 100ms | 500万 | 50万 |
