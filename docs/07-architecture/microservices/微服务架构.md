## 📋 执行摘要

微服务架构是一种将单体应用拆分为一组小型、独立部署服务的架构风格。每个服务围绕业务能力构建，通过轻量级通信机制（通常是HTTP/REST或消息队列）交互，实现独立开发、部署和扩展。

## 一、核心概念

### 1.1 定义与原理

**微服务（Microservices）** 是一种架构风格，将应用程序构建为一组小型服务的集合，每个服务：

- 运行在自己的进程中
- 通过轻量级机制（HTTP API或消息）通信
- 围绕业务能力构建
- 可独立部署
- 可使用不同技术栈

### 1.2 微服务设计原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 一个服务只做一件事，做好一件事 |
| **独立部署** | 服务可独立构建、测试、部署、扩容 |
| **故障隔离** | 单个服务故障不影响整体系统 |
| **去中心化** | 数据管理去中心化，技术栈多样化 |
| **基础设施自动化** | CI/CD、自动化测试、监控 |
| **演进式设计** | 服务可重构、合并、拆分 |

### 1.3 适用场景评估

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 大型团队（>30人） | ⭐⭐⭐⭐⭐ | 团队自治，独立交付 |
| 高频迭代业务 | ⭐⭐⭐⭐⭐ | 独立部署，快速上线 |
| 多租户SaaS | ⭐⭐⭐⭐⭐ | 服务级别扩展 |
| 高可用要求 | ⭐⭐⭐⭐ | 故障隔离，局部降级 |
| 小型团队（<10人） | ⭐⭐ | 运维复杂度高 |
| 业务边界模糊 | ⭐⭐ | 拆分困难，耦合严重 |
| 强事务一致性需求 | ⭐⭐ | 分布式事务复杂 |

## 二、服务拆分策略

### 2.1 拆分方法

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **领域驱动设计(DDD)** | 按业务领域/限界上下文拆分 | 业务复杂系统 |
| **业务能力拆分** | 按业务功能模块拆分 | 功能边界清晰 |
| **数据边界拆分** | 按数据所有权拆分 | 数据耦合度低 |
| **读写分离(CQRS)** | 读服务和写服务分离 | 读写比悬殊 |
| **前后端分离** | UI服务和API服务分离 | 多端接入 |

### 2.3 拆分粒度决策

| 粒度 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| **粗粒度** (5-10个服务) | 运维简单，事务易处理 | 团队冲突，部署耦合 | 中型系统 |
| **中粒度** (20-50个服务) | 团队自治，独立部署 | 复杂度适中 | 大型系统 |
| **细粒度** (100+服务) | 极致灵活，精准扩展 | 运维噩梦，调用链长 | 超大型系统 |

**指导原则**：

- **两个披萨团队**：一个服务由两个披萨能吃饱的团队维护（6-10人）
- **康威定律**：组织沟通结构决定系统架构
- **独立演进**：服务可独立重构、替换、废弃

### 5.2 断路器实现

**Resilience4j 示例（Java）**：

```java
// 配置断路器
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)          // 失败率阈值50%
    .slowCallRateThreshold(80)         // 慢调用阈值80%
    .slowCallDurationThreshold(Duration.ofSeconds(2))
    .permittedNumberOfCallsInHalfOpenState(10)
    .slidingWindowSize(100)            // 滑动窗口大小
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .build();

CircuitBreaker circuitBreaker = CircuitBreaker.of("orderService", config);

// 使用断路器保护调用
Supplier<String> decorated = CircuitBreaker
    .decorateSupplier(circuitBreaker, () -> orderService.call());

// 降级处理
try {
    String result = decorated.get();
} catch (CallNotPermittedException ex) {
    // 断路器打开，返回降级结果
    return "Fallback: Order service unavailable";
}
```

| 模式 | 说明 | 与断路器关系 |
|------|------|--------------|
| **重试(Retry)** | 失败后自动重试 | 重试耗尽后触发断路 |
| **降级(Fallback)** | 失败时返回默认值 | 断路器打开后的行为 |
| **限流(Rate Limit)** | 限制请求速率 | 防止过载，配合断路 |
| **隔离(Bulkhead)** | 资源池隔离 | 防止故障扩散 |
| **超时(Timeout)** | 设置调用超时 | 快速失败，触发断路 |

## 六、系统对比

### 6.1 微服务框架对比

| 特性 | Spring Cloud | Dubbo | gRPC | Istio |
|------|--------------|-------|------|-------|
| **通信协议** | HTTP/REST | Dubbo协议/gRPC | HTTP/2 + Protobuf | HTTP/HTTP2/gRPC |
| **服务发现** | Eureka/Consul | Nacos/ZK | 需额外注册中心 | K8s Service |
| **负载均衡** | Ribbon | 内置 | 客户端LB | Envoy代理 |
| **熔断限流** | Hystrix/Sentinel | Sentinel | 需额外实现 | 原生支持 |
| **链路追踪** | Sleuth+Zipkin | SkyWalking | OpenTelemetry | Jaeger/Zipkin |
| **配置中心** | Config Server | Nacos | - | K8s ConfigMap |
| **网关** | Gateway/Zuul | - | - | Ingress Gateway |
| **语言支持** | Java为主 | Java为主 | 多语言 | 多语言 |
| **部署方式** | 独立部署 | 独立部署 | 独立部署 | K8s Sidecar |

### 6.2 通信协议选择

| 协议 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **REST/HTTP** | 简单、通用、易调试 | 开销大、无类型 | 外部API、浏览器 |
| **gRPC** | 高性能、强类型、流支持 | 浏览器需代理 | 内部服务通信 |
| **GraphQL** | 按需获取、聚合查询 | 缓存复杂 | 复杂数据查询 |
| **消息队列** | 异步解耦、削峰填谷 | 实时性差 | 事件驱动 |
| **Dubbo** | Java生态完善 | 跨语言弱 | Java微服务 |

### 7.2 Docker Compose 本地开发

```yaml
# docker-compose.yml
version: '3.8'

services:
  # API 网关
  gateway:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
    depends_on:
      - user-service
      - order-service

  # 用户服务
  user-service:
    build: ./user-service
    environment:
      - DB_HOST=user-db
      - REDIS_HOST=redis
    depends_on:
      - user-db
      - redis

  # 订单服务
  order-service:
    build: ./order-service
    environment:
      - DB_HOST=order-db
      - USER_SERVICE_URL=http://user-service:8080
    depends_on:
      - order-db
      - user-service

  # 数据库
  user-db:
    image: postgres:15
    environment:
      POSTGRES_DB: userdb
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password

  order-db:
    image: mysql:8
    environment:
      MYSQL_DATABASE: orderdb
      MYSQL_ROOT_PASSWORD: password

  # Redis缓存
  redis:
    image: redis:7-alpine

  # 注册中心
  consul:
    image: consul:1.16
    ports:
      - "8500:8500"

  # 链路追踪
  jaeger:
    image: jaegertracing/all-in-one:1.47
    ports:
      - "16686:16686"
```

### 7.3 最佳实践

| 实践 | 说明 |
|------|------|
| **数据库独立** | 每个服务拥有自己的数据库 |
| **API版本化** | URL或Header版本控制 (/v1/, /v2/) |
| **异步优先** | 服务间优先使用消息队列 |
| **契约测试** | 消费者驱动契约测试(CDC) |
| **监控可观测** | 日志、指标、追踪三支柱 |
| **配置外部化** | 配置与代码分离，支持动态刷新 |
| **健康检查** | 实现/health端点 |
| **优雅关闭** | 处理SIGTERM，完成请求再退出 |
