## 📋 执行摘要

MySQL分库分表是解决单机数据库性能瓶颈和容量限制的核心方案，通过水平拆分将数据分布到多个节点，实现存储和计算能力的水平扩展。本文档详细阐述分库分表的核心概念、拆分策略、路由算法以及ShardingSphere等中间件的实战应用。

## 一、核心概念

### 1.1 分库分表类型

```
┌─────────────────────────────────────────────────────────────┐
│                    分库分表类型分类                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  垂直拆分 vs 水平拆分：                                      │
│                                                            │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  垂直分库 (Vertical Sharding)                         │ │
│  │                                                       │ │
│  │  单库架构                    垂直分库后                 │ │
│  │  ┌─────────────┐            ┌─────────┐ ┌─────────┐   │ │
│  │  │ user表      │            │用户库   │ │订单库   │   │ │
│  │  │ order表     │    ──→     ├─────────┤ ├─────────┤   │ │
│  │  │ product表   │            │user表   │ │order表  │   │ │
│  │  │ log表       │            │profile表│ │pay表    │   │ │
│  │  └─────────────┘            └─────────┘ └─────────┘   │ │
│  │                                                       │ │
│  │  特点：                                                │ │
│  │  ├─ 按业务模块拆分                                     │ │
│  │  ├─ 单表结构不变                                       │ │
│  │  ├─ 解决库表过多问题                                   │ │
│  │  └─ 跨库Join需应用层处理                               │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  水平分表 (Horizontal Sharding)                       │ │
│  │                                                       │ │
│  │  单表架构                    水平分表后                 │ │
│  │  ┌─────────────┐            ┌─────────┬─────────┐     │ │
│  │  │ order表     │            │order_0  │ order_1 │     │ │
│  │  │ (1亿条)     │    ──→     │(0-5kw)  │(5kw-1亿)│     │ │
│  │  │             │            ├─────────┼─────────┤     │ │
│  │  │             │            │order_2  │ order_3 │     │ │
│  │  │             │            │(1-1.5亿)│(1.5-2亿)│     │ │
│  │  └─────────────┘            └─────────┴─────────┘     │ │
│  │                                                       │ │
│  │  特点：                                                │ │
│  │  ├─ 按数据行拆分                                       │ │
│  │  ├─ 表结构相同                                         │ │
│  │  ├─ 解决单表过大问题                                   │ │
│  │  └─ 需要路由算法                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  垂直分表 (Normalization)：                                  │
│  ├─ 将宽表拆分为多个关联表                                    │
│  ├─ 例：user → user_basic + user_profile + user_setting      │
│  └─ 冷热分离，常用字段和blob/text分离                          │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 分库分表场景判断

| 场景 | 建议方案 | 原因 |
|------|----------|------|
| 单表 > 5000万行 | 水平分表 | 性能下降，维护困难 |
| 单库 > 100张表 | 垂直分库 | 管理复杂，元数据膨胀 |
| 单库数据 > 500GB | 水平分库 | 存储和备份压力 |
| 写QPS > 5000 | 水平分库 | 单实例写入瓶颈 |
| 读QPS > 20000 | 读写分离+水平分表 | 读压力分散 |

## 二、技术细节

### 2.1 分片策略详解

```
┌─────────────────────────────────────────────────────────────┐
│                    分片策略详解                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 范围分片 (Range Sharding)                              │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  按时间范围：                                          │ │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐     │ │
│  │  │2024-Q1  │ │2024-Q2  │ │2024-Q3  │ │2024-Q4  │     │ │
│  │  │order_1  │ │order_2  │ │order_3  │ │order_4  │     │ │
│  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘     │ │
│  │                                                       │ │
│  │  按ID范围：                                            │ │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐                 │ │
│  │  │ 1-1M   │ │1M-2M    │ │2M-3M    │ ...             │ │
│  │  │ shard0  │ │ shard1  │ │ shard2  │                 │ │
│  │  └─────────┘ └─────────┘ └─────────┘                 │ │
│  │                                                       │ │
│  │  优点：                                                │ │
│  │  ├─ 支持范围查询                                       │ │
│  │  ├─ 扩容简单（新增分片即可）                            │ │
│  │  └─ 数据迁移方便                                       │ │
│  │                                                       │ │
│  │  缺点：                                                │ │
│  │  ├─ 热点问题（新数据集中在最后一个分片）                 │ │
│  │  └─ 数据分布不均                                       │ │
│  │                                                       │ │
│  │  适用：时序数据、日志、历史归档                          │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. 哈希分片 (Hash Sharding)                               │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  shard = hash(key) % N                                │ │
│  │                                                       │ │
│  │  例：hash(user_id) % 4                                │ │
│  │                                                       │ │
│  │  user_id=1  → hash=12345 → 12345 % 4 = 1 → shard_1  │ │
│  │  user_id=2  → hash=67890 → 67890 % 4 = 2 → shard_2  │ │
│  │  user_id=3  → hash=11111 → 11111 % 4 = 3 → shard_3  │ │
│  │                                                       │ │
│  │  优点：                                                │ │
│  │  ├─ 数据分布均匀                                       │ │
│  │  ├─ 无热点问题                                         │ │
│  │  └─ 点查性能好                                         │ │
│  │                                                       │ │
│  │  缺点：                                                │ │
│  │  ├─ 范围查询需扫描全部分片                              │ │
│  │  ├─ 扩容复杂（需迁移数据）                              │ │
│  │  └─ 难以实现范围分页                                   │ │
│  │                                                       │ │
│  │  适用：用户数据、订单数据等均匀访问                      │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. 一致性哈希 (Consistent Hashing)                        │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  原理：                                                │ │
│  │  ├─ 将节点和数据映射到哈希环                           │ │
│  │  ├─ 数据顺时针找到最近的节点                           │ │
│  │  └─ 扩容只影响相邻节点                                 │ │
│  │                                                       │ │
│  │  虚拟节点：                                            │ │
│  │  ├─ 每个物理节点对应多个虚拟节点                        │ │
│  │  └─ 解决数据倾斜问题                                   │ │
│  │                                                       │ │
│  │  详见：一致性哈希.md                                    │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  4. 标签分片 (Tag/Enum Sharding)                           │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  按枚举值分片：                                         │ │
│  │  ├─ region=CN → shard_cn                              │ │
│  │  ├─ region=US → shard_us                              │ │
│  │  └─ region=EU → shard_eu                              │ │
│  │                                                       │ │
│  │  适用：多租户、地域分区                                 │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  5. 复合分片策略                                           │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  分库 + 分表：                                          │ │
│  │  ├─ 先按user_id % 8 分库（8个库）                       │ │
│  │  └─ 再按order_id % 128 分表（每库128表）                │ │
│  │                                                       │ │
│  │  时间 + 哈希：                                          │ │
│  │  ├─ 先按时间分表（按月）                                │ │
│  │  └─ 再按user_id哈希分片                                 │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 四、实践指南

### 4.1 ShardingSphere配置示例

```yaml
# application-sharding.yml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1, ds2, ds3
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/db0
        username: root
        password: xxx
      ds1:
        jdbc-url: jdbc:mysql://localhost:3306/db1
        # ...
      ds2:
        jdbc-url: jdbc:mysql://localhost:3306/db2
        # ...
      ds3:
        jdbc-url: jdbc:mysql://localhost:3306/db3
        # ...
    
    rules:
      sharding:
        tables:
          t_order:
            # 实际数据节点
            actual-data-nodes: ds${0..3}.t_order_${0..15}
            
            # 分库策略
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: db-hash
            
            # 分表策略
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: table-hash
            
            # 主键生成策略
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake
        
        # 分片算法
        sharding-algorithms:
          db-hash:
            type: INLINE
            props:
              algorithm-expression: ds${user_id % 4}
          table-hash:
            type: INLINE
            props:
              algorithm-expression: t_order_${order_id % 16}
        
        # 主键生成器
        key-generators:
          snowflake:
            type: SNOWFLAKE
            props:
              worker-id: ${WORKER_ID:0}
    
    props:
      sql-show: true
```

### 4.3 常见问题

**Q1: 分库分表后如何做关联查询？**
A: 1) 相同分片键的数据放在同一分片（绑定表）；2) 宽表冗余字段；3) 异构索引表；4) 应用层组装；5) ES/HBase辅助查询。

**Q2: 如何选择分片数量？**
A: 预估未来3-5年数据量，按单表500万行计算。建议分片数为2^n，便于扩容。初期可部署少节点，逻辑分片多，后期迁移。

**Q3: 扩容如何平滑进行？**
A: 1) 逻辑分片 > 物理分片，初期多逻辑分片映射到少节点；2) 使用一致性哈希减少迁移；3) 双写+增量同步+校验。

**Q4: 全局唯一ID如何生成？**
A: 推荐使用雪花算法（趋势递增）或号段模式（严格递增）。避免UUID（无序）和自增ID（分片困难）。
