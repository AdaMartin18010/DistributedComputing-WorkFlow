# 事务隔离级别详解

## 概述

事务隔离级别（Transaction Isolation Level）是数据库管理系统（DBMS）中用于控制并发事务之间相互影响程度的重要机制。不同的隔离级别在数据一致性、并发性能和系统开销之间提供不同的权衡。SQL-92标准定义了四个隔离级别，从最低的读未提交到最高的串行化，每个级别都对应着不同的事务并发问题。

---

## 一、并发事务问题

在深入讨论隔离级别之前，我们需要先了解并发事务可能引发的三种典型问题。

### 1.1 脏读（Dirty Read）

**定义**：一个事务读取了另一个尚未提交事务修改的数据。

```
┌─────────────────────────────────────────────────────────────┐
│                      脏读示意图                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   时间轴 ───────────────────────────────────────────────→   │
│                                                             │
│   事务A                      事务B                          │
│   ─────                      ─────                          │
│   BEGIN;                                                    │
│   UPDATE accounts                                           │
│   SET balance = 800                                         │
│   WHERE id = 'A';  ──────────────────────────┐              │
│     (未提交)                                  │              │
│                                              ↓              │
│                              BEGIN;                         │
│                              SELECT balance                 │
│                              FROM accounts                  │
│                              WHERE id = 'A';                │
│                              返回: 800  ←── 脏读！           │
│                                                              │
│   ROLLBACK;  ←── 事务A回滚                                  │
│   (balance恢复为1000)                                        │
│                                                              │
│                              COMMIT;                        │
│                              (基于错误数据做了决策)            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**危害**：

- 读取到可能被回滚的"临时"数据
- 基于脏数据做出错误业务决策
- 破坏业务逻辑的正确性

### 1.2 不可重复读（Non-Repeatable Read）

**定义**：在同一个事务内，两次读取同一行数据却得到了不同的结果。

```
┌─────────────────────────────────────────────────────────────┐
│                   不可重复读示意图                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   事务A                      事务B                          │
│   ─────                      ─────                          │
│   BEGIN;                                                    │
│                                                             │
│   SELECT balance                                            │
│   FROM accounts                                             │
│   WHERE id = 'A';                                           │
│   返回: 1000  ←── 第一次读取                                │
│                                │                            │
│                                ↓                            │
│                              BEGIN;                         │
│                              UPDATE accounts                │
│                              SET balance = 800              │
│                              WHERE id = 'A';                │
│                              COMMIT;                        │
│                                │                            │
│   SELECT balance              │                             │
│   FROM accounts               │                             │
│   WHERE id = 'A';             │                             │
│   返回: 800  ←── 第二次读取，数据变了！                      │
│                               (与第一次结果不一致)            │
│                                                             │
│   COMMIT;                                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**与脏读的区别**：

- 脏读读取的是未提交的数据
- 不可重复读读取的是已提交的修改
- 不可重复读是"合法"但"不希望"的读取

### 1.3 幻读（Phantom Read）

**定义**：在同一个事务内，两次执行相同的查询条件，返回的结果集行数不同（多了或少了一些行）。

```
┌─────────────────────────────────────────────────────────────┐
│                      幻读示意图                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   事务A                      事务B                          │
│   ─────                      ─────                          │
│   BEGIN;                                                    │
│                                                             │
│   SELECT * FROM orders                                      │
│   WHERE status = 'PENDING';                                 │
│   返回: [Order#1, Order#2]  ←── 第一次查询，2条记录          │
│                                │                            │
│                                ↓                            │
│                              BEGIN;                         │
│                              INSERT INTO orders             │
│                              VALUES ('Order#3', 'PENDING'); │
│                              COMMIT;                        │
│                                │                            │
│   SELECT * FROM orders        │                             │
│   WHERE status = 'PENDING';   │                             │
│   返回: [Order#1, Order#2,    │                             │
│          Order#3]             │                             │
│   多了一条记录！ ←── 幻读！     │                             │
│                               (像出现了"幻影")               │
│                                                             │
│   COMMIT;                                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**与不可重复读的区别**：

- 不可重复读：同一行数据内容变化
- 幻读：结果集的行数变化（新增或删除了行）

## 二、四种隔离级别

SQL-92标准定义了四种隔离级别，每种级别解决不同的问题。

### 2.1 隔离级别速查表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现机制 | 性能 |
|---------|------|-----------|------|---------|------|
| READ UNCOMMITTED | ✓ | ✓ | ✓ | 无锁读取 | 最高 |
| READ COMMITTED | ✗ | ✓ | ✓ | 行级锁+MVCC | 高 |
| REPEATABLE READ | ✗ | ✗ | ✓ | 间隙锁+MVCC | 中 |
| SERIALIZABLE | ✗ | ✗ | ✗ | 范围锁/串行执行 | 最低 |

> ✓ = 可能发生，✗ = 不会发生

### 2.2 READ UNCOMMITTED（读未提交）

**定义**：最低的隔离级别，允许事务读取其他事务尚未提交的数据修改。

```
┌─────────────────────────────────────────────────────────────┐
│                    读未提交示意                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   读取操作不需要加任何锁，直接读取最新数据（包括未提交的）      │
│                                                             │
│   ┌─────────┐         ┌─────────┐                          │
│   │ 事务A   │         │ 事务B   │                          │
│   │ 写锁   │ ←────X──→ │ 无锁   │  (读不阻塞写，写不阻塞读)   │
│   └────┬────┘         └────┬────┘                          │
│        │                   │                                │
│        │                   └──→ 直接读取内存中的最新值        │
│        │                       (不管是否已提交)               │
│        │                                                    │
│        └──→ 修改数据但不影响其他事务的读取                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Java代码示例**：

```java
public class ReadUncommittedDemo {

    public void demonstrateReadUncommitted() throws SQLException {
        Connection conn = dataSource.getConnection();

        // 设置读未提交隔离级别
        conn.setTransactionIsolation(Connection.TRANSACTION_READ_UNCOMMITTED);
        conn.setAutoCommit(false);

        try {
            // 查询账户余额
            String sql = "SELECT balance FROM accounts WHERE id = ?";
            try (PreparedStatement stmt = conn.prepareStatement(sql)) {
                stmt.setString(1, "account_001");
                ResultSet rs = stmt.executeQuery();

                if (rs.next()) {
                    double balance = rs.getDouble("balance");
                    // 此时读到的可能是其他事务未提交的修改
                    System.out.println("读取余额（可能包含未提交数据）: " + balance);
                }
            }

            conn.commit();
        } catch (Exception e) {
            conn.rollback();
            throw e;
        }
    }
}
```

**适用场景**：

- 对数据一致性要求极低的统计报表
- 大数据量粗略估算
- 非关键业务的临时查询

**警告**：生产环境极少使用，仅用于特殊场景。

### 2.3 READ COMMITTED（读已提交）

**定义**：事务只能读取已经提交的数据，避免脏读，但可能出现不可重复读和幻读。

**实现原理（MVCC）**：

```
┌─────────────────────────────────────────────────────────────┐
│              读已提交 - MVCC实现原理                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   数据存储结构（每行包含隐藏字段）：                           │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  id  │  balance  │  DB_TRX_ID  │  DB_ROLL_PTR  │   │  │
│   │  A   │   1000    │    TX_100   │    指向Undo    │   │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   Undo日志链：                                               │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐             │
│   │ balance  │    │ balance  │    │ balance  │             │
│   │ = 800    │ ←──│ = 900    │ ←──│ = 1000   │             │
│   │ (TX_103) │    │ (TX_102) │    │ (TX_100) │             │
│   └──────────┘    └──────────┘    └──────────┘             │
│        ↑                                                    │
│        └── 当前事务ID: TX_105                               │
│            只读取已提交的数据版本（TX_102及之前）              │
│                                                             │
│   读取流程：                                                 │
│   1. 找到数据行                                              │
│   2. 检查DB_TRX_ID是否已提交                                  │
│   3. 若未提交，沿DB_ROLL_PTR查找上一个已提交版本              │
│   4. 返回已提交版本的数据                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Oracle实现示例**：

```java
public class ReadCommittedOracleDemo {

    /**
     * Oracle默认使用READ COMMITTED
     * 每次查询都获取最新的已提交快照
     */
    public void oracleReadCommittedBehavior() throws SQLException {
        Connection conn = dataSource.getConnection();
        // Oracle默认就是READ COMMITTED
        // conn.setTransactionIsolation(Connection.TRANSACTION_READ_COMMITTED);

        conn.setAutoCommit(false);

        try {
            // 第一次查询
            double balance1 = queryBalance(conn, "account_001");
            System.out.println("T1时刻余额: " + balance1);

            // 模拟其他事务提交修改
            simulateOtherTransactionCommit();

            // 第二次查询（同一事务内）
            double balance2 = queryBalance(conn, "account_001");
            System.out.println("T2时刻余额: " + balance2);

            // 在Oracle中，balance2可能不等于balance1（不可重复读）
            System.out.println("两次读取结果" + (balance1 == balance2 ? "相同" : "不同"));

            conn.commit();
        } catch (Exception e) {
            conn.rollback();
            throw e;
        }
    }

    private double queryBalance(Connection conn, String accountId) throws SQLException {
        String sql = "SELECT balance FROM accounts WHERE id = ?";
        try (PreparedStatement stmt = conn.prepareStatement(sql)) {
            stmt.setString(1, accountId);
            ResultSet rs = stmt.executeQuery();
            if (rs.next()) {
                return rs.getDouble("balance");
            }
            return 0;
        }
    }
}
```

**优缺点分析**：

| 优点 | 缺点 |
|-----|------|
| 避免脏读，保证基本数据质量 | 不可重复读，同一事务内数据可能变化 |
| 并发性能较好 | 幻读问题 |
| 实现简单高效 | 某些业务场景下需要额外处理 |

**适用场景**：

- 大多数OLTP系统的默认选择
- 对实时性要求高的查询
- 单次读取操作

### 2.4 REPEATABLE READ（可重复读）

**定义**：在同一个事务内，多次读取同一行数据结果保持一致，避免不可重复读，但可能出现幻读。

**MySQL InnoDB实现**：

```
┌─────────────────────────────────────────────────────────────┐
│              可重复读 - MySQL实现原理                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   事务快照（Read View）：                                     │
│                                                             │
│   事务开始时创建一致性快照，整个事务期间使用同一快照            │
│                                                             │
│   Read View结构：                                           │
│   ┌─────────────────────────────────────────┐              │
│   │  creator_trx_id: 当前事务ID             │              │
│   │  m_ids: 活跃事务ID列表                   │              │
│   │  min_trx_id: 最小活跃事务ID              │              │
│   │  max_trx_id: 下一个分配的事务ID          │              │
│   └─────────────────────────────────────────┘              │
│                                                             │
│   可见性判断规则：                                           │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                                                     │  │
│   │  1. trx_id == creator_trx_id → 可见（当前事务修改）  │  │
│   │                                                     │  │
│   │  2. trx_id < min_trx_id → 可见（已提交事务）         │  │
│   │                                                     │  │
│   │  3. trx_id >= max_trx_id → 不可见（未来事务）        │  │
│   │                                                     │  │
│   │  4. min_trx_id <= trx_id < max_trx_id:             │  │
│   │     - 在m_ids列表中 → 不可见（未提交）               │  │
│   │     - 不在m_ids列表中 → 可见（已提交）               │  │
│   │                                                     │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**间隙锁（Gap Lock）防止幻读**：

```java
public class RepeatableReadDemo {

    /**
     * MySQL InnoDB REPEATABLE READ通过间隙锁部分解决幻读
     */
    public void mysqlRepeatableRead() throws SQLException {
        Connection conn = dataSource.getConnection();
        conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
        conn.setAutoCommit(false);

        try {
            // 查询时使用间隙锁锁定范围
            // 锁定 balance BETWEEN 1000 AND 2000 的记录及间隙
            String sql = "SELECT * FROM accounts " +
                        "WHERE balance BETWEEN 1000 AND 2000 " +
                        "FOR UPDATE";  // 加排他锁，包括间隙锁

            try (Statement stmt = conn.createStatement();
                 ResultSet rs = stmt.executeQuery(sql)) {

                List<Account> accounts = new ArrayList<>();
                while (rs.next()) {
                    accounts.add(new Account(
                        rs.getString("id"),
                        rs.getDouble("balance")
                    ));
                }
                System.out.println("查询到 " + accounts.size() + " 条记录");
            }

            // 此时其他事务无法在这个范围内插入新记录
            // (间隙锁阻止了INSERT操作)

            // 再次查询，结果集相同
            try (Statement stmt = conn.createStatement();
                 ResultSet rs = stmt.executeQuery(sql)) {
                // 结果集与第一次相同，无幻读
            }

            conn.commit();
        } catch (Exception e) {
            conn.rollback();
            throw e;
        }
    }

    /**
     * 演示MySQL中RR的幻读问题（特殊情况下）
     */
    public void demonstratePhantomReadInMySQL() throws SQLException {
        Connection conn = dataSource.getConnection();
        conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
        conn.setAutoCommit(false);

        try {
            // 第一次查询（不加锁）
            String query = "SELECT COUNT(*) FROM orders WHERE status = 'PENDING'";
            int count1 = executeCountQuery(conn, query);
            System.out.println("第一次查询: " + count1 + " 条记录");

            // 其他事务插入新记录并提交
            simulateInsertByOtherTransaction();

            // 第二次查询（使用当前读 - 会读到新数据）
            String update = "UPDATE orders SET status = 'PROCESSING' " +
                           "WHERE status = 'PENDING'";
            int updated = executeUpdate(conn, update);
            System.out.println("更新了 " + updated + " 条记录");
            // updated 可能大于 count1！（幻读）

            conn.commit();
        } catch (Exception e) {
            conn.rollback();
            throw e;
        }
    }
}
```

**快照读 vs 当前读**：

| 类型 | SQL示例 | 行为 | 幻读风险 |
|-----|---------|------|---------|
| 快照读 | `SELECT * FROM t` | 读取快照版本 | 无 |
| 当前读 | `SELECT ... FOR UPDATE` | 读取最新版本并加锁 | 有 |
| 当前读 | `UPDATE/DELETE/INSERT` | 读取最新版本 | 有 |

### 2.5 SERIALIZABLE（串行化）

**定义**：最高的隔离级别，完全避免脏读、不可重复读和幻读，通过强制事务串行执行实现。

**实现方式**：

```
┌─────────────────────────────────────────────────────────────┐
│                   串行化实现方式                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   方式1：两阶段锁（2PL）+ 范围锁                              │
│                                                             │
│   ┌─────────┐         ┌─────────┐                          │
│   │ 事务A   │         │ 事务B   │                          │
│   │         │         │         │                          │
│   │ 锁定    │ ←────X──→│ 等待   │  (范围锁互斥)             │
│   │ 范围R   │         │         │                          │
│   │         │         │         │                          │
│   │ 执行    │         │ 阻塞    │                          │
│   │ 查询    │         │         │                          │
│   │         │         │         │                          │
│   │ 释放锁  │ ───────→│ 获得锁  │                          │
│   │ COMMIT  │         │ 执行    │                          │
│   └─────────┘         └─────────┘                          │
│                                                             │
│   方式2：串行执行队列（如SQLite）                             │
│                                                             │
│   ┌─────────────────────────────────────┐                  │
│   │           串行执行队列               │                  │
│   │  ┌─────┐   ┌─────┐   ┌─────┐       │                  │
│   │  │ TX1 │ → │ TX2 │ → │ TX3 │ → ... │                  │
│   │  └─────┘   └─────┘   └─────┘       │                  │
│   └─────────────────────────────────────┘                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Java代码示例**：

```java
public class SerializableDemo {

    /**
     * 严格串行化事务
     */
    public void strictSerializableTransaction() throws SQLException {
        Connection conn = dataSource.getConnection();
        conn.setTransactionIsolation(Connection.TRANSACTION_SERIALIZABLE);
        conn.setAutoCommit(false);

        try {
            // 1. 查询并锁定整个范围
            String query = "SELECT * FROM inventory WHERE product_id = ?";
            try (PreparedStatement stmt = conn.prepareStatement(query)) {
                stmt.setString(1, "PROD_001");
                ResultSet rs = stmt.executeQuery();

                int currentStock = 0;
                if (rs.next()) {
                    currentStock = rs.getInt("stock");
                }

                // 2. 检查库存
                if (currentStock < 10) {
                    throw new InsufficientStockException("库存不足");
                }

                // 3. 扣减库存
                String update = "UPDATE inventory SET stock = stock - ? " +
                               "WHERE product_id = ?";
                try (PreparedStatement updateStmt = conn.prepareStatement(update)) {
                    updateStmt.setInt(1, 10);
                    updateStmt.setString(2, "PROD_001");
                    updateStmt.executeUpdate();
                }

                // 4. 创建订单
                createOrder(conn, "PROD_001", 10);
            }

            conn.commit();
            System.out.println("订单创建成功");

        } catch (SQLException e) {
            // 串行化冲突处理
            if (e.getErrorCode() == 1213) {  // MySQL死锁错误码
                System.err.println("串行化冲突，事务被回滚: " + e.getMessage());
            }
            conn.rollback();
            throw e;
        }
    }
}
```

## 三、隔离级别对比与选择

### 3.1 详细对比表

| 特性 | RU | RC | RR | SERIALIZABLE |
|-----|----|----|----|-------------|
| 脏读 | ✓ | ✗ | ✗ | ✗ |
| 不可重复读 | ✓ | ✓ | ✗ | ✗ |
| 幻读 | ✓ | ✓ | △ | ✗ |
| 并发度 | 最高 | 高 | 中 | 最低 |
| 锁开销 | 无 | 低 | 中 | 高 |
| 适用场景 | 统计/日志 | OLTP | 金融/核心 | 极端严格 |

> △ = MySQL中部分解决，但当前读仍可能幻读

### 3.2 各数据库默认隔离级别

| 数据库 | 默认隔离级别 | 特点 |
|-------|------------|------|
| MySQL (InnoDB) | REPEATABLE READ | 通过MVCC和间隙锁实现 |
| Oracle | READ COMMITTED | 仅支持RC和SERIALIZABLE |
| PostgreSQL | READ COMMITTED | MVCC实现 |
| SQL Server | READ COMMITTED | 支持快照隔离 |
| SQLite | SERIALIZABLE | 串行执行 |

```
│                                                             │
│   开始                                                       │
│    │                                                        │
│    ▼                                                        │
```

### 4.1 锁超时与死锁

```java
public class TransactionBestPractices {

    /**
     * 设置锁等待超时
     */
    public void setLockTimeout() throws SQLException {
        Connection conn = dataSource.getConnection();

        // MySQL: 设置锁等待超时时间（秒）
        try (Statement stmt = conn.createStatement()) {
            stmt.execute("SET innodb_lock_wait_timeout = 10");
        }

        // PostgreSQL: 设置语句超时
        try (Statement stmt = conn.createStatement()) {
            stmt.execute("SET statement_timeout = '10s'");
        }
    }

    /**
     * 死锁检测与重试
     */
    public void executeWithDeadlockRetry(RetryableTransaction transaction) {
        int maxRetries = 3;
        int retryCount = 0;

        while (retryCount < maxRetries) {
            try {
                transaction.execute();
                return;  // 成功，退出
            } catch (SQLException e) {
                // MySQL死锁错误码: 1213
                // PostgreSQL死锁错误码: 40P01
                if (isDeadlockError(e) && retryCount < maxRetries - 1) {
                    retryCount++;
                    long backoff = (long) (Math.pow(2, retryCount) * 100);  // 指数退避
                    try {
                        Thread.sleep(backoff);
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        throw new RuntimeException(ie);
                    }
                } else {
                    throw new TransactionException("事务执行失败", e);
                }
            }
        }
    }

    @FunctionalInterface
    public interface RetryableTransaction {
        void execute() throws SQLException;
    }
}
```

### 4.2 隔离级别与性能优化

```java
public class IsolationOptimization {

    /**
     * 根据操作类型选择合适的隔离级别
     */
    public void optimizedTransaction() throws SQLException {
        Connection conn = dataSource.getConnection();

        try {
            // 1. 只读查询 - 使用低隔离级别
            conn.setTransactionIsolation(Connection.TRANSACTION_READ_COMMITTED);
            conn.setReadOnly(true);  // 提示数据库优化

            List<ReportData> reportData = generateReport(conn);

            // 2. 关键更新 - 提升隔离级别
            conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
            conn.setReadOnly(false);
            conn.setAutoCommit(false);

            updateCriticalData(conn, reportData);

            conn.commit();
        } catch (Exception e) {
            conn.rollback();
            throw e;
        }
    }

    /**
     * 批量操作使用较低隔离级别
     */
    public void batchOperationWithLowIsolation() throws SQLException {
        Connection conn = dataSource.getConnection();
        // 大批量数据处理，允许一定程度的非严格一致性
        conn.setTransactionIsolation(Connection.TRANSACTION_READ_COMMITTED);
        conn.setAutoCommit(false);

        try {
            // 批量插入
            String sql = "INSERT INTO logs (level, message, created_at) VALUES (?, ?, ?)";
            try (PreparedStatement stmt = conn.prepareStatement(sql)) {
                for (int i = 0; i < 10000; i++) {
                    stmt.setString(1, "INFO");
                    stmt.setString(2, "Log message " + i);
                    stmt.setTimestamp(3, new Timestamp(System.currentTimeMillis()));
                    stmt.addBatch();

                    if (i % 1000 == 0) {
                        stmt.executeBatch();  // 分批提交，减少锁持有时间
                    }
                }
                stmt.executeBatch();
            }

            conn.commit();
        } catch (Exception e) {
            conn.rollback();
            throw e;
        }
    }
}
```

## 五、总结

事务隔离级别是数据库并发控制的核心机制，理解各级别的特点对于设计高并发系统至关重要：

1. **READ UNCOMMITTED**：性能最高但问题最多，仅用于特殊场景
2. **READ COMMITTED**：大多数系统的默认选择，平衡一致性和性能
3. **REPEATABLE READ**：MySQL默认，通过MVCC实现较好的并发性能
4. **SERIALIZABLE**：最安全但性能最差，用于极端严格场景

在实际应用中，应该根据业务需求选择合适的隔离级别，必要时可以通过应用层补偿来弥补数据库隔离级别的不足。同时，合理的索引设计、事务粒度和锁策略也是保证数据一致性和系统性能的关键因素。
