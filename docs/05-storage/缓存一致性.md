## 📋 执行摘要

缓存一致性是分布式系统中的核心挑战，涉及缓存与数据库之间的数据同步问题。双写一致性、最终一致性、强一致性等策略各有取舍。本文档详细分析缓存不一致的根本原因，并提供从延迟双删到分布式事务的多种解决方案。

## 一、核心概念

### 1.1 缓存不一致场景

```
┌─────────────────────────────────────────────────────────────┐
│                    缓存不一致场景分类                        │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 并发读写导致的不一致                                      │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 时刻  线程A(写)       线程B(写)       线程C(读)        │ │
│  │ T1    更新DB:v=1                               │ │
│  │ T2                    更新DB:v=2               │ │
│  │ T3                    删缓存                   │ │
│  │ T4    删缓存                                   │ │
│  │ T5                                    读缓存miss│ │
│  │ T6                                    读DB:v=1 │ │
│  │ T7                                    写缓存v=1│ │
│  │                                              │ │
│  │ 结果: 缓存v=1, DBv=2, 不一致!                  │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  2. 主从延迟导致的不一致                                      │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ ┌────────┐    ┌────────┐    ┌────────┐               │ │
│  │ │ Client │───→│ Master │───→│ Slave  │               │ │
│  │ └────────┘    └────────┘    └────┬───┘               │ │
│  │    ↓                  ↓          │                   │ │
│  │ 写数据               复制延迟     ↓                   │ │
│  │    ↓                  (100ms)  读数据                 │ │
│  │ 删缓存                  ↓      读到旧值               │ │
│  │    ↓                缓存加载   回填缓存               │ │
│  │ 新缓存                旧值      旧值长期存在           │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  3. 缓存失效导致的不一致                                      │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 1. 缓存过期被删除                                        │ │
│  │ 2. 并发请求同时查询DB                                    │ │
│  │ 3. 一个请求先回填旧值                                    │ │
│  │ 4. 另一个请求后回填旧值                                  │ │
│  │ 5. 中间DB更新被覆盖                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 一致性模型

| 模型 | 定义 | 实现难度 | 性能影响 | 适用场景 |
|------|------|----------|----------|----------|
| **强一致性** | 读写操作都反映最新值 | 高 | 大 | 金融交易、库存扣减 |
| **会话一致性** | 同一会话内读写一致 | 中 | 中 | 用户操作反馈 |
| **最终一致性** | 无更新时最终达到一致 | 低 | 小 | 大部分业务场景 |
| **弱一致性** | 不保证一致性 | 最低 | 最小 | 非关键数据缓存 |

### 2.1 双写一致性方案

#### 方案一：延迟双删（Cache Aside优化）

```
┌─────────────────────────────────────────────────────────────┐
│                    延迟双删流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  写操作：                                                    │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌───────┐ │
│  │ Client  │────→│  App    │────→│  Cache  │────→│  DB   │ │
│  │         │     │         │     │         │     │       │ │
│  └─────────┘     └────┬────┘     └────┬────┘     └───┬───┘ │
│                       │               │               │     │
│  Step 1: 删除缓存      │──────────────→│               │     │
│                       │   del(key)    │               │     │
│                       │               │               │     │
│  Step 2: 更新数据库    │───────────────────────────────→     │
│                       │          update                 │     │
│                       │←───────────────────────────────     │
│                       │               │               │     │
│  Step 3: 再次删除缓存  │──────────────→│               │     │
│                       │   del(key)    │               │     │
│                       │   [延迟500ms] │               │     │
│                       │   [异步执行]  │               │     │
│                       │               │               │     │
│  延迟时间计算：                                             │
│  - 主从复制延迟 + 业务读操作耗时 + 安全余量                   │
│  - 通常设置为 200ms ~ 1000ms                                │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

```java
@Service
public class DelayedDoubleDeleteService {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @Autowired
    private UserRepository userRepository;
    
    private ScheduledExecutorService scheduler = 
        Executors.newScheduledThreadPool(10);
    
    /**
     * 延迟双删更新
     */
    public void updateUser(User user) {
        String key = "user:" + user.getId();
        
        // 1. 先删缓存
        redisTemplate.delete(key);
        
        // 2. 更新数据库
        userRepository.save(user);
        
        // 3. 延迟再次删除
        scheduler.schedule(() -> {
            redisTemplate.delete(key);
            // 可选：第二次删除失败的重试机制
        }, 500, TimeUnit.MILLISECONDS);
    }
}
```

```java
@Component
public class CacheConsistencyService {
    
    @Autowired
    private KafkaTemplate<String, CacheDeleteMessage> kafkaTemplate;
    
    @Autowired
    private UserRepository userRepository;
    
    /**
     * 更新数据并发送缓存删除消息
     */
    @Transactional
    public void updateUser(User user) {
        // 1. 更新数据库
        userRepository.save(user);
        
        // 2. 发送缓存删除消息（事务提交后发送）
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronization() {
                @Override
                public void afterCommit() {
                    CacheDeleteMessage msg = new CacheDeleteMessage();
                    msg.setKey("user:" + user.getId());
                    msg.setTimestamp(System.currentTimeMillis());
                    kafkaTemplate.send("cache-delete-topic", msg);
                }
            }
        );
    }
}

@Component
public class CacheDeleteConsumer {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @KafkaListener(topics = "cache-delete-topic")
    public void onMessage(ConsumerRecord<String, CacheDeleteMessage> record) {
        CacheDeleteMessage msg = record.value();
        
        // 删除缓存，带重试
        boolean success = deleteWithRetry(msg.getKey(), 3);
        
        if (!success) {
            // 记录日志，告警，人工介入
            log.error("缓存删除失败: {}", msg.getKey());
        }
    }
    
    private boolean deleteWithRetry(String key, int maxRetries) {
        for (int i = 0; i < maxRetries; i++) {
            try {
                redisTemplate.delete(key);
                return true;
            } catch (Exception e) {
                if (i == maxRetries - 1) return false;
                try {
                    Thread.sleep(100 * (i + 1));
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                }
            }
        }
        return false;
    }
}
```

### 2.2 分布式锁方案（强一致性）

```java
@Component
public class DistributedLockCacheService {
    
    @Autowired
    private RedissonClient redisson;
    
    @Autowired
    private UserRepository userRepository;
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    /**
     * 使用分布式锁保证强一致性
     */
    public void updateUserWithLock(Long userId, Consumer<User> updater) {
        String lockKey = "lock:user:" + userId;
        RLock lock = redisson.getLock(lockKey);
        
        try {
            // 获取锁，最多等待10秒，持有锁30秒
            boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
            if (!locked) {
                throw new RuntimeException("获取锁失败");
            }
            
            try {
                // 1. 查询最新数据
                User user = userRepository.findById(userId)
                    .orElseThrow(() -> new RuntimeException("用户不存在"));
                
                // 2. 执行业务更新
                updater.accept(user);
                
                // 3. 更新数据库
                userRepository.save(user);
                
                // 4. 更新缓存（而非删除）
                String cacheKey = "user:" + userId;
                redisTemplate.opsForValue().set(cacheKey, user, 30, TimeUnit.MINUTES);
                
            } finally {
                lock.unlock();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("获取锁被中断");
        }
    }
    
    /**
     * 读操作使用读锁优化
     */
    public User getUserWithReadLock(Long userId) {
        String lockKey = "lock:user:" + userId;
        RReadWriteLock rwLock = redisson.getReadWriteLock(lockKey);
        RLock readLock = rwLock.readLock();
        
        String cacheKey = "user:" + userId;
        
        // 1. 先尝试无锁读缓存
        User user = (User) redisTemplate.opsForValue().get(cacheKey);
        if (user != null) {
            return user;
        }
        
        try {
            // 2. 缓存未命中，加读锁查库
            readLock.lock();
            
            // 双重检查
            user = (User) redisTemplate.opsForValue().get(cacheKey);
            if (user != null) {
                return user;
            }
            
            // 3. 查询数据库
            user = userRepository.findById(userId).orElse(null);
            
            // 4. 回填缓存
            if (user != null) {
                redisTemplate.opsForValue().set(cacheKey, user, 30, TimeUnit.MINUTES);
            }
            
            return user;
        } finally {
            readLock.unlock();
        }
    }
}
```

### 4.3 常见问题

**Q1: 延迟双删的延迟时间如何确定？**
A: 延迟时间 = 主从同步延迟 + 业务读操作时间 + 安全余量(100~200ms)。建议通过监控统计平均延迟，设置P99延迟值。

**Q2: 消息队列方案如何保证消息不丢失？**
A: 1) Producer开启事务，确保DB提交后才发消息；2) Consumer手动ACK，处理成功才确认；3) 消息持久化；4) 死信队列处理失败消息。

**Q3: 分布式锁的性能瓶颈如何解决？**
A: 1) 锁粒度细化（按用户ID而非全局）；2) 锁超时自动释放；3) 看门狗续期；4) 读多写少场景使用读写锁。

**Q4: 如何发现缓存不一致问题？**
A: 1) 定期抽样对比缓存与DB；2) 业务对账机制；3) 用户反馈异常时主动校验；4) 关键数据添加版本号校验。
