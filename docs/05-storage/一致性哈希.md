## 📋 执行摘要

一致性哈希（Consistent Hashing）是一种特殊的哈希算法，在分布式系统中广泛应用于负载均衡、数据分片和缓存路由。其核心优势是在节点增减时只需重新定位少量数据，有效解决传统哈希取模带来的全量迁移问题。

## 一、核心概念

### 1.1 传统哈希 vs 一致性哈希

```
┌─────────────────────────────────────────────────────────────┐
│              传统哈希 vs 一致性哈希对比                      │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  传统哈希（取模）：                                          │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  node = hash(key) % N                                 │ │
│  │                                                       │ │
│  │  N=3时:                                               │ │
│  │  Key1 → hash(Key1)%3 = 1 → Node 1                    │ │
│  │  Key2 → hash(Key2)%3 = 2 → Node 2                    │ │
│  │  Key3 → hash(Key3)%3 = 0 → Node 0                    │ │
│  │                                                       │ │
│  │  扩容到N=4时:                                          │ │
│  │  Key1 → hash(Key1)%4 = 2 → Node 2 (变更!)            │ │
│  │  Key2 → hash(Key2)%4 = 3 → Node 3 (变更!)            │ │
│  │  Key3 → hash(Key3)%4 = 0 → Node 0 (不变)             │ │
│  │                                                       │ │
│  │  问题: 节点变化时，几乎所有Key都需要重新映射             │ │
│  │        (约 1 - N/(N+1) = 75% 的数据需要迁移)          │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  一致性哈希：                                                │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  哈希环: [0, 2^32-1]                                  │ │
│  │                                                       │ │
│  │  Node映射到环上:                                      │ │
│  │  ├─ Node1: hash("Node1") → 位置100                    │ │
│  │  ├─ Node2: hash("Node2") → 位置1000                   │ │
│  │  └─ Node3: hash("Node3") → 位置5000                   │ │
│  │                                                       │ │
│  │  Key映射到环上，顺时针找到最近的Node:                    │ │
│  │  ├─ Key1: hash(Key1)=200 → 顺时针→ Node2 (1000)      │ │
│  │  ├─ Key2: hash(Key2)=800 → 顺时针→ Node2 (1000)      │ │
│  │  └─ Key3: hash(Key3)=3000 → 顺时针→ Node3 (5000)     │ │
│  │                                                       │ │
│  │  环的结构:                                            │ │
│  │       0                                               │ │
│  │       │                                               │ │
│  │  ┌────┴────┐                                          │ │
│  │  │  Node1  │ ← 位置100                                 │ │
│  │  └────┬────┘                                          │ │
│  │       │                                               │ │
│  │  ┌────┴────┐                                          │ │
│  │  │  Node2  │ ← 位置1000                                │ │
│  │  └────┬────┘                                          │ │
│  │       │                                               │ │
│  │  ┌────┴────┐                                          │ │
│  │  │  Node3  │ ← 位置5000                                │ │
│  │  └────┬────┘                                          │ │
│  │       │                                               │ │
│  │      2^32-1                                           │ │
│  │                                                       │ │
│  │  添加Node4 (位置2500):                                │ │
│  │  ├─ 只影响 Node2 (1000) ~ Node4 (2500) 之间的Key       │ │
│  │  └─ 仅约 25% 的数据需要迁移                             │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 虚拟节点

```
┌─────────────────────────────────────────────────────────────┐
│                    虚拟节点（Virtual Nodes）                 │
│                    解决数据倾斜问题                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  问题：                                                      │
│  ├─ 物理节点少时，哈希分布可能不均                          │ │
│  ├─ 节点宕机时，其负载全部转移到下一个节点                   │ │
│  └─ 不同节点性能差异无法体现                                │ │
│                                                            │
│  解决方案 - 虚拟节点：                                        │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  每个物理节点对应多个虚拟节点                            │ │
│  │                                                       │ │
│  │  Physical Node A:                                     │ │
│  │  ├─ Virtual A1: hash("A#1") → 位置 100                │ │
│  │  ├─ Virtual A2: hash("A#2") → 位置 800                │ │
│  │  ├─ Virtual A3: hash("A#3") → 位置 1500               │ │
│  │  └─ ... (共150个虚拟节点)                              │ │
│  │                                                       │ │
│  │  Physical Node B:                                     │ │
│  │  ├─ Virtual B1: hash("B#1") → 位置 300                │ │
│  │  ├─ Virtual B2: hash("B#2") → 位置 1200               │ │
│  │  └─ ...                                               │ │
│  │                                                       │ │
│  │  效果：                                                │ │
│  │  ├─ 数据分布更均匀（大数定律）                          │ │
│  │  ├─ 节点宕机时，负载分散到多个节点                      │ │
│  │  ├─ 性能好的节点可配置更多虚拟节点                       │ │
│  │  └─ 推荐虚拟节点数: 100-200/物理节点                    │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 二、技术细节

### 2.1 算法实现

```java
public class ConsistentHash<T> {
    private final TreeMap<Long, T> circle = new TreeMap<>();
    private final int virtualNodes;
    private final HashFunction hashFunction;

    public ConsistentHash(int virtualNodes, Collection<T> nodes) {
        this.virtualNodes = virtualNodes;
        this.hashFunction = Hashing.murmur3_128();
        for (T node : nodes) {
            add(node);
        }
    }

    public void add(T node) {
        // 为每个物理节点添加虚拟节点
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hashFunction.hashString(
                node.toString() + "#" + i, StandardCharsets.UTF_8
            ).asLong();
            circle.put(hash, node);
        }
    }

    public void remove(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hashFunction.hashString(
                node.toString() + "#" + i, StandardCharsets.UTF_8
            ).asLong();
            circle.remove(hash);
        }
    }

    public T get(Object key) {
        if (circle.isEmpty()) {
            return null;
        }
        long hash = hashFunction.hashString(
            key.toString(), StandardCharsets.UTF_8
        ).asLong();

        // 顺时针找到最近的节点
        Map.Entry<Long, T> entry = circle.ceilingEntry(hash);
        if (entry == null) {
            // 环尾，回到头部
            entry = circle.firstEntry();
        }
        return entry.getValue();
    }
}
```

### 2.2 应用场景

```
┌─────────────────────────────────────────────────────────────┐
│                    一致性哈希应用场景                        │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 分布式缓存（Dynamo, Cassandra）：                        │ │
│  ├─ Key路由到对应的缓存节点                                  │ │
│  ├─ 节点增减时最小化缓存失效                                 │ │
│  └─ 配合副本因子保证可用性                                   │ │
│                                                            │
│  2. 负载均衡（Nginx, HAProxy）：                             │ │
│  ├─ 会话保持（同一客户端路由到同一后端）                      │ │
│  ├─ 后端节点动态增减                                        │ │
│  └─ 平滑权重调整                                            │ │
│                                                            │
│  3. 分布式存储（Ceph, Swift）：                              │ │
│  ├─ 对象路由到存储节点                                       │ │
│  ├─ 磁盘/节点故障时数据迁移                                  │ │
│  └─ 扩容时均匀分布新数据                                     │ │
│                                                            │
│  4. 分库分表：                                               │ │
│  ├─ 数据路由到分片                                          │ │
│  ├─ 平滑扩容（减少数据迁移）                                 │ │

│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 最佳实践

```
┌─────────────────────────────────────────────────────────────┐
│                    一致性哈希最佳实践                        │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 虚拟节点数：                                             │ │
│  ├─ 推荐：150-200个/物理节点                                │ │
│  ├─ 节点少时增加虚拟节点数                                  │ │
│  └─ 内存允许可适当增加                                      │ │
│                                                            │
│  2. 哈希算法：                                               │ │
│  ├─ 推荐：MurmurHash, FNV-1a                              │ │
│  ├─ 避免：Java默认hashCode（分布不均）                       │ │
│  └─ 考虑：64位哈希减少碰撞                                  │ │
│                                                            │
│  3. 节点权重：                                               │ │
│  ├─ 性能好的节点配置更多虚拟节点                             │ │
│  └─ 实现加权一致性哈希                                      │ │
│                                                            │
│  4. 故障处理：                                               │ │
│  ├─ 节点临时故障：跳过该节点，访问下一个                      │ │
│  ├─ 使用Hinted Handoff暂存数据                              │ │
│  └─ 节点恢复后数据移交（Handoff）                            │ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```
