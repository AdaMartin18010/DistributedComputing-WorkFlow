## 📋 执行摘要

存储引擎是数据库系统的核心组件，不同存储引擎针对特定工作负载进行优化。B-Tree系列（InnoDB）优化读取和事务，LSM-Tree系列（RocksDB）优化写入吞吐，哈希引擎优化点查，列式引擎优化分析。本文档全面对比主流存储引擎的架构、性能特征和适用场景。

### 2.2 详细架构对比

#### B-Tree vs LSM-Tree 写入路径对比

```
┌─────────────────────────────────────────────────────────────┐
│               B-Tree (InnoDB) 写入流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 写入Redo Log（顺序写，WAL）                              │
│       ↓                                                    │
│  2. 修改Buffer Pool中的页                                    │
│     ├─ 页不在内存：从磁盘加载 → 修改                         │
│     └─ 页在内存：直接修改                                   │
│       ↓                                                    │
│  3. 脏页刷盘（后台或Checkpoint触发）                         │
│     ├─ 随机写（修改B+树结构）                                │
│     ├─ 可能触发页分裂（大量随机I/O）                         │
│     └─ Double Write保护                                     │
│       ↓                                                    │
│  4. 写入Binlog（复制用）                                     │
│                                                            │
│  特征：                                                      │
│  ├─ 读取路径简单，延迟低                                     │
│  ├─ 写入路径复杂，随机I/O多                                  │
│  └─ 事务和崩溃恢复完善                                       │
│                                                            │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│               LSM-Tree (RocksDB) 写入流程                     │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 写入WAL（顺序写）                                        │
│       ↓                                                    │
│  2. 写入MemTable（内存跳表）                                 │
│       ↓                                                    │
│  3. MemTable达到阈值 → Flush到L0（顺序写）                   │
│       ↓                                                    │
│  4. 后台Compaction（多路归并，顺序I/O）                       │
│     ├─ L0→L1: 全量排序                                      │
│     ├─ LN→LN+1: 部分合并                                    │
│     └─ 清理过期版本                                         │
│       ↓                                                    │
│  5. SSTable文件管理（分层有序）                              │
│                                                            │
│  特征：                                                      │
│  ├─ 写入路径简单，纯顺序I/O                                  │
│  ├─ 读取路径复杂，需多层查找                                 │
│  └─ 空间放大和读放大需调优                                   │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 四、选型决策指南

### 4.1 决策树

```
存储引擎选型决策树：

开始
  │
  ├── 是否需要持久化？
  │     ├── 否 → Redis / Memcached（内存缓存）
  │     └── 是 → 继续
  │
  ├── 主要工作负载类型？
  │     ├── OLTP（高并发事务）
  │     │     ├── 是否需要SQL接口？
  │     │     │     ├── 是 → MySQL/InnoDB 或 PostgreSQL
  │     │     │     └── 否 → TiKV / FoundationDB
  │     │     └── 写入量极大？
  │     │           ├── 是 → RocksDB 底层引擎
  │     │           └── 否 → InnoDB
  │     │
  │     ├── OLAP（分析查询）
  │     │     ├── 实时性要求高？
  │     │     │     ├── 是 → ClickHouse / Apache Doris
  │     │     │     └── 否 → Snowflake / BigQuery
  │     │     └── 数据规模？
  │     │           ├── PB级 → Presto/Trino + 数据湖
  │     │           └── TB级 → ClickHouse / StarRocks
  │     │
  │     ├── HTAP（混合负载）
  │     │     └── TiDB / CockroachDB / SingleStore
  │     │
  │     └── 时序数据
  │           ├── 高频写入 → TDengine / TimescaleDB
  │           └── 复杂查询 → InfluxDB IOx / ClickHouse
  │
  └── 是否需要分布式？
        ├── 是 → 继续分布式特性评估
        └── 否 → 单机引擎选型
```

### 5.2 数据一致性策略

| 场景 | 一致性模型 | 实现方式 |
|------|------------|----------|
| 缓存-DB | 最终一致 | Cache Aside + 过期时间 + 延迟双删 |
| 主从复制 | 最终一致 | 异步复制 + 读写分离 |
| 分布式事务 | 强一致 | 2PC / Saga / TCC |
| 跨机房同步 | 最终一致 | 异步复制 + 冲突检测 |

### 5.3 常见问题

**Q1: 如何在B-Tree和LSM-Tree之间选择？**
A: 关键看读写比例和延迟要求：

- 读写比 > 10:1，读延迟敏感 → B-Tree
- 写入量大，可接受读延迟波动 → LSM-Tree
- 混合负载 → 考虑RocksDB的调优或混合架构

**Q2: 为什么Redis不适合做主存储？**
A: 1) 内存成本高；2) 持久化有窗口期（AOF重写/RDB快照）；3) 单线程处理能力有上限；4) 复杂查询能力弱。建议仅作缓存/计数/会话。

**Q3: ClickHouse为什么不适合OLTP？**
A: 1) 不支持事务；2) 单条写入性能差；3) 更新删除是异步的；4) 行级锁不存在。仅适合批量写入的分析场景。

**Q4: 如何评估存储引擎的扩展性？**
A: 关注：1) 是否原生支持分片；2) 扩缩容是否需要停机；3) 数据重平衡效率；4) 跨分片事务支持度。
