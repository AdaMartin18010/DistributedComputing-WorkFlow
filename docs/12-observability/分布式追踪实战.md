# 分布式追踪实战

## 概述

分布式追踪是解决微服务架构下请求链路可视化的关键技术。通过上下文传播机制，追踪系统能够还原请求在多个服务间的完整调用路径，帮助快速定位性能瓶颈和故障根因。本文深入讲解分布式追踪的核心原理和实战技巧。

## 上下文传播机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    分布式上下文传播流程                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  客户端                                  服务端                          │
│    │                                      │                             │
│    │  1. 创建根Span                       │                             │
│    │     • TraceID: abc123               │                             │
│    │     • SpanID: span001               │                             │
│    │     • Flags: 1 (采样)               │                             │
│    │                                      │                             │
│    │  2. 注入传播上下文                    │                             │
│    │     • HTTP Headers                  │                             │
│    │     • gRPC Metadata                 │                             │
│    │     • Message Properties            │                             │
│    │                                      │                             │
│    │ ────────────────────────────────────►│                             │
│    │  traceparent: 00-abc123-span001-01  │                             │
│    │  tracestate: rojo=00f067            │                             │
│    │  baggage: userId=12345              │                             │
│    │                                      │                             │
│    │                                      │  3. 提取传播上下文          │
│    │                                      │     • 解析traceparent       │
│    │                                      │     • 恢复TraceID           │
│    │                                      │                             │
│    │                                      │  4. 创建子Span              │
│    │                                      │     • ParentID: span001     │
│    │                                      │     • SpanID: span002       │
│    │                                      │                             │
│    │ ◄────────────────────────────────────│                             │
│    │  5. 响应携带上下文                    │                             │
│    │                                      │                             │
└─────────────────────────────────────────────────────────────────────────┘
```

## 实战场景配置

### 场景1：HTTP服务间传播

```yaml
# http-propagation.yaml
# Spring Boot HTTP传播配置
spring:
  sleuth:
    propagation:
      type: W3C  # W3C Trace Context
    
    baggage:
      remote-fields:
        - user-id
        - request-id
        - tenant-id
      local-fields:
        - session-id
        
  web:
    client:
      # HTTP Client自动传播
      auto-config: true
      
# 手动传播示例
tracing:
  http:
    # 注入上下文到请求头
    inject:
      type: outgoing
      headers:
        - traceparent
        - baggage
      
    # 从请求头提取上下文
    extract:
      type: incoming
      headers:
        - traceparent
        - baggage
```

```java
// 手动HTTP传播示例
@RestController
public class OrderController {
    
    @Autowired
    private Tracer tracer;
    
    @GetMapping("/orders/{id}")
    public Order getOrder(@PathVariable String id, 
                          @RequestHeader HttpHeaders headers) {
        // 从传入请求提取上下文
        Context extractedContext = extractContext(headers);
        
        // 创建子Span
        Span span = tracer.spanBuilder("getOrder")
            .setParent(extractedContext)
            .setSpanKind(SpanKind.SERVER)
            .startSpan();
        
        try (Scope scope = span.makeCurrent()) {
            // 调用下游服务
            HttpHeaders outgoingHeaders = new HttpHeaders();
            injectContext(Context.current(), outgoingHeaders);
            
            ResponseEntity<Payment> response = restTemplate.exchange(
                "http://payment-service/payments/" + id,
                HttpMethod.GET,
                new HttpEntity<>(outgoingHeaders),
                Payment.class
            );
            
            return new Order(id, response.getBody());
        } finally {
            span.end();
        }
    }
    
    private void injectContext(Context context, HttpHeaders headers) {
        W3CTraceContextPropagator propagator = W3CTraceContextPropagator.getInstance();
        propagator.inject(context, headers, HttpHeaders::set);
    }
    
    private Context extractContext(HttpHeaders headers) {
        W3CTraceContextPropagator propagator = W3CTraceContextPropagator.getInstance();
        return propagator.extract(Context.current(), headers, HttpHeaders::get);
    }
}
```

```java
// gRPC客户端拦截器
public class TracingClientInterceptor implements ClientInterceptor {
    
    private final TextMapPropagator propagator;
    
    @Override
    public <ReqT, RespT> ClientCall<ReqT, RespT> interceptCall(
            MethodDescriptor<ReqT, RespT> method,
            CallOptions callOptions,
            Channel next) {
        
        return new ForwardingClientCall.SimpleForwardingClientCall<>(
            next.newCall(method, callOptions)) {
            
            @Override
            public void start(Listener<RespT> responseListener, Metadata headers) {
                // 注入追踪上下文到Metadata
                Context current = Context.current();
                propagator.inject(current, headers, Metadata::put);
                
                super.start(responseListener, headers);
            }
        };
    }
}

// gRPC服务端拦截器
public class TracingServerInterceptor implements ServerInterceptor {
    
    @Override
    public <ReqT, RespT> ServerCall.Listener<ReqT> interceptCall(
            ServerCall<ReqT, RespT> call,
            Metadata headers,
            ServerCallHandler<ReqT, RespT> next) {
        
        // 从Metadata提取上下文
        Context extractedContext = propagator.extract(
            Context.current(), headers, Metadata::get
        );
        
        // 创建Span
        Span span = tracer.spanBuilder(call.getMethodDescriptor().getFullMethodName())
            .setParent(extractedContext)
            .setSpanKind(SpanKind.SERVER)
            .startSpan();
        
        try (Scope scope = span.makeCurrent()) {
            return Contexts.interceptCall(extractedContext, call, headers, next);
        } finally {
            span.end();
        }
    }
}
```

### 场景3：消息队列传播

```yaml
# messaging-propagation.yaml
messaging:
  kafka:
    tracing:
      # 生产者配置
      producer:
        enabled: true
        headers:
          - traceparent
          - baggage
          - kafka.traceId
          
      # 消费者配置
      consumer:
        enabled: true
        headers:
          - traceparent
          - baggage
        mode: extract  # extract | extract_and_continue
```

```java
// Kafka消息传播
@Component
public class TracingKafkaProducer {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    @Autowired
    private TextMapPropagator propagator;
    
    public void send(String topic, String message) {
        // 获取当前Span
        Span currentSpan = Span.current();
        
        // 创建消息头
        Headers headers = new RecordHeaders();
        
        // 注入追踪上下文到消息头
        propagator.inject(Context.current(), headers, 
            (carrier, key, value) -> carrier.add(key, value.getBytes()));
        
        // 发送消息
        ProducerRecord<String, String> record = new ProducerRecord<>(
            topic, null, null, message, headers
        );
        
        kafkaTemplate.send(record);
    }
}

@Component
public class TracingKafkaConsumer {
    
    @KafkaListener(topics = "orders")
    public void consume(ConsumerRecord<String, String> record) {
        // 从消息头提取上下文
        Context extractedContext = propagator.extract(
            Context.current(),
            record.headers(),
            (carrier, key) -> {
                Header header = carrier.lastHeader(key);
                return header != null ? new String(header.value()) : null;
            }
        );
        
        // 创建消费者Span
        Span span = tracer.spanBuilder("process-order")
            .setParent(extractedContext)
            .setSpanKind(SpanKind.CONSUMER)
            .setAttribute("messaging.system", "kafka")
            .setAttribute("messaging.destination", "orders")
            .startSpan();
        
        try (Scope scope = span.makeCurrent()) {
            processMessage(record.value());
        } finally {
            span.end();
        }
    }
}
```

### 场景4：异步/线程传播

```yaml
# async-propagation.yaml
async:
  executor:
    tracing:
      enabled: true
      mode: propagate  # propagate | wrap
      
  reactor:
    tracing:
      enabled: true
      context-propagation: true
```

```java
// 线程池上下文传播
@Configuration
public class TracingExecutorConfig {
    
    @Bean
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        
        // 包装为传播上下文
        executor.setTaskDecorator(new ContextPropagatingTaskDecorator());
        
        return executor;
    }
}

// 任务装饰器
public class ContextPropagatingTaskDecorator implements TaskDecorator {
    
    @Override
    public Runnable decorate(Runnable runnable) {
        // 捕获当前上下文
        Context context = Context.current();
        
        return () -> {
            // 在新线程中恢复上下文
            try (Scope scope = context.makeCurrent()) {
                runnable.run();
            }
        };
    }
}

// CompletableFuture传播
public void asyncOperation() {
    Context context = Context.current();
    
    CompletableFuture.supplyAsync(() -> {
        try (Scope scope = context.makeCurrent()) {
            // 上下文已传播到异步线程
            return doSomething();
        }
    }, executor);
}
```

### 场景5：数据库追踪

```yaml
# database-tracing.yaml
database:
  tracing:
    enabled: true
    capture:
      - query
      - parameters
      - rows_affected
      - duration
    sanitization:
      enabled: true  # 脱敏敏感参数
```

```java
// JDBC追踪
@Component
public class TracingDataSource extends DelegatingDataSource {
    
    @Autowired
    private Tracer tracer;
    
    @Override
    public Connection getConnection() throws SQLException {
        Connection connection = super.getConnection();
        return new TracingConnection(connection, tracer);
    }
}

// 追踪连接
public class TracingConnection implements Connection {
    
    private final Connection delegate;
    private final Tracer tracer;
    
    @Override
    public PreparedStatement prepareStatement(String sql) throws SQLException {
        return new TracingPreparedStatement(
            delegate.prepareStatement(sql), 
            tracer, 
            sql
        );
    }
}

// 追踪Statement
public class TracingPreparedStatement implements PreparedStatement {
    
    private final String sql;
    private final Tracer tracer;
    
    @Override
    public ResultSet executeQuery() throws SQLException {
        Span span = tracer.spanBuilder("SQL Query")
            .setAttribute("db.system", "mysql")
            .setAttribute("db.statement", sql)
            .setSpanKind(SpanKind.CLIENT)
            .startSpan();
        
        try (Scope scope = span.makeCurrent()) {
            long start = System.nanoTime();
            ResultSet rs = delegate.executeQuery();
            long duration = System.nanoTime() - start;
            
            span.setAttribute("db.duration_ms", duration / 1_000_000.0);
            return rs;
        } catch (Exception e) {
            span.recordException(e);
            span.setStatus(StatusCode.ERROR, e.getMessage());
            throw e;
        } finally {
            span.end();
        }
    }
}
```

## 常见问题排查

```yaml
# troubleshooting.yaml
troubleshooting:
  # 问题1: 链路断裂
  broken_trace:
    symptoms:
      - 追踪在特定服务处断开
      - 部分Span缺失
    causes:
      - 传播头丢失
      - 采样不一致
      - 时钟偏移
    solutions:
      - 检查所有传播点
      - 统一采样策略
      - 使用NTP同步时间
      
  # 问题2: 上下文丢失
  lost_context:
    symptoms:
      - 异步操作无关联
      - 线程池上下文丢失
    causes:
      - 未正确传播上下文
      - 使用原生线程池
    solutions:
      - 使用ContextPropagatingTaskDecorator
      - 包装Callable/Runnable
      
  # 问题3: 循环调用
  circular_call:
    symptoms:
      - 追踪图出现循环
      - Span深度异常
    causes:
      - 服务间循环依赖
      - 缓存击穿
    solutions:
      - 检测循环调用
      - 设置最大深度限制
```

## 最佳实践

1. **统一标准**：全链路使用W3C Trace Context
2. **采样一致**：确保上下游采样策略兼容
3. **传播检查**：定期检查传播头完整性
4. **异步处理**：确保异步操作正确传播上下文
5. **日志关联**：日志中输出trace_id便于关联
