## 一、核心概念

### 1.1 定义与原理

**云原生（Cloud Native）**是一种构建和运行应用程序的方法，它充分利用云计算的弹性、分布式和自动化特性。云原生技术使组织能够在现代动态环境（如公有云、私有云和混合云）中构建和运行可扩展的应用。

**核心原理**：

### 1.2 关键特性

- **容器化封装**：应用及其依赖打包为标准化容器镜像
- **动态编排**：自动化的部署、扩展和管理容器化应用
- **微服务架构**：将单体应用拆分为独立部署的服务
- **声明式API**：通过声明式配置管理基础设施
- **可观测性**：全面的日志、监控和追踪能力
- **DevOps文化**：开发与运维的深度融合

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|

| 弹性伸缩需求 | ⭐⭐⭐⭐⭐ | 自动扩缩容应对流量波动 |
| 多环境一致性 | ⭐⭐⭐⭐⭐ | 开发/测试/生产环境一致 |
| 遗留系统现代化 | ⭐⭐⭐⭐ | 逐步迁移至云原生架构 |

### 3.3 CRI-O详解

**设计目标**：

- 专为Kubernetes设计的最小化容器运行时
- 仅实现CRI接口，无额外功能
- 资源占用少，启动速度快

│    ▼                    ▼                    ▼                 │

### 3.4 运行时对比

| 特性 | containerd | CRI-O | Docker |
|------|------------|-------|--------|
| **定位** | 通用容器运行时 | K8s专用运行时 | 完整容器平台 |
| **CRI支持** | 是（通过cri插件） | 原生支持 | 需dockershim（已废弃） |
| **镜像支持** | OCI/Docker | OCI/Docker | Docker/OCI |
| **资源占用** | 中 | 低 | 高 |
| **启动速度** | 快 | 最快 | 中等 |
| **额外功能** | 有（ctr工具） | 最小化 | 丰富（compose等） |
| **K8s默认** | v1.20+默认 | 可选 | 已弃用 |
| **适用场景** | 通用生产环境 | K8s专用环境 | 开发/遗留系统 |

### 4.2 主流CNI插件对比

### 7.3 ArgoCD vs Flux对比

| 特性 | ArgoCD | Flux |
|------|--------|------|
| **架构** | 单体应用 | 模块化组件 |
| **UI** | 内置Web UI | 需配合Weave GitOps |
| **多源支持** | Git、Helm、Kustomize | Git、OCI、Helm、Kustomize |
| **镜像自动更新** | 需Argo Image Updater | 原生支持（ImagePolicy） |
| **通知功能** | 内置 | 需单独配置 |
| **多租户** | 项目隔离 | Namespace隔离 |
| **适用场景** | 企业级、需要UI | GitOps原生、与Flux生态整合 |

### 7.4 GitOps配置示例

```yaml
# ArgoCD Application 配置
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/gitops-repo.git
    targetRevision: HEAD
    path: overlays/production
    kustomize:
      namePrefix: prod-
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true        # 自动删除Git中不存在的资源
      selfHeal: true     # 自动修复漂移
      allowEmpty: false
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas  # 忽略副本数差异（支持HPA）
---
# Flux GitRepository + Kustomization
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: myapp
  namespace: flux-system
spec:
  interval: 1m
  url: https://github.com/org/gitops-repo.git
  ref:
    branch: main
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: myapp
  namespace: flux-system
spec:
  interval: 10m
  targetNamespace: production
  sourceRef:
    kind: GitRepository
    name: myapp
  path: ./overlays/production
  prune: true
  wait: true
  timeout: 5m
```

### 8.3 常见问题

**Q1: containerd和CRI-O如何选择？**
A: 对于标准K8s环境，两者都可行。containerd生态更成熟、工具链更丰富；CRI-O更轻量、专为K8s设计。K8s 1.20+默认使用containerd。

**Q2: Calico和Cilium哪个更好？**
A: Calico成熟稳定，BGP模式性能优异；Cilium基于eBPF，功能更先进（L7感知、可观测性更强）。新项目推荐Cilium，已有Calico可继续维护。

**Q3: 是否需要服务网格？**
A: 如果已有成熟的微服务治理方案（如Spring Cloud），可暂缓引入。新建项目或需要统一流量管理、安全通信时，建议引入服务网格。

**Q4: GitOps是否适用于所有场景？**
A: GitOps最适合声明式基础设施和K8s应用。对于需要频繁交互式变更或有状态服务（如数据库主从切换），可能需要配合其他工具。
