# 数据库查询优化：索引与执行计划

## 核心概念

### 1. SQL执行流程

理解SQL执行流程是优化的基础，只有知道查询如何执行，才能找到优化点。

```
┌─────────────────────────────────────────────────────────────┐
│                    SQL执行流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   SQL语句                                                   │
│      ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  1. 解析器 (Parser)                                  │   │
│  │     - 词法分析、语法分析                             │   │
│  │     - 生成解析树                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│      ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  2. 预处理器 (Preprocessor)                          │   │
│  │     - 语义检查                                       │   │
│  │     - 权限验证                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│      ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  3. 优化器 (Optimizer)                               │   │
│  │     - 生成执行计划                                   │   │
│  │     - 成本估算                                       │   │
│  │     - 选择最优计划                                   │   │
│  └─────────────────────────────────────────────────────┘   │
│      ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  4. 执行引擎 (Execution Engine)                      │   │
│  │     - 按执行计划操作数据                             │   │
│  │     - 访问存储引擎                                   │   │
│  └─────────────────────────────────────────────────────┘   │
│      ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  5. 存储引擎 (Storage Engine)                        │   │
│  │     - InnoDB: 行锁、事务、MVCC                       │   │
│  │     - MyISAM: 表锁、全文索引                         │   │
│  └─────────────────────────────────────────────────────┘   │
│      ↓                                                      │
│   返回结果集                                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 索引原理

索引是数据库优化的核心手段，理解索引结构有助于正确使用。

**B+树索引结构**：
```
┌─────────────────────────────────────────────────────────────┐
│                   B+树索引结构                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    [30]                                     │
│                   /    \                                    │
│              [10,20]    [40,50]                            │
│             /   |   \   /    |    \                         │
│     [1,3,5] [12,15] [25] [35,38] [42,45] [55,60]           │
│       |       |       |      |       |       |              │
│      数据页   数据页   数据页  数据页   数据页   数据页       │
│                                                             │
│  特点：                                                      │
│  1. 非叶子节点只存储键值，不存储数据                        │
│  2. 叶子节点包含所有键值和对应数据指针                      │
│  3. 叶子节点按顺序链接，支持范围查询                        │
│  4. 所有叶子节点在同一层，保证O(logN)查询复杂度             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**索引类型对比**：

| 索引类型 | 结构 | 适用场景 | 特点 |
|----------|------|----------|------|
| B+树索引 | B+树 | 等值、范围查询 | 最常用，支持排序 |
| 哈希索引 | 哈希表 | 精确匹配 | O(1)查询，不支持范围 |
| 全文索引 | 倒排表 | 文本搜索 | 支持分词搜索 |
| 空间索引 | R树 | 地理位置 | 支持空间范围查询 |

### 3. 执行计划分析

执行计划展示了优化器选择的查询执行方式。

**关键指标**：
```
┌─────────────────────────────────────────────────────────────┐
│                  执行计划关键字段                           │
├──────────────┬──────────────────────────────────────────────┤
│    id        │  执行顺序标识                                │
│    select_type│ 查询类型（SIMPLE/PRIMARY/SUBQUERY等）       │
│    table     │  访问的表                                    │
│    type      │  访问类型（性能由好到差）：                  │
│              │  system > const > eq_ref > ref > range >    │
│              │  index > ALL                                │
│    possible_keys│ 可能使用的索引                              │
│    key       │  实际使用的索引                              │
│    rows      │  预估扫描行数                                │
│    filtered  │  过滤百分比                                  │
│    Extra     │  额外信息：                                  │
│              │  Using index（覆盖索引）                    │
│              │  Using where（回表过滤）                    │
│              │  Using filesort（文件排序）                 │
│              │  Using temporary（临时表）                  │
└──────────────┴──────────────────────────────────────────────┘
```

## 实践方法

### 索引设计最佳实践

**1. 索引创建原则**：
```sql
-- 原则1：选择性高的列建索引
-- 选择性 = 不同值数量 / 总行数
-- 接近1的选择性最优
SELECT 
    COUNT(DISTINCT status) / COUNT(*) as selectivity
FROM orders;
-- status只有几个值，选择性低，不适合单独建索引

-- 原则2：最左前缀原则
-- 联合索引 (a, b, c) 可以用于：
-- - WHERE a = ?
-- - WHERE a = ? AND b = ?
-- - WHERE a = ? AND b = ? AND c = ?
-- 不能用于：
-- - WHERE b = ?  (跳过a)
-- - WHERE b = ? AND c = ?  (跳过a)

-- 正确创建联合索引
CREATE INDEX idx_user_time ON orders(user_id, created_at);

-- 有效查询
SELECT * FROM orders WHERE user_id = 100;
SELECT * FROM orders WHERE user_id = 100 AND created_at > '2024-01-01';

-- 无效查询（无法使用索引）
SELECT * FROM orders WHERE created_at > '2024-01-01';
```

**2. 覆盖索引设计**：
```sql
-- 场景：频繁查询用户的订单数量和总金额
-- 错误设计：只建user_id索引
CREATE INDEX idx_user ON orders(user_id);

-- 查询需要回表获取amount
SELECT user_id, COUNT(*), SUM(amount) 
FROM orders 
WHERE user_id = 100;

-- 优化：覆盖索引
CREATE INDEX idx_user_amount ON orders(user_id, amount);

-- 查询变为覆盖索引扫描
EXPLAIN SELECT user_id, COUNT(*), SUM(amount) 
FROM orders 
WHERE user_id = 100;
-- Extra: Using index
```

**3. 索引维护**：
```sql
-- 查看索引使用情况
SELECT 
    table_schema,
    table_name,
    index_name,
    rows_selected,
    rows_inserted,
    rows_deleted
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL;

-- 查找冗余索引
SELECT 
    t.TABLE_SCHEMA,
    t.TABLE_NAME,
    s.INDEX_NAME,
    s.COLUMN_NAME
FROM information_schema.STATISTICS s
JOIN information_schema.TABLES t ON s.TABLE_SCHEMA = t.TABLE_SCHEMA 
    AND s.TABLE_NAME = t.TABLE_NAME
WHERE t.TABLE_SCHEMA = 'mydb'
ORDER BY t.TABLE_SCHEMA, t.TABLE_NAME, s.INDEX_NAME, s.SEQ_IN_INDEX;

-- 删除未使用索引
DROP INDEX idx_unused ON orders;
```

### SQL优化技巧

**1. 避免索引失效**：
```sql
-- 场景1：隐式类型转换
-- 错误：phone是字符串，用数字比较
SELECT * FROM users WHERE phone = 13800138000;  -- 索引失效
-- 正确：
SELECT * FROM users WHERE phone = '13800138000';

-- 场景2：函数操作列
-- 错误：
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-01';
-- 正确：范围查询
SELECT * FROM orders 
WHERE created_at >= '2024-01-01 00:00:00' 
  AND created_at < '2024-01-02 00:00:00';

-- 场景3：前导模糊匹配
-- 错误：
SELECT * FROM users WHERE name LIKE '%张三%';
-- 优化：使用全文索引或倒排表
SELECT * FROM users WHERE MATCH(name) AGAINST('张三');

-- 场景4：OR条件
-- 错误：
SELECT * FROM orders WHERE user_id = 100 OR status = 1;
-- 优化：使用UNION
SELECT * FROM orders WHERE user_id = 100
UNION ALL
SELECT * FROM orders WHERE status = 1 AND user_id != 100;
```

**2. 分页优化**：
```sql
-- 深度分页问题
-- 传统分页，越往后越慢
SELECT * FROM orders 
ORDER BY created_at DESC 
LIMIT 1000000, 10;  -- 扫描1000010行，返回10行

-- 优化1：延迟关联
SELECT o.* 
FROM orders o
JOIN (
    SELECT id 
    FROM orders 
    ORDER BY created_at DESC 
    LIMIT 1000000, 10
) tmp ON o.id = tmp.id;

-- 优化2：记录上次位置
SELECT * FROM orders 
WHERE created_at < '2024-01-01 12:00:00'  -- 上次最后一条的时间
ORDER BY created_at DESC 
LIMIT 10;

-- 优化3：覆盖索引分页
SELECT id, created_at 
FROM orders 
ORDER BY created_at DESC 
LIMIT 1000000, 10;
```

**3. JOIN优化**：
```sql
-- 优化原则：小表驱动大表

-- 错误：大表驱动
SELECT * FROM big_table b 
JOIN small_table s ON b.id = s.big_id;

-- 正确：小表驱动（连接顺序由优化器决定，但应保证小表有过滤条件）
SELECT * FROM small_table s 
JOIN big_table b ON b.id = s.big_id
WHERE s.status = 1;  -- 小表先过滤

-- 确保JOIN字段有索引
CREATE INDEX idx_big_id ON big_table(id);
CREATE INDEX idx_small_big_id ON small_table(big_id);

-- 避免JOIN过多表（超过3个考虑数据冗余或应用层组装）
```

**4. 子查询优化**：
```sql
-- 错误：相关子查询
SELECT * FROM users u
WHERE age > (SELECT AVG(age) FROM users WHERE city = u.city);

-- 优化：JOIN改写
SELECT u.* 
FROM users u
JOIN (
    SELECT city, AVG(age) as avg_age 
    FROM users 
    GROUP BY city
) avg_table ON u.city = avg_table.city
WHERE u.age > avg_table.avg_age;

-- 错误：IN子查询
SELECT * FROM orders WHERE user_id IN (
    SELECT id FROM users WHERE status = 0
);

-- 优化：EXISTS或JOIN
SELECT o.* FROM orders o
WHERE EXISTS (
    SELECT 1 FROM users u 
    WHERE u.id = o.user_id AND u.status = 0
);
-- 或
SELECT o.* FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.status = 0;
```

### 执行计划分析实践

**1. EXPLAIN分析流程**：
```sql
-- 步骤1：获取执行计划
EXPLAIN ANALYZE SELECT * FROM orders 
WHERE user_id = 100 AND created_at > '2024-01-01';

-- 步骤2：检查type列
-- 期望：至少range，最好是ref或eq_ref
-- 避免：ALL（全表扫描）

-- 步骤3：检查key列
-- 确认使用了期望的索引

-- 步骤4：检查rows列
-- 预估扫描行数，评估性能

-- 步骤5：检查Extra列
-- 避免：Using filesort, Using temporary
-- 期望：Using index（覆盖索引）
```

**2. 执行计划解读示例**：
```sql
-- 示例查询
EXPLAIN SELECT 
    u.name,
    COUNT(o.id) as order_count,
    SUM(o.amount) as total_amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.status = 1
  AND o.created_at > '2024-01-01'
GROUP BY u.id;

-- 可能的执行计划
+----+-------------+-------+--------+---------------+---------+---------+------------------+-------+---------------------------------+
| id | select_type | table | type   | possible_keys | key     | key_len | ref              | rows  | Extra                           |
+----+-------------+-------+--------+---------------+---------+---------+------------------+-------+---------------------------------+
| 1  | SIMPLE      | u     | ref    | PRIMARY,idx_status | idx_status | 4     | const            | 10000 | Using where; Using temporary    |
| 1  | SIMPLE      | o     | ref    | idx_user_time | idx_user_time | 8       | db.u.id          | 5     | Using where                     |
+----+-------------+-------+--------+---------------+---------+---------+------------------+-------+---------------------------------+

-- 问题分析：
-- 1. Using temporary：使用了临时表
-- 2. users表使用status索引而非主键
-- 3. 可能缺少合适的联合索引

-- 优化方案：
CREATE INDEX idx_user_order ON orders(user_id, created_at, amount);
-- 或考虑反范式设计，在users表冗余订单统计
```

## 案例数据

### 电商订单查询优化

**优化前**：
```sql
-- 查询用户近一年订单
SELECT * FROM orders 
WHERE user_id = 100 
  AND created_at > DATE_SUB(NOW(), INTERVAL 1 YEAR)
ORDER BY created_at DESC
LIMIT 20;

-- 执行计划
-- type: ALL (全表扫描)
-- rows: 1,000,000
-- Extra: Using where; Using filesort
-- 执行时间: 2.5秒
```

**优化过程**：
```sql
-- 1. 创建联合索引
CREATE INDEX idx_user_created ON orders(user_id, created_at);

-- 2. 优化后执行计划
-- type: range
-- key: idx_user_created
-- rows: 100
-- Extra: Using where; Using index condition
-- 执行时间: 5ms

-- 3. 进一步优化：覆盖索引
CREATE INDEX idx_user_created_cover ON orders(user_id, created_at, id, status, amount);

-- 执行计划
-- Extra: Using index (覆盖索引，无需回表)
-- 执行时间: 2ms
```

**优化效果**：
| 指标 | 优化前 | 优化后 | 改进 |
|------|--------|--------|------|
| 执行时间 | 2500ms | 2ms | 99.9%↓ |
| 扫描行数 | 100万 | 20 | 99.9%↓ |
| 查询类型 | ALL | range | 显著提升 |
| CPU使用 | 高 | 极低 | 大幅下降 |

### 报表查询优化

**场景：销售统计报表**

**优化前SQL**：
```sql
SELECT 
    DATE(o.created_at) as sale_date,
    p.category_id,
    COUNT(DISTINCT o.id) as order_count,
    SUM(oi.quantity) as total_quantity,
    SUM(oi.quantity * oi.price) as total_amount
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE o.created_at >= '2024-01-01'
  AND o.status = 1
GROUP BY DATE(o.created_at), p.category_id;
-- 执行时间: 120秒
```

**优化方案**：
```sql
-- 1. 创建物化视图（或定时统计表）
CREATE TABLE daily_sales_stats (
    stat_date DATE,
    category_id INT,
    order_count INT,
    total_quantity INT,
    total_amount DECIMAL(15,2),
    PRIMARY KEY (stat_date, category_id),
    INDEX idx_date (stat_date)
);

-- 2. 定时任务更新统计
INSERT INTO daily_sales_stats
SELECT 
    DATE(o.created_at),
    p.category_id,
    COUNT(DISTINCT o.id),
    SUM(oi.quantity),
    SUM(oi.quantity * oi.price)
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE o.created_at >= CURDATE() - INTERVAL 1 DAY
  AND o.created_at < CURDATE()
  AND o.status = 1
GROUP BY DATE(o.created_at), p.category_id
ON DUPLICATE KEY UPDATE
    order_count = VALUES(order_count),
    total_quantity = VALUES(total_quantity),
    total_amount = VALUES(total_amount);

-- 3. 查询改为读取统计表
SELECT * FROM daily_sales_stats 
WHERE stat_date >= '2024-01-01'
ORDER BY stat_date, category_id;
-- 执行时间: 50ms
```

**总结**：数据库查询优化需要系统性的方法。通过理解执行计划、合理设计索引、改写低效SQL、利用统计信息，可以大幅提升查询性能。对于复杂报表，应结合物化视图或应用层聚合来降低数据库压力。
