# 多级缓存架构：L1/L2/L3/分布式

## 核心概念

### 1. 缓存层次模型

多级缓存架构通过在不同层级部署缓存，平衡访问速度与数据一致性。

**计算机体系结构中的缓存层次**：
```
┌─────────────────────────────────────────────────────────────┐
│                   计算机缓存层次                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   CPU核心                                                   │
│      ↓                                                      │
│   L1缓存 (32-64KB)                                         │
│      - 指令缓存 + 数据缓存                                   │
│      - 访问延迟：1-3个时钟周期                              │
│      ↓                                                      │
│   L2缓存 (256KB-512KB)                                     │
│      - 每个核心独立                                         │
│      - 访问延迟：10-20个时钟周期                            │
│      ↓                                                      │
│   L3缓存 (8-64MB)                                          │
│      - 多核心共享                                           │
│      - 访问延迟：40-60个时钟周期                            │
│      ↓                                                      │
│   主内存 (RAM)                                             │
│      - 访问延迟：100-300个时钟周期                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**分布式应用缓存层次**：
```
┌─────────────────────────────────────────────────────────────┐
│                   应用缓存层次                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   用户请求                                                  │
│      ↓                                                      │
│   L1: 本地内存缓存 (Caffeine/FreeCache)                    │
│      - 访问延迟：~1μs                                       │
│      - 容量：100MB-1GB                                      │
│      - 命中率：20-30%                                       │
│      ↓ 未命中                                               │
│   L2: 进程内共享缓存 (BigCache)                            │
│      - 访问延迟：~5μs                                       │
│      - 容量：1-10GB                                         │
│      - 命中率：30-50%                                       │
│      ↓ 未命中                                               │
│   L3: 分布式缓存 (Redis/Memcached)                         │
│      - 访问延迟：~1ms (网络RTT)                            │
│      - 容量：10GB-1TB                                       │
│      - 命中率：60-80%                                       │
│      ↓ 未命中                                               │
│   数据源 (DB/外部服务)                                     │
│      - 访问延迟：10-100ms                                   │
│                                                             │
│   组合命中率：1 - (1-0.25)×(1-0.4)×(1-0.7) = 86.5%        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 缓存穿透、击穿、雪崩

**缓存问题矩阵**：

| 问题 | 描述 | 触发条件 | 影响 | 解决方案 |
|------|------|----------|------|----------|
| 穿透 | 查询不存在数据，直达DB | 恶意攻击、数据确实不存在 | DB压力大 | 布隆过滤器、空值缓存 |
| 击穿 | 热点key过期，大量请求打DB | 高并发+热点key过期 | DB瞬时高负载 | 互斥锁、逻辑过期 |
| 雪崩 | 大量key同时过期 | 批量设置相同过期时间 | DB全面崩溃 | 随机过期、多级缓存 |

**缓存击穿防护**：
```
┌─────────────────────────────────────────────────────────────┐
│                   缓存击穿防护 - 互斥锁                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   请求1：获取key → 缓存未命中 → 获取锁 → 查询DB → 写入缓存  │
│                                                   ↓         │
│   请求2：获取key → 缓存未命中 → 等待锁 ───────────┘         │
│                                                             │
│   请求3：获取key → 缓存未命中 → 等待锁 ───────────┘         │
│                                                             │
│   实现要点：                                                │
│   1. 使用Redis SETNX实现分布式锁                            │
│   2. 锁超时防止死锁                                         │
│   3. 双重检查避免重复查询                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. 缓存一致性模型

**一致性级别**：
```
┌─────────────────────────────────────────────────────────────┐
│                   缓存一致性级别                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   强一致性 (Strong Consistency)                            │
│      - 读写都通过协调器                                     │
│      - 性能最差                                             │
│      - 适用：金融交易                                       │
│                                                             │
│   最终一致性 (Eventual Consistency)                        │
│      - 允许短暂不一致                                       │
│      - 性能好                                               │
│      - 适用：大多数互联网应用                               │
│                                                             │
│   弱一致性 (Weak Consistency)                              │
│      - 不保证一致性                                         │
│      - 性能最好                                             │
│      - 适用：非关键数据（计数、统计）                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 实践方法

### 多级缓存实现

**1. 本地缓存（L1）**：
```go
package cache

import (
    "time"
    "github.com/allegro/bigcache/v3"
    "github.com/patrickmn/go-cache"
)

// 本地缓存接口
type LocalCache interface {
    Get(key string) (interface{}, bool)
    Set(key string, value interface{}, ttl time.Duration)
    Delete(key string)
}

// 基于go-cache的实现（简单场景）
type GoCache struct {
    cache *cache.Cache
}

func NewGoCache(defaultExpiration, cleanupInterval time.Duration) *GoCache {
    return &GoCache{
        cache: cache.New(defaultExpiration, cleanupInterval),
    }
}

func (c *GoCache) Get(key string) (interface{}, bool) {
    return c.cache.Get(key)
}

func (c *GoCache) Set(key string, value interface{}, ttl time.Duration) {
    c.cache.Set(key, value, ttl)
}

func (c *GoCache) Delete(key string) {
    c.cache.Delete(key)
}

// 基于BigCache的实现（高性能、避免GC压力）
type BigCacheStore struct {
    cache *bigcache.BigCache
}

func NewBigCacheStore() (*BigCacheStore, error) {
    config := bigcache.Config{
        Shards:             256,               // 分片数
        LifeWindow:         10 * time.Minute,  // 默认过期时间
        CleanWindow:        5 * time.Minute,   // 清理间隔
        MaxEntriesInWindow: 1000 * 10 * 60,    // 预估条目数
        MaxEntrySize:       500,               // 最大条目大小
        StatsEnabled:       true,
    }
    
    cache, err := bigcache.New(config)
    if err != nil {
        return nil, err
    }
    
    return &BigCacheStore{cache: cache}, nil
}

func (c *BigCacheStore) Get(key string) ([]byte, error) {
    return c.cache.Get(key)
}

func (c *BigCacheStore) Set(key string, value []byte) error {
    return c.cache.Set(key, value)
}
```

**2. 分布式缓存（L2/L3）**：
```go
// Redis缓存实现
type RedisCache struct {
    client *redis.Client
}

func NewRedisCache(addr string) *RedisCache {
    client := redis.NewClient(&redis.Options{
        Addr:         addr,
        PoolSize:     100,
        MinIdleConns: 10,
        MaxRetries:   3,
    })
    
    return &RedisCache{client: client}
}

func (c *RedisCache) Get(ctx context.Context, key string) (string, error) {
    return c.client.Get(ctx, key).Result()
}

func (c *RedisCache) Set(ctx context.Context, key string, value interface{}, ttl time.Duration) error {
    return c.client.Set(ctx, key, value, ttl).Err()
}

// 批量操作（Pipeline优化）
func (c *RedisCache) MGet(ctx context.Context, keys []string) ([]interface{}, error) {
    return c.client.MGet(ctx, keys...).Result()
}

func (c *RedisCache) MSet(ctx context.Context, kv map[string]interface{}, ttl time.Duration) error {
    pipe := c.client.Pipeline()
    for k, v := range kv {
        pipe.Set(ctx, k, v, ttl)
    }
    _, err := pipe.Exec(ctx)
    return err
}
```

**3. 多级缓存管理器**：
```go
// MultiLevelCache 多级缓存
type MultiLevelCache struct {
    l1       LocalCache           // 本地缓存
    l2       *RedisCache          // Redis缓存
    hotKeys  *HotKeyDetector      // 热点key检测
    metrics  *CacheMetrics        // 监控
}

// Get 多级缓存读取
type CacheResult struct {
    Value      interface{}
    Level      string  // L1/L2/DB
    CacheHit   bool
}

func (m *MultiLevelCache) Get(ctx context.Context, key string) (*CacheResult, error) {
    // 1. 查询L1本地缓存
    if val, ok := m.l1.Get(key); ok {
        m.metrics.RecordHit("L1")
        return &CacheResult{Value: val, Level: "L1", CacheHit: true}, nil
    }
    
    // 2. 查询L2 Redis缓存
    val, err := m.l2.Get(ctx, key)
    if err == nil {
        // 回填L1
        m.l1.Set(key, val, m.calculateL1TTL(key))
        m.metrics.RecordHit("L2")
        return &CacheResult{Value: val, Level: "L2", CacheHit: true}, nil
    }
    
    if err != redis.Nil {
        return nil, err  // Redis错误
    }
    
    // 3. 查询数据库（带防击穿）
    val, err = m.loadFromDBWithProtection(ctx, key)
    if err != nil {
        return nil, err
    }
    
    // 回填各级缓存
    m.l2.Set(ctx, key, val, m.calculateL2TTL(key))
    m.l1.Set(key, val, m.calculateL1TTL(key))
    
    m.metrics.RecordMiss()
    return &CacheResult{Value: val, Level: "DB", CacheHit: false}, nil
}

// 防击穿保护
func (m *MultiLevelCache) loadFromDBWithProtection(ctx context.Context, key string) (interface{}, error) {
    lockKey := "lock:" + key
    
    // 尝试获取分布式锁
    locked, err := m.l2.client.SetNX(ctx, lockKey, "1", 10*time.Second).Result()
    if err != nil {
        return nil, err
    }
    
    if !locked {
        // 未获取到锁，等待后重试
        time.Sleep(100 * time.Millisecond)
        return m.l2.Get(ctx, key)
    }
    
    // 获取锁成功，确保释放
    defer m.l2.client.Del(ctx, lockKey)
    
    // 双重检查
    if val, err := m.l2.Get(ctx, key); err == nil {
        return val, nil
    }
    
    // 查询数据库
    return m.loadFromDB(key)
}
```

### 缓存更新策略

**1. Cache-Aside（旁路缓存）**：
```go
// 读流程
data, err := cache.Get(key)
if err != nil {
    data, err = db.Query(key)
    if err != nil {
        return err
    }
    cache.Set(key, data, ttl)
}
return data

// 写流程
err := db.Update(key, newData)
if err != nil {
    return err
}
cache.Delete(key)  // 删除而非更新，避免并发问题
```

**2. Write-Through（直写）**：
```go
// 同步写入缓存和DB
func (c *WriteThroughCache) Set(ctx context.Context, key string, value interface{}) error {
    // 先写DB
    if err := c.db.Set(key, value); err != nil {
        return err
    }
    
    // 再写缓存
    return c.cache.Set(ctx, key, value, ttl)
}
```

**3. Write-Behind（异步写）**：
```go
// 先写缓存，异步写DB
type WriteBehindCache struct {
    cache    Cache
    dbWriter chan WriteOp
}

type WriteOp struct {
    Key   string
    Value interface{}
}

func (c *WriteBehindCache) Set(key string, value interface{}) {
    // 立即写缓存
    c.cache.Set(key, value, ttl)
    
    // 异步写DB
    select {
    case c.dbWriter <- WriteOp{Key: key, Value: value}:
    default:
        // 队列满，记录日志或丢弃
    }
}

// 后台写入协程
func (c *WriteBehindCache) dbWriterLoop() {
    batch := make([]WriteOp, 0, 100)
    ticker := time.NewTicker(1 * time.Second)
    
    for {
        select {
        case op := <-c.dbWriter:
            batch = append(batch, op)
            if len(batch) >= 100 {
                c.flushBatch(batch)
                batch = batch[:0]
            }
            
        case <-ticker.C:
            if len(batch) > 0 {
                c.flushBatch(batch)
                batch = batch[:0]
            }
        }
    }
}
```

## 案例数据

### 电商商品详情缓存架构

**场景**：日均1亿PV的商品详情页

**架构设计**：
```
用户请求
    ↓
CDN (静态资源) ← 缓存命中率 95%
    ↓
Nginx Proxy Cache (URL缓存) ← 命中率 30%
    ↓
API Gateway
    ↓
L1: Caffeine (商品基础信息) ← 命中率 40%
L1: 本地Guava Cache (库存) ← 命中率 20%
    ↓
L2: Redis Cluster (商品数据) ← 命中率 70%
    - 主从复制，读写分离
    - 分片：16个master，每master 2 slave
    ↓
L3: Tair/Elasticache (热点数据) ← 命中率 80%
    ↓
MySQL + ES (搜索)
```

**缓存策略**：
```
商品基础信息：
  - L1 TTL: 5分钟
  - L2 TTL: 30分钟
  - 更新：Cache-Aside，更新后删除缓存

库存信息：
  - L1 TTL: 1分钟（极短，保证最终一致）
  - L2 TTL: 5分钟
  - 更新：写DB后MQ通知刷新缓存

价格信息：
  - L1 TTL: 30秒
  - L2 TTL: 2分钟
  - 更新：定时任务同步
```

**效果**：
```
- 平均RT: 15ms
- P99 RT: 50ms
- 数据库QPS: 1000/秒（降低95%）
- 缓存总命中率: 99.2%
- 降级能力: 单节点故障不影响服务
```

### 多级缓存优化对比

| 方案 | L1 | L2 | L3 | 平均RT | DB压力 |
|------|----|----|----|--------|--------|
| 无缓存 | - | - | - | 200ms | 100% |
| 仅Redis | - | ✓ | - | 5ms | 20% |
| L1+L2 | ✓ | ✓ | - | 0.5ms | 10% |
| L1+L2+L3 | ✓ | ✓ | ✓ | 0.2ms | 0.8% |

---

**总结**：多级缓存架构是应对高并发读场景的利器。通过合理设计各层缓存的容量、过期策略和一致性机制，可以在保证数据新鲜度的同时，将数据库压力降到最低，实现高性能、高可用的系统架构。
