# 连接池调优

## 核心概念

### 1. 连接池原理

数据库连接是昂贵的资源，连接池通过复用连接来降低开销。

**连接开销分析**：

```
┌─────────────────────────────────────────────────────────────┐
│                   数据库连接开销                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   不使用连接池：                                             │
│   请求1: 建立TCP连接 → 认证 → 执行SQL → 关闭连接           │
│   请求2: 建立TCP连接 → 认证 → 执行SQL → 关闭连接           │
│   请求3: 建立TCP连接 → 认证 → 执行SQL → 关闭连接           │
│   每次请求耗时: 2-5ms (连接) + SQL执行时间                  │
│                                                             │
│   使用连接池：                                               │
│   初始化: 建立N个连接 → 认证 → 放入池                        │
│                                                             │
│   请求1: 从池获取连接 ──→ 执行SQL ──→ 归还连接             │
│   请求2: 从池获取连接 ──→ 执行SQL ──→ 归还连接             │
│   请求3: 从池获取连接 ──→ 执行SQL ──→ 归还连接             │
│   每次请求耗时: <1ms (获取) + SQL执行时间                   │
│                                                             │
│   收益：                                                     │
│   - 避免频繁创建/销毁连接                                   │
│   - 控制并发连接数，保护数据库                              │
│   - 连接预热，减少延迟                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**连接池核心参数**：

| 参数 | 说明 | 设置建议 |
|------|------|----------|
| minIdle | 最小空闲连接 | 保证基本连接数 |
| maxPoolSize | 最大连接数 | 根据DB容量设置 |
| connectionTimeout | 获取连接超时 | 避免无限等待 |
| idleTimeout | 空闲连接超时 | 回收闲置连接 |
| maxLifetime | 连接最大寿命 | 定期刷新连接 |
| validationTimeout | 连接验证超时 | 检测死连接 |

### 2. 连接池选型

| 连接池 | 特点 | 适用场景 |
|--------|------|----------|
| HikariCP | 性能最优，轻量 | Spring Boot默认，推荐 |
| Druid | 功能丰富，监控强 | 需要详细监控 |
| c3p0 | 老牌，稳定 | 传统项目 |
| DBCP | Apache出品 | 简单场景 |

## 实践方法

### HikariCP配置

**1. 推荐配置**：

```yaml
spring:
  datasource:
    hikari:
      # 连接池名称
      pool-name: MyHikariPool

      # 核心参数
      minimum-idle: 10           # 最小空闲连接
      maximum-pool-size: 50      # 最大连接数
      connection-timeout: 30000  # 获取连接超时（毫秒）
      idle-timeout: 600000       # 空闲连接超时（毫秒）
      max-lifetime: 1800000      # 连接最大寿命（毫秒）

      # 验证参数
      connection-test-query: SELECT 1
      validation-timeout: 5000   # 验证超时

      # 性能参数
      data-source-properties:
        cachePrepStmts: true
        prepStmtCacheSize: 250
        prepStmtCacheSqlLimit: 2048
        useServerPrepStmts: true
        useLocalSessionState: true
        rewriteBatchedStatements: true
        cacheResultSetMetadata: true
        cacheServerConfiguration: true
        elideSetAutoCommits: true
        maintainTimeStats: false
```

**2. 动态调整**：

```java
@Component
public class ConnectionPoolMonitor {

    @Autowired
    private HikariDataSource dataSource;

    @Scheduled(fixedRate = 60000)
    public void monitor() {
        HikariPoolMXBean poolMXBean = dataSource.getHikariPoolMXBean();

        int active = poolMXBean.getActiveConnections();
        int idle = poolMXBean.getIdleConnections();
        int total = poolMXBean.getTotalConnections();
        int waiting = poolMXBean.getThreadsAwaitingConnection();

        // 告警：等待线程数过多
        if (waiting > 10) {
            alert.send("连接池等待线程过多: " + waiting);
        }

        // 告警：连接池使用率过高
        double usageRate = (double) active / total;
        if (usageRate > 0.8) {
            alert.send("连接池使用率过高: " + usageRate * 100 + "%");
        }
    }
}
```

### 连接数计算

**计算公式**：

```
连接数 = ((核心数 × 2) + 有效磁盘数) × 实例数

示例：
- 服务器：8核
- 磁盘：SSD × 1
- 应用实例数：4

连接数 = ((8 × 2) + 1) × 4 = 68

调优建议：
- 初始设置：70
- 监控实际使用率
- 根据DB承载能力调整
```

**实际配置示例**：

```java
// 根据数据库类型调整
@Configuration
public class DataSourceConfig {

    @Bean
    public HikariDataSource dataSource() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");
        config.setUsername("user");
        config.setPassword("pass");

        // 根据公式计算
        int cores = Runtime.getRuntime().availableProcessors();
        int poolSize = (cores * 2) + 1;

        config.setMaximumPoolSize(poolSize);
        config.setMinimumIdle(poolSize / 2);

        // 其他配置...

        return new HikariDataSource(config);
    }
}
```

## 案例数据

**场景：微服务连接池优化**

**优化前**：

```
配置：每个服务maxPoolSize=100
服务数：20个
总连接数：2000

问题：
- 数据库最大连接数：1000
- 连接耗尽，大量等待
- 响应时间飙升
```

**优化后**：

```
配置：每个服务maxPoolSize=20
服务数：20个
总连接数：400

效果：
- 连接充足
- 响应时间稳定
- 数据库压力可控
```

## 优化前后对比

| 指标 | 优化前 | 优化后 | 改进 |
|------|--------|--------|------|
| 总连接数 | 2000 | 400 | 80%↓ |
| 连接等待 | 频繁 | 无 | 消除 |
| 响应时间P99 | 5s | 200ms | 96%↓ |
| 连接使用率 | 100% | 40% | 健康 |

---

**总结**：连接池调优需要根据实际负载和数据库承载能力来配置。过大浪费资源，过小导致等待。建议通过公式估算初始值，再根据监控数据持续调整。
