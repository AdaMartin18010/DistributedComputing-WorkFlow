# 分布式事务模式决策树

> 🌲 指导如何选择最适合业务场景的分布式事务解决方案

---

## 🎯 决策树

```mermaid
graph TD
    A[开始：选择分布式事务模式] --> B{是否需要强一致性?}

    B -->|是| C{事务执行时间?}
    B -->|否| D{是否需要立即响应?}

    %% 强一致性分支
    C -->|短事务 < 10s| E[两阶段提交 2PC]
    C -->|长事务 > 10s| F{业务类型?}

    F -->|业务流程型| G[Saga模式]
    F -->|资源预留型| H[TCC模式]

    %% 弱一致性分支
    D -->|是| I{幂等性保证?}
    D -->|否| J[本地消息表模式]

    I -->|有幂等控制| K[最大努力通知]
    I -->|无幂等控制| L{是否需要可靠投递?}

    L -->|是| M[可靠消息最终一致]
    L -->|否| N[异步通知模式]

    %% 详细说明
    E --> E1["特点：强一致性阻塞<br/>实现：XA/Seata AT<br/>适用：金融转账<br/>缺点：同步阻塞性能差"]

    G --> G1["特点：长事务拆分<br/>实现：状态机/编排式<br/>适用：订单流程/出行<br/>缺点：无隔离性需补偿"]

    H --> H1["特点：Try-Confirm-Cancel<br/>实现：业务层三阶段<br/>适用：库存扣减<br/>缺点：业务侵入大"]

    J --> J1["特点：本地事务+消息表<br/>实现：轮询发送+消费<br/>适用：订单创建<br/>缺点：需要定时扫描"]

    K --> K1["特点：多次尝试通知<br/>实现：回调+查询接口<br/>适用：支付回调<br/>缺点：不保证到达"]

    M --> M1["特点：RocketMQ事务消息<br/>实现：半消息+回查<br/>适用：可靠通知<br/>缺点：依赖消息队列"]
```

## 📊 详细对比决策表

### 决策因子权重

| 决策因子 | 权重 | 说明 |
|----------|------|------|
| 一致性要求 | 高 | 金融场景必须强一致 |
| 事务时长 | 高 | 长事务必须避免阻塞 |
| 业务侵入度 | 中 | 影响开发效率和维护成本 |
| 系统复杂度 | 中 | 影响运维难度 |
| 性能要求 | 高 | 高并发场景关键因素 |

### 模式选择矩阵

| 场景特征 | 推荐模式 | 备选方案 | 不推荐 |
|----------|----------|----------|--------|
| 短事务+强一致 | 2PC/XA | Seata AT | Saga, TCC |
| 长事务+业务流程 | Saga | 本地消息表 | 2PC |
| 资源预留+短事务 | TCC | Saga | 2PC |
| 高并发+最终一致 | 本地消息表 | Saga | 2PC |
| 外部系统集成 | 最大努力通知 | 可靠消息 | 2PC, TCC |
| 消息驱动架构 | 可靠消息最终一致 | 本地消息表 | 2PC |

## 🔍 详细模式分析

### 1. 两阶段提交 (2PC)

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者A
    participant P2 as 参与者B

    C->>P1: 准备阶段(Prepare)
    C->>P2: 准备阶段(Prepare)
    P1-->>C: 就绪/失败
    P2-->>C: 就绪/失败

    alt 全部就绪
        C->>P1: 提交(Commit)
        C->>P2: 提交(Commit)
    else 任一失败
        C->>P1: 回滚(Rollback)
        C->>P2: 回滚(Rollback)
    end
```

**适用场景**：银行转账、账户余额更新
**关键限制**：协调者单点、同步阻塞

### 2. Saga模式

```mermaid
stateDiagram-v2
    [*] --> 订单创建
    订单创建 --> 库存扣减
    库存扣减 --> 支付处理
    支付处理 --> [*]

    订单创建 --> 补偿取消: 失败
    库存扣减 --> 补偿恢复库存: 失败
    支付处理 --> 补偿退款: 失败
```

**适用场景**：电商订单、出行预订、保险理赔
**两种实现**：编排式(Choreography) vs 协调式(Orchestration)

### 3. TCC模式

```mermaid
graph LR
    A[Try阶段] --> B{成功?}
    B -->|是| C[Confirm阶段]
    B -->|否| D[Cancel阶段]
    C --> E[完成]
    D --> F[回滚]

    style A fill:#e1f5e1
    style C fill:#e1f5e1
    style D fill:#ffe1e1
```

**三阶段说明**：

- **Try**：资源预留/业务检查
- **Confirm**：确认执行业务
- **Cancel**：释放预留资源

### 4. 本地消息表

```mermaid
sequenceDiagram
    participant App as 应用服务
    participant DB as 本地数据库
    participant MQ as 消息队列
    participant Consumer as 消费者

    App->>DB: 开启本地事务
    App->>DB: 执行业务SQL
    App->>DB: 插入消息记录
    DB-->>App: 事务提交

    loop 定时扫描
        App->>DB: 查询未发送消息
        App->>MQ: 发送消息
        App->>DB: 更新消息状态
    end

    MQ->>Consumer: 投递消息
    Consumer->>DB: 消费处理
```

## 🎯 快速决策流程

```
1. 是否涉及资金/关键账务？
   ├─ 是 → 需要强一致 → 评估事务时长
   │        ├─ < 10秒 → 2PC/Seata AT
   │        └─ > 10秒 → 业务拆分后用Saga
   └─ 否 → 评估性能要求
            ├─ 极高并发 → 本地消息表/可靠消息
            └─ 一般并发 → Saga/TCC
```
