# 分布式事务框架对比

## 概述

随着微服务架构的广泛应用，分布式事务处理成为系统设计的核心挑战之一。市面上涌现出众多分布式事务框架，如Seata、DTM、ByteTCC、Himly等。本文将深入对比主流分布式事务框架的特性、实现原理、适用场景，帮助开发者选择适合自身业务的技术方案。

---

## 一、主流框架概览

### 1.1 框架对比表

| 框架 | 开发方 | 主要模式 | 协议支持 | 活跃程度 | 社区规模 |
|-----|--------|---------|---------|---------|---------|
| **Seata** | 阿里巴巴 | AT/TCC/Saga/XA | 自有协议 | 高 | 大 |
| **DTM** | 深圳数字 Mind | Saga/TCC/XA/MSG | HTTP/gRPC | 高 | 中 |
| **ByteTCC** | 个人开源 | TCC | TCC协议 | 中 | 小 |
| **Himly** | 个人开源 | TCC | TCC协议 | 低 | 小 |
| **EasyTransaction** | 个人开源 | TCC/Saga | 自有协议 | 中 | 小 |

```
├─────────────────────────────────────────────────────────────┤
```

```
│   └──────┬──────┘                                           │

│   │  查询后镜像  │  SELECT * FROM account WHERE id = 1       │

│   │  INSERT INTO│  INSERT INTO undo_log                      │

│   └─────────────┘                                           │
│                                                             │
│   回滚时：                                                   │
```

#### TCC模式

```java
/**
 * Seata TCC模式示例
 */
@LocalTCC
public interface AccountTCCService {
    
    /**
     * Try阶段：预留资源
     */
    @TwoPhaseBusinessAction(name = "accountTccAction", 
                           commitMethod = "commit", 
                           rollbackMethod = "rollback")
    boolean tryDeduct(@BusinessActionContextParameter(paramName = "accountId") 
                      String accountId,
                      @BusinessActionContextParameter(paramName = "amount") 
                      BigDecimal amount);
    
    /**
     * Confirm阶段：确认执行
     */
    boolean commit(BusinessActionContext context);
    
    /**
     * Cancel阶段：取消回滚
     */
    boolean rollback(BusinessActionContext context);
}

/**
 * 实现类
 */
@Service
public class AccountTCCServiceImpl implements AccountTCCService {
    
    @Autowired
    private AccountRepository accountRepository;
    
    @Autowired
    private FrozenRecordRepository frozenRecordRepository;
    
    @Override
    @Transactional
    public boolean tryDeduct(String accountId, BigDecimal amount) {
        // 检查余额
        Account account = accountRepository.findById(accountId)
            .orElseThrow(() -> new AccountNotFoundException(accountId));
        
        if (account.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException("余额不足");
        }
        
        // 冻结金额
        account.setBalance(account.getBalance().subtract(amount));
        account.setFrozenAmount(account.getFrozenAmount().add(amount));
        accountRepository.save(account);
        
        // 记录冻结
        FrozenRecord record = new FrozenRecord();
        record.setAccountId(accountId);
        record.setAmount(amount);
        record.setStatus(FrozenStatus.FROZEN);
        frozenRecordRepository.save(record);
        
        return true;
    }
    
    @Override
    @Transactional
    public boolean commit(BusinessActionContext context) {
        String accountId = context.getActionContext("accountId");
        
        // 确认扣减：解冻并确认
        Account account = accountRepository.findById(accountId).orElse(null);
        if (account != null) {
            account.setFrozenAmount(account.getFrozenAmount().subtract(
                new BigDecimal(context.getActionContext("amount"))));
            accountRepository.save(account);
        }
        
        // 更新记录状态
        frozenRecordRepository.updateStatusByAccountId(accountId, FrozenStatus.CONFIRMED);
        
        return true;
    }
    
    @Override
    @Transactional
    public boolean rollback(BusinessActionContext context) {
        String accountId = context.getActionContext("accountId");
        BigDecimal amount = new BigDecimal(context.getActionContext("amount"));
        
        // 回滚：解冻并恢复余额
        Account account = accountRepository.findById(accountId).orElse(null);
        if (account != null) {
            account.setBalance(account.getBalance().add(amount));
            account.setFrozenAmount(account.getFrozenAmount().subtract(amount));
            accountRepository.save(account);
        }
        
        // 更新记录状态
        frozenRecordRepository.updateStatusByAccountId(accountId, FrozenStatus.CANCELLED);
        
        return true;
    }
}
```

### 2.3 Seata特点总结

|-----|------|

```java
/**
 * DTM Java SDK 示例
 */
@Service
public class OrderService {
    
    @Autowired
    private DtmClient dtmClient;
    
    public void sagaTransaction() {
        // 创建Saga事务
        Saga saga = Saga.newSaga(dtmClient.genGid())
            .add("http://localhost:8081/order", 
                 "http://localhost:8081/orderCompensate", 
                 orderRequest)
            .add("http://localhost:8082/stock", 
                 "http://localhost:8082/stockCompensate", 
                 stockRequest)
            .add("http://localhost:8083/account", 
                 "http://localhost:8083/accountCompensate", 
                 accountRequest);
        
        // 提交事务
        saga.submit();
    }
    
    /**
     * TCC事务示例
     */
    public void tccTransaction() {
        Tcc tcc = Tcc.newTcc(dtmClient.genGid());
        
        try {
            tcc.callBranch(orderRequest, 
                "http://localhost:8081/try",
                "http://localhost:8081/confirm", 
                "http://localhost:8081/cancel");
                
            tcc.callBranch(stockRequest,
                "http://localhost:8082/try",
                "http://localhost:8082/confirm",
                "http://localhost:8082/cancel");
                
        } catch (Exception e) {
            // DTM会自动调用Cancel
            throw e;
        }
    }
    
    /**
     * 消息最终一致性示例
     */
    public void msgTransaction() {
        Msg msg = Msg.newMsg();
        
        // 两阶段消息
        msg.doAndSubmitDB(
            dtmServer + "/msg/queryPrepared",
            (connection) -> {
                // 本地事务
                return orderRepository.save(order);
            },
            "http://localhost:8082/stock/deduct"
        );
    }
}
```

### 5.2 监控告警

```yaml
# 关键监控指标
monitoring:
  seata:
    # 事务相关
    - seata_transaction_total: 事务总数
    - seata_transaction_failed_total: 失败事务数
    - seata_transaction_duration_seconds: 事务执行时间
    
    # TC相关
    - seata_tc_session_size: 当前活跃会话数
    - seata_tc_lock_size: 当前锁数量

  dtm:
    # 事务相关
    - dtm_transactions_total: 事务总数
    - dtm_transactions_failed: 失败事务数

    # 存储相关
    - dtm_storage_size: 存储大小
    - dtm_gc_count: 垃圾回收次数

# 告警规则
alerts:
  - alert: HighTransactionFailureRate
    expr: rate(seata_transaction_failed_total[5m]) > 0.1
    for: 5m
    annotations:
      summary: "事务失败率过高"
      
  - alert: LongRunningTransaction
    expr: seata_transaction_duration_seconds > 30
    for: 1m
    annotations:
      summary: "存在长事务"
```

## 六、总结

分布式事务框架的选择需要根据业务场景、技术栈、性能要求综合考虑：

| 场景 | 推荐框架 | 推荐模式 |
|-----|---------|---------|
| Java单体应用，简单CRUD | Seata | AT模式 |
| Java微服务，高并发 | Seata/DTM | TCC模式 |
| 多语言环境 | DTM | TCC/Saga |
| 长业务流程 | Seata/DTM | Saga模式 |
| 最终一致性足够 | RocketMQ/DTM | 事务消息/MSG |
| 强一致性要求 | Seata | XA模式 |

**最佳实践**：
1. 优先考虑最终一致性方案（消息事务）
2. 必要时使用TCC模式保证高性能
3. 避免过长的分布式事务
4. 完善的监控和告警机制
5. 准备好人工介入预案
