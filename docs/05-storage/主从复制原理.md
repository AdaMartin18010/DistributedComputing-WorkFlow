## 📋 执行摘要

主从复制（Master-Slave Replication）是分布式数据库最基础的复制架构，通过将主节点的数据变更同步到一个或多个从节点，实现数据冗余、读写分离和高可用。本文档深入剖析MySQL主从复制的实现原理、复制模式和相关技术细节。

## 一、核心概念

### 1.1 主从复制架构

```
┌─────────────────────────────────────────────────────────────┐
│                    主从复制架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  异步复制模式：                                              │
│  ┌─────────────┐              ┌─────────────┐              │
│  │   Master    │              │    Slave    │              │
│  │  ┌───────┐  │   Binlog     │  ┌───────┐  │              │
│  │  │ Write │──┼─────────────→│  │ Apply │  │              │
│  │  │  API  │  │  (异步)      │  │  SQL  │  │              │
│  │  └───────┘  │              │  └───────┘  │              │
│  │      ↓      │              │      ↓      │              │
│  │  ┌───────┐  │              │  ┌───────┐  │              │
│  │  │Binlog │  │              │  │Relay  │  │              │
│  │  │File   │  │              │  │Log    │  │              │
│  │  └───────┘  │              │  └───────┘  │              │
│  │      ↓      │              │      ↓      │              │
│  │  返回Client │              │  执行结果    │              │
│  │  (不等待)   │              │  (延迟)      │              │
│  └─────────────┘              └─────────────┘              │
│                                                            │
│  同步复制模式：                                              │
│  ┌─────────────┐              ┌─────────────┐              │
│  │   Master    │              │    Slave    │              │
│  │  ┌───────┐  │   Binlog     │  ┌───────┐  │              │
│  │  │ Write │──┼─────────────→│  │ Apply │  │              │
│  │  │  API  │  │              │  │  SQL  │  │              │
│  │  └───────┘  │              │  └───┬───┘  │              │
│  │      ↑      │              │      │ ACK   │              │
│  │      │      │              │      └───────┤              │
│  │  等待Slave  │              │              │              │
│  │  确认后返回 │              │              │              │
│  └─────────────┘              └─────────────┘              │
│                                                            │
│  半同步复制（Semi-Sync）：                                    │
│  ├─ 至少一个Slave确认接收（不一定执行）                       │
│  ├─ 超时降级为异步                                           │
│  └─ 平衡一致性和性能                                          │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 复制模式对比

| 模式 | 一致性 | 性能 | 可用性 | 适用场景 |
|------|--------|------|--------|----------|
| **异步** | 最终一致 | 最高 | 最高 | 读写分离、报表 |
| **半同步** | 较强 | 中等 | 中等 | 金融交易、关键业务 |
| **全同步** | 强一致 | 低 | 低 | 强一致性要求场景 |
| **组复制** | 强一致 | 中等 | 高 | 高可用集群 |

## 二、技术细节

### 2.1 MySQL复制原理

```
┌─────────────────────────────────────────────────────────────┐
│                  MySQL复制流程详解                           │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  Binlog格式：                                                │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Statement格式：                                       │ │
│  │  ├─ 记录原始SQL语句                                    │ │
│  │  ├─ 优点：日志小，可读性好                              │ │
│  │  └─ 缺点：某些语句可能不一致（NOW(), UUID()等）          │ │
│  │                                                       │ │
│  │  Row格式（推荐）：                                      │ │
│  │  ├─ 记录每行数据的变更前后镜像                          │ │
│  │  ├─ 优点：精确复制，无歧义                              │ │
│  │  └─ 缺点：日志量大（批量更新时）                         │ │
│  │                                                       │ │
│  │  Mixed格式：                                           │ │
│  │  └─ 混合使用，由MySQL决定使用哪种                        │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  复制线程模型：                                              │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Master:                                              │ │
│  │  ├─ Binlog Dump Thread: 读取Binlog发送给Slave           │ │
│  │  │   每个Slave连接对应一个Dump线程                       │ │
│  │  └─ 当Binlog有更新时，主动通知Slave                      │ │
│  │                                                       │ │
│  │  Slave:                                               │ │
│  │  ├─ I/O Thread:                                       │ │
│  │  │   ├─ 连接Master，请求Binlog                          │ │
│  │  │   ├─ 接收Binlog写入Relay Log                        │ │
│  │  │   └─ 维护master.info（连接信息）                     │ │
│  │  │                                                    │ │
│  │  └─ SQL Thread:                                       │ │
│  │      ├─ 读取Relay Log                                  │ │
│  │      ├─ 解析事件并执行                                 │ │
│  │      ├─ 维护relay-log.info（执行位置）                  │ │
│  │      └─ GTID模式下可并行执行（MTS）                     │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  并行复制（MTS - Multi-Threaded Slave）：                     │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  基于库的并行（database）：                              │ │
│  │  ├─ 不同库的事务并行执行                                │ │
│  │  └─ 同库事务串行                                       │ │
│  │                                                       │ │
│  │  基于逻辑时钟的并行（logical_clock）：                    │ │
│  │  ├─ 没有冲突的事务并行执行                              │ │
│  │  └─ 通过binlog_group_commit_sync_delay优化组提交         │ │
│  │                                                       │ │
│  │  基于WRITESET的并行：                                   │ │
│  │  └─ 分析事务修改的行，无冲突则并行                        │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  GTID复制：                                                  │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  GTID = source_id:transaction_id                      │ │
│  │  例: 3E11FA47-71CA-11E1-9E33-C80AA9429562:23           │ │
│  │                                                       │ │
│  │  优势:                                                │ │
│  │  ├─ 全局唯一标识，定位方便                              │ │
│  │  ├─ 自动定位复制位置（无需指定文件名/位置）              │ │
│  │  ├─ 支持failover时的自动恢复                            │ │
│  │  └─ 支持跳过指定GTID                                    │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 复制延迟原因与处理

```
┌─────────────────────────────────────────────────────────────┐
│                  复制延迟分析                                │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  延迟原因：                                                  │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 1. SQL Thread单线程（传统复制）                         │ │
│  │    ├─ Master并发写入，Slave串行执行                     │ │
│  │    └─ 高并发写入时延迟累积                              │ │
│  │                                                       │ │
│  │ 2. 大事务                                               │ │
│  │    ├─ DDL操作（ALTER TABLE）                            │ │
│  │    ├─ 大批量DELETE/UPDATE                               │ │
│  │    └─ 大事务阻塞后续复制                                │ │
│  │                                                       │ │
│  │ 3. 从库负载                                             │ │
│  │    ├─ 从库同时承担读查询                                │ │
│  │    └─ 资源竞争导致复制变慢                              │ │
│  │                                                       │ │
│  │ 4. 网络问题                                             │ │
│  │    └─ 带宽不足、网络抖动                                 │ │
│  │                                                       │ │
│  │ 5. 磁盘I/O                                              │ │
│  │    └─ 从库磁盘性能差                                     │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  延迟监控：                                                  │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  SHOW SLAVE STATUS;                                   │ │
│  │  ├─ Seconds_Behind_Master: 延迟秒数（估算）             │ │
│  │  ├─ Read_Master_Log_Pos: I/O线程读取位置               │ │
│  │  └─ Exec_Master_Log_Pos: SQL线程执行位置               │ │
│  │                                                       │ │
│  │  performance_schema:                                  │ │
│  │  SELECT * FROM replication_applier_status;            │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 三、实践指南

### 3.1 MySQL复制配置

```ini
# my.cnf 主库配置
[mysqld]
server-id = 1
log_bin = mysql-bin
binlog_format = ROW
binlog_row_image = FULL
sync_binlog = 1
innodb_flush_log_at_trx_commit = 1

gtid_mode = ON
enforce_gtid_consistency = ON
binlog_gtid_simple_recovery = ON

# 半同步复制
plugin_load = "rpl_semi_sync_master=semisync_master.so"
rpl_semi_sync_master_enabled = 1
rpl_semi_sync_master_timeout = 1000

# my.cnf 从库配置
[mysqld]
server-id = 2
relay_log = mysql-relay-bin
log_bin = mysql-bin
read_only = 1
log_slave_updates = 1

gtid_mode = ON
enforce_gtid_consistency = ON

# 并行复制
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 8
slave_preserve_commit_order = 1

# 半同步复制
plugin_load = "rpl_semi_sync_slave=semisync_slave.so"
rpl_semi_sync_slave_enabled = 1
```

```sql
-- 创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 从库启动复制（GTID模式）
CHANGE MASTER TO
    MASTER_HOST='master_host',
    MASTER_USER='repl',
    MASTER_PASSWORD='password',
    MASTER_AUTO_POSITION=1;

START SLAVE;

-- 查看复制状态
SHOW SLAVE STATUS\G
```

### 3.2 最佳实践

```
┌─────────────────────────────────────────────────────────────┐
│                    主从复制最佳实践                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 架构设计：                                              │ │
│  ├─ 一主多从（1主2-4从），避免过多从库                       │ │

│  2. 参数优化：                                              │ │
│  ├─ sync_binlog = 1（金融场景）/ 1000（普通）                │ │
│  ├─ innodb_flush_log_at_trx_commit = 1/2                   │ │
│  ├─ slave_parallel_workers = 4-16                          │ │
│  └─ binlog_cache_size适当增大                                │ │
│                                                            │
│  3. 监控告警：                                              │ │
│  ├─ Seconds_Behind_Master > 10s 告警                        │ │
│  ├─ Slave_IO_Running / Slave_SQL_Running 状态               │ │

│  ├─ 1062错误（重复键）：skip或replace                         │ │
```

## 四、常见问题

**Q1: 主从延迟如何优化？**
A: 1) 启用并行复制；2) 大事务拆分；3) 从库硬件升级；4) 业务读写分离，减轻从库压力。

**Q2: 如何选择binlog格式？**
A: 推荐使用ROW格式，尽管日志量大，但复制精确，无歧义。配合binlog_row_image=MINIMAL减少日志。

**Q3: GTID和传统复制如何选择？**
A: 新项目强烈推荐GTID，failover更简单。老项目迁移需考虑兼容性问题。

**Q4: 从库可以写吗？**
A: 不建议，read_only=1设置后仅super权限可写。双写会导致数据不一致。
