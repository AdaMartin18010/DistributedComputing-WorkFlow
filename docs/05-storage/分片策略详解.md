## 📋 执行摘要

数据分片（Sharding）是分布式数据库解决单机容量和性能瓶颈的核心技术。通过将数据水平拆分到多个节点，实现存储和计算能力的水平扩展。本文档系统阐述范围分片、哈希分片、列表分片等策略的原理、优缺点和适用场景。

## 一、核心概念

### 1.1 分片策略对比

```
┌─────────────────────────────────────────────────────────────┐
│                    分片策略对比                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  范围分片（Range Sharding）：                                │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Shard 1    │ Shard 2    │ Shard 3    │ Shard 4       │ │
│  │ [0-2500)    │ [2500-5000)│ [5000-7500)│ [7500-10000)  │ │
│  │  id < 2500  │ 2500-4999  │ 5000-7499  │ id >= 7500    │ │
│  │                                                       │ │
│  │  优点：                                                │ │
│  │  ├─ 支持范围查询和排序                                  │ │
│  │  ├─ 扩容简单（增加分片即可）                            │ │
│  │  └─ 数据迁移量小                                        │ │
│  │                                                       │ │
│  │  缺点：                                                │ │
│  │  ├─ 热点问题（新数据集中在最后一个分片）                 │ │
│  │  └─ 数据分布不均                                        │ │
│  │                                                       │ │
│  │  适用：时序数据、按时间分区                              │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  哈希分片（Hash Sharding）：                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  shard = hash(key) % N                                │ │
│  │                                                       │ │
│  │  Shard 0    Shard 1    Shard 2    Shard 3             │ │
│  │  hash%4=0   hash%4=1   hash%4=2   hash%4=3            │ │
│  │     ●          ●          ●          ●                │ │
│  │     ●          ●          ●          ●                │ │
│  │     ●          ●          ●          ●                │ │
│  │                                                       │ │
│  │  优点：                                                │ │
│  │  ├─ 数据分布均匀                                        │ │
│  │  ├─ 避免热点                                            │ │
│  │  └─ 点查性能好                                          │ │
│  │                                                       │ │
│  │  缺点：                                                │ │
│  │  ├─ 范围查询需全部分片扫描                              │ │
│  │  ├─ 扩容复杂（需迁移大量数据）                          │ │
│  │  └─ 难以实现范围分页                                    │ │
│  │                                                       │ │
│  │  适用：用户数据、订单数据                                │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  列表分片（List Sharding）：                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Shard CN    Shard US    Shard EU                     │ │
│  │  region='CN' │ region='US'│ region='EU'               │ │
│  │  country in  │ country in │ country in                │ │
│  │  ('CN','JP') │ ('US','CA')│ ('DE','UK')               │ │
│  │                                                       │ │
│  │  优点：                                                │ │
│  │  ├─ 按业务/地域直观划分                                  │ │
│  │  ├─ 便于数据隔离                                        │ │
│  │  └─ 支持业务规则                                        │ │
│  │                                                       │ │
│  │  适用：多租户、地域分区                                  │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 策略选择矩阵

| 维度 | 范围分片 | 哈希分片 | 列表分片 |
|------|----------|----------|----------|
| **点查性能** | 中 | 高 | 高 |
| **范围查询** | 高 | 低 | 中 |
| **数据分布** | 不均 | 均匀 | 取决于列表 |
| **热点风险** | 高 | 低 | 中 |
| **扩容难度** | 低 | 高 | 低 |
| **适用场景** | 时序数据 | 用户/订单 | 多租户/地域 |

## 二、技术细节

### 2.1 复合分片策略

```
┌─────────────────────────────────────────────────────────────┐
│                    复合分片策略                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  两级分片（分库+分表）：                                      │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  第一级：Hash分库（解决写性能）                          │ │
│  │  db = hash(user_id) % 8                                │ │
│  │                                                       │ │
│  │  第二级：Range分表（解决查询性能）                       │ │
│  │  table = order_YYYYMM                                  │ │
│  │                                                       │ │
│  │  结果：                                                │ │
│  │  ├─ db0: order_202401, order_202402, ...              │ │
│  │  ├─ db1: order_202401, order_202402, ...              │ │
│  │  └─ ...                                               │ │
│  │                                                       │ │
│  │  路由示例：                                            │ │
│  │  user_id=12345 → hash(12345)%8=3 → db3                │ │
│  │  查询2024年1月 → db3.order_202401                     │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  雪崩分片（Snowflake Sharding）：                             │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  结合Snowflake ID结构天然分片                           │ │
│  │                                                       │ │
│  │  Snowflake ID:                                        │ │
│  │  ├─ 1bit sign                                         │ │
│  │  ├─ 41bits timestamp                                  │ │
│  │  ├─ 10bits worker_id (可作为分片键)                    │ │
│  │  └─ 12bits sequence                                   │ │
│  │                                                       │ │
│  │  分片：shard = worker_id % N                          │ │
│  │  优势：ID生成和分片路由天然结合                          │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  Tag分片（Tag-based Sharding）：                              │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  按标签动态路由                                         │ │
│  │                                                       │ │
│  │  用户标签：VIP → 高性能分片                             │ │
│  │           普通 → 标准分片                               │ │
│  │           冷数据 → 归档分片                             │ │
│  │                                                       │ │
│  │  实现：用户表维护tag字段，路由时读取                      │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 分片路由

```
┌─────────────────────────────────────────────────────────────┐
│                    分片路由实现                              │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  路由方式：                                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  1. 客户端路由（Client-side）                          │ │
│  │     ├─ 应用层计算分片位置                               │ │
│  │     ├─ 直接连接目标数据库                               │ │
│  │     └─ 例：ShardingSphere-JDBC                         │ │
│  │                                                       │ │
│  │  2. 代理层路由（Proxy-side）                           │ │
│  │     ├─ 应用连接代理                                     │ │
│  │     ├─ 代理计算路由                                     │ │
│  │     └─ 例：MyCat, ShardingSphere-Proxy                 │ │
│  │                                                       │ │
│  │  3. 服务端路由（Server-side）                          │ │
│  │     ├─ 数据库内置分片                                   │ │
│  │     └─ 例：MongoDB, Elasticsearch                      │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  路由算法：                                                  │ │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  public class ShardRouter {                           │ │
│  │      public int route(Long shardKey, int shardCount) {│ │
│  │          // 哈希分片                                   │ │
│  │          return (int) (shardKey.hashCode() & 0x7FFFFFFF) │ │
│  │                 % shardCount;                         │ │
│  │      }                                                │ │
│  │                                                       │ │
│  │      public int rangeRoute(Long shardKey,             │ │
│  │                           List<Range> ranges) {       │ │
│  │          // 范围分片                                   │ │
│  │          for (int i = 0; i < ranges.size(); i++) {    │ │
│  │              if (ranges.get(i).contains(shardKey)) {  │ │
│  │                  return i;                            │ │
│  │              }                                        │ │
│  │          }                                            │ │
│  │          throw new IllegalArgumentException();        │ │
│  │      }                                                │ │
│  │  }                                                    │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```

## 三、实践指南

### 3.1 ShardingSphere配置

```yaml
# 哈希分片配置
spring:
  shardingsphere:
    rules:
      sharding:
        tables:
          t_user:
            actual-data-nodes: ds${0..7}.t_user
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: user-hash
        sharding-algorithms:
          user-hash:
            type: INLINE
            props:
              algorithm-expression: ds${user_id % 8}

# 范围分片配置
sharding-algorithms:
  order-range:
    type: INLINE
    props:
      algorithm-expression: |
        ds${order_id >= 0 && order_id < 1000000 ? 0 :
           order_id >= 1000000 && order_id < 2000000 ? 1 :
           order_id >= 2000000 && order_id < 3000000 ? 2 : 3}
```

### 3.2 最佳实践

```
┌─────────────────────────────────────────────────────────────┐
│                    分片策略最佳实践                          │
├─────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 分片键选择：                                             │ │
│  ├─ 高频查询条件                                           │ │
│  ├─ 数据分布均匀                                           │ │
│  ├─ 不可变更（避免数据迁移）                                │ │
│  └─ 例：user_id, order_id, tenant_id                       │ │
│                                                            │
│  2. 分片数设计：                                             │ │
│  ├─ 初期逻辑分片 > 物理节点（便于扩容）                      │ │
│  ├─ 分片数建议为2的幂（便于一致性哈希）                      │ │
│  └─ 预估数据量，单分片控制在合理大小                         │ │
│                                                            │
│  3. 避免跨分片查询：                                         │ │
│  ├─ 相同user_id的数据路由到同一分片                         │ │
│  ├─ 绑定表（Binding Tables）配置                            │ │
│  └─ 应用层组装跨分片结果                                    │ │
│                                                            │
│  4. 数据倾斜处理：                                           │ │
│  ├─ 热点数据单独分片                                        │ │
│  ├─ 使用salting分散热点                                     │ │
│  └─ 定期检查分片数据分布                                    │ │
│                                                            │
└─────────────────────────────────────────────────────────────┘
```
