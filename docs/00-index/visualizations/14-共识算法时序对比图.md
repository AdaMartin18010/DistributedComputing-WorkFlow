# 共识算法时序对比图

> ⏱️ 使用Mermaid SequenceDiagram对比主流共识算法的消息流程与时序特征

---

## 1️⃣ Raft正常流程

```mermaid
sequenceDiagram
    autonumber
    participant C as Client
    participant L as Leader
    participant F1 as Follower 1
    participant F2 as Follower 2

    Note over C,F2: 阶段1: 日志复制
    C->>L: 发送命令 (Write x=1)
    activate L
    L->>L: 追加到本地日志<br/>[Index=5, Term=3]

    L->>F1: AppendEntries<br/>prevLogIndex=4, prevLogTerm=3<br/>entries=[{5,3,x=1}]
    L->>F2: AppendEntries<br/>prevLogIndex=4, prevLogTerm=3<br/>entries=[{5,3,x=1}]

    activate F1
    activate F2
    F1->>F1: 验证prevLog匹配<br/>追加新条目
    F2->>F2: 验证prevLog匹配<br/>追加新条目

    F1-->>L: AppendEntriesResponse<br/>success=true, matchIndex=5
    F2-->>L: AppendEntriesResponse<br/>success=true, matchIndex=5
    deactivate F1
    deactivate F2

    Note over L: 阶段2: 提交确认
    L->>L: 多数派确认<br/>commitIndex=5
    L->>L: 应用到状态机

    L-->>C: 响应成功
    deactivate L

    Note over L,F2: 阶段3: 心跳传播提交
    L->>F1: AppendEntries<br/>heartbeat + commitIndex=5
    L->>F2: AppendEntries<br/>heartbeat + commitIndex=5

    F1->>F1: 应用Index=5到状态机
    F2->>F2: 应用Index=5到状态机
```

**关键特征**：

- **消息轮次**：2轮（Leader→Followers→Leader）
- **延迟**：2×RTT + 本地处理
- **消息复杂度**：O(n)，n为Follower数量

## 2️⃣ Paxos两阶段流程

```mermaid
sequenceDiagram
    autonumber
    participant P as Proposer
    participant A1 as Acceptor 1
    participant A2 as Acceptor 2
    participant A3 as Acceptor 3
    participant L as Learner

    Note over P,L: 阶段1a: Prepare阶段
    P->>P: 生成Proposal Number: n

    P->>A1: Prepare(n)
    P->>A2: Prepare(n)
    P->>A3: Prepare(n)

    activate A1
    activate A2
    activate A3

    A1->>A1: 如果n>maxPromised<br/>Promise不再接受<n
    A2->>A2: 同上
    A3->>A3: 同上

    A1-->>P: Promise(n, lastAcceptedValue)
    A2-->>P: Promise(n, lastAcceptedValue)
    A3-->>P: Promise(n, lastAcceptedValue)

    deactivate A1
    deactivate A2
    deactivate A3

    Note over P: 阶段1b: 值选择<br/>选择多数派的值或新值

    Note over P,L: 阶段2a: Accept阶段
    P->>A1: Accept(n, v)
    P->>A2: Accept(n, v)
    P->>A3: Accept(n, v)

    activate A1
    activate A2
    activate A3

    A1->>A1: 如果n>=maxPromised<br/>接受值v
    A2->>A2: 同上
    A3->>A3: 同上

    A1-->>P: Accepted(n, v)
    A2-->>P: Accepted(n, v)
    A3-->>P: Accepted(n, v)

    deactivate A1
    deactivate A2
    deactivate A3

    Note over P: 获得多数派Accepted<br/>共识达成

    Note over P,L: 阶段2b: 学习阶段
    A1->>L: Accepted通知
    A2->>L: Accepted通知
    A3->>L: Accepted通知

    L->>L: 收到多数派Accepted<br/>学习已选值v
```

**关键特征**：

- **消息轮次**：2-3轮（Prepare→Promise→Accept→Accepted）
- **延迟**：2×RTT（最优）~ 3×RTT（含冲突）
- **消息复杂度**：O(n²)，多Proposer时冲突

## 3️⃣ PBFT三阶段流程

```mermaid
sequenceDiagram
    autonumber
    participant C as Client
    participant P as Primary (Leader)
    participant R1 as Replica 1
    participant R2 as Replica 2
    participant R3 as Replica 3

    Note over C,R3: 阶段1: Pre-Prepare
    C->>P: Request <o>
    activate P
    P->>P: 分配序号n<br/>digest(d) = Hash(o)

    P->>R1: <<PRE-PREPARE, v, n, d>, o>
    P->>R2: <<PRE-PREPARE, v, n, d>, o>
    P->>R3: <<PRE-PREPARE, v, n, d>, o>
    deactivate P

    Note over C,R3: 阶段2: Prepare
    activate R1
    activate R2
    activate R3

    R1->>R1: 验证：签名/视图/序号/摘要
    R2->>R2: 同上
    R3->>R3: 同上

    R1->>P: <PREPARE, v, n, d, i>
    R1->>R2: <PREPARE, v, n, d, i>
    R1->>R3: <PREPARE, v, n, d, i>

    R2->>P: <PREPARE, v, n, d, i>
    R2->>R1: <PREPARE, v, n, d, i>
    R2->>R3: <PREPARE, v, n, d, i>

    R3->>P: <PREPARE, v, n, d, i>
    R3->>R1: <PREPARE, v, n, d, i>
    R3->>R2: <PREPARE, v, n, d, i>

    Note over R1,R3: 每个节点收集<br/>2f个Prepare<br/>进入prepared状态

    Note over C,R3: 阶段3: Commit
    R1->>P: <COMMIT, v, n, d, i>
    R1->>R2: <COMMIT, v, n, d, i>
    R1->>R3: <COMMIT, v, n, d, i>

    R2->>P: <COMMIT, v, n, d, i>
    R2->>R1: <COMMIT, v, n, d, i>
    R2->>R3: <COMMIT, v, n, d, i>

    R3->>P: <COMMIT, v, n, d, i>
    R3->>R1: <COMMIT, v, n, d, i>
    R3->>R2: <COMMIT, v, n, d, i>

    Note over R1,R3: 收集2f+1个Commit<br/>进入committed-local<br/>可执行操作

    R1->>R1: 执行操作o<br/>发送Reply给Client
    R2->>R2: 执行操作o
    R3->>R3: 执行操作o

    R1-->>C: <REPLY, v, t, c, i, r>
    deactivate R1
    deactivate R2
    deactivate R3

    P->>P: 执行操作o
    P-->>C: <REPLY, v, t, c, i, r>
```

**关键特征**：

- **消息轮次**：3轮（Pre-Prepare→Prepare→Commit）
- **延迟**：3×RTT
- **消息复杂度**：O(n²)，全互联广播
- **容错**：3f+1节点容忍f个拜占庭故障

## 4️⃣ 算法时序对比总览

```mermaid
sequenceDiagram
    autonumber
    participant C as Client
    participant N1 as Node 1
    participant N2 as Node 2
    participant N3 as Node 3

    Note over C,N3: === Raft ===
    C->>N1: Request
    N1->>N2: AppendEntries
    N1->>N3: AppendEntries
    N2-->>N1: Success
    N3-->>N1: Success
    N1-->>C: Response

    Note over C,N3: === Paxos ===
    C->>N1: Propose
    N1->>N2: Prepare
    N1->>N3: Prepare
    N2-->>N1: Promise
    N3-->>N1: Promise
    N1->>N2: Accept
    N1->>N3: Accept
    N2-->>N1: Accepted
    N3-->>N1: Accepted
    N1-->>C: Response

    Note over C,N3: === PBFT ===
    C->>N1: Request
    N1->>N2: Pre-Prepare
    N1->>N3: Pre-Prepare
    N2->>N1: Prepare
    N2->>N3: Prepare
    N3->>N1: Prepare
    N3->>N2: Prepare
    N2->>N1: Commit
    N2->>N3: Commit
    N3->>N1: Commit
    N3->>N2: Commit
    N1-->>C: Response
```

## 📊 时序特征对比表

| 特征 | Raft | Paxos | PBFT |
|------|------|-------|------|
| **消息轮次** | 2轮 | 2-3轮 | 3轮 |
| **最优延迟** | 2×RTT | 2×RTT | 3×RTT |
| **消息复杂度** | O(n) | O(n²) | O(n²) |
| **广播方式** | Leader→All | All→All | All↔All |
| **故障模型** | CFT | CFT | BFT |
| **容错节点** | 2f+1 | 2f+1 | 3f+1 |
| **领导者** | 有 | 可选 | 有(Primary) |

## 🎯 时序优化策略

```mermaid
graph LR
    subgraph "优化方向"
        A[减少RTT] --> B[流水线]
        A --> C[批量提交]

        D[减少消息数] --> E[领导者聚合]
        D --> F[树形广播]

        G[并行处理] --> H[管道化]
        G --> I[投机执行]
    end
```

### 优化技术对比

| 优化技术 | 适用算法 | 效果 | 复杂度 |
|----------|----------|------|--------|
| **流水线** | Raft | 减少RTT等待 | 低 |
| **批量提交** | Raft/Paxos | 摊薄开销 | 低 |
| **Fast Paxos** | Paxos | 1轮无冲突 | 中 |
| **链式BFT** | HotStuff | O(n)消息 | 高 |
| **投机执行** | Zyzzyva | 减少延迟 | 高 |
