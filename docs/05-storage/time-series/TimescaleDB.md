# TimescaleDB 详解

TimescaleDB 是基于 PostgreSQL 构建的开源时序数据库，它继承了 PostgreSQL 的完整 SQL 支持和生态系统，同时针对时序数据进行了专门优化，实现了关系型数据库的灵活性与时序数据库性能的最佳平衡。

## 1. 架构概述

TimescaleDB 采用**超级表 (Hypertable)** 和**分块 (Chunk)** 的架构设计：


## 2. 核心概念

### 2.1 Hypertable (超级表)

Hypertable 是面向用户的抽象，呈现为一张普通表，但实际上由多个 Chunk 组成：

```
┌─────────────────────────────────────────────────────────────────┐
│                   Hypertable 抽象层                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   用户视角 (单表):                                               │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  SELECT * FROM conditions                               │   │
│   │  WHERE time > '2024-01-01' AND location = 'office';     │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│   TimescaleDB 自动转换:                                          │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  查询重写 → 确定涉及的 Chunks → 并行/串行执行 → 合并结果   │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│         ┌────────────────────┼────────────────────┐              │
│         ▼                    ▼                    ▼              │
│   ┌─────────────┐      ┌─────────────┐      ┌─────────────┐      │
│   │  Chunk 1    │      │  Chunk 2    │      │  Chunk 3    │      │
│   │  (time <    │      │  (time >=   │      │  (time >=   │      │
│   │   '2024-01')│      │  '2024-01'  │      │  '2024-02') │      │
│   │             │      │  AND <      │      │             │      │
│   │             │      │  '2024-02') │      │             │      │
│   └─────────────┘      └─────────────┘      └─────────────┘      │
│                                                                  │
│   特性:                                                          │
│   • 自动分区: 数据按时间维度自动分区                             │
│   • 空间分区: 可额外按 device_id/location 等分区                 │
│   • 透明查询: 用户无感知，自动路由到对应 Chunk                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 Chunk (数据分块)

Chunk 是实际的物理存储单元，每个 Chunk 是独立的 PostgreSQL 表：

```
┌─────────────────────────────────────────────────────────────────┐
│                   Chunk 存储结构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Chunk 参数配置:                                                 │
│  • chunk_time_interval: 每个 Chunk 的时间跨度 (默认7天)           │
│  • chunk_size_target: 目标 Chunk 大小 (默认约1GB)                 │
│                                                                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  Chunk: _hyper_1_1_chunk (2024年1月数据)                    │  │
│  │                                                           │  │
│  │  数据存储 (列式压缩):                                       │  │
│  │  ┌─────────────┬─────────────┬─────────────────────────┐  │  │
│  │  │    time     │  device_id  │       temperature       │  │  │
│  │  ├─────────────┼─────────────┼─────────────────────────┤  │  │
│  │  │ 2024-01-01  │     1       │        23.5             │  │  │
│  │  │ 2024-01-01  │     2       │        24.1             │  │  │
│  │  │    ...      │    ...      │         ...             │  │  │
│  │  └─────────────┴─────────────┴─────────────────────────┘  │  │
│  │                                                           │  │
│  │  索引结构:                                                  │  │
│  │  ┌─────────────────────────────────────────────────────┐  │  │
│  │  │  CREATE INDEX ON _hyper_1_1_chunk (time DESC);      │  │  │
│  │  │  CREATE INDEX ON _hyper_1_1_chunk (device_id, time);│  │  │
│  │  └─────────────────────────────────────────────────────┘  │  │
│  │                                                           │  │
│  │  压缩状态:                                                  │  │
│  │  ┌──────────────┬──────────────┬────────────────────────┐ │  │
│  │  │ is_compressed│   压缩率     │   压缩后大小           │ │  │
│  │  │    false     │     -        │      950 MB            │ │  │
│  │  └──────────────┴──────────────┴────────────────────────┘ │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 数据写入流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     数据写入流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 客户端 INSERT                                                │
│  INSERT INTO conditions(time, device_id, temperature)           │
│  VALUES ('2024-03-15 10:00:00', 1, 23.5);                       │
│                              │                                   │
│                              ▼                                   │
│  2. TimescaleDB 路由层                                           │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  • 解析时间戳: '2024-03-15'                              │    │
│  │  • 查询 chunk 元数据表                                    │    │
│  │  • 确定目标 Chunk: _hyper_1_3_chunk (March 2024)         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              │                                   │
│                              ▼                                   │
│  3. 检查 Chunk 存在性                                            │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Chunk 存在?                                              │    │
│  └─────────────────────────────────────────────────────────┘    │
│         │                    │                                   │
│        是                   否                                   │
│         │                    │                                   │
│         ▼                    ▼                                   │
│  直接写入            创建新 Chunk                                 │
│                      ┌─────────────────────────────────────┐    │
│                      │ • 创建新表 _hyper_1_N_chunk          │    │
│                      │ • 创建索引                            │    │
│                      │ • 注册到元数据表                       │    │
│                      └─────────────────────────────────────┘    │
│                              │                                   │
│                              ▼                                   │
│  4. 写入数据                                                     │
│  • 写入 Chunk 表                                                │
│  • 更新索引                                                      │
│  • 写入 WAL (保证持久性)                                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 4. 查询优化机制

### 4.1 查询计划优化

```
┌─────────────────────────────────────────────────────────────────┐
│                   查询执行优化                                   │
├─────────────────────────────────────────────────────────────────┤

│  原始查询:                                                       │
│  SELECT device_id, avg(temperature)                             │
│  FROM conditions                                                │

│                              │                                   │
│                              ▼                                   │

│                                                                  │
```


## 5. 数据压缩

```
┌─────────────────────────────────────────────────────────────────┐
│                   TimescaleDB 数据压缩                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  压缩策略配置:                                                   │
│  ALTER TABLE conditions                                         │
│  SET (timescaledb.compress,                                     │
│        timescaledb.compress_segmentby = 'device_id',            │
│        timescaledb.compress_orderby = 'time DESC');             │
│                                                                  │
│  自动压缩策略:                                                   │
│  SELECT add_compression_policy('conditions', INTERVAL '7 days'); │
│  -- 7天前的 Chunk 自动压缩                                       │
│                              │                                   │
│                              ▼                                   │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                    压缩流程                                │  │
│  │                                                           │  │
│  │  原始 Chunk (未压缩):                                      │  │
│  │  ┌──────────┬──────────┬──────────┐                       │  │
│  │  │   time   │device_id │   temp   │  100万行              │  │
│  │  └──────────┴──────────┴──────────┘  ~50 MB               │  │
│  │                                                           │  │
│  │            │                                              │  │
│  │            ▼                                              │  │
│  │  压缩算法:                                                 │  │
│  │  • 按 segmentby 列分组 (device_id)                         │  │
│  │  • 时间列: Delta-of-Delta 编码                             │  │
│  │  • 数值列: Gorilla 浮点压缩                                │  │
│  │  • 字典编码用于重复字符串                                   │  │
│  │                                                           │  │
│  │            │                                              │  │
│  │            ▼                                              │  │
│  │  压缩后 Chunk:                                             │  │
│  │  ┌─────────────────────────────────────────┐               │  │
│  │  │ 压缩数据块 (列式存储, 不可变)              │  ~2-5 MB     │  │
│  │  │ 压缩率: 90-95%                           │               │  │
│  │  └─────────────────────────────────────────┘               │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                  │
│  压缩后特性:                                                     │
│  • 透明查询: 查询自动解压缩                                       │
│  • 高效检索: 支持按 segmentby 列过滤                             │
│  • 不可变: 压缩后 Chunk 不可修改，需解压才能更新                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 7. 核心优势对比

| 特性 | TimescaleDB | InfluxDB | PostgreSQL |
|-----|-------------|----------|------------|
| SQL 支持 | 完整 | InfluxQL/Flux | 完整 |
| 时序优化 | 是 | 原生 | 无 |
| 生态系统 | PostgreSQL 生态 | InfluxData 生态 | PostgreSQL 生态 |
| 压缩率 | 90-95% | 90%+ | 无 |
| 水平扩展 | 多节点 | 集群版 | 需 Citus |
| JOIN 支持 | 完整 | 有限 | 完整 |
| 复杂查询 | 优秀 | 一般 | 优秀 |
| 学习成本 | 低 (标准 SQL) | 中 (新语言) | 低 |

## 8. 适用场景

- **已有 PostgreSQL 基础设施**: 无缝迁移，保留现有工具和技能
- **复杂查询需求**: 需要 JOIN、窗口函数、复杂聚合
- **关系型 + 时序数据**: 设备元数据与时序指标关联查询
- **金融数据**: 需要严格 ACID 保证的交易数据
- **地理时序数据**: 结合 PostGIS 的时空分析

TimescaleDB 是希望在 PostgreSQL 生态内获得时序数据库性能的理想选择，特别适合需要将时序数据与关系型业务数据结合的场景。
