## 一、理论类面试题

### 1.1 CAP定理

#### 面试题1: 什么是CAP定理？实际系统中如何权衡？

**答题思路**

```
1. 先明确定义: CAP定理的基本概念
2. 解释三个特性: C/A/P的具体含义
3. 说明权衡: 为什么只能三选二
4. 实际应用: 举例说明CP和AP系统的选择
```

**核心要点**

- **C (Consistency)**: 一致性，所有节点同时看到相同数据
- **A (Availability)**: 可用性，每个请求都能得到响应
- **P (Partition Tolerance)**: 分区容错性，网络分区时系统仍能工作
- **核心结论**: 分布式系统中P通常必须满足，实际在C和A之间权衡

**参考答案**
> CAP定理指出分布式系统无法同时满足一致性、可用性和分区容错性三个特性。在实际系统中，由于网络分区不可避免，我们通常需要在一致性和可用性之间做权衡。
>
> 选择CP的场景：金融系统、库存扣减，需要强一致性，宁可拒绝服务也不能返回错误数据。例如：HBase、MongoDB(配置为强一致)、ZooKeeper。
>
> 选择AP的场景：社交网络、内容推荐，优先考虑可用性，可以容忍短暂的不一致。例如：Cassandra、DynamoDB、CouchDB。

**关联文档**: [CAP定理专题文档](../02-THEORY/distributed-systems/CAP定理专题文档.md)

### 1.2 共识算法

#### 面试题2: Paxos和Raft的区别是什么？

**答题思路**

```
1. 共同点: 都是解决分布式共识问题的算法
2. 不同点1: 可理解性（Raft更易理解）
3. 不同点2: 工程实现（Raft更清晰）
4. 不同点3: 性能对比
5. 应用场景: 各自适合什么场景
```

**核心对比表**

| 维度 | Paxos | Raft |
|------|-------|------|
| **提出时间** | 1989年 | 2014年 |
| **可理解性** | 困难，难以实现 | 易于理解，工程友好 |
| **Leader选举** | 隐含，多轮协商 | 明确，心跳超时触发 |
| **日志复制** | 复杂 | 清晰，Leader驱动 |
| **消息复杂度** | O(n²) | O(n) |
| **工程实现** | Chubby、ZooKeeper | etcd、Consul、TiKV |

**参考答案**
> Paxos和Raft都是分布式共识算法，解决多个节点如何就某个值达成一致的问题。
>
> **主要区别**：
>
> 1. **可理解性**：Raft设计目标就是易于理解，将共识问题分解为Leader选举、日志复制、安全性三个子问题；Paxos理论优美但实现复杂。
> 2. **Leader角色**：Raft有明确的Leader概念，所有写操作通过Leader；Paxos中Leader是优化而非必需。
> 3. **工程友好**：Raft论文提供了完整的工程实现细节，因此etcd、Consul等系统采用Raft；Paxos更多在理论层面。
>
> **选择建议**：新项目推荐Raft，生态成熟、实现简单；需要多Leader写入的场景考虑Paxos变种。

**关联文档**: [Paxos算法专题文档](../02-THEORY/distributed-systems/Paxos算法专题文档.md) | [Raft算法专题文档](../02-THEORY/distributed-systems/Raft算法专题文档.md)

### 1.3 一致性模型

#### 面试题3: 强一致性和最终一致性有什么区别？如何选择？

**答题思路**

```
1. 定义解释: 两种一致性的含义
2. 实现机制: 各自如何实现
3. 优缺点对比: 性能、可用性、复杂度
4. 选择依据: 业务场景决定
```

**一致性强度对比**

```
强一致性 → 顺序一致性 → 因果一致性 → 最终一致性 → 弱一致性
   ↑           ↑            ↑              ↑            ↑
一致性强    一致性中等    一致性中等      一致性弱      无保证
延迟高      延迟中等      延迟中等       延迟低        延迟最低
可用性低    可用性中等    可用性中等      可用性高      可用性最高
```

**参考答案**
> **强一致性**：任何读取都能读到最新的写入，保证所有节点数据一致。实现方式包括：两阶段提交、Paxos/Raft共识、同步复制。优点是数据一致不会出错，缺点是延迟高、可用性受影响（网络分区时可能无法写入）。
>
> **最终一致性**：不保证立即一致，但保证在没有新写入的情况下，最终所有副本会一致。实现方式包括：异步复制、Gossip协议、向量时钟。优点是高性能、高可用，缺点是可能读到旧数据，需要处理冲突。
>
> **选择原则**：
>
> - 金融交易、库存扣减选强一致性
> - 社交Feed、商品详情选最终一致性

**关联文档**: [一致性模型专题文档](../02-THEORY/distributed-systems/一致性模型专题文档.md)

### 1.4 FLP不可能定理

#### 面试题4: FLP不可能定理说明了什么？对系统设计的启示？

**参考答案**
> FLP不可能定理（Fischer, Lynch, Paterson）证明：在异步分布式系统中，即使只有一个进程可能崩溃，也不存在确定性的共识算法。
>
> **核心启示**：
>
> 1. **超时机制**：必须使用超时来判断进程是否故障，而不能确定地知道
> 2. **部分同步假设**：实际系统采用部分同步模型，假设网络最终可靠
> 3. **概率算法**：使用随机化算法绕过FLP限制（如Paxos的Leader选举超时）
> 4. **故障检测器**：使用不完美但实用的故障检测器

**关联文档**: [FLP不可能定理专题文档](../02-THEORY/distributed-systems/FLP不可能定理专题文档.md)

### 1.5 分布式事务

#### 面试题5: 2PC和Saga模式有什么区别？

**答题思路**

```
1. 先分类: 2PC是强一致性方案，Saga是最终一致性方案
2. 讲原理: 各自的工作流程
3. 对比优缺点: 一致性、性能、复杂度
4. 给建议: 什么场景用什么
```

**对比表**

| 特性 | 2PC | Saga |
|------|-----|------|
| **一致性** | 强一致性 | 最终一致性 |
| **性能** | 低（锁资源） | 高（无锁） |
| **复杂度** | 协议复杂 | 业务复杂（需补偿） |
| **回滚** | 自动回滚 | 补偿操作 |
| **适用场景** | 短事务、强一致 | 长事务、跨服务 |

**参考答案**
> **2PC（两阶段提交）**：通过准备和提交两个阶段保证事务原子性，协调者询问所有参与者，全部同意才提交。优点是强一致性，缺点是同步阻塞、单点故障、性能低。
>
> **Saga模式**：将长事务拆分为本地事务序列，每个事务有对应的补偿操作，失败时反向执行补偿。优点是无锁、高性能，缺点是缺少隔离性、需要编写补偿逻辑。
>
> **选择建议**：短事务且需要强一致用2PC，长事务或跨多个服务用Saga。

**关联文档**: [Saga模式](../07-KNOWLEDGE/01-工作流设计模型/Saga模式.md) | [2PC与3PC](../08-transactions/2PC与3PC.md)

## 二、设计类面试题

### 2.1 秒杀系统

#### 面试题6: 如何设计一个支持10万人同时抢购的秒杀系统？

**答题思路（4S方法）**

```
Scenario（场景）: 10万人抢1000件商品
Service（服务）: 网关、秒杀服务、库存服务、订单服务
Storage（存储）: Redis、MySQL、消息队列
Scale（扩展）: 限流、削峰、异步、缓存
```

**架构设计**

```
用户 → CDN → Nginx(限流) → 网关 → 秒杀服务 → Redis(预扣库存)
                                        ↓
                                    消息队列(Kafka)
                                        ↓
                                    订单服务 → MySQL(最终订单)
```

**核心策略**

| 问题 | 解决方案 |
|------|----------|
| **超卖** | Redis原子减库存，库存扣减和下单分离 |
| **高并发** | 限流（令牌桶）、削峰（消息队列）、静态化 |
| **热点** | 库存分片、本地缓存、请求合并 |
| **公平性** | 验证码、排队、随机时间片 |

**参考答案**
> **核心思路**：流量分层过滤，将请求层层筛选，最终只让有效请求到达数据库。
>
> **具体措施**：
>
> 1. **前端层**：CDN加速、验证码防刷、按钮置灰
> 2. **接入层**：Nginx限流（每秒1万请求），URL动态化防刷
> 3. **服务层**：Redis原子扣减库存，扣减成功发消息异步创建订单
> 4. **数据层**：消息队列削峰，订单服务消费消息写数据库
>
> **关键点**：
>
> - 库存预扣在Redis，使用Lua脚本保证原子性
> - 秒杀订单单独表，避免影响普通订单
> - 失败快速返回，减少资源占用

### 2.2 即时通讯系统

#### 面试题7: 设计一个支持千万用户的即时通讯系统

**答题思路**

```
1. 明确功能: 单聊、群聊、在线状态、消息历史
2. 核心挑战: 实时性、消息可靠投递、有序性
3. 架构设计: 接入层、路由层、存储层
4. 关键技术: 长连接、消息推拉结合、消息ID
```

**架构设计**

```
客户端 → 负载均衡 → 网关集群(长连接) → 路由服务 → 业务服务
                                      ↓
                                ┌─────┴─────┐
                            Redis(在线状态)  MySQL(消息历史)
```

**关键技术点**

| 问题 | 方案 |
|------|------|
| **长连接管理** | WebSocket + Gateway集群，一致性哈希路由 |
| **消息有序** | 用户级Sequence ID，服务端分配 |
| **消息可靠** | ACK机制 + 离线消息 + 消息重传 |
| **群聊扩散** | 写扩散（小群）vs 读扩散（大群） |
| **在线状态** | Redis bitmap + 心跳检测 |

**参考答案**
> **接入层**：WebSocket长连接，Gateway集群管理连接，使用一致性哈希保证同一用户请求路由到同一节点。
>
> **消息流程**：
>
> 1. 发送方 → Gateway → 消息服务（分配Sequence ID）
> 2. 消息服务 → 推送服务 → 接收方Gateway → 接收方
> 3. 接收方ACK确认，失败进入重试队列
>
> **群聊策略**：
>
> - 500人以下群：写扩散，发送时写每个成员收件箱
> - 500人以上群：读扩散，收件箱只存一条，拉取时扩散

### 2.3 短链系统

#### 面试题8: 设计一个短链接生成服务

**答题思路**

```
1. 功能需求: 长链转短链、短链跳转、过期管理、统计
2. 核心问题: 短码生成（不重复、不易预测）、高并发跳转
3. 存储设计: 短码→长链映射、访问统计
4. 优化: 缓存、预生成、跳转优化
```

**短码生成方案**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **自增ID+Base62** | 简单、无冲突 | 可预测、不安全 |
| **雪花算法** | 趋势递增、分布式 | 较长(11位) |
| **随机+数据库唯一** | 不可预测 | 冲突检测开销 |
| **预生成池** | 高性能 | 需要维护池 |

**参考答案**
> **短码生成**：采用预生成策略，服务启动时从数据库批量获取一批短码（如1000个），内存中分配，用完再申请。短码使用62进制编码（a-zA-Z0-9），6位可表示约568亿个URL。
>
> **跳转流程**：
>
> 1. 解析短码 → Redis查询长链（缓存）
> 2. 缓存命中：301重定向（永久）
> 3. 缓存未命中：数据库查询 → 回填缓存 → 重定向
>
> **优化点**：
>
> - 布隆过滤器防止缓存穿透
> - 热点URL本地缓存
> - 异步记录访问日志

### 2.4 推荐系统

#### 面试题9: 设计一个新闻推荐系统

**答题思路**

```
1. 推荐场景: 个性化、实时性、多样性
2. 系统架构: 召回 → 粗排 → 精排 → 重排
3. 技术选型: 向量检索、机器学习、在线服务
4. 冷启动: 新用户、新内容处理
```

**推荐系统架构**

```
用户请求 → 召回层(多路) → 粗排 → 精排(模型) → 重排 → 返回
              ↓              ↓        ↓         ↓
          协同过滤       轻量模型   深度模型   多样性/规则
          内容相似       简单特征   复杂特征   业务策略
          热门内容
```

**参考答案**
> **召回层**：多路召回策略
>
> - 协同过滤：基于用户相似度（UserCF）或物品相似度（ItemCF）
> - 向量召回：用户和内容Embedding，使用HNSW近似最近邻检索
> - 热门/趋势：保证新热内容曝光
>
> **排序层**：
>
> - 粗排：轻量LR模型，快速筛选千级候选
> - 精排：深度模型（Wide&Deep/DIN），预估点击率
> - 重排：多样性控制（MMR算法）、新鲜度加权、业务规则
>
> **冷启动**：
>
> - 新用户：基于人口统计推荐、热门内容
> - 新内容：内容Embedding、探索机制（Bandit）

### 2.5 分布式ID生成

#### 面试题10: 分布式系统中如何生成唯一ID？

**方案对比**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **UUID** | 简单、去中心化 | 长(36字符)、无序 | 日志追踪 |
| **雪花算法** | 趋势递增、高性能 | 依赖时钟 | 订单ID |
| **数据库自增** | 简单、有序 | 单点、性能低 | 小规模 |
| **号段模式** | 高性能、趋势递增 | 依赖数据库 | 高并发 |

**雪花算法详解**

```
64位ID结构:
0(1bit) | 时间戳(41bit) | 机器ID(10bit) | 序列号(12bit)

- 时间戳: 毫秒级，可用约69年
- 机器ID: 支持1024个节点
- 序列号: 每毫秒每节点可生成4096个ID
```

**参考答案**
> **推荐方案**：雪花算法改进版（美团Leaf、百度UidGenerator）
>
> **原因**：
>
> 1. **趋势递增**：数据库索引友好，避免页分裂
> 2. **高性能**：本地生成，无需网络IO
> 3. **可排序**：ID包含时间信息，便于归档
> 4. **分布式**：支持多节点同时生成
>
> **时钟回拨问题**：
>
> - 方案1：等待时钟追上
> - 方案2：使用最大漂移阈值
> - 方案3：百度UidGenerator的延迟启动

### 3.1 故障排查

#### 面试题11: 系统突然CPU飙升，如何排查？

**排查步骤**

```
1. 确认现象
   └─ top命令查看CPU占用高的进程

2. 定位线程
   └─ top -Hp <pid> 查看进程内各线程CPU占用

3. 分析堆栈
   └─ jstack <pid> > thread_dump.txt
   └─ 将线程ID转16进制，在堆栈中查找

4. 常见原因
   ├─ 死循环
   ├─ 频繁GC
   ├─ 正则表达式回溯
   ├─ 大量线程竞争
   └─ 计算密集型任务

5. 解决验证
   └─ 修复后观察，确认CPU恢复正常
```

**参考答案**
> **排查流程**：
>
> 1. 使用`top`找到CPU高的Java进程
> 2. `top -Hp pid`找到占用CPU高的线程，记录线程ID
> 3. 线程ID转16进制：`printf '%x\n' 线程ID`
> 4. `jstack pid | grep -A 20 16进制线程ID`查看堆栈
> 5. 根据堆栈定位问题代码
>
> **常见原因及解决**：
>
> - **Full GC频繁**：检查内存泄漏，调整堆大小
> - **死循环**：代码逻辑问题，修复循环条件
> - **正则回溯**：优化正则，使用预编译

#### 面试题12: 数据库查询突然变慢，可能是什么原因？

**排查清单**

| 层面 | 可能原因 | 检查方法 |
|------|----------|----------|
| **SQL** | 索引失效、大查询 | EXPLAIN分析执行计划 |
| **连接** | 连接池耗尽 | 监控活跃连接数 |
| **锁** | 行锁、表锁 | `show engine innodb status` |
| **硬件** | 磁盘IO高 | `iostat` |
| **数据** | 统计信息过期 | `ANALYZE TABLE` |
| **并发** | 突发流量 | QPS监控 |

**参考答案**
> **排查步骤**：
>
> 1. **慢查询日志**：找出执行时间长的SQL
> 2. **执行计划**：EXPLAIN查看是否走索引
> 3. **锁分析**：查看是否有锁等待
> 4. **资源监控**：检查CPU、IO、内存
>
> **常见原因**：
>
> - 索引失效：函数操作、类型转换、like '%xxx'
> - 数据倾斜：某些Key数据量过大
> - 锁竞争：长事务持有锁
> - 连接池满：连接泄漏或突增流量

### 3.2 性能优化

#### 面试题13: 接口响应时间从100ms变成1s，如何优化？

**优化思路**

```
1. 定位瓶颈
   ├─ 日志打点：各阶段耗时
   ├─ APM工具：链路追踪
   └─ 压测：确定性能基线

2. 常见优化
   ├─ 数据库: 加索引、优化SQL、分库分表
   ├─ 缓存: Redis、本地缓存
   ├─ 异步: 非关键逻辑异步化
   ├─ 并发: 并行调用多个服务
   └─ 代码: 算法优化、减少循环

3. 验证效果
   └─ 性能测试确认优化效果
```

**参考答案**
> **系统化优化方法**：
>
> 1. **添加监控**：在关键路径添加耗时日志，定位慢的阶段
> 2. **数据库优化**：
>    - 慢查询加索引
>    - 避免N+1查询，使用JOIN或IN批量查询
>    - 大表考虑分库分表
> 3. **引入缓存**：
>    - 热点数据Redis缓存
>    - 本地Caffeine缓存
> 4. **异步化**：
>    - 非核心逻辑发消息异步处理
>    - 使用CompletableFuture并行调用
> 5. **连接池优化**：调整数据库连接池、HTTP连接池大小

---

#### 面试题14: 如何优化MySQL大表查询性能？

**优化策略**

| 策略 | 适用场景 | 实施方法 |
|------|----------|----------|
| **索引优化** | 查询条件固定 | 覆盖索引、联合索引、索引下推 |
| **分库分表** | 单表超千万 | 按用户ID哈希分表 |
| **读写分离** | 读多写少 | 一主多从，查询走从库 |
| **归档** | 历史数据多 | 冷数据迁移到历史表 |
| **缓存** | 热点数据 | Redis缓存常用查询 |

**参考答案**
> **分阶段优化**：
>
> 1. **索引层面**：确保WHERE、ORDER BY、JOIN字段有索引，使用覆盖索引避免回表
> 2. **SQL层面**：避免SELECT *，避免大OFFSET分页，用JOIN替代子查询
> 3. **架构层面**：
>    - 读写分离：主库写，从库读
>    - 分库分表：水平拆分分散压力
>    - 数据归档：冷数据迁移，减少表大小
> 4. **缓存层面**：热点数据Redis缓存，减轻数据库压力

## 四、答题思路与模板

### 4.1 答题框架

#### 系统设计题答题模板

```
【1. 需求分析】(2分钟)
   - 功能性需求：核心功能是什么？
   - 非功能性需求：QPS？延迟？一致性要求？

【2. 流量估算】(1分钟)
   - DAU、峰值QPS、数据存储量

【3. 高层设计】(5分钟)
   - 画出系统架构图
   - 说明各组件职责

【4. 详细设计】(10分钟)
   - 核心流程：数据如何流动？
   - 数据模型：关键表结构
   - 关键问题：如何解决一致性、高并发等问题？

【5. 扩展讨论】(2分钟)
   - 如何扩展支持10倍增长？
   - 单点故障如何处理？
```

### 4.2 关键词技巧

#### 加分关键词

| 类别 | 关键词 | 使用场景 |
|------|--------|----------|
| **架构** | 微服务、事件驱动、CQRS、分层架构 | 描述系统架构 |
| **一致性** | 最终一致性、BASE、CAP、幂等性 | 讨论数据一致性 |
| **性能** | 缓存、异步、削峰、限流、降级 | 性能优化场景 |
| **可用性** | 熔断、舱壁、故障转移、多活 | 高可用设计 |
| **扩展** | 无状态、分片、复制、负载均衡 | 扩展性讨论 |

### 4.3 常见错误

| 错误类型 | 示例 | 改进建议 |
|----------|------|----------|
| **过早优化** | 一上来就谈分库分表 | 先估算规模，再决定是否拆分 |
| **忽视边界** | 不考虑网络分区 | 分布式系统必须考虑故障场景 |
| **过度设计** | 简单问题复杂化 | 根据实际需求选择方案 |
| **缺少量化** | 不说具体数字 | 给出估算：QPS、存储、延迟 |
| **忽视运维** | 只谈开发不谈部署 | 考虑监控、告警、回滚 |
