# 慢查询优化

## 核心概念

### 1. 慢查询定义

慢查询是指执行时间超过预设阈值的SQL语句，通常是性能问题的直接表现。

**慢查询标准**：

```
┌─────────────────────────────────────────────────────────────┐
│                   慢查询判定标准                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   MySQL默认阈值：                                            │
│   - long_query_time = 10秒                                 │
│   - 建议生产环境设置为1秒或更低                             │
│                                                             │
│   不同场景标准：                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  业务类型        慢查询阈值      说明                │  │
│   ├─────────────────────────────────────────────────────┤  │
│   │  OLTP交易       > 100ms       用户交互敏感         │  │
│   │  API接口        > 200ms       用户体验要求         │  │
│   │  报表查询       > 5s          可接受较长           │  │
│   │  数据分析       > 30s         批量处理容忍         │  │
│   │  定时任务       > 60s         非实时场景           │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 慢查询产生原因

**常见原因分类**：

| 类别 | 具体原因 | 占比 |
|------|----------|------|
| 索引问题 | 无索引、索引失效、索引选择不当 | 60% |
| 查询设计 | 大表扫描、全表排序、深度分页 | 20% |
| 数据问题 | 数据量过大、统计信息过期 | 10% |
| 架构问题 | 锁竞争、并发冲突 | 5% |
| 配置问题 | 缓存不足、连接数限制 | 5% |

**索引失效场景**：

```sql
-- 1. 隐式类型转换
SELECT * FROM users WHERE phone = 13800138000;  -- phone是字符串

-- 2. 函数操作列
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-01';

-- 3. 前导模糊匹配
SELECT * FROM users WHERE name LIKE '%张三%';

-- 4. OR条件使用不当
SELECT * FROM orders WHERE user_id = 100 OR status = 1;  -- status无索引

-- 5. 数据类型不匹配
SELECT * FROM products WHERE price = '100';  -- price是DECIMAL

-- 6. 索引列运算
SELECT * FROM orders WHERE user_id + 1 = 101;

-- 7. 不等于和NOT IN
SELECT * FROM users WHERE status != 1;  -- 可能走全表

-- 8. IS NULL判断（取决于数据分布）
SELECT * FROM users WHERE email IS NULL;
```

## 实践方法

### 慢查询分析流程

**1. 收集慢查询日志**：

```bash
# MySQL配置
[mysqld]
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = 1

# 查看当前设置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
```

**2. 分析慢查询**：

```bash
# 使用mysqldumpslow
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 参数说明
# -s t: 按总时间排序
# -s l: 按锁定时间排序
# -s r: 按返回行数排序
# -t 10: 显示前10条

# 使用pt-query-digest
pt-query-digest --type slowlog /var/log/mysql/slow.log > slow_report.txt

# 输出关键指标
# - Query ID: 查询指纹
# - Response time: 响应时间分布
# - Rows sent/examined: 返回/扫描行数
```

**3. 分析执行计划**：

```sql
-- 获取执行计划
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE user_id = 100;

-- 关键字段分析
-- select_type: SIMPLE/PRIMARY/SUBQUERY/DERIVED
-- type: system > const > eq_ref > ref > range > index > ALL
-- key: 实际使用的索引
-- rows: 预估扫描行数
-- filtered: 过滤比例
-- Extra: Using index/Using where/Using filesort/Using temporary
```

### 慢查询优化技巧

**1. 添加缺失索引**：

```sql
-- 问题查询
SELECT * FROM orders
WHERE user_id = 100 AND status = 1
ORDER BY created_at DESC;

-- 分析：
-- type: ALL (全表扫描)
-- rows: 1000000
-- Extra: Using where; Using filesort

-- 优化：添加复合索引
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);

-- 优化后：
-- type: ref
-- key: idx_user_status_created
-- rows: 100
-- Extra: Using index condition
```

**2. 优化LIMIT深分页**：

```sql
-- 问题查询（深分页）
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 1000000, 10;
-- 扫描1000010行，返回10行

-- 优化1：延迟关联
SELECT o.* FROM orders o
JOIN (
    SELECT id FROM orders
    ORDER BY created_at DESC
    LIMIT 1000000, 10
) tmp ON o.id = tmp.id;

-- 优化2：记录上次位置
SELECT * FROM orders
WHERE created_at < '2024-01-01 12:00:00'
ORDER BY created_at DESC
LIMIT 10;

-- 优化3：限制分页深度
-- 业务上限制只能查看前100页
```

**3. 优化排序和临时表**：

```sql
-- 问题查询（Using filesort + Using temporary）
SELECT status, COUNT(*), AVG(amount)
FROM orders
WHERE created_at > '2024-01-01'
GROUP BY status
ORDER BY COUNT(*) DESC;

-- 优化：添加覆盖索引
CREATE INDEX idx_created_status_amount ON orders(created_at, status, amount);

-- 优化后：
-- Extra: Using index（覆盖索引）
```

**4. 优化大表JOIN**：

```sql
-- 问题查询
SELECT o.*, u.name, u.email
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at > '2024-01-01';
-- 大表JOIN，可能产生笛卡尔积

-- 优化1：确保JOIN列有索引
CREATE INDEX idx_user_id ON orders(user_id);

-- 优化2：减少返回列
SELECT o.id, o.amount, u.name  -- 只选需要的列
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at > '2024-01-01';

-- 优化3：先过滤再JOIN
SELECT o.*, u.name, u.email
FROM (
    SELECT * FROM orders
    WHERE created_at > '2024-01-01'
) o
JOIN users u ON o.user_id = u.id;
```

**5. 优化子查询**：

```sql
-- 问题查询（相关子查询）
SELECT * FROM orders o
WHERE amount > (
    SELECT AVG(amount) FROM orders WHERE user_id = o.user_id
);

-- 优化：JOIN改写
SELECT o.*
FROM orders o
JOIN (
    SELECT user_id, AVG(amount) as avg_amount
    FROM orders
    GROUP BY user_id
) avg_table ON o.user_id = avg_table.user_id
WHERE o.amount > avg_table.avg_amount;

-- 或使用窗口函数（MySQL 8.0+）
SELECT * FROM (
    SELECT *,
           AVG(amount) OVER (PARTITION BY user_id) as avg_amount
    FROM orders
) t
WHERE amount > avg_amount;
```

### 慢查询监控系统

```go
// 慢查询告警
type SlowQueryMonitor struct {
    slowQueryCounter prometheus.Counter
    slowQueryDuration prometheus.Histogram
}

func (m *SlowQueryMonitor) Record(query string, duration time.Duration) {
    m.slowQueryDuration.Observe(duration.Seconds())

    if duration > 1*time.Second {
        m.slowQueryCounter.Inc()
        // 发送告警
        alert.Send(fmt.Sprintf("Slow query detected: %s, duration: %v", query, duration))
    }
}
```

## 案例数据

### 真实慢查询优化案例

**案例1：订单查询优化**

```sql
-- 优化前：3.5秒
SELECT COUNT(*), SUM(amount)
FROM orders
WHERE user_id IN (
    SELECT id FROM users WHERE created_at > '2024-01-01'
);

-- 优化后：50ms
SELECT COUNT(*), SUM(amount)
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.created_at > '2024-01-01';
-- 添加索引后性能提升70倍
```

**案例2：报表查询优化**

```sql
-- 优化前：超时（>30秒）
SELECT
    DATE(created_at),
    COUNT(*),
    SUM(amount)
FROM orders
WHERE status = 1
GROUP BY DATE(created_at);

-- 优化方案：
-- 1. 添加汇总表（定时更新）
CREATE TABLE order_daily_stats (
    stat_date DATE PRIMARY KEY,
    order_count BIGINT,
    total_amount DECIMAL(15,2)
);

-- 2. 查询汇总表（<10ms）
SELECT * FROM order_daily_stats;
```
