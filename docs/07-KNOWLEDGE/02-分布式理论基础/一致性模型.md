# 一致性模型

## 概述

**一致性模型（Consistency Model）** 是分布式系统中定义数据一致性保证级别的规则集合。它描述了在分布式系统中，多个副本之间的数据应该如何保持一致，以及读写操作应该满足什么约束。

一致性模型形成一个层次结构，从强一致性到弱一致性，在一致性保证和性能之间存在权衡。

## 一致性模型层次

```
线性一致性 (Linearizability)
    ↓
顺序一致性 (Sequential Consistency)
    ↓
因果一致性 (Causal Consistency)
    ↓
会话一致性 (Session Consistency)
    ↓
单调读/写一致性 (Monotonic Read/Write)
    ↓
最终一致性 (Eventual Consistency)
```

## 强一致性模型

### 1. 线性一致性（Linearizability）

**定义**：线性一致性要求所有操作看起来像是在单一副本上按某种顺序原子执行，且该顺序与操作的实际时间顺序一致。

**形式化定义**：
操作历史 $H$ 是线性一致的，如果存在一个等价于该历史的串行执行 $H'$，满足：

$$
\text{Linearizable}(H) \iff \exists H': \text{Equivalent}(H, H') \land \text{Serial}(H') \land \text{TimeOrder}(H')
$$

其中 $\text{TimeOrder}$ 要求如果操作 $op_1$ 在 $op_2$ 开始前完成，则 $op_1$ 在 $H'$ 中排在 $op_2$ 之前。

**线性化点（Linearization Point）**：
每个操作都有一个线性化点，即操作在逻辑上原子执行的时刻。

```python
# 线性一致性示例伪代码
class LinearizableStore:
    def __init__(self):
        self.data = {}
        self.lock = Lock()

    def write(self, key, value):
        with self.lock:  # 原子操作
            self.data[key] = value
            return success

    def read(self, key):
        with self.lock:  # 原子操作
            return self.data.get(key)
```

**实现方式**：

- 单主复制（Primary-Backup）
- 共识算法（Paxos、Raft）
- 两阶段提交（2PC）

**复杂度分析**：

- **时间复杂度**：$O(n)$（需要同步到大多数节点）
- **消息复杂度**：$O(n)$（每个操作）
- **延迟**：较高（需要等待确认）

### 2. 顺序一致性（Sequential Consistency）

**定义**：顺序一致性要求所有进程看到的操作顺序一致，且每个进程的操作顺序与其程序顺序一致，但不要求与实时顺序一致。

**形式化定义**：

$$
\text{SequentialConsistent}(H) \iff \exists H': \text{Equivalent}(H, H') \land \text{Serial}(H') \land \text{ProgramOrder}(H')
$$

**与线性一致性的区别**：

- 线性一致性：操作顺序必须与实时顺序一致
- 顺序一致性：操作顺序只需一致，可以与实时顺序不同

```
时间线示例：

进程A: W(x,1) ──────────────>
进程B: ─────── W(x,2) ──────>

线性一致性：必须确定W(x,1)和W(x,2)的先后顺序
顺序一致性：可以任意排序，只要所有进程看到相同的顺序
```

**实现方式**：

- 全局顺序广播
- 序列号分配器

## 弱一致性模型

### 3. 因果一致性（Causal Consistency）

**定义**：因果一致性保证因果相关的操作在所有进程上以相同的顺序被观察到，允许并发操作以不同顺序被观察到。

**形式化定义**：

$$
\text{CausalConsistent}(H) \iff \forall p, q: \text{CausalOrder}(op_p, op_q) \implies \text{ObservedOrder}(op_p, op_q)
$$

**happened-before关系**：

- 如果事件 $e_1$ happened-before $e_2$，则所有进程必须先看到 $e_1$，再看到 $e_2$
- 如果 $e_1$ 和 $e_2$ 并发（无因果关系），则不同进程可以按不同顺序看到它们

**实现方式**：

- 向量时钟（Vector Clock）
- 因果广播（Causal Broadcast）

```python
# 因果一致性示例伪代码
class CausallyConsistentStore:
    def __init__(self, node_id, num_nodes):
        self.node_id = node_id
        self.vc = [0] * num_nodes  # 向量时钟
        self.data = {}

    def write(self, key, value):
        self.vc[self.node_id] += 1
        msg = Message(key, value, self.vc.copy())
        broadcast(msg)

    def receive(self, msg):
        # 更新向量时钟
        for i in range(len(self.vc)):
            self.vc[i] = max(self.vc[i], msg.vc[i])
        self.data[msg.key] = msg.value
```

**复杂度分析**：

- **时间复杂度**：$O(n)$（向量时钟更新）
- **空间复杂度**：$O(n)$（向量时钟存储）

### 4. 会话一致性（Session Consistency）

**定义**：会话一致性保证在同一个会话（Session）内的操作满足特定的一致性要求，不同会话之间可以不一致。

**保证**：

- **读你所写（Read Your Writes）**：会话内写入的数据，后续读取一定能看到
- **单调读（Monotonic Reads）**：如果会话读取了某个值，后续读取不会看到更旧的值
- **单调写（Monotonic Writes）**：会话内的写入按顺序生效
- **写后读（Writes Follow Reads）**：如果会话读取了某个写入的数据，后续写入不会在该写入之前生效

**应用场景**：

- 用户会话管理
- 购物车系统
- 在线编辑器

### 5. 单调读一致性（Monotonic Reads）

**定义**：如果一个进程读取了某个数据项的值，后续读取不会看到该数据项的更早值。

**形式化定义**：

$$
\forall r_1, r_2: (r_1 \text{ before } r_2 \land \text{Value}(r_1) = v) \implies \neg(\text{Value}(r_2) < v)
$$

### 6. 单调写一致性（Monotonic Writes）

**定义**：一个进程的写入操作按顺序被其他进程观察到。

**形式化定义**：

$$
\forall w_1, w_2: (w_1 \text{ before } w_2) \implies \text{Observed}(w_1) \text{ before } \text{Observed}(w_2)
$$

## 最终一致性

### 7. 最终一致性（Eventual Consistency）

**定义**：系统是最终一致的，如果所有副本在没有新的更新时，最终会收敛到相同的值。

**形式化定义**：

$$
\text{EventualConsistent}(S) \iff \forall r_1, r_2: \lim_{t \to \infty} P(\text{Value}_{r_1}(t) = \text{Value}_{r_2}(t)) = 1
$$

**特点**：

- 不保证实时一致性
- 保证最终所有副本一致
- 高可用性和低延迟

**实现机制**：

- 异步复制
- 反熵（Anti-Entropy）协议
- Gossip协议
- 读修复（Read Repair）

```python
# 最终一致性示例伪代码
class EventuallyConsistentStore:
    def __init__(self):
        self.local_data = {}
        self.version_vectors = {}

    def write(self, key, value):
        version = self.version_vectors.get(key, 0) + 1
        self.version_vectors[key] = version
        self.local_data[key] = (value, version)
        # 异步传播到其他节点
        async_replicate(key, value, version)

    def read(self, key):
        # 可能返回旧值
        if key in self.local_data:
            return self.local_data[key][0]
        # 尝试从其他节点读取
        return read_from_replicas(key)

    def merge(self, key, remote_value, remote_version):
        # 冲突解决：Last-Write-Wins
        local_version = self.version_vectors.get(key, 0)
        if remote_version > local_version:
            self.local_data[key] = (remote_value, remote_version)
            self.version_vectors[key] = remote_version
```

## 一致性模型对比

| 一致性模型 | 一致性强度 | 延迟 | 吞吐量 | 可用性 | 复杂度 | 典型系统 |
|-----------|-----------|------|--------|--------|--------|---------|
| **线性一致性** | 最强 | 高 | 低 | 低 | 高 | etcd、ZooKeeper、Consul |
| **顺序一致性** | 强 | 高 | 中 | 中 | 高 | 某些分布式文件系统 |
| **因果一致性** | 中等 | 中 | 高 | 高 | 中 | Cassandra（可选）、COPS |
| **会话一致性** | 中等 | 低 | 高 | 高 | 低 | DynamoDB、MongoDB |
| **单调读/写** | 弱 | 低 | 高 | 高 | 低 | 许多NoSQL数据库 |
| **最终一致性** | 最弱 | 最低 | 最高 | 最高 | 最低 | Cassandra、Riak、Voldemort |

## 形式化定理

### 定理1：线性一致性的可组合性

**表述**：如果每个对象都是线性一致的，则整个系统是线性一致的。

$$
\forall obj \in \text{Objects}: \text{Linearizable}(obj) \implies \text{Linearizable}(\text{System})
$$

**证明概要**：

1. 对于每个对象，存在线性化点
2. 将所有操作的线性化点按时间排序
3. 构造全局线性化序列
4. 验证该序列满足线性一致性要求

### 定理2：一致性模型层次关系

**表述**：一致性模型形成一个层次结构，强一致性模型蕴含弱一致性模型。

$$
\text{Linearizability} \implies \text{SequentialConsistency} \implies \text{CausalConsistency} \implies \text{EventualConsistency}
$$

**传递性**：如果系统满足线性一致性，则它也满足顺序一致性、因果一致性和最终一致性。
