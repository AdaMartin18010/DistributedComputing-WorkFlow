# 读写分离实现

## 核心概念

### 1. 读写分离的必要性

读多写少是互联网应用的典型特征，读写分离可以将读压力分散到多个从库。

**应用场景**：

```
┌─────────────────────────────────────────────────────────────┐
│                   读写分离应用场景                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   典型读多写少场景：                                         │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  场景            读:写比例    是否适合读写分离        │  │
│   ├─────────────────────────────────────────────────────┤  │
│   │  新闻/博客        100:1      ✓ 非常适合              │  │
│   │  电商商品         50:1       ✓ 适合                  │  │
│   │  社交网络         20:1       ✓ 适合                  │  │
│   │  订单系统         5:1        △ 可以考虑              │  │
│   │  支付系统         1:1        ✗ 不适合                │  │
│   │  库存系统         1:10       ✗ 不适合                │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   核心收益：                                                 │
│   1. 扩展读性能：一主多从，线性扩展读QPS                    │
│   2. 高可用：主库故障可快速切换到从库                        │
│   3. 备份分离：从库承担备份任务，不影响主库                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 主从复制原理

**MySQL主从复制流程**：

```
┌─────────────────────────────────────────────────────────────┐
│                   MySQL主从复制流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Master                    Slave                           │
│      │                         │                            │
│   1. 写操作                    │                            │
│      │                         │                            │
│   2. 写binlog                  │                            │
│      │                         │                            │
│   3. Dump线程 ────────┐        │                            │
│      │                │        │                            │
│   4. 发送binlog ─────→│        │                            │
│                       ↓        │                            │
│              ┌──────────────┐  │                            │
│              │  Relay Log   │  │                            │
│              │  (中继日志)   │  │                            │
│              └──────┬───────┘  │                            │
│                     │          │                            │
│   5. IO线程写入 ────┘          │                            │
│                                │                            │
│                     ┌──────────┘                            │
│                     ↓                                       │
│              ┌──────────────┐                               │
│              │  SQL线程回放  │                               │
│              │  重放binlog   │                               │
│              └──────────────┘                               │
│                                                             │
│   复制模式：                                                │
│   - 异步复制：Master不等待Slave确认，性能最好，可能丢数据   │
│   - 半同步复制：Master等待至少一个Slave确认，平衡性能和安全 │
│   - 全同步复制：Master等待所有Slave确认，最安全，性能最差   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**复制延迟问题**：

```
延迟来源：
1. 网络传输延迟
2. 从库IO线程写入延迟
3. 从库SQL线程重放延迟（单线程）

监控延迟：
Seconds_Behind_Master: 显示从库落后主库的秒数

缓解措施：
1. 并行复制（MySQL 5.6+）
2. 读写分离路由策略（延迟阈值）
3. 强制走主库（对一致性要求高的查询）
```

### 3. 读写分离架构

**常见架构模式**：

| 模式 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 代理模式 | 通过代理层路由 | 对应用透明 | 增加网络跳数 | 大多数场景 |
| 客户端模式 | 客户端选择数据源 | 性能最好 | 代码侵入 | 高性能要求 |
| 中间件模式 | 独立中间件 | 灵活可控 | 部署复杂 | 大型系统 |

**代理模式架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                   代理模式读写分离                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Application                                               │
│      │                                                      │
│      ▼                                                      │
│   ┌─────────────────────────────────────────────────────┐  │
│   │           Database Proxy (MyCat/ShardingSphere)     │  │
│   │                                                     │  │
│   │   ┌─────────────────────────────────────────────┐  │  │
│   │   │  SQL Parser → 路由判断 → 读写分离           │  │  │
│   │   │                                             │  │  │
│   │   │  INSERT/UPDATE/DELETE ──→ Master          │  │  │
│   │   │  SELECT ──→ Slave1/Slave2/Slave3 (负载均衡)│  │  │
│   │   └─────────────────────────────────────────────┘  │  │
│   └─────────────────────────────────────────────────────┘  │
│      │                    │                                │
│      ▼                    ▼                                │
│   ┌─────────┐         ┌─────────┐                         │
│   │ Master  │ ←──────→│ Slave 1 │                         │
│   │ (写)    │  复制   │  (读)   │                         │
│   └─────────┘         ├─────────┤                         │
│                       │ Slave 2 │                         │
│                       │  (读)   │                         │
│                       ├─────────┤                         │
│                       │ Slave 3 │                         │
│                       │  (读)   │                         │
│                       └─────────┘                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 实践方法

### 基于ShardingSphere的读写分离

**1. 配置文件**：

```yaml
spring:
  shardingsphere:
    datasource:
      names: master, slave0, slave1
      master:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://master:3306/mydb
        username: root
        password: xxx
      slave0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://slave0:3306/mydb
        username: root
        password: xxx
      slave1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://slave1:3306/mydb
        username: root
        password: xxx

    rules:
      readwrite-splitting:
        data-sources:
          ds:
            type: Static
            props:
              write-data-source-name: master
              read-data-source-names: slave0, slave1
            load-balancer-name: round-robin

        load-balancers:
          round-robin:
            type: ROUND_ROBIN
          random:
            type: RANDOM
          weight:
            type: WEIGHT
            props:
              slave0: 2
              slave1: 1
```

**2. 强制走主库**：

```java
// ShardingSphere Hint强制路由
HintManager hintManager = HintManager.getInstance();
hintManager.setWriteRouteOnly();  // 强制走主库

try {
    // 这里的查询会路由到主库
    Order order = orderDao.findById(orderId);
} finally {
    hintManager.close();
}
```

### 基于Spring的读写分离

**1. 动态数据源配置**：

```java
@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource routingDataSource(
            @Qualifier("masterDataSource") DataSource master,
            @Qualifier("slaveDataSource") DataSource slave) {

        DynamicRoutingDataSource routingDataSource = new DynamicRoutingDataSource();

        Map<Object, Object> targetDataSources = new HashMap<>();
        targetDataSources.put("master", master);
        targetDataSources.put("slave", slave);

        routingDataSource.setTargetDataSources(targetDataSources);
        routingDataSource.setDefaultTargetDataSource(master);

        return routingDataSource;
    }
}

// 动态数据源上下文
public class DataSourceContext {
    private static final ThreadLocal<String> context = new ThreadLocal<>();

    public static void setMaster() {
        context.set("master");
    }

    public static void setSlave() {
        context.set("slave");
    }

    public static String get() {
        return context.get();
    }

    public static void clear() {
        context.remove();
    }
}

// 动态数据源实现
public class DynamicRoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContext.get();
    }
}
```

**2. AOP自动切换**：

```java
@Aspect
@Component
public class DataSourceAspect {

    @Before("@annotation(master)")
    public void setMaster(Master master) {
        DataSourceContext.setMaster();
    }

    @Before("@annotation(slave)")
    public void setSlave(Slave slave) {
        DataSourceContext.setSlave();
    }

    @After("@annotation(master) || @annotation(slave)")
    public void clear() {
        DataSourceContext.clear();
    }
}

// 注解定义
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Master {}

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Slave {}

// 使用
@Service
public class OrderService {

    @Master  // 写操作走主库
    public void createOrder(Order order) {
        orderDao.insert(order);
    }

    @Slave   // 读操作走从库
    public Order getOrder(Long id) {
        return orderDao.findById(id);
    }

    @Master  // 对一致性要求高的读也走主库
    public Order getOrderForUpdate(Long id) {
        return orderDao.findById(id);
    }
}
```

### 延迟敏感处理

**1. 延迟检测与路由**：

```java
@Service
public class SmartRoutingService {

    @Autowired
    private ReplicationLagMonitor lagMonitor;

    public <T> T executeWithConsistency(Function<String, T> operation,
                                         long maxLagMs) {
        // 检查复制延迟
        long currentLag = lagMonitor.getReplicationLag();

        if (currentLag > maxLagMs) {
            // 延迟超过阈值，强制走主库
            DataSourceContext.setMaster();
            log.warn("Replication lag {}ms exceeds threshold {}ms, routing to master",
                currentLag, maxLagMs);
        } else {
            DataSourceContext.setSlave();
        }

        try {
            return operation.apply(DataSourceContext.get());
        } finally {
            DataSourceContext.clear();
        }
    }
}

// 使用
Order order = smartRoutingService.executeWithConsistency(
    ds -> orderDao.findById(orderId),
    1000  // 最多容忍1秒延迟
);
```

**2. 事务内走主库**：

```java
@Transactional
public void updateOrder(Long orderId) {
    // 事务内所有操作自动走主库
    // 因为事务需要读写一致性
    Order order = orderDao.findById(orderId);  // 走主库
    order.setStatus(1);
    orderDao.update(order);  // 走主库
}
```

## 案例数据

### 新闻网站读写分离

**场景**：新闻内容读多写少，高峰10万QPS

**架构**：

```
1主3从
- 主库：负责写操作（编辑发布）
- 从库1：用户阅读（60%流量）
- 从库2：搜索索引（20%流量）
- 从库3：数据分析（20%流量）
```

**效果**：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 读QPS | 2万 | 10万 |
| 写QPS | 100 | 100 |
| 主库CPU | 95% | 30% |
| 响应延迟 | 500ms | 20ms |

## 优化前后对比

| 指标 | 单库 | 读写分离(1主3从) | 改进 |
|------|------|------------------|------|
| 读能力 | 2万QPS | 10万QPS | 5倍 |
| 主库负载 | 95% | 30% | 68%↓ |
| 可用性 | 单点 | 高可用 | 显著提升 |

---

**总结**：读写分离是提升数据库读性能的有效手段。实现时需要关注主从延迟问题，合理设计路由策略，对一致性要求高的查询提供强制走主库的能力。
