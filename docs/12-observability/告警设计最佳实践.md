# 告警设计最佳实践

## 概述

告警是可观测性体系的"最后一公里"，设计良好的告警能够在系统异常时及时通知相关人员，同时避免告警疲劳。本文从告警分级、告警规则设计、通知渠道等方面，总结告警设计的最佳实践。

## 告警分级配置

```yaml
# alert-levels.yaml
alert_levels:
  p0_critical:
    name: "P0-紧急"
    description: "服务不可用或核心功能受损"
    criteria:
      - 服务完全不可用
      - 数据丢失风险
      - 安全事件
      - 核心支付/交易失败
    response_time: "5分钟"
    notification_channels:
      - phone_call
      - sms
      - pagerduty
      - wechat_work
    escalation:
      enabled: true
      intervals: [5m, 10m, 15m]
      escalate_to: ["team_lead", "manager", "director"]
    auto_resolve: false
    
  p1_high:
    name: "P1-重要"
    description: "主要功能受损，存在降级方案"
    criteria:
      - 服务性能严重下降(P99>2s)
      - 错误率显著升高(>5%)
      - 非核心功能不可用
      - 容量告警(>80%)
    response_time: "30分钟"
    notification_channels:
      - wechat_work
      - slack
      - email
    working_hours_only: false
    auto_resolve: true
    resolve_after: 30m
    
  p2_medium:
    name: "P2-一般"
    description: "次要问题，不影响核心功能"
    criteria:
      - 服务性能轻微下降
      - 错误率轻微升高(1-5%)
      - 单节点故障(多副本)
      - 非关键任务失败
    response_time: "4小时"
    notification_channels:
      - email
      - ticketing_system
    working_hours_only: true
    auto_resolve: true

      - 安全补丁提示
      - 容量规划提醒
      - 版本升级建议
    response_time: "24小时"
    notification_channels:
      - weekly_digest
```

## 告警规则设计

### 黄金信号告警

```yaml
# golden-signals-alerts.yaml
groups:
  - name: golden_signals
    interval: 1m
    rules:
      # P0: 服务宕机
      - alert: ServiceDown
        expr: up{job=~".*-service"} == 0
        for: 1m
        labels:
          severity: p0
          team: sre
        annotations:
          summary: "服务 {{ $labels.job }} 宕机"
          description: "服务已不可用超过1分钟"
          runbook_url: "https://wiki/runbooks/service-down"
          
      # P0: 错误率激增
      - alert: HighErrorRate
        expr: |
          sum(rate(http_requests_total{status=~"5.."}[5m])) by (service) 
          / 
          sum(rate(http_requests_total[5m])) by (service) 
          > 0.1
        for: 2m
        labels:
          severity: p0
          team: sre
        annotations:
          summary: "{{ $labels.service }} 错误率过高"
          description: "错误率: {{ $value | humanizePercentage }}"
          
      # P1: 高延迟
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99, 
            sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service)
          ) > 1
        for: 5m
        labels:
          severity: p1
          team: sre
        annotations:
          summary: "{{ $labels.service }} P99延迟过高"
          description: "P99延迟: {{ $value }}s"
          
      # P1: 流量异常
      - alert: TrafficDrop
        expr: |
          sum(rate(http_requests_total[5m])) by (service) 
          < 
          0.5 * avg_over_time(sum(rate(http_requests_total[5m])) by (service)[1h:5m])
        for: 5m
        labels:
          severity: p1
          team: sre
        annotations:
          summary: "{{ $labels.service }} 流量下降50%"
          
      # P2: 饱和度预警
      - alert: HighSaturation
        expr: |
          (1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) 
          > 0.85
        for: 10m
        labels:
          severity: p2
          team: infrastructure
        annotations:
          summary: "节点 {{ $labels.instance }} 内存使用率过高"
          description: "内存使用率: {{ $value | humanizePercentage }}"
```

### SLO告警

```yaml
# slo-alerts.yaml
groups:
  - name: slo_alerts
    interval: 5m
    rules:
      # 快速消耗错误预算告警
      - alert: SLOErrorBudgetBurnRate
        expr: |
          (
            sum(rate(http_requests_total{status=~"5.."}[1h]))
            /
            sum(rate(http_requests_total[1h]))
          ) 
          > 
          (14.4 * (1 - 0.999))  # 1小时窗口消耗2%预算
        for: 2m
        labels:
          severity: p0
          slo: availability
        annotations:
          summary: "快速消耗错误预算"
          description: "1小时内错误率{{ $value | humanizePercentage }}，超过快速燃烧阈值"
          
      # 慢速消耗错误预算告警
      - alert: SLOErrorBudgetSlowBurn
        expr: |
          (
            sum(rate(http_requests_total{status=~"5.."}[6h]))
            /
            sum(rate(http_requests_total[6h]))
          ) 
          > 
          (6 * (1 - 0.999))  # 6小时窗口消耗5%预算
        for: 15m
        labels:
          severity: p1
          slo: availability
        annotations:
          summary: "慢速消耗错误预算"
          description: "6小时内错误率持续高于预期"
```

## 告警降噪策略

```yaml
# alert-noise-reduction.yaml
noise_reduction:
  # 1. 时间聚合
  time_aggregation:
    description: "相同告警在一定时间内只发送一次"
    group_by: [alertname, service]
    group_wait: 30s
    group_interval: 5m
    repeat_interval: 12h
    
  # 2. 抑制规则
  inhibition:
    rules:
      # 高层级告警抑制低层级
      - source_match:
          severity: p0
        target_match:
          severity: p1
        equal: [service, instance]
        
      # 宕机告警抑制其他该服务的告警
      - source_match:
          alertname: ServiceDown
        target_match_re:
          alertname: .*
        equal: [service]
        
  # 3. 依赖抑制
  dependency_inhibition:
    rules:
      # 如果数据库宕机，抑制所有依赖该数据库的服务告警
      - source_match:
          alertname: DatabaseDown
        target_match:
          dependency: database
        
  # 4. 维护窗口
  maintenance_windows:
    enabled: true
    check_interval: 1m
    default_silence_duration: 1h
    
  # 5. 告警阈值调优
  threshold_tuning:
    # 使用历史数据计算动态阈值
    method: percentile
    percentile: 99.9
    lookback: 7d
    min_threshold: 0.1
```

## 最佳实践

1. **分级明确**：P0/P1/P2/P3四级分类，响应时间和渠道清晰
2. **降噪优先**：通过抑制、聚合减少无效告警
3. **可行动**：每个告警都应有明确的处理手册(runbook)
4. **自动化**：自动恢复、自动扩容减少人工介入
5. **持续优化**：定期回顾告警，调整阈值和规则
