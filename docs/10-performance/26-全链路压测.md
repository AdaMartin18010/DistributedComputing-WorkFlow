# 全链路压测

## 核心概念

### 1. 全链路压测定义

全链路压测是在生产环境或准生产环境进行的端到端压力测试，模拟真实用户行为，验证整个系统的承载能力。

**与传统压测的区别**：

| 维度 | 单模块压测 | 全链路压测 |
|------|-----------|-----------|
| 范围 | 单个服务 | 完整链路 |
| 环境 | 测试环境 | 生产/影子环境 |
| 数据 | 测试数据 | 真实数据/影子数据 |
| 覆盖 | 代码路径 | 完整业务场景 |
| 风险 | 低 | 需要严格隔离 |
| 价值 | 有限 | 最真实 |

## 实践方法

### 流量染色与隔离

**1. 流量染色**：

```java
@Component
public class ShadowTrafficFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain) {

        // 识别压测流量
        boolean isShadow = "true".equals(request.getHeader("X-Shadow-Request"));
        ShadowContext.setShadow(isShadow);

        try {
            chain.doFilter(request, response);
        } finally {
            ShadowContext.clear();
        }
    }
}

// 上下文传递
public class ShadowContext {
    private static final ThreadLocal<Boolean> context = new ThreadLocal<>();

    public static void setShadow(boolean isShadow) {
        context.set(isShadow);
    }

    public static boolean isShadow() {
        return Boolean.TRUE.equals(context.get());
    }

    public static void clear() {
        context.remove();
    }
}
```

**2. 数据路由**：

```java
@Component
public class ShadowDataSourceRouter extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        return ShadowContext.isShadow() ? "shadow" : "master";
    }
}

@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource routingDataSource(
            @Qualifier("masterDataSource") DataSource master,
            @Qualifier("shadowDataSource") DataSource shadow) {

        ShadowDataSourceRouter router = new ShadowDataSourceRouter();
        Map<Object, Object> targetDataSources = new HashMap<>();
        targetDataSources.put("master", master);
        targetDataSources.put("shadow", shadow);

        router.setTargetDataSources(targetDataSources);
        router.setDefaultTargetDataSource(master);

        return router;
    }
}
```

**3. 影子表策略**：


### 压测方案设计

**1. 压测模型**：

```yaml
# 全链路压测配置
shadow_test:
  # 影子用户
  users:
    count: 1000000
    prefix: shadow_user_

  # 影子商品
  products:
    count: 100000
    prefix: shadow_product_

  # 压测场景
  scenarios:
    - name: browse_flow
      weight: 60
      steps:
        - login
        - browse_products
        - view_detail

    - name: order_flow
      weight: 30
      steps:
        - login
        - add_cart
        - submit_order
        - pay

    - name: query_flow
      weight: 10
      steps:
        - login
        - query_orders
        - query_coupons
```

**2. 压测执行**：

```java
@Service
public class ShadowLoadTestService {

    public void executeLoadTest(LoadTestPlan plan) {
        // 1. 准备影子数据
        prepareShadowData(plan.getUsers(), plan.getProducts());

        // 2. 预热系统
        warmUp();

        // 3. 执行压测
        for (Stage stage : plan.getStages()) {
            executeStage(stage);
        }

        // 4. 清理数据
        cleanupShadowData();
    }

    private void executeStage(Stage stage) {
        // 控制压测流量
        for (int i = 0; i < stage.getDurationMinutes(); i++) {
            int qps = stage.getTargetQps();
            generateTraffic(qps);

            // 监控指标
            collectMetrics();

            // 异常检测
            if (detectAnomaly()) {
                alertAndPause();
            }
        }
    }
}
```

## 案例数据

### 电商全链路压测案例

**压测规模**：

```
压测流量：日常峰值的2倍
用户量：100万影子用户
持续时间：72小时
场景覆盖：浏览、搜索、下单、支付、售后
```

**发现的问题**：

```
1. 数据库瓶颈
   - 现象：订单库CPU 100%
   - 根因：库存扣减热点
   - 优化：分段锁 + 缓存预扣

2. 缓存雪崩
   - 现象：Redis偶发超时
   - 根因：批量key同时过期
   - 优化：随机过期时间

3. 下游限流
   - 现象：支付接口大量429
   - 根因：未做熔断降级
   - 优化：添加熔断器

4. 日志磁盘满
   - 现象：部分节点磁盘100%
   - 根因：日志未清理
   - 优化：自动归档策略
```

**优化效果**：

```
压测前承载：5万QPS
压测后承载：15万QPS
系统稳定性：99.99%
故障恢复时间：5分钟 → 30秒
```

## 优化前后对比

| 指标 | 压测前 | 压测后 | 改进 |
|------|--------|--------|------|
| 承载能力 | 5万QPS | 15万QPS | 3倍↑ |
| 问题发现数 | 上线后暴露 | 压测中发现 | 提前 |
| 系统稳定性 | 99.9% | 99.99% | 10倍↑ |
| 大促信心 | 低 | 高 | 显著提升 |

---

**总结**：全链路压测是保障大促稳定的最有效手段。通过流量染色、数据隔离、真实场景模拟，可以在不影响生产的前提下，全面验证系统承载能力，提前发现并解决潜在问题。
