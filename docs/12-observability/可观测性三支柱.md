# 可观测性三支柱

## 概述

可观测性（Observability）是指通过系统的外部输出推断其内部状态的能力。在分布式系统中，可观测性由三个核心支柱构成：**指标（Metrics）**、**日志（Logs）** 和 **追踪（Traces）**。这三个维度相互补充，共同构建了对复杂系统的全面理解。

## 三支柱详解

### 1. 指标（Metrics）

指标是随时间聚合的数值数据，用于描述系统的状态和性能特征。

**核心特性：**

- **时序性**：按时间序列存储，支持趋势分析
- **聚合性**：可汇总统计，适合大规模监控
- **高效性**：存储成本低，查询速度快

**指标类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| Counter | 单调递增的计数器 | 请求总数、错误数 |
| Gauge | 可增可减的瞬时值 | CPU使用率、内存占用 |
| Histogram | 采样分布统计 | 请求延迟分布 |
| Summary | 分位数统计 | P99延迟 |

**配置示例：**

```yaml
# Prometheus指标配置
metrics:
  enabled: true
  path: /metrics
  port: 9090

  # 自定义指标定义
  custom_metrics:
    - name: http_requests_total
      type: counter
      labels: [method, status, endpoint]
      help: "Total HTTP requests"

    - name: http_request_duration_seconds
      type: histogram
      labels: [method, endpoint]
      buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]
      help: "HTTP request latency"

    - name: active_connections
      type: gauge
      help: "Current active connections"
```

### 2. 日志（Logs）

日志是系统运行过程中产生的离散事件记录，包含时间戳和上下文信息。

**日志级别：**

- **DEBUG**：详细的调试信息
- **INFO**：常规运行信息
- **WARN**：警告信息，需要关注
- **ERROR**：错误信息，需要处理
- **FATAL**：致命错误，系统异常

**结构化日志：**

```yaml
# 日志配置
logging:
  level: INFO
  format: json
  output: stdout

  # 结构化字段
  fields:
    service: order-service
    version: v1.2.0
    environment: production

  # 采样配置
  sampling:
    enabled: true
    initial: 100
    thereafter: 100

  # 日志轮转
  rotation:
    max_size: 100MB
    max_age: 7d
    max_backups: 10
    compress: true
```

### 3. 追踪（Traces）

追踪记录了请求在分布式系统中的完整调用链路，帮助理解服务间依赖关系。

**核心概念：**

- **Trace**：一次完整请求的调用链
- **Span**：追踪中的单个操作单元
- **Span Context**：传播的上下文信息

**追踪配置：**

```yaml
# OpenTelemetry追踪配置
tracing:
  enabled: true
  service_name: payment-service

  exporter:
    type: otlp
    endpoint: http://otel-collector:4317
    timeout: 10s

  sampling:
    strategy: parentbased_traceidratio
    ratio: 0.1

  propagators:
    - tracecontext
    - baggage

  resource:
    attributes:
      deployment.environment: production
      host.name: ${HOSTNAME}
      service.namespace: ecommerce
```

## 数据关联策略

### TraceID关联

所有日志和指标都应携带TraceID，实现跨支柱关联：

```yaml
# 日志中注入TraceID
log_format: '{"time":"%t","level":"%l","trace_id":"%X{trace_id}","message":"%m"}'

# 指标中注入TraceID标签（可选）
metrics_labels:
  - trace_id  # 高基数，谨慎使用
```

### 时间窗口对齐

三者数据应使用统一的时间戳和时区：

```yaml
# 统一时间配置
timestamp:
  format: RFC3339Nano
  timezone: UTC

# 时间同步
ntp:
  enabled: true
  servers:
    - time.google.com
    - time.cloudflare.com
```

## 最佳实践

1. **指标优先**：能用指标解决的问题，不要用日志
2. **日志结构化**：统一JSON格式，便于解析和查询
3. **追踪采样**：生产环境使用自适应采样，控制存储成本
4. **统一标签**：服务名、环境、版本等标签保持一致
5. **关联打通**：TraceID贯穿三者，实现全链路可观测

## 总结

可观测性三支柱是相互补充的整体：

- **指标**告诉你系统"发生了什么"
- **日志**告诉你"为什么会发生"
- **追踪**告诉你"发生在哪里、如何传播"

只有三者协同工作，才能构建完整的可观测性体系，有效支撑分布式系统的运维和故障排查。
