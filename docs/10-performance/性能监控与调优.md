## 一、核心概念

### 1.1 定义与原理

**性能监控**是持续收集、存储和分析系统运行指标的过程。**性能调优**则是基于监控数据，识别瓶颈并优化系统配置和代码的过程。

**黄金信号**（Google SRE）：

- **延迟**（Latency）：服务响应时间
- **流量**（Traffic）：系统请求量
- **错误**（Errors）：失败请求比例
- **饱和度**（Saturation）：资源使用接近上限的程度

### 1.2 关键特性

- **可观测性**：Metrics、Logs、Traces三大支柱
- **实时性**：秒级甚至毫秒级指标采集
- **可追溯**：端到端请求链路追踪
- **可预测**：基于历史数据预测容量需求

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 线上故障排查 | ⭐⭐⭐⭐⭐ | 快速定位性能瓶颈 |
| 容量规划 | ⭐⭐⭐⭐⭐ | 预测资源需求，避免过载 |
| 版本发布评估 | ⭐⭐⭐⭐⭐ | 对比前后性能差异 |
| 持续优化 | ⭐⭐⭐⭐⭐ | 建立性能基线，持续改进 |
| 告警响应 | ⭐⭐⭐⭐ | 异常检测和自动恢复 |

## 二、技术细节

### 2.1 性能指标体系

#### 核心指标定义

| 指标 | 定义 | 计算公式 | 典型目标 |
|------|------|----------|----------|
| **QPS** | 每秒查询数 | 总请求数/时间 | 因业务而异 |
| **TPS** | 每秒事务数 | 总事务数/时间 | 因业务而异 |
| **RT** | 响应时间 | 请求处理耗时 | P99 < 100ms |
| **并发数** | 同时处理请求数 | 活跃连接数 | < 最大连接数 |
| **吞吐量** | 单位时间处理数据量 | 数据量/时间 | 最大化 |
| **错误率** | 失败请求占比 | 错误数/总数 | < 0.1% |
| **CPU利用率** | CPU使用时间占比 | 使用/总量 | 60%-80% |
| **内存使用率** | 内存使用占比 | 使用/总量 | < 80% |
| **IOPS** | 每秒IO操作数 | IO次数/时间 | 视存储而定 |
| **带宽** | 网络传输速率 | 数据量/时间 | < 80%容量 |

#### 百分位指标

```
响应时间分布示意：

0ms    P50    P90    P99    P999   max
│      │      │      │      │      │
████████░░░░░░░░░░░░▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓
↑      ↑      ↑      ↑      ↑      ↑
50%    50%    10%    1%     0.1%   最大
请求   请求   请求   请求   请求   请求
<50ms  <100ms <200ms <500ms <1s    5s

P99.9的重要性：
- 100万请求中，999,000个<1s
- 只有1,000个请求>1s
- 但这1000个用户可能有糟糕体验
```

**百分位计算**：

```java
// HDR Histogram实现
Histogram histogram = new Histogram(3600000000000L, 3);

// 记录请求延迟
histogram.recordValue(responseTimeNanos);

// 获取百分位
long p50 = histogram.getValueAtPercentile(50);
long p99 = histogram.getValueAtPercentile(99);
long p999 = histogram.getValueAtPercentile(99.9);
```

### 2.2 瓶颈分析方法

#### USE方法

```
USE方法：Utilization（使用率）、Saturation（饱和度）、Errors（错误）

系统资源检查清单：
┌─────────────────────────────────────────────────────────────┐
│ 资源类型    │ 使用率        │ 饱和度           │ 错误       │
├─────────────────────────────────────────────────────────────┤
│ CPU        │ cpu_util     │ run_queue_length │ -          │
│ Memory     │ mem_util     │ swap_usage       │ oom_kills  │
│ Disk       │ disk_util    │ io_wait          │ io_errors  │
│ Network    │ bandwidth    │ buffer_usage     │ drops      │
│ File Des.  │ fd_count     │ -                │ -          │
└─────────────────────────────────────────────────────────────┘

诊断流程：
1. 检查所有资源的使用率
2. 检查饱和度指标
3. 检查错误计数
4. 识别最紧张的资源 → 即为瓶颈
```

#### 分布式追踪

```
分布式请求链路：

[User] ──→ [API Gateway] ──→ [Service A] ──→ [Service B] ──→ [DB]
   │            │                 │               │             │
   │         15ms              25ms            30ms          5ms
   │            │                 │               │             │
   └────────────┴─────────────────┴───────────────┴─────────────┘
   │
   └── 总延迟：75ms (忽略网络开销)

Trace ID: abc123
├─ Span 1: API Gateway (15ms)
│  ├─ Span 2: Service A (25ms)
│  │  ├─ Span 3: Service B (30ms)
│  │  │  └─ Span 4: DB Query (5ms)
│  │  └─ Span 5: Redis Cache (2ms)
│  └─ Span 6: Auth Service (8ms)
└─ Span 7: Logging Service (1ms)

关键指标：
- 关键路径：1 → 2 → 3 → 4 (影响总延迟)
- 并行度：Span 6与Span 2并行执行
- 长尾分析：Span 3的P99延迟
```

**OpenTelemetry实现**：

```java
// 创建Tracer
Tracer tracer = GlobalOpenTelemetry.getTracer("my-service");

// 创建Span
Span span = tracer.spanBuilder("processOrder")

    } finally {
        dbSpan.end();
    }

    // 外部调用
    Span httpSpan = tracer.spanBuilder("payment.charge").startSpan();
    try {
        paymentClient.charge(order);
    } finally {
        httpSpan.end();
    }
} finally {
    span.end();
}
```

### 2.3 压测方法论

#### 压测类型

| 类型 | 目的 | 负载模式 | 持续时间 |
|------|------|----------|----------|
| 负载测试 | 验证系统在正常负载下表现 | 逐步增加 | 稳定期30min+ |
| 压力测试 | 找到系统崩溃点 | 持续加压 | 直到故障 |
| 稳定性测试 | 验证长期运行稳定性 | 正常负载 | 24h+ |
| 峰值测试 | 验证突发流量处理能力 | 瞬间高峰 | 短脉冲 |
| 容量测试 | 确定最大承载能力 | 梯度加压 | 每级10min |

#### 压测执行流程

```
压测流程：

1. 目标定义
   └── 目标QPS: 100,000
   └── 目标P99延迟: < 50ms
   └── 错误率: < 0.01%

2. 环境准备
   ├── 压测环境（与生产1:1或按比例）
   ├── 监控部署（全链路可观测）
   ├── 数据准备（脱敏的生产数据）
   └── 熔断预案（保护生产系统）

3. 基线测试
   └── 当前系统性能基线记录
   └── 单接口性能摸底

4. 阶梯加压
   ├── 25% 负载 (25,000 QPS)
   ├── 50% 负载 (50,000 QPS)
   ├── 75% 负载 (75,000 QPS)
   ├── 100% 负载 (100,000 QPS)
   └── 120% 负载 (压力测试)

5. 瓶颈分析
   └── CPU/Memory/IO/Network
   └── 代码热点分析
   └── 配置参数优化

6. 调优验证
   └── 优化后重新压测
   └── 对比前后指标
```

```go
// k6压测脚本
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },   //  ramp up
    { duration: '5m', target: 100 },   //  stable
    { duration: '2m', target: 200 },   //  ramp up
    { duration: '5m', target: 200 },   //  stable
    { duration: '2m', target: 0 },     //  ramp down
  ],
  thresholds: {
    http_req_duration: ['p(99)<50'],   // P99 < 50ms
    http_req_failed: ['rate<0.01'],    // 错误率 < 1%
  },
};

export default function () {
  const res = http.post('https://api.example.com/v1/order', {
    product_id: getRandomProduct(),
    quantity: Math.floor(Math.random() * 10) + 1,
  });

  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time < 50ms': (r) => r.timings.duration < 50,
  });

  sleep(1);
}
```

```java
// Micrometer + Prometheus
@Configuration
public class MetricsConfig {

    @Bean
    MeterRegistryCustomizer<MeterRegistry> metricsCommonTags() {
        return registry -> registry.config()
            .commonTags("application", "order-service")
            .commonTags("environment", "production");
    }
}

@Service
public class OrderService {
    private final MeterRegistry registry;
    private final Timer orderTimer;
    private final Counter orderCounter;

    public OrderService(MeterRegistry registry) {
        this.registry = registry;
        this.orderTimer = Timer.builder("order.processing.time")
            .description("Order processing time")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
        this.orderCounter = Counter.builder("order.created")
            .description("Total orders created")
            .register(registry);
    }

    public Order createOrder(CreateOrderRequest request) {
        return orderTimer.recordCallable(() -> {
            // 业务逻辑
            Order order = doCreateOrder(request);
            orderCounter.increment();
            return order;
        });
    }
}
```

2. **SLO/SLI定义**

   ```yaml
   # SLI (Service Level Indicator)
   - 可用性: 成功请求数 / 总请求数
   - 延迟: 请求的响应时间分布
   - 错误率: 错误请求数 / 总请求数
   ```

### 4.4 常见问题

**Q1: 如何区分系统瓶颈是CPU还是IO？**
A:

- CPU高 + IO低 → CPU瓶颈（代码优化、算法改进）
- CPU低 + IO高 → IO瓶颈（缓存、批量处理、硬件升级）
- CPU高 + IO高 → 可能是锁竞争或资源争用

**Q2: 压测结果与生产环境差异大？**
A:

- 检查数据分布是否真实
- 检查网络延迟是否一致
- 检查下游依赖是否模拟
- 检查资源规格是否一致

**Q3: 长尾延迟如何优化？**
A:

- 分析P99.9请求特征
- 检查GC停顿
- 检查资源竞争（锁、连接池）
- 检查慢查询
- 引入自适应超时和重试

## 五、形式化分析

### 5.1 排队论模型

**M/M/1队列模型**：

$$\rho = \frac{\lambda}{\mu}$$

- $\lambda$：到达率（请求/秒）
- $\mu$：服务率（请求/秒）
- $\rho$：系统利用率

**平均响应时间**：

$$E[T] = \frac{1}{\mu - \lambda} = \frac{1}{\mu(1-\rho)}$$

**关键结论**：

- 当$\rho$接近1时，延迟趋向无穷大
- 建议保持$\rho < 0.8$以获得稳定性能

### 5.2 利特尔法则（Little's Law）

$$L = \lambda \cdot W$$

- $L$：系统中平均任务数
- $\lambda$：平均到达率
- $W$：平均停留时间

**应用**：

- 已知并发数和吞吐量，估算响应时间
- 容量规划：根据目标延迟计算所需资源
