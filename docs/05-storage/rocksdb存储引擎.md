## 📋 执行摘要

RocksDB是Facebook基于Google LevelDB开发的高性能嵌入式键值存储引擎，采用LSM-Tree数据结构优化写入性能。它通过多线程Compaction、Column Family、事务支持和多种存储后端适配，成为分布式数据库（TiDB、CockroachDB、YugabyteDB）和存储系统的首选存储引擎，在处理大规模写入密集型工作负载方面表现出色。

## 一、LSM-Tree核心原理

### 1.1 LSM-Tree架构详解

RocksDB基于Log-Structured Merge Tree（LSM-Tree）设计，将随机写转换为顺序写。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    RocksDB LSM-Tree 架构                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   写入路径 (Write Path):                                                     │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   客户端写入                                                           │ │
│   │       │                                                               │ │
│   │       ▼                                                               │ │
│   │   ┌───────────┐                                                       │ │
│   │   │   WAL     │  ← Write Ahead Log (顺序写入, 用于崩溃恢复)             │ │
│   │   │   日志    │                                                       │ │
│   │   └─────┬─────┘                                                       │ │
│   │         │ (可选异步写入)                                               │ │
│   │         ▼                                                             │ │
│   │   ┌───────────┐                                                       │ │
│   │   │ MemTable  │  ← 内存中的跳表/红黑树 (Active, 可写)                  │ │
│   │   │  (跳表)   │     默认大小: 64MB                                     │ │
│   │   └─────┬─────┘                                                       │ │
│   │         │ 达到阈值 (如64MB)                                            │ │
│   │         ▼                                                             │ │
│   │   ┌───────────┐                                                       │ │
│   │   │ Immutable │  ← 冻结的MemTable, 等待刷盘                            │ │
│   │   │ MemTable  │     (只读, 后台Flush)                                  │ │
│   │   └─────┬─────┘                                                       │ │
│   │         │ 后台Flush线程                                                │ │
│   │         ▼                                                             │ │
│   │   ┌───────────┐                                                       │ │
│   │   │   Level 0 │  ← SSTable文件 (可能重叠)                              │ │
│   │   │  SSTable  │                                                       │ │
│   │   └─────┬─────┘                                                       │ │
│   │         │ Compaction触发                                               │ │
│   │         ▼                                                             │ │
│   │   ┌───────────┐                                                       │ │
│   │   │   Level 1 │  ← SSTable文件 (有序, 不重叠)                          │ │
│   │   │  SSTable  │     大小阈值: 256MB                                     │ │
│   │   └─────┬─────┘                                                       │ │
│   │         │ 继续Compaction                                               │ │
│   │         ▼                                                             │ │
│   │   ┌───────────┐                                                       │ │
│   │   │   Level 2 │  ← SSTable文件                                         │ │
│   │   │  SSTable  │     大小阈值: 2.56GB (L1 × 10)                          │ │
│   │   └─────┬─────┘                                                       │ │
│   │         ▼                                                             │ │
│   │      Level N  ← 每层大小是上一层的10倍, 最多6-7层                       │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
│   读取路径 (Read Path):                                                      │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   查询Key                                                              │ │
│   │       │                                                               │ │
│   │       ▼                                                               │ │
│   │   ┌────────────────────────────────────────────────────────────────┐  │ │
│   │   │ 1. 检查MemTable (Active)                                       │  │ │
│   │   │ 2. 检查Immutable MemTable(s)                                   │  │ │
│   │   │ 3. 检查Block Cache (LRU缓存)                                   │  │ │
│   │   │ 4. 检查Level 0 SSTables (从新到旧, 可能多个)                    │  │ │
│   │   │ 5. 检查Level 1+ SSTables (每层最多1个, 二分查找)               │  │ │
│   │   │ 6. 返回结果或NotFound                                          │  │ │
│   │   └────────────────────────────────────────────────────────────────┘  │ │
│   │                                                                       │ │
│   │   优化: Bloom Filter减少无效磁盘读取                                    │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 二、写入优化机制

### 2.1 WAL与MemTable优化

RocksDB通过多种机制优化写入性能。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    RocksDB 写入优化策略                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. WAL写入优化                                                             │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   同步模式 (Sync)                                                     │ │
│   │   write_options.sync = true                                           │ │
│   │   └── 每次写入都fsync, 最安全但最慢                                     │ │
│   │                                                                       │ │
│   │   异步模式 (Async) - 默认                                              │ │
│   │   write_options.sync = false                                          │ │
│   │   └── 写入OS缓冲区即返回, 后台定期fsync                                 │ │
│   │                                                                       │ │
│   │   完全禁用WAL (仅MemTable)                                             │ │
│   │   write_options.disableWAL = true                                     │ │
│   │   └── 最快但崩溃可能丢数据, 适用于可重算数据                             │ │
│   │                                                                       │ │
│   │   Group Commit (批量提交)                                              │ ││
│   │   ┌─────────┐  ┌─────────┐  ┌─────────┐                              │ │
│   │   │ Write 1 │  │ Write 2 │  │ Write 3 │                              │ │
│   │   └────┬────┘  └────┬────┘  └────┬────┘                              │ │
│   │        └─────────────┴─────────────┘                                  │ │
│   │                  │                                                    │ │
│   │                  ▼                                                    │ │
│   │           ┌─────────────┐                                             │ │
│   │           │ Group Commit│ 一次fsync批量提交                           │ │
│   │           │ Batch Write │                                             │ │
│   │           └─────────────┘                                             │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
│   2. MemTable实现选择                                                        │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   SkipList (默认)                                                     │ │
│   │   └── memtable = "skip_list"                                          │ │
│   │       优点: 通用, 支持并发写入                                          │ │
│   │       适用: 通用场景                                                    │ │
│   │                                                                       │ │
│   │   HashSkipList                                                        │ │
│   │   └── memtable = "hash_skip_list"                                     │ │
│   │       优点: 前缀查询快, 范围查询慢                                       │ │
│   │       适用: 点查多的场景                                                │ │
│   │                                                                       │ │
│   │   VectorMemTable                                                      │ │
│   │   └── memtable = "vector"                                             │ │
│   │       优点: 追加写入最快, 无排序开销                                     │ │
│   │       适用: 仅追加日志场景                                              │ │
│   │                                                                       │ │
│   │   HashLinkList                                                      │ │
│   │   └── memtable = "hash_link_list"                                     │ │
│   │       优点: 内存效率高                                                  │ │
│   │       适用: 内存受限场景                                                │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
│   3. 并发写入优化                                                            │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   允许多线程同时写入MemTable                                           │ │
│   │   allow_concurrent_memtable_write = true                              │ │
│   │                                                                       │ │
│   │   写入线程模型:                                                         │ │
│   │   ┌─────────┐  ┌─────────┐  ┌─────────┐                              │ │
│   │   │ Thread 1│  │ Thread 2│  │ Thread 3│                              │ │
│   │   │ 写入KeyA│  │ 写入KeyB│  │ 写入KeyC│                              │ │
│   │   └────┬────┘  └────┬────┘  └────┬────┘                              │ │
│   │        └─────────────┼─────────────┘                                  │ │
│   │                      ▼                                                │ │

│   │                                                                       │ │

│                                                                             │
```

### 2.2 写入流程与批量写入

```cpp
// RocksDB 批量写入示例
#include <rocksdb/db.h>

using namespace rocksdb;

// 打开数据库
Options options;
options.create_if_missing = true;
options.write_buffer_size = 64 * 1024 * 1024;  // 64MB MemTable
options.max_write_buffer_number = 3;
options.min_write_buffer_number_to_merge = 1;

DB* db;
Status s = DB::Open(options, "/path/to/db", &db);

// 方式1: 单条写入 (适合低频写入)
void singleWrite() {
    WriteOptions write_options;
    write_options.sync = false;  // 异步写入

    s = db->Put(write_options, "key1", "value1");
    assert(s.ok());
}

// 方式2: 批量写入 (推荐, 减少WAL写入次数)
void batchWrite() {
    WriteBatch batch;
    batch.Put("key1", "value1");
    batch.Put("key2", "value2");
    batch.Delete("key3");
    batch.Put("key4", "value4");
    // 更多操作...

    WriteOptions write_options;
    s = db->Write(write_options, &batch);
    assert(s.ok());
}

// 方式3: 异步写入回调
class WriteCallback : public WriteCallback {
public:
    void Callback(Status status) override {
        if (status.ok()) {
            std::cout << "Write succeeded" << std::endl;
        }
    }
};
```

## 三、Compaction策略

### 3.1 Leveled Compaction

RocksDB默认的Leveled Compaction策略。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Leveled Compaction 策略                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   层大小配置 (默认):                                                         │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │  Level  │ 大小阈值    │ SST文件大小  │ 说明                           │ │
│   ├─────────┼────────────┼──────────────┼────────────────────────────────┤ │
│   │  L0     │ 无限制      │ 可变         │ MemTable直接刷盘, 可能重叠      │ │
│   │  L1     │ 256MB      │ 64MB         │ 第一层有序数据                 │ │
│   │  L2     │ 2.56GB     │ 64MB         │ L1 × 10                        │ │
│   │  L3     │ 25.6GB     │ 64MB         │ L2 × 10                        │ │
│   │  L4     │ 256GB      │ 64MB         │ L3 × 10                        │ │
│   │  L5     │ 2.56TB     │ 64MB         │ L4 × 10                        │ │
│   │  L6     │ 25.6TB     │ 64MB         │ 最后一层                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
│   Compaction触发条件:                                                        │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   L0 → L1 Compaction                                                  │ │
│   │   ├── L0文件数 >= 4 (level0_file_num_compaction_trigger)              │ │
│   │   └── 或 L0文件总大小超过阈值                                          │ │
│   │                                                                       │ │
│   │   Ln → Ln+1 Compaction (n >= 1)                                       │ │
│   │   └── Ln层大小超过该层阈值                                             │ │
│   │                                                                       │ │
│   │   示例: 当L2达到2.56GB时, 触发L2 → L3 Compaction                       │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
│   Compaction过程:                                                            │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   场景: L1 → L2 Compaction                                            │ │
│   │                                                                       │ │
│   │   L1: [A-C] [D-F] [G-I]      (3个SST文件, 共300MB, 超过256MB阈值)      │ │
│   │         │     │     │                                               │ │
│   │         └─────┴─────┘                                               │ │
│   │              │                                                      │ │
│   │              ▼                                                      │ │
│   │   L2: [A-E] [F-J] [K-O]      (L2现有文件, 需要合并重叠范围)            │ │
│   │         ▲▲▲ ▲▲▲                                                │ │
│   │         └─┘ └─┘ 与L1重叠的文件                                       │ │
│   │                                                                       │ │
│   │   合并过程:                                                           │ │
│   │   1. 找出L1所有文件 [A-C] [D-F] [G-I]                                 │ │
│   │   2. 找出L2中与L1范围重叠的文件 [A-E] [F-J]                            │ │
│   │   3. 合并排序, 生成新的L2文件                                         │ │
│   │   4. 原子替换旧文件                                                   │ │
│   │                                                                       │ │
│   │   结果:                                                               │ │
│   │   L1: (空, 所有文件已合并到L2)                                         │ │
│   │   L2: [A-D] [E-H] [I-L] [M-P] ... (新的有序文件)                       │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 Universal Compaction

适用于写入密集型场景的替代策略。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Universal Compaction 策略                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   特点:                                                                      │
│   • 减少写放大 (Write Amplification)                                        │
│   • 读放大较大                                                              │
│   • 适合SSD, 适合写入密集型场景                                              │
│                                                                             │
│   文件组织:                                                                  │
│   ┌───────────────────────────────────────────────────────────────────────┐ │
│   │                                                                       │ │
│   │   不像Leveled那样严格分层, 而是按大小分组                               │ │
│   │                                                                       │ │
│   │   Size Tier 1: [4MB] [4MB] [4MB]     (小文件)                         │ │
│   │                    ↓                                                  │ │
│   │   Size Tier 2: [16MB] [16MB]         (4个小文件合并)                   │ │
│   │                    ↓                                                  │ │
│   │   Size Tier 3: [64MB] [64MB]         (4个中等文件合并)                 │ │
│   │                    ↓                                                  │ │
│   │   Size Tier 4: [256MB]               (更大的文件)                      │ │
│   │                                                                       │ │
│   └───────────────────────────────────────────────────────────────────────┘ │
│                                                                             │
│   Compaction触发:                                                            │
│   • 同一大小的文件数达到阈值 (如4个)                                          │
│   • 合并为更大的文件                                                         │
│                                                                             │
│   配置:                                                                      │
│   options.compaction_style = kCompactionStyleUniversal;                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 Compaction优化配置

```cpp
// RocksDB Compaction调优配置
Options options;

// 选择Compaction策略
options.compaction_style = kCompactionStyleLevel;  // 或 kCompactionStyleUniversal

// Leveled Compaction调优
options.write_buffer_size = 128 * 1024 * 1024;  // 128MB MemTable
options.max_write_buffer_number = 6;
options.target_file_size_base = 64 * 1024 * 1024;  // 64MB SST文件
```

## 五、性能基准与调优

### 5.1 性能基准测试

```cpp
// RocksDB 性能测试工具
#include <rocksdb/db.h>
#include <rocksdb/options.h>

using namespace rocksdb;

void benchmark() {
    DB* db;
    Options options;
    options.create_if_missing = true;
    options.IncreaseParallelism(16);  // 使用16线程
    options.OptimizeLevelStyleCompaction(512 * 1024 * 1024);  // 512MB内存

    Status s = DB::Open(options, "/tmp/rocksdb_bench", &db);

    // 写入测试
    WriteOptions write_options;
    write_options.sync = false;

    auto start = std::chrono::high_resolution_clock::now();

    for (int i = 0; i < 1000000; i++) {
        std::string key = "key" + std::to_string(i);
        std::string value = "value" + std::to_string(i);
        db->Put(write_options, key, value);
    }

    auto end = std::chrono::high_resolution_clock::now();
    auto duration = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);

    std::cout << "写入100万条耗时: " << duration.count() << "ms" << std::endl;
    std::cout << "写入吞吐量: " << 1000000.0 / duration.count() * 1000 << " ops/s" << std::endl;
}

// 典型性能指标 (SSD, 16线程):
// - 随机写入: 150,000 - 300,000 ops/s
// - 随机读取 (热数据): 300,000 - 500,000 ops/s
// - 顺序写入: 500+ MB/s
// - 写放大: 10-30x (Leveled Compaction)
// - 读放大: 1-10x
```

```cpp
Options writeHeavyOptions() {

    options.max_write_buffer_number = 6;
    options.min_write_buffer_number_to_merge = 2;
    options.compaction_style = kCompactionStyleUniversal;  // Universal
    options.compression = kLZ4;  // 快速压缩
    return options;
}

Options readHeavyOptions() {
    Options options;
    options.write_buffer_size = 64 * 1024 * 1024;
    options.max_open_files = 100000;
    options.table_factory.reset(NewBlockBasedTableFactory(
        BlockBasedTableOptions()));
    // 大Block Cache
    options.block_cache = NewLRUCache(8 * 1024 * 1024 * 1024LL);  // 8GB
    return options;
}

// 3. 内存受限 (Memory-Constrained)
Options memoryConstrainedOptions() {
    Options options;
    options.write_buffer_size = 16 * 1024 * 1024;  // 小MemTable
    options.max_write_buffer_number = 2;
    options.target_file_size_base = 16 * 1024 * 1024;
    options.max_bytes_for_level_base = 64 * 1024 * 1024;
    options.block_cache = NewLRUCache(256 * 1024 * 1024);  // 256MB
    return options;
}
```

## 六、总结

RocksDB作为现代分布式存储系统的基石，具有以下核心优势：

1. **写优化**：LSM-Tree将随机写转换为顺序写，实现高吞吐写入
2. **灵活配置**：丰富的调优参数适应不同工作负载
3. **生产验证**：被TiDB、CockroachDB、MyRocks等大规模系统采用
4. **活跃生态**：持续优化的Compaction策略和存储格式

RocksDB是构建高性能、可靠存储系统的首选嵌入式引擎。
