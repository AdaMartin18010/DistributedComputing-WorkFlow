# 搜索引擎对比

全文搜索引擎是分布式系统中的重要组件，用于处理大规模文本数据的高效索引和检索。本文对比主流搜索引擎的架构特点、适用场景和性能特征。

## 3. 功能特性对比

| 特性 | Elasticsearch | OpenSearch | Solr | Meilisearch |
|-----|---------------|------------|------|-------------|
| **协议** | HTTP/REST | HTTP/REST | HTTP/REST | HTTP/REST |
| **客户端** | 丰富多语言 | 兼容 ES | 丰富 | 较新较少 |
| **实时性** | 近实时(1s) | 近实时 | 近实时 | 近实时 |
| **分词器** | 丰富(中文IK) | 同ES | 丰富 | 基础 |
| **聚合分析** | 极强 | 同ES | 强 | 基础 |
| **机器学习** | X-Pack/收费 | 开源 ML | Learning to Rank | 无 |
| **安全** | X-Pack/收费 | 开源安全 | Kerberos/SSL | API Key |
| **SQL 支持** | 有限 | 有限 | 完善 | 无 |
| **向量搜索** | 支持(8.0+) | 支持 | 插件 | 实验性 |

## 5. 写入与查询流程对比

### 5.1 数据写入流程

```
┌─────────────────────────────────────────────────────────────────┐
│                Elasticsearch 写入流程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 客户端发送 bulk/index 请求                                  │
│                    │                                             │
│                    ▼                                             │
│  2. 协调节点 (Coordinating Node)                                │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ • 路由计算: shard = hash(routing) % num_primary_shards │ │
│     │ • routing 默认为 _id                                    │ │
│     └─────────────────────────────────────────────────────────┘ │
│                    │                                             │
│                    ▼                                             │
│  3. 主分片 (Primary Shard)                                      │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ • 验证 Mapping                                          │ │
│     │ • 分词处理                                              │ │
│     │ • 写入 Translog (持久化)                                │ │
│     │ • 写入 Segment (内存)                                   │ │
│     └─────────────────────────────────────────────────────────┘ │
│                    │                                             │
│                    ▼                                             │
│  4. 副本分片 (Replica Shard) - 同步复制                          │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ 复制到所有 replica shards 成功后返回客户端               │ │
│     │ (可配置异步复制以提升性能)                                │ │
│     └─────────────────────────────────────────────────────────┘ │
│                    │                                             │
│                    ▼                                             │
│  5. 刷新 (Refresh) - 默认 1s                                    │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ 内存中的 Segment 刷新到文件系统缓存，变为可搜索          │ │
│     │ (refresh_interval 可调优)                               │ │
│     └─────────────────────────────────────────────────────────┘ │
│                    │                                             │
│                    ▼                                             │
│  6. 刷盘 (Flush) - 默认 30min 或 Translog 满                    │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ • 持久化到磁盘                                           │ │
│     │ • 清空 Translog                                          │ │
│     │ • 后台合并 Segment                                       │ │
│     └─────────────────────────────────────────────────────────┘ │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────┐
│                分布式查询执行流程                                │

│     ┌─────────────────────────────────────────────────────────┐ │
│     │                                                         │ │

│     │     │                     │                │           │ │
│     │  合并各分片结果，取全局 Top N                           │ │

│     └─────────────────────────────────────────────────────────┘ │
│                    │                                             │
│                    ▼                                             │

│     ┌─────────────────────────────────────────────────────────┐ │
│     │                                                         │ │

│     │                                                         │ │
│     │  优化: 优先从已查询的分片获取，减少网络往返               │ │
│     │                                                         │ │
│     └─────────────────────────────────────────────────────────┘ │
│                    │                                             │
│                    ▼                                             │
│  4. 返回聚合结果给客户端                                         │
│                                                                  │
│  优化策略:                                                        │
│  • 路由查询: 使用 routing 参数定位单一分片                        │
│  • 偏好设置: preference 控制副本选择                              │
```

## 6. 选型建议

| 场景 | 推荐引擎 | 理由 |
|-----|---------|------|
| **日志分析/ELK** | Elasticsearch/OpenSearch | 生态成熟，Kibana/Grafana 集成 |
| **企业搜索** | Apache Solr | 细粒度控制，复杂查询支持好 |
| **纯开源合规** | OpenSearch/Manticore | 无商业许可风险 |
| **轻量级搜索** | Meilisearch | 部署简单，API 现代 |
| **高性能低资源** | Manticore | C++ 实现，资源占用低 |
| **向量搜索** | Elasticsearch 8+ | 原生向量支持 |

## 7. 发展趋势

1. **向量搜索融合**: 传统搜索引擎整合向量检索能力，支持语义搜索
2. **云原生架构**: 存储计算分离，更好的弹性伸缩
3. **Serverless**: 托管服务减少运维负担
4. **AI 集成**: 内置机器学习功能，智能相关性排序

选择搜索引擎时需综合考虑数据规模、查询复杂度、团队技术栈和运维能力，没有绝对的最优解，只有最适合场景的方案。
