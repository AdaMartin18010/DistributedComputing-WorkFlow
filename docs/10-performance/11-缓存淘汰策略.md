# 缓存淘汰策略：LRU/LFU/TTL

## 核心概念

### 1. 缓存淘汰的必要性

缓存容量有限，当数据量超过容量时，需要通过淘汰策略决定哪些数据应该被移除。

**缓存容量限制因素**：
```
┌─────────────────────────────────────────────────────────────┐
│                   缓存容量限制因素                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 内存成本                                               │
│      - 内存价格远高于磁盘                                   │
│      - 单机内存上限（通常几百GB）                           │
│                                                             │
│   2. GC压力（Java/Go等）                                    │
│      - 大内存导致GC时间过长                                 │
│      - 需要控制堆外内存或分片                               │
│                                                             │
│   3. 查询效率                                               │
│      - 数据量过大导致索引失效                               │
│      - 扫描时间增加                                         │
│                                                             │
│   4. 冷数据比例                                             │
│      - 80/20法则：20%数据承担80%访问                        │
│      - 保留热数据，淘汰冷数据                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 主流淘汰策略对比

**策略特性矩阵**：

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| FIFO | 先进先出 | 简单 | 不考虑访问频率 | 队列场景 |
| LRU | 最近最少使用 | 符合局部性原理 | 突发流量污染 | 一般场景 |
| LFU | 最少使用频率 | 识别真正热点 | 新数据难存活 | 稳定热点 |
| TTL | 时间到期 | 精确控制 | 需预估时间 | 时效数据 |
| Random | 随机淘汰 | 实现简单 | 效果差 | 测试场景 |
| W-TinyLFU | 频率+时效 | 综合优势 | 复杂 | 高要求场景 |

**LRU vs LFU 对比**：
```
┌─────────────────────────────────────────────────────────────┐
│              LRU vs LFU 行为对比                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   场景：缓存容量3，访问序列：A,B,C,A,D,B,E                  │
│                                                             │
│   LRU行为：                                                 │
│   1. 访问A: [A]                                             │
│   2. 访问B: [A,B]                                           │
│   3. 访问C: [A,B,C]                                         │
│   4. 访问A: [B,C,A]  (A移动到头部)                          │
│   5. 访问D: [C,A,D]  (淘汰B)                                │
│   6. 访问B: [A,D,B]  (淘汰C)                                │
│   7. 访问E: [D,B,E]  (淘汰A) ❌ 高频A被淘汰                 │
│                                                             │
│   LFU行为：                                                 │
│   1. 访问A(1): [A:1]                                        │
│   2. 访问B(1): [A:1,B:1]                                    │
│   3. 访问C(1): [A:1,B:1,C:1]                                │
│   4. 访问A(2): [B:1,C:1,A:2]                                │
│   5. 访问D(1): [C:1,A:2,D:1]  (淘汰B:1)                     │
│   6. 访问B(1): [A:2,D:1,B:1]  (淘汰C:1)                     │
│   7. 访问E(1): [A:2,B:1,E:1]  (淘汰D:1) ✓ 保留高频A         │
│                                                             │
│   结论：LRU对突发流量敏感，LFU更能识别长期热点             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. W-TinyLFU算法

Caffeine缓存使用的W-TinyLFU（Window Tiny LFU）算法结合了LRU和LFU的优点。

**架构设计**：
```
┌─────────────────────────────────────────────────────────────┐
│                   W-TinyLFU架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                  Window Cache (1%)                  │  │
│   │              新数据首先进入的区域                     │  │
│   │              使用LRU策略                             │  │
│   └──────────────────────┬──────────────────────────────┘  │
│                          │ 淘汰的数据进入Main Cache        │
│                          ↓                                 │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                   Main Cache (99%)                  │  │
│   │  ┌─────────────────┐  ┌─────────────────┐           │  │
│   │  │  Protected (80%)│  │  Probation (20%)│           │  │
│   │  │   受保护区域     │  │   候选淘汰区     │           │  │
│   │  │   LRU策略       │  │   TinyLFU策略   │           │  │
│   │  └─────────────────┘  └─────────────────┘           │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   Frequency Sketch: 使用Count-Min Sketch统计访问频率       │
│   - 概率数据结构，节省内存                                  │
│   - 频率衰减机制，防止历史热点长期占用                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 实践方法

### LRU实现

**1. 基于HashMap + 双向链表**：
```go
package cache

import "container/list"

// LRUCache 实现
type LRUCache struct {
    capacity int
    cache    map[string]*list.Element
    ll       *list.List
}

type entry struct {
    key   string
    value interface{}
}

func NewLRUCache(capacity int) *LRUCache {
    return &LRUCache{
        capacity: capacity,
        cache:    make(map[string]*list.Element),
        ll:       list.New(),
    }
}

func (c *LRUCache) Get(key string) (interface{}, bool) {
    if ele, ok := c.cache[key]; ok {
        c.ll.MoveToFront(ele)
        return ele.Value.(*entry).value, true
    }
    return nil, false
}

func (c *LRUCache) Set(key string, value interface{}) {
    if ele, ok := c.cache[key]; ok {
        c.ll.MoveToFront(ele)
        ele.Value.(*entry).value = value
        return
    }
    
    ele := c.ll.PushFront(&entry{key, value})
    c.cache[key] = ele
    
    if c.ll.Len() > c.capacity {
        c.removeOldest()
    }
}

func (c *LRUCache) removeOldest() {
    ele := c.ll.Back()
    if ele != nil {
        c.removeElement(ele)
    }
}

func (c *LRUCache) removeElement(ele *list.Element) {
    c.ll.Remove(ele)
    kv := ele.Value.(*entry)
    delete(c.cache, kv.key)
}
```

**2. 基于LinkedHashMap（Java）**：
```java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;
    
    public LRUCache(int capacity) {
        super(capacity, 0.75f, true);  // accessOrder = true
        this.capacity = capacity;
    }
    
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

### LFU实现

**1. 基于频次桶**：
```go
package cache

import "container/list"

// LFUCache 实现
type LFUCache struct {
    capacity int
    minFreq  int
    nodes    map[string]*node
    freq     map[int]*list.List
}

type node struct {
    key       string
    value     interface{}
    freq      int
    listElem  *list.Element
}

func NewLFUCache(capacity int) *LFUCache {
    return &LFUCache{
        capacity: capacity,
        minFreq:  0,
        nodes:    make(map[string]*node),
        freq:     make(map[int]*list.List),
    }
}

func (c *LFUCache) Get(key string) (interface{}, bool) {
    n, ok := c.nodes[key]
    if !ok {
        return nil, false
    }
    c.increaseFreq(n)
    return n.value, true
}

func (c *LFUCache) Set(key string, value interface{}) {
    if c.capacity == 0 {
        return
    }
    
    if n, ok := c.nodes[key]; ok {
        n.value = value
        c.increaseFreq(n)
        return
    }
    
    if len(c.nodes) >= c.capacity {
        c.removeMinFreq()
    }
    
    n := &node{key: key, value: value, freq: 1}
    c.nodes[key] = n
    c.addToFreqList(n)
    c.minFreq = 1
}

func (c *LFUCache) increaseFreq(n *node) {
    // 从旧频率列表移除
    c.freq[n.freq].Remove(n.listElem)
    if c.freq[n.freq].Len() == 0 && c.minFreq == n.freq {
        c.minFreq++
    }
    
    // 增加频率
    n.freq++
    c.addToFreqList(n)
}

func (c *LFUCache) addToFreqList(n *node) {
    if c.freq[n.freq] == nil {
        c.freq[n.freq] = list.New()
    }
    n.listElem = c.freq[n.freq].PushBack(n)
}

func (c *LFUCache) removeMinFreq() {
    l := c.freq[c.minFreq]
    ele := l.Front()
    n := ele.Value.(*node)
    delete(c.nodes, n.key)
    l.Remove(ele)
}
```

### TTL实现

**1. 基于过期时间**：
```go
package cache

import (
    "sync"
    "time"
)

// TTLCache 实现
type TTLCache struct {
    mu       sync.RWMutex
    items    map[string]*item
    stopCh   chan struct{}
}

type item struct {
    value      interface{}
    expiration int64
}

func (i *item) isExpired() bool {
    return time.Now().UnixNano() > i.expiration
}

func NewTTLCache(cleanupInterval time.Duration) *TTLCache {
    c := &TTLCache{
        items:  make(map[string]*item),
        stopCh: make(chan struct{}),
    }
    go c.cleanupLoop(cleanupInterval)
    return c
}

func (c *TTLCache) Get(key string) (interface{}, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    
    item, ok := c.items[key]
    if !ok || item.isExpired() {
        return nil, false
    }
    return item.value, true
}

func (c *TTLCache) Set(key string, value interface{}, ttl time.Duration) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    expiration := time.Now().Add(ttl).UnixNano()
    c.items[key] = &item{
        value:      value,
        expiration: expiration,
    }
}

func (c *TTLCache) cleanupLoop(interval time.Duration) {
    ticker := time.NewTicker(interval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ticker.C:
            c.deleteExpired()
        case <-c.stopCh:
            return
        }
    }
}

func (c *TTLCache) deleteExpired() {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    for key, item := range c.items {
        if item.isExpired() {
            delete(c.items, key)
        }
    }
}
```

**2. 分层过期策略**：
```go
// 不同数据类型的过期策略
type ExpirationPolicy struct {
    BaseTTL       time.Duration  // 基础过期时间
    MaxTTL        time.Duration  // 最大过期时间
    JitterPercent float64        // 随机波动百分比
}

var policies = map[string]ExpirationPolicy{
    "user_session": {
        BaseTTL:       30 * time.Minute,
        MaxTTL:        2 * time.Hour,
        JitterPercent: 0.1,  // 10%波动，防止雪崩
    },
    "product_info": {
        BaseTTL:       10 * time.Minute,
        MaxTTL:        1 * time.Hour,
        JitterPercent: 0.2,
    },
    "inventory": {
        BaseTTL:       1 * time.Minute,
        MaxTTL:        5 * time.Minute,
        JitterPercent: 0.05,
    },
}

func CalculateTTL(policyName string) time.Duration {
    policy := policies[policyName]
    jitter := time.Duration(float64(policy.BaseTTL) * 
        policy.JitterPercent * (rand.Float64()*2 - 1))
    return policy.BaseTTL + jitter
}
```

### Redis淘汰策略

**配置策略**：
```bash
# redis.conf
maxmemory 4gb
maxmemory-policy allkeys-lru  # 全局LRU
# 可选策略：
# - volatile-lru: 只对设置了TTL的key使用LRU
# - allkeys-lru: 所有key使用LRU
# - volatile-lfu: 只对设置了TTL的key使用LFU
# - allkeys-lfu: 所有key使用LFU
# - volatile-random: 随机淘汰设置了TTL的key
# - allkeys-random: 随机淘汰
# - volatile-ttl: 淘汰即将过期的key
# - noeviction: 不淘汰，写操作报错

# LFU相关配置
lfu-log-factor 10
lfu-decay-time 1  # 每分钟衰减一次
```

**近似LRU实现**：
```
Redis使用近似LRU而非精确LRU：

1. 每个key对象记录24bit的lru字段
2. 随机采样N个key（默认5个）
3. 淘汰其中最久未访问的

优点：
- O(1)时间复杂度
- 内存占用小
- 效果接近真实LRU（90%+）
```

## 案例数据

### 不同淘汰策略对比测试

**测试场景**：缓存容量1000，访问100000次，遵循Zipf分布（80/20法则）

| 策略 | 命中率 | 实现复杂度 | 内存开销 |
|------|--------|------------|----------|
| Random | 52% | 低 | 低 |
| FIFO | 58% | 低 | 低 |
| LRU | 78% | 中 | 中 |
| LFU | 82% | 中 | 高 |
| W-TinyLFU | 85% | 高 | 中 |

**结论**：
- 简单场景：LRU足够
- 热点识别：LFU更优
- 综合性能：W-TinyLFU最佳

### 实际业务优化案例

**场景：新闻推荐系统**

**优化前**：
```
策略：简单LRU
问题：突发热点（如突发新闻）导致缓存污染
现象：
- 正常时段命中率：85%
- 热点时段命中率：45%
- 数据库压力暴增
```

**优化方案**：
```
采用W-TinyLFU（Caffeine）：
1. Window Cache保护新数据
2. 频率衰减防止历史热点
3. 频次统计识别真实热点
```

**优化后**：
```
- 正常时段命中率：88%
- 热点时段命中率：82%
- 数据库压力稳定
```

---

**总结**：选择合适的缓存淘汰策略对系统性能至关重要。LRU适用于大多数场景，LFU适合识别长期热点，W-TinyLFU在复杂场景下表现最佳。在实际应用中，还需要结合TTL策略和随机过期来防止缓存雪崩。
