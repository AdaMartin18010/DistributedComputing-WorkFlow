# 容器镜像优化

## 一、概述

容器镜像优化是云原生应用开发中的关键环节。一个经过优化的镜像可以显著降低存储成本、加速部署速度、减少攻击面。本文将深入探讨容器镜像的构建原理、优化技术和最佳实践。

## 二、镜像分层原理

### 2.1 镜像分层结构

Docker 镜像采用分层存储机制，每一层代表 Dockerfile 中的一条指令：

```
┌─────────────────────────────────────────────────────────────┐
│  Layer N: 容器读写层 (Container Layer)                      │
│  - 可写层，容器运行时修改                                     │
│  - 容器删除后数据丢失                                         │
├─────────────────────────────────────────────────────────────┤
│  Layer 4: COPY ./app /app                                   │
│  - 应用代码层                                                │
│  - 经常变动                                                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: RUN npm ci                                        │
│  - 依赖层                                                   │
│  - 中等变动频率                                              │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: COPY package*.json ./                             │
│  - 依赖配置文件                                              │
│  - 较少变动                                                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: FROM node:18-alpine                               │
│  - 基础镜像层                                                │
│  - 很少变动                                                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 0: FROM scratch (基础系统层)                         │
│  - Alpine/Debian 等基础系统                                  │
│  - 几乎不变                                                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 分层缓存机制

```
构建流程：
                    缓存命中                缓存未命中
    FROM node:18-alpine ──► 使用本地缓存 ──► 拉取基础镜像
           │
           ▼
    COPY package*.json ./ ──► 文件未变化 ──► 重新执行
           │                    │
           │                    ▼
           │               后续层全部重建
           │
           ▼（文件无变化，使用缓存）
    RUN npm ci ──► 使用缓存层
           │
           ▼
    COPY ./app /app ──► 代码变化 ──► 重新执行
           │                           │
           ▼                           ▼
    CMD ["node", "index.js"]      最终镜像
```

### 2.3 分层最佳实践

```dockerfile
# ❌ 不好的实践 - 缓存失效频繁
FROM node:18
COPY . /app
WORKDIR /app
RUN npm install
CMD ["node", "index.js"]

# ✅ 好的实践 - 优化缓存命中率
FROM node:18-alpine
WORKDIR /app

# 1. 先复制依赖配置（不经常变动）
COPY package*.json ./

# 2. 安装依赖（依赖变动频率中等）
RUN npm ci --only=production && \
    npm cache clean --force

# 3. 最后复制应用代码（经常变动）
COPY . .

# 4. 非 root 用户运行
USER node

CMD ["node", "index.js"]
```

## 三、多阶段构建

### 3.1 多阶段构建原理

多阶段构建允许在一个 Dockerfile 中使用多个 FROM 语句，每个阶段可以独立构建，最终只将需要的产物复制到最终镜像。

```
┌─────────────────────────────────────────────────────────────┐
│                   Multi-stage Build                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ Stage 1: Builder                                       │  │
│  │  ┌─────────────┐                                      │  │
│  │  │ Build Tools │ 包含完整的构建环境                       │  │
│  │  │ - gcc       │ 编译器、开发库                          │  │
│  │  │ - node-gyp  │ 体积较大                               │  │
│  │  │ - python    │                                      │  │
│  │  └─────────────┘                                      │  │
│  │  Output: 编译产物 (二进制文件/静态资源)                   │  │
│  └───────────────────────────────────────────────────────┘  │
│                           │                                  │
│                           │ COPY --from=builder              │
│                           ▼                                  │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ Stage 2: Production                                    │  │
│  │  ┌─────────────┐                                      │  │
│  │  │ Runtime Only│ 仅包含运行时依赖                        │  │
│  │  │ - 基础系统  │ 体积小巧                               │  │
│  │  │ - 编译产物  │                                      │  │
│  │  └─────────────┘                                      │  │
│  │  Size: ~20MB vs ~500MB                                │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Node.js 多阶段构建示例

```dockerfile
# ========== 阶段 1: 依赖安装 ==========
FROM node:18-alpine AS deps
WORKDIR /app

# 安装构建依赖
RUN apk add --no-cache libc6-compat

# 复制依赖文件
COPY package*.json ./
RUN npm ci --only=production

# ========== 阶段 2: 构建 ==========
FROM node:18-alpine AS builder
WORKDIR /app

# 复制依赖
COPY --from=deps /app/node_modules ./node_modules
COPY . .

# 构建应用
RUN npm run build

# ========== 阶段 3: 生产运行 ==========
FROM node:18-alpine AS runner
WORKDIR /app

ENV NODE_ENV=production
ENV PORT=3000

# 创建非 root 用户
RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

# 仅复制必要文件
COPY --from=builder --chown=nextjs:nodejs /app/dist ./dist
COPY --from=builder --chown=nextjs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nextjs:nodejs /app/package.json ./package.json

USER nextjs

EXPOSE 3000

ENV NEXT_TELEMETRY_DISABLED=1

CMD ["node", "dist/main.js"]
```

### 3.3 Go 多阶段构建示例

```dockerfile
# ========== 阶段 1: 构建 ==========
FROM golang:1.21-alpine AS builder

# 安装构建工具
RUN apk add --no-cache git ca-certificates tzdata

WORKDIR /build

# 复制依赖文件（利用缓存）
COPY go.mod go.sum ./
RUN go mod download && go mod verify

# 复制源码
COPY . .

# 构建静态二进制文件
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build \
    -ldflags='-w -s -extldflags "-static"' \
    -a -installsuffix cgo \
    -o /app/server \
    ./cmd/server

# ========== 阶段 2: 生产运行 ==========
FROM scratch

# 从 builder 复制 CA 证书
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /usr/share/zoneinfo /usr/share/zoneinfo

# 复制二进制文件
COPY --from=builder /app/server /server

# 非 root 用户（需要提前在 scratch 中创建）
# 使用 distroless 或创建最小文件系统

EXPOSE 8080

ENTRYPOINT ["/server"]
```

### 3.4 Java 多阶段构建示例

```dockerfile
# ========== 阶段 1: 依赖缓存 ==========
FROM maven:3.9-eclipse-temurin-17-alpine AS deps
WORKDIR /build

# 单独复制 pom.xml 以利用缓存
COPY pom.xml .
RUN mvn dependency:go-offline

# ========== 阶段 2: 构建 ==========
FROM maven:3.9-eclipse-temurin-17-alpine AS builder
WORKDIR /build

# 从 deps 阶段复制缓存的依赖
COPY --from=deps /root/.m2 /root/.m2
COPY pom.xml .
COPY src ./src

RUN mvn clean package -DskipTests

# ========== 阶段 3: 生产运行 ==========
FROM eclipse-temurin:17-jre-alpine

# 安装安全更新
RUN apk update && \
    apk upgrade && \
    apk add --no-cache curl && \
    rm -rf /var/cache/apk/*

WORKDIR /app

# 创建非 root 用户
RUN addgroup -S spring -g 1000 && \
    adduser -S spring -u 1000 -G spring

# 复制 JAR 文件
COPY --from=builder --chown=spring:spring /build/target/*.jar app.jar

USER spring:spring

EXPOSE 8080

# JVM 优化参数
ENV JAVA_OPTS="-XX:+UseContainerSupport \
               -XX:MaxRAMPercentage=75.0 \
               -XX:+UseG1GC \
               -XX:MaxGCPauseMillis=200 \
               -Djava.security.egd=file:/dev/./urandom"

HEALTHCHECK --interval=30s --timeout=3s --start-period=60s --retries=3 \
    CMD curl -f http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

## 四、基础镜像选择

### 4.1 基础镜像对比

| 镜像 | 大小 | 特点 | 适用场景 |
|------|------|------|----------|
| scratch | ~0 MB | 空镜像，无shell | Go 静态编译 |
| distroless | ~20 MB | Google 维护，最小运行时 | 生产环境 |
| alpine | ~5 MB | musl libc，busybox | 通用轻量 |
| debian:slim | ~80 MB | glibc，完整工具链 | 兼容性要求 |
| ubuntu | ~80 MB | 广泛支持 | 开发/企业 |
| wolfi | ~10 MB | Chainguard，安全优先 | 安全敏感 |
| chainguard | ~10 MB | 零 CVE 基础镜像 | 高安全要求 |

### 4.2 各语言推荐基础镜像

```dockerfile
# Go - 推荐 scratch 或 distroless
FROM gcr.io/distroless/static-debian11

# Rust - 推荐 distroless 或 scratch
FROM gcr.io/distroless/cc-debian11

# Python - 推荐 slim 或 alpine
FROM python:3.11-slim-bookworm

# Node.js - 推荐 alpine 或 distroless
FROM node:18-alpine

# Java - 推荐 eclipse-temurin JRE
FROM eclipse-temurin:17-jre-alpine

# .NET - 推荐 aspnet runtime
FROM mcr.microsoft.com/dotnet/aspnet:8.0-alpine

# Ruby - 推荐 alpine
FROM ruby:3.2-alpine
```

### 4.3 使用 Chainguard 镜像

```dockerfile
# Chainguard 镜像 - 零 CVE，最小攻击面
FROM cgr.dev/chainguard/node:latest

# 已经是非 root 用户 (UID 65532)
WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY . .

# 无需 USER 指令，默认就是非 root
EXPOSE 3000

CMD ["node", "index.js"]
```

## 五、镜像瘦身技术

### 5.1 合并 RUN 指令

```dockerfile
# ❌ 不好的实践 - 产生多个层
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y vim
RUN rm -rf /var/lib/apt/lists/*

# ✅ 好的实践 - 单层，清理缓存
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        ca-certificates && \
    rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*
```

### 5.3 删除构建依赖

```dockerfile
# Python 示例 - 使用虚拟环境并清理
FROM python:3.11-slim as builder

WORKDIR /app

# 安装构建依赖
RUN apt-get update && apt-get install -y --no-install-recommends gcc

# 创建虚拟环境
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# 安装依赖
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 生产镜像
FROM python:3.11-slim

WORKDIR /app

# 仅复制虚拟环境
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# 复制应用代码
COPY . .

CMD ["python", "app.py"]
```

## 七、性能优化

### 7.1 构建缓存优化

```dockerfile
# 最佳实践：将不经常变动的指令放在前面
FROM node:18-alpine

WORKDIR /app

# 1. 系统依赖（很少变化）
RUN apk add --no-cache dumb-init

# 2. 项目依赖配置（较少变化）
COPY package*.json ./

# 3. 依赖安装（中等变化）
RUN npm ci --only=production

# 4. 应用代码（经常变化）
COPY . .

# 5. 运行时配置
EXPOSE 3000
USER node

ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "index.js"]
```

### 7.2 并行构建

```dockerfile
# syntax=docker/dockerfile:1

# 使用 BuildKit 并行构建
FROM node:18-alpine AS base
WORKDIR /app

# 并行下载依赖和复制配置文件
FROM base AS deps
COPY package*.json ./
RUN npm ci --only=production

FROM base AS config
COPY config/ ./config/

# 最终镜像并行复制
FROM base AS runner
COPY --from=deps /app/node_modules ./node_modules
COPY --from=config /app/config ./config
COPY . .
```

### 7.3 BuildKit 高级特性

```bash
# 启用 BuildKit
export DOCKER_BUILDKIT=1

# 缓存挂载 - 在构建期间保持缓存
docker build \
  --cache-from type=local,src=.cache \
  --cache-to type=local,dest=.cache,mode=max \
  -t myimage:latest .

# 使用 --mount 缓存依赖
RUN --mount=type=cache,target=/root/.npm \
    npm ci

# 使用 --mount 挂载秘密（不进入镜像层）
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

# 使用 --ssh 挂载 SSH 密钥
RUN --mount=type=ssh,id=github \
    git clone git@github.com:private/repo.git
```

```dockerfile
# 使用 BuildKit 缓存挂载
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./

# 使用缓存挂载保持 npm 缓存
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production

COPY . .

# 使用缓存挂载编译
RUN --mount=type=cache,target=/app/.cache \
    npm run build

CMD ["node", "dist/index.js"]
```

## 八、镜像大小分析

### 8.1 使用 dive 分析

```bash
# 安装 dive
wget https://github.com/wagoodman/dive/releases/download/v0.11.0/dive_0.11.0_linux_amd64.deb
sudo dpkg -i dive_0.11.0_linux_amd64.deb

# 分析镜像
dive myimage:latest

# CI 模式 - 检查效率
dive --ci --lowestEfficiency=0.9 myimage:latest
```

### 8.2 各层大小分析

```bash
# 查看镜像历史
docker history myimage:latest

# 查看每层大小
docker history --format "table {{.Size}}\t{{.CreatedBy}}" myimage:latest

# 导出镜像分析
docker save myimage:latest | tar -tf - | xargs -I {} sh -c 'echo "{}" && docker save myimage:latest | tar -xO -f - "{}" | wc -c'
```

### 8.3 镜像优化前后对比

| 优化阶段 | 镜像大小 | 减少比例 | 主要优化手段 |
|----------|----------|----------|--------------|
| 原始镜像 | 1.2 GB | - | 未优化 |
| 使用 Alpine | 350 MB | 70% | 基础镜像替换 |
| 多阶段构建 | 120 MB | 90% | 分离构建/运行 |
| 清理缓存 | 85 MB | 93% | 删除构建依赖 |
| 使用 distroless | 45 MB | 96% | 最小运行时 |

## 九、总结

容器镜像优化的核心原则：

1. **选择合适的基础镜像**：优先使用 alpine、distroless 或 chainguard
2. **使用多阶段构建**：分离构建环境和运行环境
3. **优化层缓存**：合理安排指令顺序，减少缓存失效
4. **删除不必要文件**：使用 .dockerignore，清理缓存
5. **安全加固**：使用非 root 用户，最小权限原则
6. **持续分析**：使用 dive、trivy 等工具定期检查

通过这些优化技术，通常可以将镜像大小减少 90% 以上，同时提高安全性和部署效率。
