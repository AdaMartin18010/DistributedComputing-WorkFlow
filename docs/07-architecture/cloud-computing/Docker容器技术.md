## 📋 执行摘要

Docker容器技术通过操作系统级虚拟化实现应用及其依赖的打包与隔离，提供轻量级、可移植、高效的运行环境，是云原生架构的基石技术。

## 一、核心概念

### 1.1 定义与原理

**容器（Container）** 是一种操作系统级虚拟化技术，它将应用程序及其所有依赖（代码、运行时、系统工具、库、配置等）打包在一起，形成一个标准化、可移植的运行单元。

**核心原理**：

- **Namespace 隔离**：为容器提供独立的进程、网络、文件系统视图
- **Cgroups 限制**：控制容器的资源使用（CPU、内存、IO等）
- **UnionFS 分层存储**：通过镜像分层实现高效存储和快速部署

### 1.2 关键特性

| 特性 | 说明 |
|------|------|
| **轻量级** | 共享宿主机内核，无完整OS开销，启动毫秒级 |
| **可移植** | 一次构建，到处运行，环境一致性保障 |
| **隔离性** | 进程、网络、文件系统隔离，安全沙箱 |
| **高效性** | 资源利用率高，密度可达VM的10-100倍 |
| **版本控制** | 镜像分层支持版本管理和增量更新 |

### 1.3 容器 vs 虚拟机对比

| 维度 | Docker容器 | 传统虚拟机 |
|------|------------|------------|
| **架构** | 共享宿主机内核，用户空间隔离 | 完整OS + Hypervisor，硬件虚拟化 |
| **启动速度** | 秒级/毫秒级 | 分钟级 |
| **资源占用** | MB级，共享内核 | GB级，每个VM独立OS |
| **性能** | 接近原生（~100%） | 有虚拟化开销（~70-80%） |
| **密度** | 单主机可运行数千容器 | 单主机通常运行数十VM |
| **隔离级别** | 进程级隔离 | 硬件级隔离，更强安全性 |
| **镜像大小** | 小（MB级） | 大（GB级） |
| **适用场景** | 微服务、CI/CD、开发测试 | 多OS需求、强隔离要求、遗留系统 |

## 二、Namespace 与 Cgroups

### 2.1 Linux Namespace 隔离机制

Namespace 是 Linux 内核提供的一种资源隔离机制，让每个容器拥有独立的系统资源视图。

| Namespace | 隔离资源 | 说明 |
|-----------|----------|------|
| **PID** | 进程ID | 容器内进程ID独立编号，PID 1为容器主进程 |
| **NET** | 网络 | 独立的网络设备、IP、端口、路由表 |
| **IPC** | 进程间通信 | 隔离信号量、消息队列、共享内存 |
| **MNT** | 挂载点 | 独立的文件系统层次结构 |
| **UTS** | 主机名/域名 | 独立的hostname和NIS domain |
| **USER** | 用户/组ID | 用户ID映射，容器内root映射到宿主机普通用户 |
| **CGROUP** | Cgroup根目录 | 隐藏其他容器的cgroup信息 |
| **TIME** | 系统时间 | 独立的boot时间和 monotonic 时间（Linux 5.6+） |

```bash
# 查看进程的Namespace
ls -la /proc/<pid>/ns/

# 创建新Namespace的示例
unshare --pid --fork --mount-proc /bin/bash
```

### 2.2 Cgroups 资源限制

Cgroups（Control Groups）是 Linux 内核的资源管理机制，用于限制、记录和隔离进程组的资源使用。

**Cgroups v1 子系统**：

| 子系统 | 功能 |
|--------|------|
| **cpu** | CPU时间分配（shares、quota、period） |
| **cpuacct** | CPU使用统计 |
| **cpuset** | 绑定特定CPU核心和内存节点 |
| **memory** | 内存限制（limit、swap、kernel memory） |
| **blkio** | 块设备IO限制 |
| **devices** | 设备访问控制（白名单/黑名单） |
| **freezer** | 暂停/恢复进程组 |
| **net_cls** | 网络流量标记（用于tc） |
| **net_prio** | 网络接口优先级 |
| **pids** | 进程数量限制 |
| **hugetlb** | 大页内存限制 |

**Cgroups v2 改进**：

- 统一层级结构，简化管理
- 更好的资源分配模型
- 增强的委派和根权限控制
- 统一的控制器接口

```bash
# Docker资源限制示例
docker run -d \
  --memory="512m" \
  --memory-swap="1g" \
  --cpus="1.5" \
  --cpu-shares=512 \
  --pids-limit=100 \
  --device-read-bps /dev/sda:10mb \
  nginx
```

## 三、容器镜像与分层设计

### 3.1 镜像分层架构

Docker 镜像采用**联合文件系统（UnionFS）**技术，由多个只读层（Layer）叠加组成。

```
┌─────────────────────────────────────────┐
│         Container Layer (RW)            │  ← 容器可写层
│    （运行时修改，容器删除后丢失）          │
├─────────────────────────────────────────┤
│         Image Layer N (RO)              │  ← 应用层
│         (Application Layer)             │
├─────────────────────────────────────────┤
│         Image Layer 2 (RO)              │  ← 依赖层
│         (Dependencies Layer)            │
├─────────────────────────────────────────┤
│         Image Layer 1 (RO)              │  ← 基础层
│         (Base Image Layer)              │
├─────────────────────────────────────────┤
│         Bootfs (Kernel)                 │  ← 宿主机内核
└─────────────────────────────────────────┘
```

**分层优势**：

- **共享存储**：相同基础层可被多个镜像共享
- **快速构建**：仅重建变更层，利用缓存
- **高效传输**：仅需传输差异层
- **版本管理**：每层有唯一ID，可追溯

### 3.2 镜像构建优化

```dockerfile
# 多阶段构建示例
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o main .

FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /root/
COPY --from=builder /app/main .
CMD ["./main"]
```

**最佳实践**：

1. **使用官方基础镜像**：安全性高，维护好
2. **最小化镜像层**：合并RUN指令，使用多阶段构建
3. **利用构建缓存**：将不常变更的指令放前面
4. **使用 .dockerignore**：排除不必要的文件
5. **选择合适的基础镜像**：alpine、distroless等

### 3.3 镜像仓库与分发

| 仓库类型 | 代表 | 特点 |
|----------|------|------|
| **公共仓库** | Docker Hub | 官方镜像，社区丰富 |
| **云厂商** | ECR、ACR、GCR | 与云服务集成，高可用 |
| **私有仓库** | Harbor、Nexus | 企业级安全、镜像扫描 |
| **本地缓存** | 本地daemon | 构建加速，离线可用 |

- **distribution-spec**：镜像分发协议标准

## 六、实践指南

### 6.1 Dockerfile 最佳实践

```dockerfile
# ✅ 好的实践
FROM node:18-alpine
WORKDIR /app

# 先复制依赖文件，利用缓存
COPY package*.json ./
RUN npm ci --only=production

# 再复制应用代码
COPY . .

# 非root用户运行
USER node

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:3000/health || exit 1

EXPOSE 3000
CMD ["node", "server.js"]
```

### 6.2 容器安全实践

| 实践 | 说明 |
|------|------|
| **最小镜像** | 使用alpine或distroless减少攻击面 |
| **非root运行** | 避免容器内使用root用户 |
| **只读文件系统** | `--read-only` 限制写入 |
| **资源限制** | 设置CPU、内存、IO上限 |
| **安全扫描** | 使用Trivy、Clair扫描镜像漏洞 |
| **Capabilities** | 仅授予必要权限，`--cap-drop=ALL` |
| **Seccomp** | 使用seccomp配置文件限制系统调用 |

### 6.3 常见问题

**Q1: 容器和宿主机时区不一致？**

```bash
# 挂载本地时区文件
docker run -v /etc/localtime:/etc/localtime:ro ...
```

**Q2: 如何查看容器实际资源使用？**

```bash
docker stats <container_id>
docker system df -v  # 查看磁盘使用
```

**Q3: 容器内应用无法访问外部网络？**

- 检查DNS配置：`docker run --dns 8.8.8.8`
- 检查防火墙规则
- 确认bridge网络配置
