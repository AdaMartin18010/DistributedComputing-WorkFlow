# 性能指标详解：QPS/TPS/RT/吞吐量

## 核心概念

### 1. QPS（Queries Per Second）

QPS表示每秒查询率，是衡量系统处理查询请求能力的核心指标。

**定义与计算**：

```
QPS = 总请求数 / 总时间(秒)
```

**分类维度**：

- **业务QPS**：面向用户的实际业务请求
- **接口QPS**：单个API端点的处理能力
- **数据库QPS**：数据库层面的查询操作数

**实际应用示例**：

```
场景：电商商品详情页
- 页面QPS：10,000/秒
- 其中商品信息API：8,000/秒
- 库存查询API：6,000/秒（部分缓存）
- 推荐API：4,000/秒（异步加载）
```

### 2. TPS（Transactions Per Second）

TPS表示每秒事务数，强调业务的完整性和原子性。

**与QPS的区别**：

| 维度 | QPS | TPS |
|------|-----|-----|
| 关注点 | 请求次数 | 业务完整性 |
| 典型场景 | 查询接口 | 下单/支付 |
| 复杂度 | 单次操作 | 多步骤原子操作 |
| 失败回滚 | 不涉及 | 必须支持 |

**电商下单TPS计算**：

```
一个订单事务包含：
1. 创建订单记录
2. 扣减库存
3. 扣减用户余额/优惠券
4. 记录订单日志

TPS = 成功完成的订单数 / 时间(秒)
```

### 3. RT（Response Time）

响应时间是用户体验的核心指标，通常关注百分位值。

**百分位定义**：

- **P50（中位数）**：50%请求低于此值
- **P90**：90%请求低于此值
- **P99**：99%请求低于此值
- **P99.9**：尾部延迟，反映系统稳定性

**RT分解模型**：

```
总RT = 客户端网络延迟
     + 服务端处理时间
     + 下游服务调用时间
     + 数据库查询时间
     + 响应网络延迟

示例：
┌─────────────────────────────────────────┐
│  DNS解析:           5ms                 │
│  TCP握手:          15ms                 │
│  TLS握手:          30ms                 │
│  服务端处理:       50ms                 │
│  ├─ 权限校验:      5ms                  │
│  ├─ 业务逻辑:      20ms                 │
│  ├─ DB查询:        15ms                 │
│  └─ 缓存读取:      10ms                 │
│  网络传输:         20ms                 │
│  ─────────────────────────              │
│  总RT:            120ms                 │
└─────────────────────────────────────────┘
```

### 4. 吞吐量（Throughput）

吞吐量是综合性指标，表示单位时间内系统成功处理的请求/数据量。

**不同维度的吞吐量**：

```
网络吞吐量：Mbps/Gbps
磁盘吞吐量：MB/s
系统吞吐量：请求数/秒
应用吞吐量：业务处理量/秒
```

**吞吐量与并发关系**：

```
吞吐量 = 并发用户数 / 平均响应时间

示例：
- 并发用户：1000
- 平均RT：200ms
- 吞吐量 = 1000 / 0.2 = 5000 请求/秒
```

### 指标采集方案

**1. 全链路指标采集架构**：

```

└─────────────┘    └─────────────┘    └──────┬──────┘
                                             │
                              ┌──────────────┼──────────────┐
                              ↓              ↓              ↓
                        ┌─────────┐    ┌─────────┐    ┌─────────┐
                        │   DB    │    │  Cache  │    │  MQ     │
                        │(SlowLog)│    │ (Stats) │    │(Metrics)│
                        └─────────┘    └─────────┘    └─────────┘
                              │              │              │
                              └──────────────┴──────────────┘
                                             │
                                        ┌────┴────┐
                                        │Prometheus│
                                        │  + Grafana│
                                        └─────────┘
```

**2. 实时指标计算代码示例**：

```java
@Component
public class PerformanceMetricsCollector {

    private final MeterRegistry registry;

    // QPS计数器
    private final Counter requestCounter;

    // RT分布直方图
    private final DistributionSummary responseTimeSummary;

    // TPS滑动窗口
    private final SlidingWindowCounter transactionCounter;

    public void recordRequest(long responseTimeMs, boolean success) {
        requestCounter.increment();
        responseTimeSummary.record(responseTimeMs);

        if (success) {
            transactionCounter.increment();
        }
    }

    // 获取实时QPS
    public double getCurrentQps() {
        return requestCounter.count() / getTimeWindowSeconds();
    }

    // 获取百分位RT
    public double getPercentileRT(double percentile) {
        return responseTimeSummary.percentile(percentile);
    }
}
```

### 性能基线建立

**1. 分级性能目标**：

```
┌─────────────────────────────────────────────────────┐
│              性能等级划分                           │
├──────────┬──────────┬──────────┬──────────────────┤
│  等级    │  QPS     │  P99 RT  │  适用场景        │
├──────────┼──────────┼──────────┼──────────────────┤
│  S级     │  50K+    │  <10ms   │  核心支付链路    │
│  A级     │  20K+    │  <50ms   │  核心查询接口    │
│  B级     │  5K+     │  <100ms  │  普通业务接口    │
│  C级     │  1K+     │  <500ms  │  后台管理接口    │
└──────────┴──────────┴──────────┴──────────────────┘
```

**2. 容量阈值设定**：

```yaml
performance:
  thresholds:
    qps:
      warning: 8000      # 80%容量预警
      critical: 9000     # 90%容量告警
    rt:
      p50_max: 50ms
      p99_max: 200ms
      p999_max: 500ms
    error_rate:
      warning: 0.1%      # 千分之一错误率
      critical: 1%       # 百分之一错误率
```

## 工具使用

### 1. Prometheus + Grafana

**指标采集配置**：

```yaml
# prometheus.yml
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'application'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['app:8080']

  - job_name: 'nginx'
    static_configs:
      - targets: ['nginx:9113']
```

**Grafana仪表板配置**：

```json
{
  "dashboard": {
    "title": "Performance Metrics",
    "panels": [
      {
        "title": "QPS",
        "targets": [{
          "expr": "rate(http_requests_total[1m])"
        }]
      },
      {
        "title": "P99 Latency",
        "targets": [{
          "expr": "histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))"
        }]
      }
    ]
  }
}
```

## 优化前后对比

### 案例：商品详情页性能优化

**优化前指标**：

```
QPS:        2,000/秒
P50 RT:     350ms
P99 RT:     2,500ms
错误率:     2.5%
CPU使用率:  85%
内存使用:   12GB
```

**优化措施**：

1. 引入Redis缓存（命中率90%+）
2. 数据库查询优化（索引+减少N+1查询）
3. 接口异步化（库存、推荐异步加载）
4. CDN静态资源加速

**优化后指标**：

```
QPS:        15,000/秒   (↑ 650%)
P50 RT:     25ms        (↓ 93%)
P99 RT:     120ms       (↓ 95%)
错误率:     0.05%       (↓ 98%)
CPU使用率:  45%         (↓ 47%)
内存使用:   4GB         (↓ 67%)
```

### 性能优化ROI分析

| 优化项 | 投入成本 | 性能提升 | ROI |
|--------|----------|----------|-----|
| 缓存引入 | 2人周 | QPS+500% | 极高 |
| 数据库索引 | 3人日 | RT-60% | 极高 |
| 异步化改造 | 1人月 | RT-40% | 高 |
| 代码重构 | 2人月 | CPU-30% | 中 |
| 硬件升级 | 50万 | QPS+100% | 低 |

**总结**：性能指标是系统健康的晴雨表。建立全面的指标体系，结合业务特点设定合理目标，通过持续监控和优化，才能在保证用户体验的同时实现资源的高效利用。
