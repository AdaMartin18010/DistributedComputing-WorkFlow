# 日志聚合架构

## 概述

日志聚合是将分散在各处的日志数据集中收集、处理和存储的过程，是构建统一可观测性平台的基础。合理的日志聚合架构能够支撑海量日志数据的实时处理、高效存储和快速查询，为故障排查和业务分析提供数据基础。

## 核心组件配置

### 1. Vector (高性能收集器)

```yaml
# vector.yaml
# 数据源配置
sources:
  kubernetes_logs:
    type: kubernetes_logs
    pod_annotation_fields:
      container_image: "container_image"
      container_name: "container_name"
      pod_name: "pod_name"
      pod_namespace: "pod_namespace"
      
  file_logs:
    type: file
    include:
      - /var/log/**/*.log
    exclude:
      - /var/log/**/*.gz
    read_from: beginning
    
  syslog:
    type: syslog
    address: 0.0.0.0:514
    mode: udp

# 转换处理
transforms:
  parse_json:
    type: remap
    inputs:
      - kubernetes_logs
      - file_logs
    source: |
      # 解析JSON日志
      if is_json(.message) {
        parsed = parse_json!(.message)
        . = merge(., parsed)
      }
      
      # 添加元数据
      .environment = "production"
      .host = get_hostname!()
      
      # 提取日志级别
      if exists(.level) {
        .severity = upcase!(.level)
      } else {
        .severity = "INFO"
      }
      
      # 脱敏处理
      if exists(.password) {
        .password = "[REDACTED]"
      }
      
  filter_errors:
    type: filter
    inputs:
      - parse_json
    condition: |
      .severity == "ERROR" || .severity == "FATAL"

# 输出目标
sinks:
  # 发送到Loki
  loki:
    type: loki
    inputs:
      - parse_json
    endpoint: http://loki:3100
    encoding:
      codec: json
    labels:
      environment: "{{ environment }}"
      service: "{{ service_name }}"
      severity: "{{ severity }}"
      host: "{{ host }}"
    out_of_order_action: accept
    
  # 发送到Elasticsearch
  elasticsearch:
    type: elasticsearch
    inputs:
      - filter_errors
    endpoint: http://elasticsearch:9200
    index: "logs-{{ .environment }}-%Y.%m.%d"
    bulk:
      index: "logs"
    encoding:
      codec: json
      
  # 发送到Kafka
  kafka:
    type: kafka
    inputs:
      - parse_json
    bootstrap_servers: kafka:9092
    topic: "logs-{{ .severity | lower }}"
    encoding:
      codec: json
      
  # 发送到S3 (归档)
  s3_archive:
    type: aws_s3
    inputs:
      - parse_json
    bucket: logs-archive
    region: us-east-1
    key_prefix: "logs/date=%Y-%m-%d/"
    encoding:
      codec: json
    compression: gzip
    
  # 控制台输出(调试)
  console:
    type: console
    inputs:
      - parse_json
    encoding:
      codec: json
```

### 2. Fluentd (处理中间件)

```yaml
# fluentd.conf
# 源配置
<source>
  @type forward
  port 24224
  bind 0.0.0.0
</source>

<source>
  @type tail
  path /var/log/containers/*.log
  pos_file /var/log/fluentd-docker.pos
  tag kubernetes.*
  <parse>
    @type json
    time_key time
    time_format %Y-%m-%dT%H:%M:%S.%NZ
  </parse>
</source>

# 过滤器
<filter kubernetes.**>
  @type kubernetes_metadata
  @id filter_kube_metadata
</filter>

<filter kubernetes.**>
  @type grep
  <regexp>
    key $.kubernetes.container_name
    pattern ^(?!fluentd).*
  </regexp>
</filter>

<filter kubernetes.**>
  @type record_transformer
  <record>
    environment ${ENVIRONMENT}
    cluster ${CLUSTER_NAME}
  </record>
</filter>

# 解析日志
<filter kubernetes.**>
  @type parser
  format json
  key_name log
  reserve_data true
  <parse>
    @type json
  </parse>
</filter>

# 输出配置
<match kubernetes.**>
  @type copy
  
  # 发送到Elasticsearch
  <store>
    @type elasticsearch
    host elasticsearch
    port 9200
    index_name logs-${tag}-%Y.%m.%d
    type_name _doc
    
    <buffer tag, time>
      @type file
      path /var/log/fluentd-buffers/kubernetes.system.buffer
      timekey 3600
      timekey_use_utc true
      chunk_limit_size 256m
      queue_limit_length 128
      flush_interval 5s
      retry_max_interval 30
      retry_forever true
    </buffer>
  </store>
  
  # 发送到S3归档
  <store>
    @type s3
    s3_bucket logs-archive
    s3_region us-east-1
    path logs/%Y/%m/%d/
    
    <buffer time>
      @type file
      path /var/log/fluentd-buffers/s3.buffer
      timekey 3600
      timekey_wait 10m
      chunk_limit_size 256m
    </buffer>
    
    <format>
      @type json
    </format>
  </store>
</match>
```

## 日志处理模式

```yaml
# log-patterns.yaml
patterns:
  # 模式1: 边车收集器
  sidecar:
    description: 每个Pod运行收集器容器
    use_case: Kubernetes环境
    components:
      - fluent-bit (DaemonSet)
      - vector (Sidecar)
    pros:
      - 低延迟
      - 本地解析
    cons:
      - 资源开销
      
  # 模式2: 节点级收集
  node_level:
    description: 每个节点运行收集器
    use_case: 大规模集群
    components:
      - fluent-bit (DaemonSet)
      - promtail (DaemonSet)
    pros:
      - 资源高效
      - 易于管理
    cons:
      - 需要挂载宿主机日志
      
  # 模式3: 应用直写
  direct_write:
    description: 应用直接发送到存储
    use_case: 云原生应用
    components:
      - OTLP协议
      - 结构化日志库
    pros:
      - 无代理
      - 精准控制
    cons:
      - 代码侵入
      
  # 模式4: 日志代理
  log_agent:
    description: 应用写入文件，代理收集
    use_case: 传统应用
    components:
      - filebeat
      - vector
    pros:
      - 零侵入
      - 可靠传输
    cons:
      - 磁盘IO
      - 延迟较高
```

## 最佳实践

1. **结构化日志**：统一JSON格式，便于解析和查询
2. **日志分级**：DEBUG/INFO/WARN/ERROR/FATAL合理分级
3. **上下文注入**：trace_id、span_id、user_id等关键字段
4. **敏感脱敏**：自动脱敏密码、Token等敏感信息
5. **容量规划**：根据日志量选择合适的存储方案
