# 术语歧义消解表

> 🔍 多义词辨析 | 同义词对照 | 语境指南

---

## 一、多义词辨析

### 1. "一致性"的不同含义

| 语境 | 定义 | 关键区别 | 示例 |
|:---|:---|:---|:---|
| **ACID一致性** | 事务执行前后，数据库从一个有效状态转换到另一个有效状态，所有完整性约束不被违反 | 关注数据正确性、约束满足 | 外键约束、唯一性约束、触发器 |
| **CAP一致性** | 所有节点在同一时间看到相同的数据，等同于原子一致性 | 关注副本间数据同步 | 分布式KV存储、数据库复制 |
| **最终一致性** | 如果系统在给定时间内没有新的更新，最终所有访问将返回最后更新的值 | 允许暂时的不一致 | Cassandra、DNS、S3 |
| **缓存一致性** | 多核CPU缓存间的数据一致性，确保所有核心看到相同的内存值 | 硬件层面的缓存同步 | MESI协议、CPU缓存 |
| **会话一致性** | 同一客户端会话内保证特定的一致性保证（读己之写、单调读等） | 限定在单个客户端会话 | 移动应用、边缘计算 |

**消解建议**：
- 在事务语境中指ACID一致性
- 在分布式系统复制语境中指CAP一致性
- 明确标注"最终一致性"或"缓存一致性"以区分

---

### 2. "分区"的不同含义

| 语境 | 定义 | 相关概念 | 处理方式 |
|:---|:---|:---|:---|
| **网络分区** | 网络故障导致系统被分割为无法通信的子集 | CAP定理、脑裂、分区容忍性 | Quorum机制、隔离策略 |
| **数据分区** | 将数据水平分割存储在不同节点 | Sharding、分片键、数据分布 | 范围分片、哈希分片、一致性哈希 |
| **Kafka分区** | 消息队列的并行单位，Topic的逻辑子集 | 消费者组、偏移量、并行度 | 增加分区提高并行度 |
| **硬盘分区** | 物理存储设备的逻辑划分 | 文件系统、挂载点 | fdisk、parted |
| **分区表** | 数据库中按分区键划分的大表 | 分区裁剪、分区交换 | MySQL分区、Hive分区 |

**消解建议**：
- 分布式系统语境通常指"网络分区"
- 数据库语境根据上下文区分"数据分区"和"表分区"
- Kafka语境明确指"Kafka分区"

---

### 3. "复制"的不同含义

| 语境 | 定义 | 关键特性 | 代表系统 |
|:---|:---|:---|:---|
| **数据复制** | 在多个节点维护相同数据副本 | 容错、可用性、读写分离 | MySQL主从、Redis复制 |
| **状态机复制** | 在多个副本上执行相同操作序列 | 确定性、容错、共识 | Raft、PBFT、ZooKeeper |
| **日志复制** | 将操作日志复制到多个节点 | 顺序保证、容错恢复 | Kafka、Paxos日志 |
| **虚拟内存复制** | Copy-on-Write机制，延迟物理复制 | 写时复制、内存优化 | fork()、容器镜像 |
| **DNA复制** | 生物学概念（分布式计算中偶见比喻） | - | - |

**消解建议**：
- 数据库语境通常指"数据复制"
- 共识算法语境指"状态机复制"
- 明确区分"同步复制"和"异步复制"

---

### 4. "Leader"的不同含义

| 语境 | 定义 | 选举方式 | 职责 |
|:---|:---|:---|:---|
| **Raft Leader** | Raft共识中的主节点 | 选举产生 | 处理写请求、复制日志 |
| **Kafka Leader** | Partition的主副本 | Controller指派 | 处理该Partition的读写 |
| **数据库Leader** | 主从复制中的主节点 | 手动配置或自动切换 | 处理写操作、复制到从节点 |
| **团队Leader** | 组织管理角色 | 任命或选举 | 团队协调、决策 |

**消解建议**：
- 明确上下文（如"Raft Leader"、"Kafka Leader"）
- 数据库语境可用"主节点"替代避免歧义

---

### 5. "提交"的不同含义

| 语境 | 定义 | 阶段 | 回滚可能性 |
|:---|:---|:---|:---:|
| **事务提交** | 事务的最终确认，使修改永久生效 | 事务最后阶段 | 不可 |
| **2PC提交** | 两阶段提交协议的第二阶段 | Phase 2 | 不可 |
| **Git提交** | 将变更保存到本地仓库 | 版本控制操作 | 可（reset/revert） |
| **代码提交** | 将代码变更提交到代码库 | 开发流程 | 可 |

**消解建议**：
- 数据库语境指"事务提交"
- 分布式事务语境可能指"2PC提交"
- 开发流程语境指"Git/代码提交"

---

### 6. "事务"的不同含义

| 语境 | 定义 | 特性 | ACID支持 |
|:---|:---|:---|:---:|
| **数据库事务** | 数据库操作的逻辑单元 | 原子性、隔离性 | ✅ |
| **分布式事务** | 跨多节点的原子操作 | 2PC/Saga/TCC实现 | ✅（弱） |
| **业务事务** | 业务流程中的业务单元 | 长运行、可能需要补偿 | 部分 |
| **区块链事务** | 区块链上的状态变更操作 | 不可篡改、需确认 | 最终 |

**消解建议**：
- 明确区分"本地事务"和"分布式事务"
- 区块链语境明确为"区块链事务"

---

### 7. "快照"的不同含义

| 语境 | 定义 | 用途 | 一致性 |
|:---|:---|:---|:---|
| **分布式快照** | 捕获分布式系统的全局状态 | 检查点恢复、调试 | 一致性割集 |
| **数据库快照** | 某一时刻数据库状态的只读副本 | 备份、报表、测试 | 崩溃一致 |
| **存储快照** | 存储卷的时间点副本 | 备份、克隆 | 崩溃一致 |
| **VM快照** | 虚拟机的完整状态保存 | 回滚、迁移 | 完整 |
| **Chandy-Lamport快照** | 特定算法实现的分布式快照 | 全局状态检测 | 一致性 |

**消解建议**：
- 分布式系统语境可能指"Chandy-Lamport快照"
- 数据库/存储语境指常规"快照"

---

### 8. "时钟"的不同含义

| 语境 | 定义 | 精度 | 用途 |
|:---|:---|:---:|:---|
| **物理时钟** | 真实时间（墙上时钟） | 纳秒/毫秒 | 超时、日志时间戳 |
| **逻辑时钟** | 捕获事件因果关系的计数器 | 整数 | 因果排序、并发检测 |
| **向量时钟** | 扩展的逻辑时钟，使用向量 | 向量 | 完整因果检测 |
| **TrueTime** | Spanner的物理时钟API | 不确定性区间 | 外部一致性 |
| **混合逻辑时钟** | 结合物理和逻辑时钟 | 时间戳+计数器 | CockroachDB使用 |

**消解建议**：
- 默认指"物理时钟"
- 分布式排序语境指"逻辑时钟"
- Spanner语境指"TrueTime"

---

### 9. "隔离"的不同含义

| 语境 | 定义 | 级别/类型 | 问题 |
|:---|:---|:---|:---|
| **事务隔离** | 并发事务间的隔离程度 | 读未提交到可串行化 | 脏读、幻读等 |
| **网络隔离** | 网络层面的访问控制 | VLAN、安全组 | 网络分区 |
| **资源隔离** | 系统资源的分离 | cgroups、容器 | 资源争抢 |
| **故障隔离** | 防止故障扩散 | 舱壁模式 | 级联故障 |

**消解建议**：
- 数据库语境指"事务隔离级别"
- 安全/容器语境指"网络/资源隔离"
- 可靠性语境指"故障隔离"

---

### 10. "Token"的不同含义

| 语境 | 定义 | 用途 | 有效期 |
|:---|:---|:---|:---:|
| **JWT Token** | 编码用户信息的JSON令牌 | 认证授权 | 可配置 |
| **CSRF Token** | 防止跨站请求伪造的令牌 | 安全验证 | 会话级 |
| **访问令牌** | OAuth中的访问凭证 | API访问 | 短期 |
| **刷新令牌** | 用于获取新访问令牌的凭证 | 续期 | 长期 |
| **Fencing Token** | 分布式锁的递增序号 | 防止延迟客户端问题 | 单次有效 |
| **令牌桶** | 限流算法中的虚拟令牌 | 流量控制 | 持续产生 |
| **区块链Token** | 加密代币 | 价值转移 | 永久 |

**消解建议**：
- 认证语境指"JWT/访问/刷新 Token"
- 分布式锁语境指"Fencing Token"
- 限流语境指"令牌桶"

## 二、同义词对照表

### 1. 架构组件同义词

| 术语1 | 术语2 | 使用偏好 | 说明 |
|:---|:---|:---|:---|
| Leader | Primary | 共识/存储用Leader | Leader更常见于Raft/Redis，Primary常见于数据库 |
| Leader | Master | 数据库用Primary | Master被认为有奴隶制含义，逐渐弃用 |
| Follower | Replica | 一般等价 | Follower强调角色，Replica强调副本 |
| Follower | Secondary | 数据库语境 | MongoDB用Secondary |
| Slave | Replica | 避免使用Slave | Slave有负面含义，建议用Replica |
| Worker | Executor | 计算框架 | Spark用Executor，其他常用Worker |
| Node | Instance | 云语境 | Instance更常用于云服务 |
| Node | Peer | P2P网络 | Peer强调对等性 |
| Cluster | Fleet | 一般等价 | Fleet更常用于容器编排 |
| Shard | Partition | 一般等价 | 分片/分区，Shard更常用于数据库 |

### 2. 操作同义词

| 术语1 | 术语2 | 使用偏好 | 说明 |
|:---|:---|:---|:---|
| Commit | Persist | 事务用Commit | Commit强调确认，Persist强调持久化 |
| Rollback | Abort | 一般等价 | Abort更常用于协议描述 |
| Compensate | Undo | Saga用Compensate | Compensate指业务补偿，Undo指技术回滚 |
| Retry | Resubmit | 一般等价 | Resubmit更强调重新提交 |
| Replicate | Copy | 分布式用Replicate | Replicate强调持续同步，Copy是单次 |
| Heartbeat | Keepalive | 一般等价 | Keepalive更常用于TCP/连接 |
| Failover | Switchover | 故障用Failover | Failover是自动/故障，Switchover是手动/计划 |
| Scale out | Horizontal scaling | 完全等价 | 水平扩展 |
| Scale up | Vertical scaling | 完全等价 | 垂直扩展 |

### 3. 概念同义词

| 术语1 | 术语2 | 使用偏好 | 说明 |

| Availability | Uptime | 用Availability | Uptime是度量，Availability是属性 |
| Durability | Persistence | 用Durability | Durability是ACID属性，Persistence是一般持久化 |

## 三、易混淆术语对比

### 1. 一致性模型对比

```
线性一致性  vs  顺序一致性  vs  因果一致性
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
线性一致性：与全局实时序一致，最强
顺序一致性：所有进程看到一致顺序，但不要求与实时序一致
因果一致性：只有因果关系的事件需有序

关系：线性一致性 > 顺序一致性 > 因果一致性 > 最终一致性
```

### 2. 共识算法对比

```
Paxos  vs  Raft  vs  PBFT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Paxos：理论正确，难实现，无领导者概念
Raft：易理解，强领导者，功能等价于Multi-Paxos
PBFT：处理拜占庭故障，需要n≥3f+1

选择：
- 崩溃容错 → Paxos/Raft
- 易理解性优先 → Raft
- 拜占庭容错 → PBFT/HotStuff
```

### 3. 复制模式对比

```
主从复制  vs  多主复制  vs  无主复制
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
主从复制：单主写，多从读，简单但有瓶颈
多主复制：多主写，需冲突解决，适合地理分布
无主复制：无主节点，Quorum读写，高可用但需读修复

一致性：主从 > 多主 > 无主（通常）
可用性：无主 > 多主 > 主从（通常）
```

### 4. 事务模式对比

```
2PC  vs  Saga  vs  TCC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2PC：强一致性，阻塞，短事务
Saga：最终一致，长事务，补偿机制
TCC：预留资源，业务侵入，高性能

选择：
- 强一致性必需 → 2PC
- 长业务流程 → Saga
- 高并发短事务 → TCC
```

### 5. 计算模式对比

```
批处理  vs  流处理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
批处理：处理 bounded 数据，高吞吐，高延迟
流处理：处理 unbounded 数据，低延迟，持续处理

代表：
- 批处理：MapReduce, Spark
- 流处理：Flink, Kafka Streams
- 混合：Spark Structured Streaming（微批）
```

### 6. 部署策略对比

```
蓝绿部署  vs  金丝雀发布  vs  滚动更新
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
蓝绿部署：两套环境切换，快速回滚，资源成本高
金丝雀发布：渐进发布，风险控制，需要监控
滚动更新：逐个替换，资源友好，回滚慢

选择：
- 关键系统，资源充足 → 蓝绿部署
- 需要验证新功能 → 金丝雀发布
- 常规更新 → 滚动更新
```

## 四、语境依赖术语

以下术语含义强烈依赖上下文，使用时需明确语境：

| 术语 | 可能含义 | 建议用法 |
|:---|:---|:---|
| **Node** | 集群节点、Kubernetes节点、图节点、链表节点 | 明确前缀："K8s Node"、"集群Node" |
| **Cluster** | 数据库集群、K8s集群、计算集群 | 明确类型："Redis Cluster"、"K8s Cluster" |
| **Master** | 数据库主节点、K8s控制平面、主分支 | 避免使用，改用Primary/Control Plane |
| **Controller** | K8s控制器、MVC控制器、硬件控制器 | 明确上下文 |
| **Scheduler** | K8s调度器、操作系统调度器、任务调度器 | 明确前缀 |
| **Proxy** | 反向代理、正向代理、数据库代理 | 明确类型 |
| **Client** | 应用程序客户端、数据库客户端、API客户端 | 明确服务 |
| **Session** | HTTP会话、数据库会话、TCP会话 | 明确协议 |
| **Lock** | 数据库锁、分布式锁、文件锁 | 明确类型 |
| **Queue** | 消息队列、任务队列、优先队列 | 明确用途 |
