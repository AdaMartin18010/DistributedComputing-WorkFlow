## 二、容量预估与性能计算

### 2.1 容量预估方法论

#### 容量预估公式

```
【基础公式】
峰值QPS = 日活用户 × 日均访问次数 ÷ 86400 × 峰值系数

【示例计算】
- 日活用户(DAU): 1000万
- 人均访问次数: 30次/天
- 峰值系数: 5 (峰值是平均的5倍)

平均QPS = 10,000,000 × 30 ÷ 86400 ≈ 3,472 QPS
峰值QPS = 3,472 × 5 ≈ 17,361 QPS

考虑30%冗余:
设计峰值QPS = 17,361 × 1.3 ≈ 22,570 QPS
```

#### 各层容量估算

| 层次 | 容量指标 | 计算依据 | 设计值 |
|------|----------|----------|--------|
| **CDN** | 带宽 | 页面大小×PV | 100Gbps |
| **Nginx** | 并发连接 | 峰值QPS×连接保持时间 | 50,000 |
| **应用服务** | 实例数 | 峰值QPS÷单机QPS | 50台 (每台500QPS) |
| **Redis** | 内存 | 热数据大小×1.5 | 100GB |
| **MySQL** | 连接数 | 应用实例×连接池 | 2,000 |
| **消息队列** | 吞吐量 | 峰值写入量×2 | 100,000 msg/s |

### 2.2 性能计算模型

#### 延迟预算分配

```
【目标】端到端延迟 < 200ms (P99)

延迟预算分配:
├─ 网络传输: 20ms (10%)
├─ 负载均衡: 5ms (2.5%)
├─ 网关层: 10ms (5%)
├─ 业务逻辑: 80ms (40%)
│  ├─ 缓存读取: 5ms
│  ├─ 数据库查询: 30ms
│  ├─ 外部服务调用: 30ms
│  └─ 业务处理: 15ms
├─ 消息队列: 20ms (10%)
└─ 预留缓冲: 65ms (32.5%)
```

#### 并发度计算

```
【利特尔法则】
并发数 = 吞吐量 × 平均响应时间

【示例】
目标吞吐量: 10,000 QPS
平均响应时间: 0.1秒

所需并发数 = 10,000 × 0.1 = 1,000

考虑线程池配置:
- 核心线程数: 200
- 最大线程数: 1,000
- 队列容量: 5,000
```

## 三、架构设计模式

### 3.1 分层架构

#### 经典四层架构

```
┌─────────────────────────────────────────────────────────┐
│                     接入层 (Access Layer)                │
│  DNS → CDN → 负载均衡 (LVS/Nginx/HAProxy)               │
├─────────────────────────────────────────────────────────┤
│                     网关层 (Gateway Layer)               │
│  限流 → 鉴权 → 路由 → 协议转换 → 熔断                    │
├─────────────────────────────────────────────────────────┤
│                     业务层 (Service Layer)               │
│  订单服务 → 库存服务 → 支付服务 → 用户服务               │
├─────────────────────────────────────────────────────────┤
│                     数据层 (Data Layer)                  │
│  Redis → MySQL → MongoDB → Elasticsearch → Kafka        │
└─────────────────────────────────────────────────────────┘
```

#### 分层架构设计原则

| 原则 | 说明 | 实践建议 |
|------|------|----------|
| **单一职责** | 每层只负责特定功能 | 接入层只做流量接入，不做业务逻辑 |
| **依赖倒置** | 上层依赖下层，下层不依赖上层 | 通过接口解耦层间依赖 |
| **关注点分离** | 技术细节与业务逻辑分离 | 网关处理横切关注点 |
| **数据隐藏** | 下层数据对上层透明 | 数据库结构不暴露给上层 |

### 3.2 微服务架构

#### 微服务拆分策略

```
【拆分维度】
├─ 按业务领域 (DDD)
│  └─ 订单域、库存域、支付域、用户域
├─ 按数据归属
│  └─ 订单数据、库存数据、支付数据
├─ 按功能特性
│  └─ 读服务、写服务、计算服务
└─ 按团队组织
   └─ 订单团队、支付团队、风控团队

【拆分粒度决策树】
是否需要独立部署?
├── 是 → 是否需要独立扩展?
│   ├── 是 → 是否需要独立技术栈?
│   │   ├── 是 → 独立微服务
│   │   └── 否 → 考虑共享库
│   └── 否 → 考虑模块划分
└── 否 → 保持单体或模块
```

#### 微服务通信模式

| 通信方式 | 适用场景 | 优点 | 缺点 |
|----------|----------|------|------|
| **同步HTTP** | 实时性要求高的查询 | 简单、通用 | 阻塞、级联故障 |
| **同步gRPC** | 内部服务高性能调用 | 高效、强类型 | 需要Stub生成 |
| **异步消息** | 事件驱动、削峰填谷 | 解耦、可靠 | 最终一致性 |
| **异步事件** | 状态变更通知 | 松耦合 | 顺序保证复杂 |

### 3.3 事件驱动架构

#### 事件驱动架构模式

```
【基础架构】
生产者 → 事件总线 → 消费者
   ↓         ↓          ↓
订单服务   Kafka    库存服务
           /    \     物流服务
      支付服务  通知服务

【模式1: 事件通知】
订单创建 → OrderCreatedEvent → 库存扣减
                              → 短信通知
                              → 数据分析

【模式2: 事件溯源】
状态 = fold(初始状态, [事件1, 事件2, ...事件n])

【模式3: CQRS】
命令模型(写) ←→ 事件存储 ←→ 查询模型(读)
                    ↓
                投影更新
```

#### 事件驱动设计要点

| 要点 | 设计建议 |
|------|----------|
| **事件设计** | 事件包含完整上下文，避免消费者回查 |
| **事件顺序** | 使用分区键保证同一实体的事件顺序 |
| **事件去重** | 消费者幂等处理，支持至少一次语义 |
| **事件版本** | 事件Schema支持向前/向后兼容 |
| **死信队列** | 处理失败事件，支持重试和人工介入 |

## 四、数据一致性设计

### 4.1 一致性模型选择

#### 一致性强度对比

| 一致性级别 | 描述 | 延迟 | 可用性 | 适用场景 |
|------------|------|------|--------|----------|
| **强一致性** | 读操作返回最新写入 | 高 | 低 | 金融交易、库存扣减 |
| **顺序一致性** | 操作按程序顺序执行 | 中 | 中 | 分布式锁 |
| **因果一致性** | 因果相关的操作有序 | 中 | 中 | 社交网络、评论 |
| **最终一致性** | 最终所有副本一致 | 低 | 高 | 商品详情、用户资料 |
| **弱一致性** | 不保证一致性 | 最低 | 最高 | 日志收集、监控 |

### 4.2 分布式事务方案

#### 事务方案选型决策树

```
是否需要强一致性?
├── 是 → 事务执行时间?
│   ├── 短事务(<1s) → 2PC/TCC
│   ├── 中事务(<1min) → TCC/Saga
│   └── 长事务(>1min) → Saga
└── 否 → 最终一致性方案
    ├── 消息队列 + 本地事务
    ├── 最大努力通知
    └── 定时对账补偿
```

#### 事务方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| **2PC** | 强一致性 | 低 | 中 | 短事务、强一致要求 |
| **TCC** | 强一致性 | 中 | 高 | 电商交易、需要预留资源 |
| **Saga** | 最终一致性 | 高 | 中 | 长事务、跨服务编排 |
| **本地消息表** | 最终一致性 | 高 | 中 | 异步场景、消息可靠投递 |
| **最大努力通知** | 弱一致性 | 最高 | 低 | 对账、通知类场景 |

## 五、可用性设计

### 5.1 多活架构设计

#### 多活架构模式

```
【同城双活】
        用户流量
           ↓
    ┌─────┴─────┐
    ↓           ↓
  机房A       机房B
  (主)        (备)
    ↑           ↑
    └─────┬─────┘
        数据库
      (主从同步)

【异地多活】
        全局负载均衡 (GSLB)
              ↓
    ┌─────────┼─────────┐
    ↓         ↓         ↌
  北京       上海       深圳
  ├─用户服务  ├─用户服务  ├─用户服务
  ├─订单服务  ├─订单服务  ├─订单服务
  └─数据库    └─数据库    └─数据库
   (主从)     (主从)     (主从)
```

#### 多活数据同步策略

| 策略 | 描述 | 一致性 | 适用场景 |
|------|------|--------|----------|
| **主从复制** | 一主多从，异步复制 | 最终一致 | 读多写少 |
| **双主复制** | 双主互备，双向同步 | 冲突风险 | 双活架构 |
| **分片多活** | 数据分片，各机房负责不同分片 | 强一致 | 单元化架构 |
| **CRDT** | 无冲突复制数据类型 | 自动合并 | 协同编辑 |

### 5.2 灾备设计

#### 灾备等级 (RTO/RPO)

| 等级 | 名称 | RTO | RPO | 架构 |
|------|------|-----|-----|------|
| Tier 1 | 基础备份 | 数天 | 24小时 | 定期冷备 |
| Tier 2 | 冷备 | 数小时 | 数小时 | 备用数据中心 |
| Tier 3 | 温备 | 1小时 | 数分钟 | 热备节点 |
| Tier 4 | 热备 | 分钟级 | 近零 | 实时同步 |
| Tier 5 | 双活 | 近零 | 零 | 双活架构 |

#### 故障转移策略

```
【故障检测】
健康检查 → 心跳超时 → 故障判定 → 触发切换

【切换流程】
1. 故障检测 (10s)
2. 故障确认 (20s)
3. 流量切换 (5s)
4. 数据一致性检查 (30s)
5. 服务恢复确认

【回切策略】
- 自动回切: 故障恢复后自动切换回原节点
- 手动回切: 人工确认后回切
- 不自动回切: 保持当前状态，避免震荡
```

## 六、扩展性设计

### 6.1 水平扩展策略

#### 无状态服务扩展

```
【扩展模型】
用户请求 → 负载均衡器 → 服务实例池
                           ↓
                    ┌───┬───┬───┬───┐
                    │S1 │S2 │S3 │S4 │
                    └───┴───┴───┴───┘
                     ↑   ↑   ↑   ↑
                    可水平扩展的无状态服务

【关键设计】
- 会话外置: Session存储在Redis
- 配置外置: 配置中心统一管理
- 日志外置: 日志收集到ELK
- 状态外置: 业务状态存储在数据库
```

#### 有状态服务扩展

| 扩展方式 | 描述 | 复杂度 | 适用场景 |
|----------|------|--------|----------|
| **数据分片** | 按Key分片到不同节点 | 中 | 缓存、数据库 |
| **一致性哈希** | 环形哈希空间分配 | 中 | 缓存集群 |
| **范围分片** | 按数据范围分片 | 低 | 时序数据 |
| **列表分片** | 按列表元素分配 | 低 | 消息队列 |

### 6.2 数据分片策略

#### 分片策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **哈希分片** | 分布均匀，简单 | 范围查询困难 | 点查为主 |
| **范围分片** | 范围查询高效 | 热点风险 | 时序数据 |
| **列表分片** | 灵活控制 | 维护复杂 | 数据迁移 |
| **复合分片** | 结合优点 | 实现复杂 | 复杂场景 |

#### 分片键选择原则

```
【选择原则】
1. 高频查询条件: 最常用的查询字段
2. 数据均匀: 避免数据倾斜
3. 业务稳定性: 不经常变更的字段
4. 单调性避免: 避免自增ID导致热点

【示例】
电商订单表:
- 用户ID: 好 (高频查询，分布均匀)
- 订单ID: 中 (点查高效，范围查询困难)
- 时间: 差 (热点严重)
- 商家ID: 中 (商家间分布不均)
```

【纵深防御体系】

### 7.2 认证授权设计

#### 认证方案对比

| 方案 | 描述 | 适用场景 |
|------|------|----------|
| **JWT** | 无状态Token，自包含信息 | 微服务内部认证 |
| **OAuth2** | 第三方授权标准 | 开放API、第三方登录 |
| **OIDC** | 基于OAuth2的身份层 | 单点登录 |

#### 权限模型

```
【RBAC模型】
用户 → 角色 → 权限 → 资源

【ABAC模型】
访问决策 = f(主体属性, 资源属性, 环境属性, 操作)

【混合模型】
- 粗粒度: RBAC (角色)
- 细粒度: ABAC (数据范围、时间等)
```

## 八、成本优化

### 8.1 成本构成分析

```
【云服务成本构成】
计算成本 (40%): EC2/ECS/容器实例
存储成本 (30%): 块存储/对象存储/数据库
网络成本 (15%): 带宽/CDN/跨区流量
其他成本 (15%): 日志/监控/安全

【优化优先级】
1. 计算资源优化 (ROI最高)
2. 存储分层策略
3. 网络流量优化
4. 预留实例/Spot实例
```

### 8.2 成本优化策略

| 策略 | 具体措施 | 预期节省 |
|------|----------|----------|
| **自动伸缩** | 根据负载自动调整实例数 | 30-50% |
| **Spot实例** | 使用抢占式实例处理批处理任务 | 60-90% |
| **存储分层** | 热数据SSD，冷数据OSS | 50-70% |
| **缓存优化** | 提高缓存命中率，减少数据库查询 | 20-30% |
| **压缩优化** | 启用HTTP压缩，减少传输量 | 50-80% |
| **CDN优化** | 提高缓存命中率，减少回源 | 30-50% |

### 9.1 案例背景

**系统名称**: 分布式电商订单系统
**业务场景**: 支持百万级并发的B2C电商平台
**核心功能**: 商品浏览、购物车、订单、支付、物流

### 9.2 需求分析

```
【功能性需求】
- 用户注册/登录/管理
- 商品浏览/搜索/推荐
- 购物车管理
- 订单创建/支付/查询
- 物流跟踪

【非功能性需求】
- 峰值QPS: 100,000
- 平均响应: <200ms
- 可用性: 99.99%
- 数据一致性: 订单强一致，库存最终一致
- 扩展性: 支持10倍增长
```

### 9.3 架构设计

#### 整体架构图

```
┌───────────────────────────────────────────────────────────────────┐
│                          接入层                                    │
│  DNS → CDN(静态资源) → GSLB → 边界网关(限流/防刷)                 │
└───────────────────────────────────────────────────────────────────┘
                              ↓
┌───────────────────────────────────────────────────────────────────┐
│                          网关层                                    │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐                 │
│  │ 认证网关 │ │ 限流网关 │ │ 路由网关 │ │ 协议转换 │                 │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘                 │
└───────────────────────────────────────────────────────────────────┘
                              ↓
┌───────────────────────────────────────────────────────────────────┐
│                          业务层 (微服务)                           │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐     │
│  │ 用户服务 │ │ 商品服务 │ │ 订单服务 │ │ 库存服务 │ │ 支付服务 │     │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘     │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐                             │
│  │ 物流服务 │ │ 营销服务 │ │ 搜索服务 │                             │
│  └─────────┘ └─────────┘ └─────────┘                             │
└───────────────────────────────────────────────────────────────────┘
                              ↓
┌───────────────────────────────────────────────────────────────────┐
│                          数据层                                    │
│  Redis集群(缓存)  MySQL集群(事务)  ES(搜索)  Kafka(消息)          │
│  MongoDB(商品)   TiDB(订单历史)   HBase(日志)                     │
└───────────────────────────────────────────────────────────────────┘
```

#### 核心服务设计

**订单服务核心流程**

```
【订单创建流程】
1. 参数校验 → 2. 库存预扣(TCC) → 3. 创建订单(本地事务) 
→ 4. 发送创建事件 → 5. 返回订单号

【订单状态机】
待支付 → 已支付 → 已发货 → 已签收 → 已完成
  ↓        ↓        ↓
已取消   退款中   已退货

【补偿设计】
- 支付超时: 释放库存，取消订单
- 支付失败: 退款，释放库存
- 发货失败: 退款，取消订单
```

### 9.4 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| **服务划分** | DDD领域划分 | 6个核心领域，清晰边界 |
| **订单一致性** | TCC事务 | 强一致要求，支持回滚 |
| **库存一致性** | Redis+DB最终一致 | 高性能，可接受短暂不一致 |
| **数据分片** | 订单按用户ID哈希 | 用户维度查询为主 |
| **缓存策略** | 多级缓存 | L1本地，L2 Redis，L3 DB |
| **消息队列** | Kafka | 高吞吐，顺序保证 |

### 9.5 性能优化措施

```
【读写分离】
- 写: 主库
- 读: 从库 + 缓存

【缓存策略】
- 商品详情: CDN + Redis，TTL 1小时
- 库存: Redis + 本地缓存，TTL 5分钟
- 用户会话: Redis，TTL 30分钟

【异步处理】
- 订单创建后: 异步发送通知、积分计算
- 支付成功: 异步更新统计、触发营销

【数据库优化】
- 订单表: 按用户ID分32库，每库32表
- 索引优化: 覆盖索引减少回表
- 读写分离: 一主两从
```

### 9.6 高可用设计

```
【多活架构】
- 同城双活: 同城两个机房
- 异地灾备: 异地一个灾备中心
- 数据同步: 主从复制 + Canal同步

【故障隔离】
- 舱壁模式: 服务隔离，故障不扩散
- 熔断降级: Hystrix熔断，降级策略
- 限流: 网关层 + 服务层双重限流

【监控告警】
- 指标: QPS/延迟/错误率/饱和度
- 链路: 分布式追踪
- 日志: ELK集中收集
- 告警: 分级告警，自动恢复
```
