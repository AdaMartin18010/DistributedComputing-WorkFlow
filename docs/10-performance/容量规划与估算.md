## 📋 执行摘要

容量规划是系统架构设计的核心环节，通过科学的估算方法预测系统资源需求，确保系统在预期负载下稳定运行，同时避免资源浪费。

---

## 一、核心概念

### 1.1 定义与原理

容量规划（Capacity Planning）是指根据业务增长预测，评估系统所需的计算、存储、网络等资源，并制定相应的扩展策略的过程。

**核心原理**：
- **需求预测**：基于历史数据和业务增长趋势预测未来负载
- **资源建模**：建立业务指标与系统资源消耗的映射关系
- **安全边际**：预留缓冲容量应对突发流量和预测误差
- **成本控制**：在满足性能要求的前提下优化资源投入

### 1.2 关键特性

- **前瞻性**：提前规划，避免系统成为业务瓶颈
- **数据驱动**：基于真实数据和科学模型进行估算
- **动态调整**：根据实际运行情况持续优化规划
- **成本效益**：平衡性能需求与资源成本

### 1.3 适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 新业务上线 | ⭐⭐⭐⭐⭐ | 评估初始资源配置需求 |
| 业务增长期 | ⭐⭐⭐⭐⭐ | 预测扩容时机和规模 |
| 大促/活动 | ⭐⭐⭐⭐⭐ | 评估峰值流量承载能力 |
| 系统重构 | ⭐⭐⭐⭐ | 重新评估资源需求模型 |
| 成本优化 | ⭐⭐⭐⭐ | 识别资源浪费，优化配置 |

## 二、技术细节

### 2.1 流量估算方法

#### 2.1.1 基于历史数据的趋势预测

```
预估流量 = 历史峰值 × 增长率 × 季节系数 × 活动系数

示例：
- 历史峰值：10万 QPS
- 年增长率：50%
- 双十一系数：3倍
- 预估峰值 = 10万 × 1.5 × 3 = 45万 QPS
```

#### 2.1.2 漏斗模型估算

```
日活跃用户(DAU) → 功能渗透率 → 操作频次 → 峰值系数

示例：
- DAU: 1000万
- 功能渗透率: 30%
- 人均操作: 5次/天
- 峰值系数: 5（按二八法则，20%时间产生80%流量）

日请求量 = 1000万 × 30% × 5 = 1500万
峰值QPS = 1500万 × 80% ÷ (24×3600×20%) ≈ 694 QPS
```

#### 2.1.3 业务指标推导

| 业务指标 | 计算方法 | 示例 |
|---------|---------|------|
| 订单量推导 | 订单QPS × 页面转化漏斗 | 1万单/秒 × 10页面 = 10万 PV/s |
| 用户行为 | DAU × 功能使用频次 ÷ 时间 | 100万 × 10次 ÷ 86400秒 |
| 第三方依赖 | 主业务QPS × 调用倍数 | 1万 × 5次外部调用 = 5万 |

### 2.2 存储容量规划

#### 2.2.1 存储需求计算公式

```
总存储 = 数据量 × 副本数 × (1 + 冗余率) × 增长周期

分项计算：
- 原始数据：日均新增 × 保留天数
- 索引数据：原始数据 × 索引系数(通常0.3-0.5)
- 日志数据：日均日志 × 保留天数
- 备份数据：全量数据 × 备份保留份数
```

#### 2.2.2 存储类型规划

| 存储类型 | 适用场景 | 容量规划要点 |
|---------|---------|-------------|
| 热存储(SSD) | 高频访问数据 | 存放7-30天活跃数据 |
| 温存储(SATA) | 中期归档 | 存放3-12个月数据 |
| 冷存储(对象) | 历史归档 | 存放1年以上数据 |
| 归档存储 | 合规留存 | 按法规要求长期保存 |

#### 2.2.3 存储增长模型

```
当前容量：10TB
日增长：100GB
年增长率：200%

1年后容量需求 = 10TB + 100GB×365 + 10TB×200%
              = 10TB + 36.5TB + 20TB
              = 66.5TB

建议预留30%缓冲：66.5TB × 1.3 = 86.5TB
```

### 2.3 带宽需求计算

#### 2.3.1 带宽计算公式

```
带宽需求 = 峰值QPS × 平均响应大小 × 8(bit) × 并发系数

示例：
- 峰值QPS: 10,000
- 平均响应: 50KB
- 并发系数: 1.5（考虑突发和重传）

带宽 = 10,000 × 50KB × 8 × 1.5
     = 6,000,000 Kbps
     = 6 Gbps
```

#### 2.3.2 内外网带宽区分

| 流量类型 | 计算方法 | 说明 |
|---------|---------|------|
| 外网入口 | 用户请求 × 请求大小 | 用户上传、表单提交 |
| 外网出口 | 响应QPS × 响应大小 | 页面、API响应 |
| 内网东西向 | 服务间调用 × 数据量 | 微服务通信 |
| 内网南北向 | 数据库/缓存访问 | 数据层交互 |

### 2.4 QPS与并发计算

#### 2.4.1 基础公式

```
QPS (每秒查询数) = 并发数 ÷ 平均响应时间(RT)

推导公式：
并发数 = QPS × 平均RT
系统承载QPS = 单机QPS × 机器数量
```

#### 2.4.2 并发度估算

| 场景 | 估算方法 | 示例 |
|-----|---------|------|
| Web应用 | 日PV ÷ (86400 × 页面平均停留时间) | 1000万PV ÷ 10秒 = 100万并发 |
| API服务 | 峰值QPS × 平均RT | 1万 × 0.1秒 = 1000并发 |
| 长连接 | 在线用户数 × 在线率 | 100万 × 10% = 10万连接 |

#### 2.4.3 线程池配置估算

```
核心线程数 = CPU核数 × (1 + 等待时间/计算时间)
最大线程数 = 核心线程数 × 2
队列容量 = 核心线程数 × 平均RT × 容忍等待倍数

示例（4核CPU，IO密集型，RT=100ms）：
- 核心线程 = 4 × (1 + 90/10) = 40
- 最大线程 = 80
- 队列容量 = 40 × 0.1 × 10 = 400
```

### 2.5 扩容策略

#### 2.5.1 扩容触发条件

| 指标 | 阈值 | 扩容策略 |
|-----|------|---------|
| CPU使用率 | >70%持续5分钟 | 水平扩容 |
| 内存使用率 | >80%持续10分钟 | 垂直扩容或水平扩容 |
| 响应时间P99 | 超过SLA 20% | 紧急扩容 |
| 错误率 | >0.1% | 检查+扩容 |
| 连接数 | >80% | 提前扩容 |

#### 2.5.2 扩容模式

```
┌─────────────────────────────────────────────────────┐
│                  扩容策略矩阵                        │
├─────────────┬─────────────────┬────────────────────┤
│    维度     │    垂直扩容      │     水平扩容        │
├─────────────┼─────────────────┼────────────────────┤
│ 实施速度    │ 慢（需重启）     │ 快（秒级）          │
│ 成本效率    │ 低（资源闲置）   │ 高（按需使用）      │
│ 技术复杂度  │ 低              │ 高（需分布式支持）   │
│ 适用场景    │ 单体应用        │ 微服务/无状态服务   │
│ 上限限制    │ 单机物理上限    │ 理论上无上限        │
└─────────────┴─────────────────┴────────────────────┘
```

#### 2.5.3 自动扩缩容策略

```yaml
# Kubernetes HPA 示例
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-service
  minReplicas: 3
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Percent
        value: 100
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
```

## 四、实践指南

### 4.1 容量规划Checklist

```
□ 收集历史流量数据（至少3个月）
□ 分析业务增长趋势
□ 识别流量峰值模式（日/周/季节性）
□ 列出重大业务活动日历
□ 评估单点性能上限
□ 计算资源需求（CPU/内存/存储/带宽）
□ 设计容灾和冗余方案
□ 制定扩容触发条件和流程
□ 建立容量监控和告警
□ 定期复盘和优化模型
```

### 4.2 最佳实践

1. **保守预估，激进扩容**
   - 预估时保留20-30%安全余量
   - 扩容决策要果断，避免被动救火

2. **分层规划**
   - 接入层、应用层、数据层分别评估
   - 识别系统瓶颈点

3. **压测验证**
   - 任何容量规划都需经过压测验证
   - 建立容量基线并定期回归

4. **成本意识**
   - 区分核心与非核心服务的资源投入
   - 利用Spot实例、自动缩容降低成本

5. **文档化**
   - 记录容量决策依据
   - 保持规划与实际的对照分析

### 4.3 常见问题

**Q1: 没有历史数据如何规划？**
A: 采用同类业务对比法，参考行业基准数据，同时设置更频繁的监控和调整机制。

**Q2: 如何应对黑天鹅事件？**
A: 建立熔断降级机制，预留紧急扩容通道，核心服务采用多活架构。

**Q3: 容量规划多久更新一次？**
A: 常规业务每季度评估，快速增长期每月评估，大促前专项评估。

## 五、形式化分析

### 5.1 排队论模型

**M/M/c 模型**适用于多数Web服务容量规划：

```
λ = 到达率（请求/秒）
μ = 服务率（请求/秒/服务器）
c = 服务器数量
ρ = λ/(cμ) < 1 （系统稳定条件）

平均等待时间 Wq = (C(c,ρ) × ρ) / (λ × (1-ρ))
其中 C(c,ρ) 为Erlang C公式
```

### 5.2 容量利用率与延迟关系

```
当利用率 ρ 接近1时，延迟急剧上升：

平均响应时间 = 服务时间 / (1 - ρ)

ρ = 50%  →  2倍服务时间
ρ = 80%  →  5倍服务时间  
ρ = 90%  →  10倍服务时间
ρ = 95%  →  20倍服务时间
```

**建议**：生产环境保持CPU利用率低于70%，为突发流量预留空间。
