# 混沌工程

## 概述

混沌工程（Chaos Engineering）是通过在生产环境中有意注入故障，来验证系统韧性和可靠性的实践方法。由Netflix首创，现已成为构建高可用分布式系统的标准实践。核心理念是"通过主动制造故障来避免故障"。

## 混沌工程原则

├─────────────────────────────────────────────────────────────────────────┤

│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  1. 建立稳定状态的假设                                            │   │
│  │     • 定义系统的正常行为指标                                      │   │
│  │     • 如：错误率 < 1%，P99延迟 < 500ms                           │   │
│  │     • 稳定状态应是可测量的、可自动化的                            │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  2. 多样化真实世界的事件                                          │   │
│  │     • 模拟真实故障场景                                            │   │
│  │     • 硬件故障：服务器宕机、磁盘损坏                              │   │
│  │     • 网络故障：延迟、丢包、分区                                  │   │
│  │     • 依赖故障：数据库超时、缓存失效                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  3. 在生产环境中运行                                              │   │
│  │     • 测试环境与生产环境永远有差异                                │   │
│  │     • 真实用户流量才能暴露真实问题                                │   │
│  │     • 需严格控制爆炸半径                                          │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  4. 持续自动化运行                                                │   │
│  │     • 手动实验无法规模化                                          │   │
│  │     • 集成到CI/CD流水线                                           │   │
│  │     • 定期自动执行，持续验证                                      │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  5. 最小化爆炸半径                                                │   │
│  │     • 使用金丝雀环境限制影响范围                                  │   │
│  │     • 可随时终止实验                                              │   │
│  │     • 准备回滚和应急预案                                          │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

## 实验场景设计

```yaml
# chaos-scenarios.yaml
scenarios:
  # 场景1: 单点故障恢复
  single_point_failure:
    description: "验证单Pod故障时的自动恢复能力"
    target: "order-service"
    experiment:
      - type: pod-kill
        count: 1
        expected:
          - 服务自动重启
          - 流量自动转移
          - 错误率<1%
          - 恢复时间<30s
          
  # 场景2: 网络分区
  network_partition:
    description: "验证脑裂场景下的数据一致性"
    target: "database-cluster"
    experiment:
      - type: network-partition
        mode: partial
        duration: 5m
        expected:
          - 选出新主节点
          - 数据不丢失
          - 自动恢复后同步
          
  # 场景3: 级联故障
  cascading_failure:
    description: "验证熔断和降级机制"
    steps:
      - inject: database-latency
        value: 5s
      - check: circuit-breaker-trigger
      - check: fallback-activated
      - check: user-experience-degraded
    expected:
      - 熔断器打开
      - 降级服务响应
      - 核心功能可用
      
  # 场景4: 资源耗尽
  resource_exhaustion:
    description: "验证资源限制和弹性伸缩"
    steps:
      - inject: memory-stress
        value: 95%
      - check: oom-killer
      - check: hpa-scaling
    expected:
      - Pod重启
      - HPA自动扩容
      - 服务持续可用
      
  # 场景5: 依赖故障
  dependency_failure:
    description: "验证第三方依赖故障时的优雅降级"
    steps:
      - inject: dns-failure
        target: "third-party-api.com"
      - check: timeout-handling
      - check: cache-hit-rate
    expected:
      - 超时控制
      - 缓存兜底
      - 用户无感知
```

## 稳态检测

```yaml
# steady-state.yaml
steady_state:
  # HTTP健康检查
  http_health:
    endpoint: http://service/health
    interval: 5s
    timeout: 3s
    expected_status: 200
    
  # 指标检查
  metrics_check:
    error_rate:
      query: |
        sum(rate(http_requests_total{status=~"5.."}[1m]))
        /
        sum(rate(http_requests_total[1m]))
      threshold: 0.01  # 1%
      
    latency_p99:
      query: |
        histogram_quantile(0.99,
          sum(rate(http_request_duration_seconds_bucket[1m])) by (le)
        )
      threshold: 1.0  # 1s
      
    availability:
      query: |
        up{job="order-service"}
      threshold: 0.9
      
  # 自定义检查脚本
  custom_check:
    script: |
      #!/bin/bash
      # 检查关键业务流程
      curl -s http://service/api/orders | jq '.status' | grep -q "success"
      exit $?
    interval: 10s
```

## 最佳实践

1. **从开发环境开始**：先在非生产环境验证实验
2. **控制爆炸半径**：使用金丝雀环境、限流机制
3. **可观测性先行**：确保有完整的监控和告警
4. **准备应急预案**：随时可终止实验、快速回滚
5. **持续改进**：根据实验结果优化系统和流程
