# Go性能优化：GC/协程/内存池

## 核心概念

### 1. Go内存管理

Go具有自动垃圾回收机制，但不当的内存使用仍会导致性能问题。

**Go内存布局**：
```
┌─────────────────────────────────────────────────────────────┐
│                    Go内存结构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 堆内存 (Heap)                        │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │          spans (mheap)                       │   │   │
│  │  │  按大小划分的内存块，用于对象分配            │   │   │
│  │  │  - mspan: 内存管理基本单元                   │   │   │
│  │  │  - size class: 67个大小类别                  │   │   │
│  │  └─────────────────────────────────────────────┘   │   │

│  │  │          mcentral (中心缓存)                 │   │   │

│  │  └─────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────┘   │

│  ┌─────────────────────────────────────────────────────┐   │
│  │           GC堆（三色标记）                          │   │

│  └─────────────────────────────────────────────────────┘   │
```

**GC演进**：
```
Go 1.0-1.4: STW标记清除
Go 1.5: 并发标记清除，100ms停顿
Go 1.6-1.7: 优化，20-50ms停顿
Go 1.8: 亚毫秒停顿（<1ms）
Go 1.9+: 并发栈收缩
Go 1.18+: 软内存限制
```

### 2. Goroutine调度

Go使用GMP模型实现高效的协程调度。

**GMP模型**：
```
G (Goroutine) - 用户态轻量级线程
M (Machine)   - OS线程
P (Processor) - 逻辑处理器，运行G的上下文

┌─────────────────────────────────────────────────────────────┐
│                      GMP调度模型                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   全局队列                      每个P的本地队列             │
│   ┌─────┐  ┌─────┐              ┌─────┐ ┌─────┐ ┌─────┐   │
│   │ G1  │  │ G2  │              │ G3  │ │ G4  │ │ G5  │   │
│   └─────┘  └─────┘              └──┬──┘ └──┬──┘ └──┬──┘   │
│                                      │       │       │     │
│   ┌──────────────────────────────────┼───────┼───────┘     │
│   │                                  │       │             │
│   ▼                                  ▼       ▼             │
│  ┌────┐   ┌────┐   ┌────┐         ┌────┐ ┌────┐ ┌────┐    │
│  │ P0 │   │ P1 │   │ P2 │         │ M0 │ │ M1 │ │ M2 │    │
│  └──┬─┘   └──┬─┘   └──┬─┘         └────┘ └────┘ └────┘    │
│     │        │        │                                    │
│     └────────┴────────┘                                    │
│              │                                             │
│              ▼                                             │
│         与CPU核心数相同                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘

调度策略：
1. work stealing：P空闲时从其他P偷取G
2. handoff：M阻塞时，P创建或复用M
3. sysmon：监控线程，处理GC、netpoll等
```

### 3. 内存逃逸分析

Go编译器决定对象分配在栈还是堆上。

**逃逸场景**：
```go
// 1. 返回指针
func escape1() *int {
    x := 1
    return &x  // x逃逸到堆
}

// 2. 闭包引用
func escape2() func() int {
    x := 1
    return func() int {  // x逃逸到堆
        return x
    }
}

// 3. 接口调用
func escape3() interface{} {
    x := 1
    return x  // 装箱，逃逸到堆
}

// 4. 切片扩容超出栈限制
func escape4() []int {
    s := make([]int, 0, 100)  // 小切片可能在栈
    s = append(s, 1)
    return s  // 返回后逃逸
}

// 5. 大对象（>32KB直接分配在堆）
func escape5() {
    _ = make([]byte, 65536)  // 大对象，堆分配
}
```

```go
func batchProcess(items []Item) {

        if end > len(items) {
            end = len(items)
        }
        processBatch(items[i:end])
    }
}
```

### Goroutine优化

**1. Goroutine池化**：
```go
// Worker Pool模式
type Pool struct {
    workers   int
    jobQueue  chan func()
    wg        sync.WaitGroup
}

func NewPool(workers int) *Pool {
    p := &Pool{
        workers:  workers,
        jobQueue: make(chan func(), 1000),
    }
    p.start()
    return p
}

func (p *Pool) start() {
    for i := 0; i < p.workers; i++ {
        p.wg.Add(1)
        go func() {
            defer p.wg.Done()
            for job := range p.jobQueue {
                job()
            }
        }()
    }
}

func (p *Pool) Submit(job func()) {
    p.jobQueue <- job
}

func (p *Pool) Shutdown() {
    close(p.jobQueue)
    p.wg.Wait()
}

// 使用
pool := NewPool(100)
for i := 0; i < 10000; i++ {
    pool.Submit(func() {
        doWork()
    })
}
pool.Shutdown()
```

**2. 避免Goroutine泄漏**：
```go
// 错误示例：Goroutine可能永远阻塞
func leaky() {
    ch := make(chan int)
    go func() {
        ch <- doWork()  // 如果doWork panic或超时，goroutine泄漏
    }()
    return <-ch
}

// 正确示例：使用context控制生命周期
func safe(ctx context.Context) (int, error) {
    ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()
    
    ch := make(chan int, 1)
    go func() {
        ch <- doWork()
    }()
    
    select {
    case result := <-ch:
        return result, nil
    case <-ctx.Done():
        return 0, ctx.Err()
    }
}
```

**3. 同步优化**：
```go
// 1. 使用sync.Map代替map+锁（特定场景）
var cache sync.Map

cache.Store(key, value)
value, ok := cache.Load(key)
cache.Delete(key)

// 2. 使用原子操作代替锁（简单计数）
type Counter struct {
    count int64
}

func (c *Counter) Inc() {
    atomic.AddInt64(&c.count, 1)
}

func (c *Counter) Get() int64 {
    return atomic.LoadInt64(&c.count)
}

// 3. 读写分离锁
var (
    mu     sync.RWMutex
    config Config
)

// 读多写少场景
func GetConfig() Config {
    mu.RLock()
    defer mu.RUnlock()
    return config
}

func SetConfig(c Config) {
    mu.Lock()
    defer mu.Unlock()
    config = c
}
```

**3. 内存池实现**：
```go
// 高性能字节缓冲区池
type BufferPool struct {
    pool sync.Pool
    size int
}

func NewBufferPool(size int) *BufferPool {
    return &BufferPool{
        pool: sync.Pool{
            New: func() interface{} {
                return make([]byte, size)
            },
        },
        size: size,
    }
}

func (p *BufferPool) Get() []byte {
    return p.pool.Get().([]byte)
}

func (p *BufferPool) Put(b []byte) {
    if cap(b) >= p.size {
        p.pool.Put(b[:p.size])
    }
}

// 对象池（带类型安全包装）
type ObjectPool[T any] struct {
    pool sync.Pool
}

func NewObjectPool[T any](newFunc func() T) *ObjectPool[T] {
    return &ObjectPool[T]{
        pool: sync.Pool{
            New: func() interface{} {
                return newFunc()
            },
        },
    }
}

func (p *ObjectPool[T]) Get() T {
    return p.pool.Get().(T)
}

func (p *ObjectPool[T]) Put(x T) {
    p.pool.Put(x)
}

// 使用
var requestPool = NewObjectPool(func() *http.Request {
    return &http.Request{}
})
```

### 高并发API服务优化

**优化前状况**：
```
场景：10万QPS的API网关
问题：
- 内存占用：8GB且持续增长
- GC停顿：50ms，每分钟触发
- P99延迟：200ms
- Goroutine数：50万
```

**优化过程**：
```go
// 1. 优化前：每次请求分配buffer
func handleRequest(w http.ResponseWriter, r *http.Request) {
    buf := make([]byte, 4096)  // 每次分配
    // 处理...
}

// 优化后：使用sync.Pool
var bufPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 4096)
    },
}

func handleRequest(w http.ResponseWriter, r *http.Request) {
    buf := bufPool.Get().([]byte)
    defer bufPool.Put(buf)
    // 处理...
}

// 2. 优化前：每个请求一个goroutine
for req := range requests {
    go process(req)  // 大量goroutine
}

// 优化后：有界worker pool
type Server struct {
    sem chan struct{}
}

func (s *Server) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    s.sem <- struct{}{}        // 获取令牌
    defer func() { <-s.sem }() // 释放令牌
    
    process(w, r)
}

// 3. 优化字符串处理
// 优化前
func joinStrings(items []string) string {
    var result string
    for _, item := range items {
        result += item + ","  // 多次分配
    }
    return result
}

// 优化后
func joinStrings(items []string) string {
    var b strings.Builder
    b.Grow(estimateSize(items))  // 预分配
    for i, item := range items {
        if i > 0 {
            b.WriteByte(',')
        }
        b.WriteString(item)
    }
    return b.String()
}
```

**优化效果**：
```
- 内存占用：8GB → 2GB（稳定）
- GC停顿：50ms → 2ms
- GC频率：60次/分 → 2次/分
- P99延迟：200ms → 15ms
- Goroutine数：50万 → 1000
```
