# 并发优化：锁优化与无锁编程

## 核心概念

### 1. 并发模型

**线程模型对比**：

| 模型 | 特点 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 多线程 | OS线程 | 真正并行 | 上下文切换开销大 | CPU密集型 |
| 协程 | 用户态线程 | 轻量 | 不能跨核 | IO密集型 |
| 事件循环 | 单线程+回调 | 简单 | 回调地狱 | Node.js |
| Actor | 消息传递 | 易扩展 | 学习曲线 | 分布式 |

**并发性能指标**：

```
┌─────────────────────────────────────────────────────────────┐
│                   并发性能指标                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   吞吐量（Throughput）：单位时间完成的工作量                │
│   - 受限于：锁竞争、上下文切换、内存带宽                    │
│                                                             │
│   延迟（Latency）：单个操作的响应时间                       │
│   - 受限于：同步开销、资源等待                              │
│                                                             │
│   可扩展性（Scalability）：增加资源后的性能提升             │
│   - 理想：线性扩展（2倍资源→2倍性能）                       │
│   - 实际：受Amdahl定律限制                                  │
│                                                             │
│   Amdahl定律：                                              │
│   加速比 = 1 / ((1-P) + P/N)                                │
│   P: 可并行部分占比，N: 处理器数量                          │
│   例：P=90%，N=8，加速比 = 1 / (0.1 + 0.9/8) = 4.7         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 锁优化

**锁粒度**：

```
┌─────────────────────────────────────────────────────────────┐
│                   锁粒度对比                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   粗粒度锁：                                                │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  lock(map)                                          │  │
│   │  ├─ read key1                                       │  │
│   │  ├─ write key2                                      │  │
│   │  └─ delete key3                                     │  │
│   │  unlock(map)                                        │  │
│   └─────────────────────────────────────────────────────┘  │
│   优点：实现简单                                            │
│   缺点：并发度低                                            │
│                                                             │
│   细粒度锁：                                                │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  lock(bucket[key1.hash]) ──→ read key1 ──→ unlock  │  │
│   │  lock(bucket[key2.hash]) ──→ write key2 ──→ unlock │  │
│   │  lock(bucket[key3.hash]) ──→ delete key3 ──→ unlock│  │
│   └─────────────────────────────────────────────────────┘  │
│   优点：并发度高                                            │
│   缺点：实现复杂，可能死锁                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**锁类型选择**：

| 锁类型 | 特点 | 适用场景 |
|--------|------|----------|
| synchronized | JVM内置 | 简单同步 |
| ReentrantLock | 可重入、可中断 | 需要灵活控制 |
| ReadWriteLock | 读共享写独占 | 读多写少 |
| StampedLock | 乐观读 | 读多且冲突少 |
| Semaphore | 信号量 | 限流控制 |

## 实践方法

### 锁优化技巧

**1. 减小锁粒度**：

```java
// 优化前：全局锁
public class GlobalCache {
    private final Map<String, Object> cache = new HashMap<>();
    private final Object lock = new Object();

    public Object get(String key) {
        synchronized (lock) {
            return cache.get(key);
        }
    }

    public void put(String key, Object value) {
        synchronized (lock) {
            cache.put(key, value);
        }
    }
}

// 优化后：分段锁
public class SegmentCache {
    private static final int SEGMENT_COUNT = 16;
    private final Segment[] segments = new Segment[SEGMENT_COUNT];

    private static class Segment {
        private final Map<String, Object> map = new HashMap<>();
        private final Lock lock = new ReentrantLock();
    }

    private Segment getSegment(String key) {
        return segments[key.hashCode() % SEGMENT_COUNT];
    }

    public Object get(String key) {
        Segment segment = getSegment(key);
        segment.lock.lock();
        try {
            return segment.map.get(key);
        } finally {
            segment.lock.unlock();
        }
    }
}

// 优化后：ConcurrentHashMap（Java内置分段锁）
public class OptimizedCache {
    private final ConcurrentHashMap<String, Object> cache =
        new ConcurrentHashMap<>();

    public Object get(String key) {
        return cache.get(key);
    }

    public void put(String key, Object value) {
        cache.put(key, value);
    }
}
```

**2. 读写分离锁**：

```java
public class ReadWriteCache {
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();
    private final Map<String, Object> cache = new HashMap<>();

    // 读操作使用读锁（并发）
    public Object get(String key) {
        readLock.lock();
        try {
            return cache.get(key);
        } finally {
            readLock.unlock();
        }
    }

    // 写操作使用写锁（独占）
    public void put(String key, Object value) {
        writeLock.lock();
        try {
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }

    // 读多写少场景性能提升显著
    // 读锁可以并发，写锁独占
}
```

**3. 乐观锁（CAS）**：

```java
public class CASCounter {
    private final AtomicLong count = new AtomicLong(0);

    public void increment() {
        long current;
        long next;
        do {
            current = count.get();
            next = current + 1;
        } while (!count.compareAndSet(current, next));
        // CAS成功则退出，失败则重试
    }

    public long get() {
        return count.get();
    }
}

// Go实现
type CASCounter struct {
    count int64
}

func (c *CASCounter) Increment() {
    for {
        old := atomic.LoadInt64(&c.count)
        new := old + 1
        if atomic.CompareAndSwapInt64(&c.count, old, new) {
            return
        }
        // CAS失败，重试
    }
}
```

### 无锁编程

**1. 无锁队列**：

```java
// Java Disruptor RingBuffer（无锁队列）
public class DisruptorExample {
    public static void main(String[] args) {
        // 创建RingBuffer，大小必须是2的幂
        RingBuffer<Event> ringBuffer = new RingBuffer<>(
            Event::new,
            new SingleProducerSequencer(1024, new BusySpinWaitStrategy())
        );

        // 生产者
        EventProducer producer = new EventProducer(ringBuffer);

        // 消费者
        EventProcessor processor = new EventProcessor(ringBuffer);

        // 无锁并发，性能极高
    }
}

// Go channel（有锁但高度优化）
func producer(ch chan<- int) {
    for i := 0; i < 1000000; i++ {
        ch <- i
    }
    close(ch)
}

func consumer(ch <-chan int, wg *sync.WaitGroup) {
    defer wg.Done()
    for v := range ch {
        process(v)
    }
}
```

## 案例数据

### 计数器性能对比

| 实现 | 单线程 | 10线程 | 100线程 |
|------|--------|--------|---------|
| synchronized | 100M/s | 5M/s | 0.5M/s |
| ReentrantLock | 100M/s | 8M/s | 1M/s |
| AtomicLong | 150M/s | 20M/s | 5M/s |
| LongAdder | 150M/s | 100M/s | 80M/s |

**结论**：高并发场景下，LongAdder性能最佳。

### 锁优化效果

**场景：库存扣减**

**优化前**：

```java
public synchronized boolean deduct(String productId, int quantity) {
    int stock = getStock(productId);
    if (stock >= quantity) {
        setStock(productId, stock - quantity);
        return true;
    }
    return false;
}
// TPS: 500
```

**优化后**：

```java
public boolean deduct(String productId, int quantity) {
    while (true) {
        int stock = stockCache.get(productId);
        if (stock < quantity) {
            return false;
        }
        if (stockCache.compareAndSet(productId, stock, stock - quantity)) {
            return true;
        }
    }
}
// TPS: 8000
```

**效果**：

| 指标 | 优化前 | 优化后 | 改进 |
|------|--------|--------|------|
| TPS | 500 | 8000 | 16倍↑ |
| 平均延迟 | 20ms | 1ms | 95%↓ |

---

**总结**：并发优化需要根据实际情况选择合适的同步机制。低竞争用CAS，高竞争用分段锁，读多写少用读写锁。无锁编程可以带来极致性能，但实现复杂，需要谨慎使用。
